# 17 — 分步实现指南：从零到可部署的完整路径

> 本文档是一份**可执行的开发手册**。每一步都包含：要做什么、怎么做、做完后如何部署验证。
> 不是理论文档——是你真正动手时的 checklist。

---

## 总览

```
Step 0  项目骨架搭建             ← 能编译通过，能启动空服务
Step 1  最小可调度闭环（本地执行）  ← 任务按 Cron 被调度、CAS 抢占、本地执行
Step 2  远程执行 + gRPC 通信      ← 调度器 → 执行器跨进程通信
Step 3  Kafka 异步状态上报         ← 执行器进度实时上报到调度器
Step 4  补偿机制（V1）            ← 重试、重调度、超时中断
Step 5  DAG 工作流引擎            ← DSL 声明多任务编排，事件驱动
Step 6  分片任务                  ← 大任务拆分为多个子任务并行执行
Step 7  分库分表 + 分布式调度（V2）← 多节点部署，每节点处理部分分片
Step 8  智能调度 + 负载检查       ← Prometheus 驱动节点选择
Step 9  可观测性 + Docker 部署    ← 监控大盘 + 一键部署
```

**核心原则**：每一步做完都能 `make run_scheduler` 跑起来，看到真实效果。

```go
// internal/service/scheduler/scheduler.go
func (s *Scheduler) scheduleLoop() {
    ticker := time.NewTicker(s.config.ScheduleInterval)
    defer ticker.Stop()
    for {
        select {
        case <-s.ctx.Done():
            return
        case <-ticker.C:
            s.scheduleOnce()
        }
    }
}

func (s *Scheduler) scheduleOnce() {
    // 1. 查询到期的可调度任务
    tasks, _ := s.taskSvc.SchedulableTasks(ctx, preemptedTimeout, batchSize)

    // 2. 逐个执行
    for _, task := range tasks {
        s.runner.Run(ctx, task)
    }

    // 3. 更新每个任务的 NextTime（计算下一次 Cron 触发时间）
}
```

## Step 2：远程执行 + gRPC 通信

### 目标

**调度器和执行器分离为两个独立进程，通过 gRPC 通信。**  
调度器下发任务 → 执行器执行 → 执行器返回结果。

### 新增依赖

- etcd（服务发现）
- Protobuf + buf（代码生成）

## Step 4：补偿机制（V1 版本）

### 目标

系统具备自愈能力：执行失败自动重试、节点故障自动重调度、任务超时自动中断。

### 要做的事

#### 4.1 Retry 补偿器

```go
// internal/compensator/retry.go
// 定时扫描状态为 FAILED_RETRYABLE 且到达 next_retry_time 的执行记录
func (r *RetryCompensator) Start(ctx context.Context) {
    for {
        select {
        case <-ctx.Done():
            return
        default:
            execs := r.execSvc.FindRetryableExecutions(ctx, r.config.BatchSize)
            for _, exec := range execs {
                r.runner.Retry(ctx, exec)
            }
            time.Sleep(r.config.MinDuration) // 防空转
        }
    }
}
```

**重试策略**：指数退避 `InitialInterval × 2^(retries-1)`，上限 `MaxInterval`

#### 4.2 Reschedule 补偿器

```go
// 扫描状态为 FAILED_RESCHEDULED 的执行记录
// 排除故障节点，选择新节点重新调度
func (r *RescheduleCompensator) processOne(ctx context.Context, exec domain.TaskExecution) {
    // ctx 注入 WithExcludedNodeID(exec.ExecutorNodeID)
    r.runner.Reschedule(ctx, exec)
}
```

#### 4.3 Interrupt 补偿器

```go
// 扫描运行时间超过 Deadline 的执行记录
// 向执行器发送 Interrupt RPC
func (i *InterruptCompensator) processOne(ctx context.Context, exec domain.TaskExecution) {
    client := i.grpcClients.Get(exec.ExecutorNodeID)
    client.Interrupt(ctx, &executorv1.InterruptRequest{Eid: exec.ID})
}
```

## Step 5：DAG 工作流引擎

### 目标

支持通过 DSL 声明多任务编排：顺序、并行、条件分支。**事件驱动后继任务执行。**

### 要做的事

#### 5.1 设计 DSL 语法

```
# 顺序
A -> B -> C

# 并行（AND，全部完成才触发后继）
A && B -> C

# 并行（OR，任一完成即触发后继）
A || B -> C

# 条件分支
A -> B ? C : D     # B 成功走 C，失败走 D

# 组合
(A && B) -> C -> D ? E : F
```

#### 5.2 ANTLR4 语法文件

```antlr
// internal/dsl/parser/TaskPlan.g4
grammar TaskPlan;

plan: expr EOF;
expr: expr '->' expr        # Sequence
    | expr '&&' expr        # ParallelAnd
    | expr '||' expr        # ParallelOr
    | expr '?' expr ':' expr # Conditional
    | '(' expr ')'          # Grouped
    | TASK_NAME             # Single
    ;
TASK_NAME: [a-zA-Z_][a-zA-Z0-9_]*;
WS: [ \t\r\n]+ -> skip;
```

#### 5.3 实现 AST 到 DAG 的转换

```go
// internal/dsl/parser/visitor.go
// ANTLR4 Visitor 遍历 AST，构建 PlanNode 图
type PlanVisitor struct {
    tasks map[string]*PlanNode
}

func (v *PlanVisitor) VisitSequence(ctx *SequenceContext) interface{} {
    left := v.Visit(ctx.Expr(0))
    right := v.Visit(ctx.Expr(1))
    // left 的所有出口节点 → right 的所有入口节点
    connect(left.exits, right.entries)
}
```

#### 5.4 实现 PlanTaskRunner

```go
// internal/service/runner/plan_task_runner.go
func (r *PlanTaskRunner) Run(ctx context.Context, task domain.Task) error {
    // 1. 创建 Plan 执行记录（RUNNING）
    // 2. 获取 Plan（解析 DSL → 构建 DAG）
    // 3. 找到根节点（没有前置依赖的节点）
    // 4. 并行启动所有根节点
    for _, root := range plan.Roots() {
        go r.runNode(ctx, root)
    }
}

// NextStep 由 Kafka 完成事件驱动调用
func (r *PlanTaskRunner) NextStep(ctx context.Context, event Event) {
    // 1. 获取当前节点的后继节点
    // 2. 检查后继节点的前置依赖是否全部满足（CheckPre）
    // 3. 满足则启动执行
}
```

#### 5.5 完成事件消费者

```go
// 消费 Kafka 完成事件 → 驱动 DAG 后继节点
func (c *CompleteConsumer) Consume(msg *sarama.ConsumerMessage) {
    var event Event
    json.Unmarshal(msg.Value, &event)
    if event.PlanID > 0 {
        r.planRunner.NextStep(ctx, event)
    }
}
```

#### 5.6 新增表/字段

```sql
-- Plan 相关字段直接加到 task 表
ALTER TABLE task ADD COLUMN type VARCHAR(20) DEFAULT 'NORMAL';  -- NORMAL / PLAN
ALTER TABLE task ADD COLUMN exec_expr TEXT;  -- DSL 表达式
```

## Step 6：分片任务

### 目标

一个大任务拆成多个子任务，分发到多个执行器节点并行执行。

### 要做的事

#### 6.1 分片规则

```go
// internal/domain/task.go
type ShardingRule struct {
    Type   string            // "Range" / "WeightedDynamicRange"
    Params map[string]string // totalNums, step 等

```

**Range 分片**：totalNums=10, step=1000 → 10 个子任务，每个处理 [0,1000), [1000,2000), ...

#### 6.2 NormalTaskRunner 支持分片

```go
func (r *NormalTaskRunner) handleShardingTask(ctx context.Context, task domain.Task) {
    // 1. 创建父执行记录
    // 2. 按分片规则计算每个分片的参数
    // 3. 创建 N 个子执行记录
    // 4. 每个子执行记录在独立 goroutine 中通过 Invoker 执行
    for i, params := range shardParams {
        go r.invoker.Run(ctx, task, childExec[i])
    }
}
```

#### 6.3 Sharding 补偿器

```go
// 聚合所有子任务结果
// 当所有子任务都到达终态时，判定父任务状态
func (s *ShardingCompensator) aggregate(ctx context.Context, parentExecID int64) {
    children := s.execSvc.FindByParentID(ctx, parentExecID)
    allDone := true
    allSuccess := true
    for _, child := range children {
        if !child.IsTerminalStatus() { allDone = false }
        if child.Status != ExecStatusSuccess { allSuccess = false }
    }
    if allDone {
        if allSuccess {
            // 父任务标记 SUCCESS
        } else {
            // 父任务标记 FAILED
        }
    }
}
```

```sql

-- NULL=普通任务, 0=分片父任务, >0=分片子任务(指向父ID)
```

## FAQ

### Q: Step 之间的代码会不会频繁重构？

**domain 层几乎不动。** 从 Step 1 开始定义的 Task、TaskExecution 结构体，到 Step 9 核心字段都没变过。变的主要是：
- DAO 层（加字段、改分片逻辑）
- Service 层（加补偿、加事件驱动）
- 基础设施层（从单机到分布式）

这就是 DDD 分层的价值。

### Q: 能不能跳过某些步骤？

可以，但有依赖：
- 跳过 Step 5（DAG）：完全可以，不影响基础调度能力
- 跳过 Step 6（分片）：可以，只是没有大任务拆分能力
- 不能跳过 Step 1-3：这是核心链路
- 不能跳过 Step 4（补偿）：没有自愈能力的调度系统不完整

### Q: 面试时怎么讲？

按 Step 讲故事：

> "我先做了最小调度闭环（Step 1），验证了 CAS 抢占的正确性。
> 然后分离了调度器和执行器（Step 2），引入了 gRPC 通信。
> 在做补偿机制（Step 4）的时候，发现重试和中断可能同时处理同一个任务——
> 所以我设计了严格的状态机来防止并发冲突。
> 后面做分库分表（Step 7）时，领域层零改动，只改了 DAO 层——
> 这证明了一开始坚持 DDD 分层是值得的。"

这比"我设计了一个分布式任务调度系统"有说服力 100 倍。
