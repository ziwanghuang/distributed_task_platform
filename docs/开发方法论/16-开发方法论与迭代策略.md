# 16 — 开发方法论与迭代策略

> 本文档记录本项目从零到一的开发思路：为什么选择"框架先行、分步迭代"，每个阶段做了什么、验证了什么、踩了什么坑。

---

## 核心原则

**框架先行，实现分步，每步可验证。**

- **框架先行**：DDD 分层、模块边界、核心接口在动手写第一行业务代码前就确定——这些是承重结构，后面改的代价极大
- **实现分步**：功能按优先级和依赖关系拆成 Phase，每个 Phase 交付一个可独立运行、可独立验证的增量
- **每步可验证**：不攒功能，做完一个就部署上去跑，用真实场景发现问题——Phase N 暴露的问题会纠偏 Phase N+1 的设计

### 为什么不一步到位？

分布式系统有太多"你以为想清楚了，实际上没有"的地方：

- 你在设计阶段觉得"重试 3 次就够了"，但上了 gRPC 后发现网络分区会导致连续超时，需要指数退避
- 你觉得"单表够用"，但补偿器全表扫描在 10 万条记录时就拖慢了调度循环
- 你觉得"DAG 用 JSON 描述就行"，但写了几个复杂编排后发现 JSON 嵌套 5 层根本没法维护，才引入了 DSL

**一步到位的风险：所有假设都在最后一刻才被验证，而那时修改成本最高。**

## 第零步：确定架构骨架

在写任何业务代码之前，先确定这些不可变的东西：

### 分层结构

```
cmd/            ← 启动入口，只做组装
ioc/            ← 依赖注入（Wire），所有组件在这里接线
internal/
  domain/       ← 领域模型，纯业务概念，不依赖任何基础设施
  repository/   ← 数据访问抽象 + DAO 实现
  service/      ← 业务编排
  compensator/  ← 补偿策略
  event/        ← 事件系统
pkg/            ← 可复用的基础组件，与业务无关
```

**关键决策**：domain 层绝对不 import 任何基础设施包（不 import gorm、不 import kafka）。这保证了后面加分库分表、换消息队列时，domain 层零改动。

### 核心接口

在 Phase 1 就定义好，后续只加实现不改接口：

```go
// 任务抢占 —— Phase 1 定义，贯穿全项目
type TaskAcquirer interface {
    Acquire(ctx context.Context, task domain.Task) error
    Renew(ctx context.Context, nodeID string) error
    Release(ctx context.Context, task domain.Task) error
}

// 任务调用 —— Phase 1 只有 LocalInvoker，Phase 2 加 GRPCInvoker
type Invoker interface {
    Run(ctx context.Context, task domain.Task, exec domain.TaskExecution) error
}

// 节点选择 —— Phase 1 返回固定节点，Phase 5 接入 Prometheus
type Picker interface {
    Pick(ctx context.Context, task domain.Task) (string, error)
}

// 负载检查 —— Phase 1 直接返回 true，Phase 6 接入三级检查
type LoadChecker interface {
    Check(ctx context.Context) (time.Duration, bool)
}
```

**设计意图**：接口在早期就定义，但实现可以从最简单的 stub 开始。这样每个 Phase 只需要"替换实现"，不需要改调用方。

### 数据库 Schema

只建 Phase 1 需要的表和字段，后续按需 ALTER：

```sql
-- Phase 1：最小字段集
CREATE TABLE tasks (
    id BIGINT PRIMARY KEY,
    name VARCHAR(255),
    cron_expr VARCHAR(100),
    status TINYINT,           -- ACTIVE / PREEMPTED / INACTIVE
    schedule_node_id VARCHAR(64),
    version BIGINT,           -- CAS 乐观锁
    next_time BIGINT,
    ...
);

-- Phase 2 追加：gRPC 配置
ALTER TABLE tasks ADD COLUMN grpc_config JSON;

-- Phase 4 追加：DAG 支持
ALTER TABLE tasks ADD COLUMN exec_expr TEXT;
ALTER TABLE tasks ADD COLUMN type TINYINT;

-- Phase 5 追加：分片
-- 不是 ALTER，而是新建分片表 + 数据迁移
```

## Phase 1 — 最小可调度闭环

**目标**：证明"CAS 抢占 + Cron 调度 + 本地执行"这条路走得通。

### 做了什么

| 模块 | 实现 |
|------|------|
| Scheduler | 单机版，`scheduleLoop` 定时拉取任务 |
| TaskAcquirer | CAS `UPDATE ... SET version = version+1 WHERE version = ?` |
| Runner | NormalTaskRunner，抢占后本地执行 |
| Invoker | LocalInvoker，直接调用函数 |
| Picker | 固定返回当前节点 ID |
| LoadChecker | 空实现，始终返回可调度 |
| 存储 | 单库单表 MySQL |

### 验证点

- [x] 任务按 Cron 表达式准时触发
- [x] 多个 Scheduler 实例不会重复抢占同一任务
- [x] 抢占超时后任务自动释放（renewLoop 停止 → version 超时 → 重新可调度）
- [x] 执行完成后任务状态正确流转

### 发现的问题

1. **CAS 冲突率**：当任务多、节点少时，`UPDATE ... WHERE version = ?` 大量空更新。→ 决策：Phase 2 加 `batchSize` 控制批量抢占数，减少无效竞争
2. **续约间隔**：最初设 30s，发现节点宕机后任务要等 30s+ 才能被重新调度。→ 调整为 5s 续约 + 10min 超时窗口
3. **NextTime 计算**：Cron 表达式解析需要时区处理，最初忘了 UTC 转换导致定时偏移

## Phase 2 — 远程执行 + 状态上报

**目标**：调度器和执行器分离，通过 gRPC 通信，Kafka 异步上报。

### 做了什么

| 模块 | 变更 |
|------|------|
| Invoker | 新增 GRPCInvoker + Dispatcher（按协议路由） |
| gRPC | 定义 `executor.proto`（Execute/Interrupt/Query） |
| 状态上报 | Kafka Producer（执行器端） + Consumer（调度器端） |
| 服务发现 | etcd 注册中心 + 自定义 Resolver |
| Picker | 简单轮询（还没接 Prometheus） |

### 验证点

- [x] 调度器通过 gRPC 下发任务到远程执行器
- [x] 执行器宕机后，调度器感知并标记任务为可重试
- [x] Kafka 上报不丢消息（ack=all + 重试）
- [x] etcd Watch 能在秒级感知节点上下线

### 发现的问题

1. **gRPC 连接池**：最初每次调用都新建连接，QPS 一上去就 `too many open files`。→ 引入连接池 + 长连接复用
2. **Kafka 消费顺序**：同一个 TaskExecution 的多次状态上报可能乱序到达。→ 用 TaskID 作为 partition key 保证单任务有序
3. **Proto 版本兼容**：执行器和调度器可能版本不一致。→ proto 字段只加不删，用 `optional` 保证向前兼容

## Phase 3 — 补偿机制

**目标**：系统具备自愈能力，各种故障场景都能自动恢复。

### 做了什么

| 补偿器 | 触发条件 | 处理方式 |
|--------|---------|---------|
| Retry | 执行器返回 `FAILED_RETRYABLE` | 指数退避重试，`InitialInterval × 2^(n-1)`，不超过 MaxInterval |
| Reschedule | 执行器返回 `FAILED_RESCHEDULED` | 排除故障节点，选择新节点重新调度 |
| Interrupt | 执行时间超过 Deadline | 向执行器发 `Interrupt` RPC，强制终止 |
| ~~Sharding~~ | Phase 4 才需要 | — |

全部先实现 V1（单机循环扫描），够用。

### 验证点

- [x] 网络抖动场景：任务自动重试 3 次后成功
- [x] 节点宕机场景：任务被重调度到其他健康节点
- [x] 任务卡死场景：超过 Deadline 后被中断，状态标记为 FAILED
- [x] 重试耗尽场景：达到 MaxRetries 后状态变为最终 FAILED，不再重试

### 发现的问题

1. **补偿器互相干扰**：一个超时任务可能同时被 Retry 和 Interrupt 捡到。→ 用状态机严格限制转换路径，RUNNING 只能转 FAILED_*，不能跳过
2. **补偿器空转**：没有需要处理的记录时，`FindRetryableExecutions` 仍然每 10s 查一次库。→ 加 `minDuration` 防自旋，无记录时 sleep 更久
3. **指数退避溢出**：重试次数大时 `2^n` 溢出。→ 加 `MaxInterval` 上限，并在 Strategy 里用 `min()` 兜底

## Phase 4 — DAG 工作流引擎

**目标**：支持多任务编排，通过 DSL 声明依赖关系。

### 技术选型决策

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| JSON 描述 | 简单、通用 | 嵌套深、可读性差、不支持条件分支 | ❌ |
| YAML 描述 | 比 JSON 好读 | 仍然需要自定义语法 | ❌ |
| 自定义 DSL + ANTLR4 | 语法简洁、可扩展、编译时校验 | 学习成本高 | ✅ |

最终选择 ANTLR4：

```
# DSL 语法直观
(A && B) -> C -> D ? E : F
```

### 做了什么

| 模块 | 实现 |
|------|------|
| DSL 语法 | `.g4` 文法文件，支持 `->` / `&&` / `\|\|` / `? :` |
| Parser | ANTLR4 Lexer + Parser + Visitor → AST |
| DAG 构建 | AST → PlanNode 图（Pre/Next 双向链接） |
| PlanTaskRunner | 执行根节点 → 完成事件触发后继 → CheckPre 判断是否可执行 |
| 事件驱动 | Kafka CompleteEvent → PlanTaskRunner.NextStep |

### 验证点

- [x] `A -> B -> C`：顺序执行
- [x] `A && B -> C`：A、B 并行，全部成功后触发 C
- [x] `A || B -> C`：A、B 任一成功即触发 C
- [x] `A -> B ? C : D`：B 成功走 C，失败走 D
- [x] 嵌套组合：`(A && B) -> C -> (D || E) ? F : G`

### 发现的问题

1. **DAG 环检测**：用户写了 `A -> B -> A`，死循环。→ 在 `NewAstPlan` 时加拓扑排序检测
2. **并行任务超时**：`A && B` 中 A 完成了但 B 超时被中断，CheckPre 永远不满足。→ AND 节点在子任务全部终态（包括 FAILED）后判断，不无限等待
3. **事件乱序**：CompleteEvent 可能比 Execution 创建还先到达。→ NextStep 里加重试循环（max 10 次，500ms 间隔）

## Phase 5 — 分布式增强

**目标**：支持多节点部署，补偿器不重复处理，调度智能化。

### 做了什么

| 模块 | 变更 |
|------|------|
| 分库分表 | 2 DB × 2 task 表 + 2 DB × 4 execution 表 |
| ID 生成 | Snowflake 变体，高位嵌入分片路由信息 |
| V2 补偿器 | ShardingLoopJob + 分布式锁（每个分片独占） |
| V2 调度器 | ShardingLoopJob 驱动，资源信号量控制并发 |
| 智能调度 | Prometheus 查 CPU/内存/任务数 → TopN 随机选节点 |
| 负载检查 | 三级：令牌桶 → 数据库响应 → 集群负载对比 |

### V1 → V2 的核心区别

```
V1: while(true) { scanAllTasks(); sleep(); }
    ↓
V2: ShardingLoopJob {
      for each shard in Broadcast() {
        if acquireLock(shard) && acquireSemaphore() {
          go scanShard(shard)
        }
      }
    }
```

- 每个分片一把锁，多节点不冲突
- 资源信号量限制并发分片数，防止 OOM
- `bizTimeout(50s) < lockTTL(1min)`，确保锁不会在业务执行中过期

### 验证点

- [x] 3 个调度节点部署，每个分片只被一个节点处理
- [x] 节点宕机后，其持有的分片锁自动过期，被其他节点接管
- [x] Snowflake ID 在分片迁移后仍能正确路由
- [x] TopN 选择器在节点负载不均时能有效均衡

### 发现的问题

1. **分片路由热点**：ID 分布不均导致某些分片数据量远大于其他。→ Snowflake 的 sequence 部分保证均匀分布
2. **锁续约风暴**：100 个分片 × 每 10s 续约 = 10 QPS 打 Redis/etcd。→ 批量续约 + 错开续约时间
3. **Prometheus 查询延迟**：PromQL 查询 P99 偶尔到 2s，影响调度决策。→ 加本地缓存（30s TTL），降级为上次查询结果

## Phase 6 — 可观测性 + 生产化

**目标**：可监控、可部署、可运维。

### 做了什么

- Prometheus 指标：调度延迟、CAS 冲突率、补偿器执行次数、gRPC 延迟等
- Grafana 面板：预配置核心监控大盘
- Docker 多阶段构建：编译镜像 + alpine 运行镜像（~15MB）
- docker-compose 全套编排：一键启动所有依赖
- 健康检查 + 优雅停机

## 经验总结

### ✅ 做对了的事

1. **DDD 分层从第一天就坚持**：domain 层从未 import 过基础设施包，Phase 5 加分库分表时 domain 零改动
2. **接口先行**：`Invoker`、`Picker`、`LoadChecker` 在 Phase 1 就定义，后面只是换实现
3. **V1 先跑通再优化为 V2**：补偿器和调度器都先做单机版验证逻辑正确，再重构为分布式版
4. **每个 Phase 都有明确的验证标准**：不是"写完了"就算完，而是"验证通过了"才进入下一阶段

### ❌ 踩过的坑

1. **过早优化**：Phase 1 就想做连接池，其实单机版根本不需要——浪费了时间
2. **Proto 字段删改**：早期改过 proto 字段类型，导致已部署的执行器解析失败——后来严格遵守只加不删
3. **补偿器测试不充分**：Phase 3 只测了正常重试路径，没测"重试过程中节点宕机"这种组合场景——Phase 5 才暴露
4. **文档滞后**：Phase 1-3 几乎没写文档，到 Phase 5 回头补的时候很多设计决策已经记不清了

### 核心心得

> **分布式系统的复杂度不是设计出来的，是跑出来的。**
>
> 你在白板上画的架构图和实际跑起来的系统之间，隔着一百个你没想到的边界条件。
> 唯一的办法是：尽早跑起来，尽早发现问题，尽早修正方向。

## 适用场景

这套方法论不只适用于任务调度平台，任何有以下特征的系统都适用：

- **多模块协作**：模块之间有依赖关系，不能全部并行开发
- **基础设施复杂**：涉及多种中间件（MQ、缓存、注册中心），每种都可能出问题
- **分布式场景**：需要处理网络分区、节点故障、数据一致性
- **长期演进**：不是一次性项目，需要持续迭代

对于简单的 CRUD 系统，直接撸就行，不需要这么讲究。
