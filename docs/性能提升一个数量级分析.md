# 性能提升一个数量级：优化分析与方案

> 本文基于当前代码（V1/V2 调度器、DAO 层、Kafka 事件处理、补偿器、gRPC 通信等）的全面分析，识别出制约系统吞吐量的关键瓶颈，并给出具体优化方案。
>
> **当前性能**：单节点 ~10 tasks/s（被 LimiterChecker 10 req/s 硬限流 + 串行分发双重卡死）
>
> **优化目标**：单节点 1,000-3,000 tasks/s，3 节点集群 5,000+ tasks/s，**提升两个数量级**

---

## 二、P0 优化：消除致命瓶颈

### 2.1 调度热路径：串行 → 并行 Worker Pool

**当前问题**

V1 和 V2 调度器中，任务分发都是串行执行的：

```go
// scheduler.go L169-179 / schedulerv2.go L156-166
for i := range tasks {
    s.runner.Run(ctx, tasks[i])  // 每个 Run 涉及 gRPC 调用
}
```

每个 `Run()` 内部需要完成：
1. 通过 LoadChecker 检查负载
2. 通过 gRPC 将任务分发到 Worker 节点
3. 等待 Worker 确认接收

假设单次 gRPC 调用平均耗时 5ms，100 个任务串行分发 = **500ms**。这还不算 LoadChecker 的 Prometheus 查询开销。

**优化方案：Worker Pool 并发分发**

```go
// 优化后的调度循环
func (s *Scheduler) scheduleOneLoop(ctx context.Context) error {
    tasks, err := s.svc.FindSchedulableTasks(ctx, s.config.BatchSize)
    if err != nil || len(tasks) == 0 {
        return err
    }

    // 使用带并发限制的 Worker Pool
    g, gCtx := errgroup.WithContext(ctx)
    g.SetLimit(s.config.MaxConcurrentDispatch) // 建议 20-50

    for i := range tasks {
        task := tasks[i]
        g.Go(func() error {
            return s.runner.Run(gCtx, task)
        })
    }

    return g.Wait()
}
```

**关键参数**：
- `MaxConcurrentDispatch`: 控制并发分发数，建议初始值 20，根据 Worker 节点数和网络状况调整
- 配合 gRPC 连接池使用，避免连接争用

**预期收益**：100 个任务从 500ms 降到 ~25ms（20 并发），**提升 20 倍**

### 2.2 核心热查询：拆分 OR 条件

**当前问题**

`FindSchedulableTasks` 是整个系统最热的查询：

```go
// task.go L119-124
db.Where("next_time <= ? AND (status = ? OR (status = ? AND utime <= ?))",
    now, domain.TaskStatusWaiting,
    domain.TaskStatusRunning, preemptedTime,
).Order("next_time ASC").Limit(batchSize)
```

这个 `OR` 条件导致 MySQL 无法在单次 B-Tree 扫描中利用复合索引 `idx_next_time_status_utime(next_time, status, utime)`。MySQL 的执行计划通常会退化为：
- 要么 `index_merge`（两次索引扫描 + 合并），开销大
- 要么放弃 `status` 部分，只用 `next_time` 前缀，扫描过多行

**优化方案：拆分为两个独立查询 + UNION**

```go
func (dao *TaskDAO) FindSchedulableTasks(ctx context.Context,
    now time.Time, preemptedTime time.Time, batchSize int) ([]Task, error) {

    // 查询 1：等待调度的任务（精确匹配 status=WAITING）
    // 能完美利用索引 (next_time, status)
    var waitingTasks []Task
    err := dao.db.WithContext(ctx).
        Where("next_time <= ? AND status = ?", now, domain.TaskStatusWaiting).
        Order("next_time ASC").
        Limit(batchSize).
        Find(&waitingTasks).Error
    if err != nil {
        return nil, err
    }

    // 查询 2：可抢占的任务（精确匹配 status=RUNNING + utime 条件）
    // 能完美利用索引 (status, utime) 或 (next_time, status, utime)
    remaining := batchSize - len(waitingTasks)
    if remaining <= 0 {
        return waitingTasks, nil
    }

    var preemptTasks []Task
    err = dao.db.WithContext(ctx).
        Where("next_time <= ? AND status = ? AND utime <= ?",
            now, domain.TaskStatusRunning, preemptedTime).
        Order("next_time ASC").
        Limit(remaining).
        Find(&preemptTasks).Error
    if err != nil {
        return waitingTasks, err // 降级：至少返回 waiting 的
    }

    return append(waitingTasks, preemptTasks...), nil
}
```

**配套索引优化**：

```sql
-- 为拆分后的查询专门优化索引
-- 查询 1 用：覆盖 (status, next_time) 的等值+范围
CREATE INDEX idx_status_next_time ON tasks (status, next_time);

-- 查询 2 用：覆盖 (status, utime, next_time)
-- 原有的 idx_next_time_status_utime 可保留，但新增一个更精确的
CREATE INDEX idx_status_utime_next_time ON tasks (status, utime, next_time);
```

**为什么拆分比 OR 快**：
- 单个查询可以精确走某一个索引的 B-Tree 路径，不需要 `index_merge`
- MySQL 优化器对等值条件 + 范围条件的复合索引利用率接近 100%
- 两个查询的总扫描行数远小于 OR 导致的全范围扫描

**预期收益**：热查询延迟从 ~10ms 降到 ~2ms（取决于数据量），**提升 3-5 倍**

### 2.3 task_executions 表：补全缺失索引

**当前问题**

`TaskExecution` 结构体没有任何 GORM 索引标签：

```go
// task_execution.go - 没有任何 index tag
type TaskExecution struct {
    Id                       int64
    Tid                      int64      // 无索引
    Status                   string     // 无索引
    NextRetryTime            int64      // 无索引
    ShardingParentId         int64      // 无索引
    Deadline                 int64      // 无索引
    // ...
}
```

而系统中有至少 5 个高频查询依赖这些字段：

| 查询 | 使用字段 | 调用方 |
|------|---------|--------|
| `FindRetryableExecutions` | status + next_retry_time | 重试补偿器，每 1s 执行 |
| `FindTimeoutExecutions` | status + deadline | 超时中断补偿器，每 1s 执行 |
| `FindShardingParents` | sharding_parent_id + status | 分片补偿器，每 1s 执行 |
| `FindReschedulableExecutions` | status + progress | 重调度补偿器，每 1s 执行 |
| `SetRunningState` 中的 SELECT | id (有主键索引，这个 OK) | 调度器 |

**没有索引意味着：每秒 4 次全表扫描**。当 executions 表达到百万级，每次扫描耗时可能 > 1s，直接拖垮整个系统。

**优化方案：添加关键索引**

```go
type TaskExecution struct {
    Id                       int64  `gorm:"primaryKey;autoIncrement"`
    Tid                      int64  `gorm:"index:idx_tid"`
    Status                   string `gorm:"index:idx_status_next_retry;index:idx_status_deadline;index:idx_status_progress"`
    NextRetryTime            int64  `gorm:"index:idx_status_next_retry"`
    ShardingParentId         int64  `gorm:"index:idx_sharding_parent_status"`
    Deadline                 int64  `gorm:"index:idx_status_deadline"`
    Progress                 int32  `gorm:"index:idx_status_progress"`
    // ...
}
```

**对应的 SQL（如果不用 GORM AutoMigrate）**：

```sql
-- 重试补偿器：FindRetryableExecutions
CREATE INDEX idx_status_next_retry ON task_executions (status, next_retry_time);

-- 超时中断：FindTimeoutExecutions
CREATE INDEX idx_status_deadline ON task_executions (status, deadline);

-- 分片补偿：FindShardingParents
CREATE INDEX idx_sharding_parent_status ON task_executions (sharding_parent_id, status);

-- 重调度：FindReschedulableExecutions
CREATE INDEX idx_status_progress ON task_executions (status, progress);

-- 通用查询：按 task_id 查
CREATE INDEX idx_tid ON task_executions (tid);
```

**预期收益**：补偿器查询从全表扫描（可能 > 1s）降到索引查询（< 5ms），**提升 100-1000 倍**

## 三、P1 优化：突破高优瓶颈

### 3.1 Kafka 分区扩展 + 批量消费

**当前问题**

```yaml
# config.yaml
kafka:
  topics:
    - name: execution_report
      partitions: 1    # 只有 1 个分区！
    - name: execution_batch_report
      partitions: 1    # 只有 1 个分区！
```

单分区意味着：
- 同一 topic 的消息只能被一个 consumer 消费
- 即使部署多个消费者实例，也只有一个在工作
- 消费吞吐完全受限于单线程处理速度

同时，消费者是单条消费模式：

```go
// mqx/consumer.go - 单条消费
select {
case message, ok := <-mqChan:
    if !ok { return }
    // 逐条处理
}
```

**优化方案**

**Step 1：扩展 Kafka 分区**

```yaml
kafka:
  topics:
    - name: execution_report
      partitions: 8     # 扩展到 8 个分区
    - name: execution_batch_report
      partitions: 4     # 扩展到 4 个分区
```

分区数建议 = min(消费者实例数 × 2, Worker 节点数)。8 个分区可以支撑 8 个并行消费者。

**Step 2：实现批量消费**

```go
// mqx/batch_consumer.go - 新增批量消费能力
type BatchConsumer[T any] struct {
    topic      string
    batchSize  int           // 批量大小，建议 50-100
    flushInterval time.Duration // 最大等待时间，建议 100ms
    consumer   sarama.Consumer
    fn         func(ctx context.Context, messages []T) error
}

func (c *BatchConsumer[T]) Start(ctx context.Context) error {
    mqChan, err := c.consumer.Consume(ctx, c.topic)
    if err != nil {
        return err
    }

    batch := make([]T, 0, c.batchSize)
    timer := time.NewTimer(c.flushInterval)

    for {
        select {
        case <-ctx.Done():
            return c.flush(ctx, batch) // 优雅退出，处理剩余
        case msg, ok := <-mqChan:
            if !ok {
                return c.flush(ctx, batch)
            }
            var t T
            if err := json.Unmarshal(msg.Value, &t); err != nil {
                continue
            }
            batch = append(batch, t)
            if len(batch) >= c.batchSize {
                _ = c.flush(ctx, batch)
                batch = batch[:0]
                timer.Reset(c.flushInterval)
            }
        case <-timer.C:
            if len(batch) > 0 {
                _ = c.flush(ctx, batch)
                batch = batch[:0]
            }
            timer.Reset(c.flushInterval)
        }
    }
}

func (c *BatchConsumer[T]) flush(ctx context.Context, batch []T) error {
    if len(batch) == 0 {
        return nil
    }
    return c.fn(ctx, batch)
}
```

**Step 3：消费端批量入库**

```go
// 配合 HandleReports 改为批量处理
func (svc *ExecutionService) HandleReportsBatch(
    ctx context.Context, reports []domain.ExecutionReport) error {
    // 按状态分组，批量更新
    groups := groupByStatus(reports)
    g, gCtx := errgroup.WithContext(ctx)
    for status, group := range groups {
        status, group := status, group
        g.Go(func() error {
            return svc.repo.BatchUpdateStatus(gCtx, status, group)
        })
    }
    return g.Wait()
}
```

**预期收益**：
- 分区扩展：消费并行度从 1 提升到 8，**吞吐 8x**
- 批量消费：减少 DB 写入次数（100 条 1 次 vs 100 次），**写入效率 3-5x**
- 综合：消息处理吞吐 **提升 10-20 倍**

### 3.2 事件驱动调度：消除固定轮询延迟

**当前问题**

```go
// scheduler.go L158-163
for {
    // ...
    if len(tasks) == 0 {
        time.Sleep(s.config.ScheduleInterval) // 固定睡 10 秒！
    }
}
```

当有新任务入库时，最坏情况要等 10 秒才会被调度。这不仅是延迟问题，也是资源浪费——空闲时每 10 秒做一次无意义的 DB 查询。

**优化方案：Channel 通知 + 指数退避**

```go
type Scheduler struct {
    // 新增：外部通知 channel
    wakeupCh chan struct{}
    // ...
}

func NewScheduler(/* ... */) *Scheduler {
    return &Scheduler{
        wakeupCh: make(chan struct{}, 1), // 带 buffer，避免发送阻塞
        // ...
    }
}

// 外部调用：新任务创建时通知调度器
func (s *Scheduler) NotifyNewTask() {
    select {
    case s.wakeupCh <- struct{}{}:
    default: // channel 满了说明已经有通知在等，不需要重复发
    }
}

func (s *Scheduler) scheduleLoop(ctx context.Context) {
    backoff := time.Second // 初始退避 1 秒
    maxBackoff := s.config.ScheduleInterval // 最大退避 10 秒

    for {
        select {
        case <-ctx.Done():
            return
        default:
        }

        tasks, err := s.scheduleOneLoop(ctx)
        if err != nil {
            time.Sleep(backoff)
            backoff = min(backoff*2, maxBackoff)
            continue
        }

        if len(tasks) > 0 {
            backoff = time.Second // 有任务，重置退避
            continue             // 立即进入下一轮，不等待
        }

        // 无任务：等待通知或超时
        select {
        case <-ctx.Done():
            return
        case <-s.wakeupCh:
            backoff = time.Second // 收到通知，重置退避
        case <-time.After(backoff):
            backoff = min(backoff*2, maxBackoff) // 指数退避
        }
    }
}
```

**在任务创建时集成通知**：

```go
// service 层
func (svc *TaskService) CreateTask(ctx context.Context, task domain.Task) error {
    err := svc.repo.Create(ctx, task)
    if err != nil {
        return err
    }
    // 通知调度器有新任务
    svc.scheduler.NotifyNewTask()
    return nil
}
```

**预期收益**：
- 新任务调度延迟：从最坏 10s 降到 **< 100ms**
- 空闲时 DB 压力：从固定 0.1 QPS 降到几乎为 0
- 繁忙时：连续调度无等待，最大化吞吐

### 3.3 批量 CAS Acquire：减少事务开销

**当前问题**

```go
// task.go L137-170 - 每个任务一个事务
func (dao *TaskDAO) Acquire(ctx context.Context, task Task) (Task, error) {
    return dao.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
        // UPDATE tasks SET ... WHERE id = ? AND status = ?
        // SELECT * FROM tasks WHERE id = ?
        // 一个事务 = 一次网络往返 + 事务开销
    })
}
```

100 个任务 = 100 个独立事务 = 100 次网络往返（假设 1ms/次 = 100ms），加上事务管理开销。

**优化方案：批量 CAS + Pipeline**

```go
func (dao *TaskDAO) BatchAcquire(ctx context.Context,
    tasks []Task, nodeID string) ([]Task, error) {

    // 方案 A：单事务批量 CAS（简单高效）
    var acquired []Task
    err := dao.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
        ids := make([]int64, len(tasks))
        for i, t := range tasks {
            ids[i] = t.Id
        }

        // 批量 UPDATE：一条 SQL 搞定
        now := time.Now().UnixMilli()
        result := tx.Model(&Task{}).
            Where("id IN ? AND (status = ? OR (status = ? AND utime <= ?))",
                ids,
                domain.TaskStatusWaiting,
                domain.TaskStatusRunning,
                now-int64(dao.preemptedTimeout.Milliseconds()),
            ).
            Updates(map[string]interface{}{
                "status":           domain.TaskStatusRunning,
                "schedule_node_id": nodeID,
                "utime":           now,
            })

        if result.Error != nil {
            return result.Error
        }

        if result.RowsAffected == 0 {
            return nil
        }

        // 批量查询被成功 acquire 的任务
        return tx.Where("id IN ? AND schedule_node_id = ? AND utime = ?",
            ids, nodeID, now).
            Find(&acquired).Error
    })

    return acquired, err
}
```

**预期收益**：100 个事务 → 1 个事务（含 2 条 SQL），**事务开销降低 50-100 倍**

## 四、P2 优化：释放系统潜力

### 4.1 MySQL 连接池调优

**当前问题**

```yaml
# config.yaml - 没有连接池配置
mysql:
  dsn: "root:root@tcp(localhost:3306)/distributed_task_platform?timeout=1s&readTimeout=3s&writeTimeout=3s"
```

Go 的 `database/sql` 默认：
- `MaxOpenConns = 0`（无限制，实际受 MySQL `max_connections` 限制）
- `MaxIdleConns = 2`（极少！每次新请求大概率要新建连接）
- `ConnMaxLifetime = 0`（连接永不过期）

在高并发下，每次请求都可能触发新建 TCP 连接 + MySQL 握手，延迟 2-5ms/次。

**优化方案**

```go
// ioc/db.go 中初始化数据库连接后，添加连接池配置
func InitDB(cfg config.MySQLConfig) *gorm.DB {
    db, err := gorm.Open(mysql.Open(cfg.DSN))
    // ...

    sqlDB, _ := db.DB()

    // 连接池关键参数
    sqlDB.SetMaxOpenConns(50)          // 最大打开连接数
    sqlDB.SetMaxIdleConns(25)          // 最大空闲连接数（建议 = MaxOpen/2）
    sqlDB.SetConnMaxLifetime(30 * time.Minute) // 连接最大存活时间
    sqlDB.SetConnMaxIdleTime(5 * time.Minute)  // 空闲连接最大存活时间

    return db
}
```

**参数计算依据**：
- `MaxOpenConns = 50`：假设 4 核 MySQL 实例，建议 `(core_count * 2) + effective_spindle_count`，SSD 场景约 50
- `MaxIdleConns = 25`：保持一半连接空闲复用
- 实际应根据部署规模调整：`MaxOpenConns = TotalMySQLMaxConns / AppInstanceCount`

**配置文件新增字段**：

```yaml
mysql:
  dsn: "root:root@tcp(localhost:3306)/distributed_task_platform?timeout=1s&readTimeout=3s&writeTimeout=3s"
  maxOpenConns: 50
  maxIdleConns: 25
  connMaxLifetimeMinutes: 30
  connMaxIdleTimeMinutes: 5
```

**预期收益**：消除连接建立开销，高并发下延迟降低 **30-50%**

### 4.2 速率限制器参数调优

**当前问题**

```yaml
# config.yaml
loadChecker:
  limiter:
    rateLimit: 10.0   # 每秒 10 个请求！
    burstSize: 20      # 突发 20 个
```

速率限制器的作用是保护系统不被过载，但 **10 req/s** 的限制太低了。当调度器并发分发任务后，很快就会被限流。

**优化方案**

```yaml
loadChecker:
  limiter:
    rateLimit: 200.0   # 提升到 200 req/s
    burstSize: 500      # 突发 500，应对批量调度
```

注意：速率限制器应该是**最后一道防线**，而不是正常流量的瓶颈。实际值应根据 Worker 集群的处理能力动态调整。

更进一步，可以实现自适应限流：

```go
// 自适应速率限制器
type AdaptiveRateLimiter struct {
    limiter    *rate.Limiter
    baseRate   float64
    maxRate    float64
    successCnt atomic.Int64
    failureCnt atomic.Int64
}

func (l *AdaptiveRateLimiter) Adjust() {
    success := l.successCnt.Swap(0)
    failure := l.failureCnt.Swap(0)

    if failure == 0 && success > 0 {
        // 全部成功，提高速率
        newRate := math.Min(l.limiter.Limit()*1.1, l.maxRate)
        l.limiter.SetLimit(rate.Limit(newRate))
    } else if failure > success/10 {
        // 失败率 > 10%，降低速率
        newRate := math.Max(l.limiter.Limit()*0.8, l.baseRate)
        l.limiter.SetLimit(rate.Limit(newRate))
    }
}
```

### 4.3 MQ 消费端并发处理

**当前问题**

即使 Kafka 分区扩展了，单个 consumer 内部仍然是串行处理：

```go
// report_event_consumer.go - 收到消息后同步处理
func (c *ReportEventConsumer) Consume(ctx context.Context, msg *sarama.ConsumerMessage) error {

    // 调用 execSvc.HandleReports - 同步等待 DB 操作完成
```

**优化方案：消费端 Worker Pool**

```go
type ConcurrentConsumer struct {
    workerPool chan struct{} // 并发控制信号量
    fn         func(ctx context.Context, msg *sarama.ConsumerMessage) error
}

func NewConcurrentConsumer(concurrency int, fn func(ctx context.Context, msg *sarama.ConsumerMessage) error) *ConcurrentConsumer {
    return &ConcurrentConsumer{
        workerPool: make(chan struct{}, concurrency),
        fn:         fn,
    }
}

func (c *ConcurrentConsumer) Consume(ctx context.Context, msg *sarama.ConsumerMessage) {
    c.workerPool <- struct{}{} // 获取 worker 槽位
    go func() {
        defer func() { <-c.workerPool }() // 释放槽位
        if err := c.fn(ctx, msg); err != nil {
            // 错误处理：重试或记录
        }
    }()
}
```

## 五、P3 优化：精细化打磨

### 5.1 补偿器：从轮询到事件驱动

**当前问题**

所有补偿器（重试、重调度、超时中断、分片）都是固定 1s 间隔轮询：

```go
// retry.go L55-60
for {
    // 查询可重试的 executions
    // 处理
    // 至少等 1s
}
```

当 executions 表没有需要补偿的数据时，这些轮询完全是浪费。4 个补偿器 × 每秒 1 次 = **4 QPS 的无效 DB 查询**。

**优化方案：延迟队列 + 事件触发**

```go
// 使用 Redis Sorted Set 作为延迟队列
type DelayedRetryQueue struct {
    rdb redis.Cmdable
    key string
}

// 在 execution 状态变为 FailedRetryable 时，写入延迟队列
func (q *DelayedRetryQueue) Enqueue(ctx context.Context,
    executionID int64, nextRetryTime time.Time) error {
    return q.rdb.ZAdd(ctx, q.key, redis.Z{
        Score:  float64(nextRetryTime.UnixMilli()),
        Member: executionID,
    }).Err()
}

// 补偿器改为：从延迟队列拉取到期的 execution
func (q *DelayedRetryQueue) Dequeue(ctx context.Context,
    now time.Time, batchSize int) ([]int64, error) {
    results, err := q.rdb.ZRangeByScore(ctx, q.key, &redis.ZRangeBy{
        Min:   "-inf",
        Max:   strconv.FormatInt(now.UnixMilli(), 10),
        Count: int64(batchSize),
    }).Result()
    // ...
}
```

**预期收益**：
- 无需补偿时：DB 查询降为 0
- 需要补偿时：精确触发，延迟更低（从最坏 1s 降到毫秒级）

### 5.2 SetRunningState：合并两次 DB 操作

**当前问题**

```go
// task_execution.go L275-308
func (dao *TaskExecutionDAO) SetRunningState(ctx context.Context, eid int64, ...) error {
    // 第一次查询：获取 TaskMaxExecutionSeconds 用于计算 deadline
    var exec TaskExecution
    dao.db.Where("id = ?", eid).First(&exec)

    // 第二次操作：更新状态
    dao.db.Where("id = ? AND status = ?", eid, "WAITING").
        Updates(map[string]interface{}{
            "status":   "RUNNING",
            "deadline": time.Now().Add(exec.TaskMaxExecutionSeconds),
        })
}
```

两次 DB 操作可以合并。

**优化方案：使用 SQL 表达式计算**

```go
func (dao *TaskExecutionDAO) SetRunningState(ctx context.Context,
    eid int64, nodeID string) error {
    now := time.Now().UnixMilli()

    // 单条 SQL 完成：利用 SQL 表达式直接计算 deadline
    result := dao.db.WithContext(ctx).
        Model(&TaskExecution{}).
        Where("id = ? AND status = ?", eid, "WAITING").
        Updates(map[string]interface{}{
            "status":        "RUNNING",
            "worker_node_id": nodeID,
            "start_time":    now,
            // 使用 SQL 表达式引用自身字段
            "deadline": gorm.Expr("? + task_max_execution_seconds * 1000", now),
            "utime":   now,
        })

    if result.RowsAffected == 0 {
        return fmt.Errorf("execution %d 状态不是 WAITING，无法设为 RUNNING", eid)
    }
    return result.Error
}
```

**预期收益**：每次状态变更少一次 DB 查询，在高频调度下效果显著。

### 5.3 Snowflake ID 优化：分段预分配

**当前问题**

```go
// snowflake.go - 全局原子序列号
var sequence atomic.Int64

func (g *Generator) Generate(bizID int64) int64 {
    seq := sequence.Add(1)  // 全局争用
    // ...
}
```

所有 goroutine 竞争同一个 `atomic.Int64`，高并发下 CAS 重试增多。

**优化方案：Per-Goroutine 序列号预分配**

```go
type Generator struct {
    seqPool sync.Pool // 每个 goroutine 持有一段序列号
}

type seqBlock struct {
    current int64
    end     int64
}

var globalSeq atomic.Int64

func (g *Generator) Generate(bizID int64) int64 {
    block := g.seqPool.Get().(*seqBlock)
    defer g.seqPool.Put(block)

    if block.current >= block.end {
        // 批量分配 256 个序列号
        start := globalSeq.Add(256)
        block.current = start - 256
        block.end = start
    }

    seq := block.current
    block.current++

    // ... 组装 ID
}
```

**预期收益**：全局原子操作频率降低 256 倍，高并发下减少 CPU 缓存行竞争。

### 7.2 端到端吞吐量推算（单节点）

要计算真实 QPS，需要把调度链路的每个环节串起来算：

**当前系统（单节点吞吐上限）**

```
调度链路：FindSchedulable → Acquire(逐个) → LoadCheck → Run(逐个gRPC)
           ~10ms           ~100ms         ~100ms*    ~500ms
                                          (限流等待)

单轮耗时 = 10 + 100 + 100 + 500 = ~710ms
单轮任务 = 100（batchSize）
有任务时不 sleep，直接下一轮

理论吞吐 = 100 / 0.71s ≈ 140 tasks/s

但！LimiterChecker = 10 req/s, burst 20
实际每秒只能通过 10 个任务的 LoadCheck
→ 实际吞吐天花板：~10 tasks/s ← 被限流器硬卡
```

**\* 注意**：当前 LimiterChecker 是在 `runner.Run()` 里逐个检查的，10 req/s 的 Token Bucket 意味着第 20 个任务之后就要等令牌，极端情况下 100 个任务要等 ~10 秒才能全部通过。

**优化后（单节点吞吐上限）**

```
优化后链路：FindSchedulable → BatchAcquire → 并发 Run(Worker Pool)
            ~2ms              ~2ms           ~30ms (20并发, 5轮)

单轮耗时 = 2 + 2 + 30 = ~34ms

batchSize 从 100 → 500（放大配合）
单轮任务 = 500
速率限制 = 1000 req/s, burst 2000（不再是瓶颈）

理论吞吐 = 500 / 0.034s ≈ 14,700 tasks/s（CPU/网络饱和前的理论值）

实际受限于：
1. MySQL 写入能力（BatchAcquire 500行/次 ≈ 5ms）
2. gRPC Worker 节点处理能力
3. 连接池大小（50 连接 × 20 QPS/连接 = 1000 DB QPS）
4. 网络带宽

保守估计单节点实际吞吐：1,000 - 3,000 tasks/s
```

**batchSize 放大是关键一环**：

当前 batchSize=100 是因为串行处理下 100 个已经要 600ms+，放大没意义。但并发化后单轮只要 30ms，完全可以放大到 500 甚至 1000：

```yaml
# 优化后的调度器配置
scheduler:
  batchSize: 500              # 100 → 500
  scheduleInterval: 5000000000  # 10s → 5s（仅作为最大退避）
  maxConcurrentDispatch: 50   # 新增：并发分发数
  maxConcurrentTasks: 5000    # 1000 → 5000

loadChecker:
  limiter:
    rateLimit: 1000.0         # 10 → 1000
    burstSize: 2000           # 20 → 2000
```

### 7.3 水平扩展能力

当前架构通过 etcd 做服务发现 + 分布式锁实现多节点调度，天然支持水平扩展：

| 调度节点数 | 单节点吞吐（优化后） | 集群总吞吐 | 扩展效率 |
|-----------|-------------------|-----------|---------|
| 1 | 1,000-3,000 tasks/s | 1,000-3,000 tasks/s | - |
| 3 | 1,000-3,000 tasks/s | 2,500-7,500 tasks/s | ~83%* |
| 5 | 1,000-3,000 tasks/s | 4,000-12,000 tasks/s | ~80%* |
| 10 | 1,000-3,000 tasks/s | 7,000-20,000 tasks/s | ~70%* |

*\* 扩展效率随节点数递减，主要受限于：*
- *MySQL 锁竞争（BatchAcquire 的 CAS 冲突率随节点数增加）*
- *etcd 分布式锁开销*
- *Kafka broker 吞吐上限*

**V2 Sharding 架构的优势在这里体现**：任务分库分表后，不同调度节点可以锁定不同分片，CAS 冲突率大幅下降，扩展效率可达 90%+。

### 7.4 瓶颈转移预判

优化到 1000+ QPS 后，新的瓶颈不再是调度器本身，而是：

| 新瓶颈 | 出现条件 | 应对方案 |
|--------|---------|---------|
| MySQL 写入 IOPS | > 5,000 tasks/s | 分库分表（V2 已支持）、SSD、读写分离 |
| Worker 节点处理能力 | Worker CPU 饱和 | 水平扩容 Worker 集群 |
| Kafka broker 吞吐 | > 10,000 msg/s | 扩 broker 节点、增加分区 |
| 网络带宽 | gRPC payload 大 | 压缩、protobuf 优化 |
| etcd 性能 | > 10 个调度节点竞争 | V2 Sharding 减少锁范围 |

### 7.5 总结

| | 当前 | 优化后（单节点） | 优化后（3 节点集群） |
|--|------|-----------------|-------------------|
| **调度吞吐** | ~10 tasks/s | 1,000-3,000 tasks/s | 2,500-7,500 tasks/s |
| **调度延迟** | 最坏 10s | < 100ms | < 100ms |
| **QPS 量级** | 10¹ | 10³ | 10³-10⁴ |
| **提升倍数** | 基准 | **100-300x** | **250-750x** |

**结论**：当前系统的核心瓶颈是 LimiterChecker 的 10 req/s 硬限流 + 串行分发，并非架构性缺陷。通过配置调优（限流参数、batchSize、连接池）+ 代码优化（并发分发、批量 CAS、索引补全），**单节点即可从 ~10 tasks/s 提升到 1000-3000 tasks/s，提升两个数量级**。配合 V2 Sharding 架构的水平扩展，集群吞吐可达万级 TPS。
