## 1. 平台概述

分布式任务调度平台是一个企业级的任务调度中台，提供以下核心能力：

```
┌──────────────────────────────────────────────────────────────┐
│                        接入层                                 │
│              gRPC SDK  /  测试脚本  /  管理后台               │
└────────────────────────────┬─────────────────────────────────┘
                             │
         ┌───────────────────┼───────────────────┐
         │                   │                   │
    ┌────▼─────┐       ┌────▼─────┐       ┌────▼─────┐
    │Scheduler │       │Scheduler │       │Scheduler │
    │   Node 1 │       │   Node 2 │       │   Node N │
    └────┬─────┘       └────┬─────┘       └────┬─────┘
         │ CAS 乐观锁抢占   │  Prometheus 智能选节点 │
         └────────┬─────────┴─────────────────────────┘
                  │ gRPC 双向通信
     ┌────────────┼──────────────────┐
     │            │                  │
┌────▼────┐ ┌────▼────┐      ┌──────▼──────┐
│Executor │ │Executor │      │  Executor   │
│ (Spark) │ │ (Flink) │ ...  │  (自定义)    │
└─────────┘ └─────────┘      └─────────────┘
                  │
     ┌────────────┼────────────┐
     │            │            │
┌────▼───┐ ┌─────▼────┐ ┌────▼────┐
│ MySQL  │ │  Kafka   │ │  etcd   │
│(分库分表)│ │ (事件流) │ │ (注册)  │
└────────┘ └──────────┘ └─────────┘
```

**核心功能清单：**

| # | 功能 | 说明 |
|---|------|------|
| 1 | 普通任务调度 | Cron 表达式定时触发，CAS 抢占保证唯一执行 |
| 2 | 分片任务 | 大任务拆分为多个子任务并行执行（Range / 权重动态分片） |
| 3 | DAG 工作流 | ANTLR4 DSL 定义任务依赖，支持顺序/并行/选择/条件分支 |
| 4 | 智能调度 | Prometheus 实时指标驱动 TopN 节点选择 |
| 5 | 三级负载检查 | 令牌桶限流 → 数据库响应检查 → 集群负载均衡 |
| 6 | 四重补偿 | 重试 / 重调度 / 中断 / 分片聚合 |
| 7 | 异步状态上报 | gRPC 直接上报 + Kafka 异步消息双链路 |
| 8 | 服务发现 | etcd Lease + Watch，秒级节点感知 |
| 9 | 分库分表 | 2 库 × 多表，Snowflake ID 内嵌分片路由 |
| 10 | 可观测性 | Prometheus + Grafana + 分布式追踪 |

## 2. 任务管理

### 2.1 任务类型

平台支持两种任务类型：

| 任务类型 | TaskType 值 | 说明 | Runner |
|---------|------------|------|--------|
| **普通任务** | `normal` | 单个任务直接执行，支持分片 | `NormalTaskRunner` |
| **工作流任务** | `plan` | DAG 工作流编排，包含多个子任务 | `PlanTaskRunner` |

**执行方式：**

| 执行方式 | ExecutionMethod 值 | 说明 |
|---------|-------------------|------|
| **本地执行** | `LOCAL` | 在调度器进程内直接执行（测试用） |
| **远程执行** | `REMOTE` | 通过 gRPC 调用远端执行器 |

**调度策略：**

| 调度策略 | SchedulingStrategy 值 | 说明 |
|---------|----------------------|------|
| **CPU 优先** | `CPU_PRIORITY` | PromQL 查询 CPU 空闲率最高的节点 |
| **内存优先** | `MEMORY_PRIORITY` | PromQL 查询可用内存最大的节点 |

### 2.2 任务状态机

```
                  ┌──────────────┐
                  │    ACTIVE    │ ← 初始状态 / 执行完成后复位
                  │  （可调度）    │
                  └──────┬───────┘
                         │ CAS 抢占成功
                         ▼
                  ┌──────────────┐
                  │  PREEMPTED   │ ← 被某个调度节点抢占
                  │  （已抢占）    │
                  └──────┬───────┘
                    ┌────┴────┐
                    │         │
                    ▼         ▼
             执行完成复位    手动停止
                    │         │
                    ▼         ▼
             ┌──────────┐  ┌──────────┐
             │  ACTIVE  │  │ INACTIVE │
             └──────────┘  │ （已停止） │
                           └──────────┘
```

**执行记录状态机：**

```
  PREPARE ──────────────────→ RUNNING
    │                           │
    │ 超时未启动                  ├─→ SUCCESS（成功）
    ▼                           │
  中断补偿器处理                  ├─→ FAILED（不可恢复失败）
                               │
                               ├─→ FAILED_RETRYABLE ──→ 重试补偿器 ──→ PREPARE
                               │
                               └─→ FAILED_RESCHEDULED ──→ 重调度补偿器 ──→ PREPARE
```

### 2.3 任务创建示例

#### 示例 1：创建普通远程任务

```go
package main

import (
    "time"
    "gitee.com/flycash/distributed_task_platform/internal/domain"
    "gitee.com/flycash/distributed_task_platform/internal/repository/dao"
    "gitee.com/flycash/distributed_task_platform/pkg/sqlx"
)

func createNormalTask(taskDAO dao.TaskDAO) {
    task := dao.Task{
        Name:     "user_data_sync",
        CronExpr: "0 */30 * * * ?",    // 每 30 分钟执行一次
        Type:     domain.NormalTaskType.String(),   // "normal"
        ExecutionMethod: domain.TaskExecutionMethodRemote.String(), // "REMOTE"
        GrpcConfig: sqlx.JSONColumn[domain.GrpcConfig]{
            Valid: true,
            Val: domain.GrpcConfig{
                ServiceName: "dataSyncService",  // etcd 中注册的服务名
                Params: map[string]string{
                    "source": "mysql",
                    "target": "elasticsearch",
                },
            },
        },
        RetryConfig: sqlx.JSONColumn[domain.RetryConfig]{
            Valid: true,
            Val: domain.RetryConfig{
                MaxRetries:      3,
                InitialInterval: 30000,   // 30 秒
                MaxInterval:     300000,  // 5 分钟
            },
        },
        ScheduleParams: sqlx.JSONColumn[map[string]string]{
            Valid: true,
            Val: map[string]string{
                "batch_size": "1000",
            },
        },
        MaxExecutionSeconds: 3600,  // 最长 1 小时
        Status:  domain.TaskStatusActive.String(),
        Version: 1,
        NextTime: time.Now().Add(3 * time.Second).UnixMilli(),
    }
    _, err := taskDAO.Create(ctx, task)
    // handle err
}
```

#### 示例 2：创建本地执行任务（测试用）


## 3. 调度引擎

### 3.1 V1 调度器（单机全量）

V1 调度器适合中小规模场景，单机全量扫描数据库：

```
调度循环：
  ┌──→ 查询可调度任务（status=ACTIVE, next_time <= now）
  │         │
  │         ▼
  │    批量 CAS 抢占（UPDATE ... WHERE version = ?）
  │         │
  │         ▼
  │    通过 Runner 分发执行（Normal / Plan）
  │         │
  │         ▼
  │    负载检查（是否需要降频）
  │         │
  │         ▼
  └── sleep(scheduleInterval) ──┘
```

**关键配置：**

```yaml
scheduler:
  batchSize: 100              # 单次抢占最大任务数
  scheduleInterval: 10s       # 调度循环间隔
  preemptedTimeout: 10m       # 抢占超时（超时后其他节点可重新抢占）
  renewInterval: 5s           # 续约间隔
  maxConcurrentTasks: 1000    # 最大并发任务数
```

### 3.2 V2 调度器（分布式分片）

V2 调度器通过 `ShardingLoopJob` + 分布式锁实现分片调度，适合大规模场景：

```
                    ┌────────────────────────┐
                    │   ShardingLoopJob      │
                    │   框架层               │
                    └───────┬────────────────┘
                            │
            ┌───────────────┼───────────────┐
            ▼               ▼               ▼
       ┌─────────┐    ┌─────────┐    ┌─────────┐
       │ 分片 0  │    │ 分片 1  │    │ 分片 N  │
       │ task_0  │    │ task_1  │    │ task_N  │
       └────┬────┘    └────┬────┘    └────┬────┘
            │              │              │
            ▼              ▼              ▼
       分布式锁保护     分布式锁保护     分布式锁保护
       scheduleOneLoop scheduleOneLoop  scheduleOneLoop
```

**V1 vs V2 对比：**

| 特性 | V1 | V2 |
|------|----|----|
| 数据扫描范围 | 全量扫描 | 分片扫描 |
| 并发控制 | 单机 WaitGroup | 分布式锁 + 资源信号量 |
| 水平扩展 | 不支持 | 自动瓜分分片 |
| 适用场景 | 任务数 < 1 万 | 任务数 10 万+ |

### 3.3 CAS 乐观锁抢占

任务抢占使用数据库乐观锁，保证多节点下同一任务只被一个调度器处理：

```sql
-- 抢占任务
UPDATE task
SET status = 'PREEMPTED',
    schedule_node = :nodeID,
    version = version + 1
WHERE id = :taskID
  AND status = 'ACTIVE'
  AND version = :currentVersion;

-- 返回 affected_rows = 1 → 抢占成功
-- 返回 affected_rows = 0 → 被其他节点抢走了
```

### 3.4 任务续约机制

调度器抢占任务后，通过定时续约防止任务被其他节点误以为超时：

```
续约循环（每 5s）：
  UPDATE task
  SET utime = NOW()
  WHERE schedule_node = :nodeID
    AND status = 'PREEMPTED';
```

如果调度器宕机，续约停止，超过 `preemptedTimeout`（默认 10 分钟）后，其他调度器可重新抢占该任务。

## 4. 智能调度

### 4.1 Prometheus 驱动节点选择

调度器通过 PromQL 实时查询执行器集群的资源指标，选择最优节点：

```
                   ┌─────────────────┐
                   │   Scheduler     │
                   │                 │
                   │  PrometheusPicker│
                   └────────┬────────┘
                            │ PromQL Query
                            ▼
                   ┌─────────────────┐
                   │   Prometheus    │
                   │                 │
                   │  executor_cpu_  │
                   │  idle_percent   │
                   │  executor_mem_  │
                   │  available_bytes│
                   └────────┬────────┘
                            │ TopN 结果
                            ▼
                   Random(TopN) → nodeID
```

### 4.2 TopN 随机算法

避免所有调度器都选同一个"最优"节点（羊群效应）：

1. PromQL 查询 `topk(N, avg_over_time(metric[window]))` 获取前 N 个候选节点
2. 从 N 个候选中**随机选择 1 个**
3. 如果智能调度失败，fallback 到默认的轮询策略

### 4.3 调度策略配置

```yaml
# 智能调度配置
intelligentScheduling:
  jobName: "executors"         # Prometheus 中执行节点的 job 名称
  topNCandidates: 3            # TopN 候选节点数
  timeWindow: 30s              # 指标查询时间窗口
  queryTimeout: 5s             # Prometheus 查询超时

# 任务创建时指定策略
# SchedulingStrategy: "CPU_PRIORITY"   → 查询 executor_cpu_idle_percent
# SchedulingStrategy: "MEMORY_PRIORITY" → 查询 executor_memory_available_bytes
```

**PromQL 示例：**

```promql
# CPU 优先策略
topk(3,
  avg_over_time(executor_cpu_idle_percent{job="executors"}[30s])
  AND ON(instance) up == 1
)

# 内存优先策略
topk(3,
  avg_over_time(executor_memory_available_bytes{job="executors"}[30s])
  AND ON(instance) up == 1
)
```

## 5. 负载检查器

### 5.1 三级检查架构

```
           调度循环
              │
              ▼
    ┌──────────────────┐
    │ CompositeChecker │ ← AND 模式：全部通过才放行
    │                  │
    │  ┌─────────────┐ │
    │  │  Level 1    │ │  令牌桶限流（硬限制）
    │  │  Limiter    │ │  10 QPS / burst=20
    │  └──────┬──────┘ │
    │         │ PASS   │
    │  ┌──────▼──────┐ │
    │  │  Level 2    │ │  数据库响应检查
    │  │  Database   │ │  阈值 100ms
    │  └──────┬──────┘ │
    │         │ PASS   │
    │  ┌──────▼──────┐ │
    │  │  Level 3    │ │  集群负载均衡检查
    │  │  Cluster    │ │  超过均值 20% 降频
    │  └─────────────┘ │
    └──────────────────┘
              │
              ▼
      shouldSchedule = true/false
      sleepDuration（如需退避）
```

### 5.2 令牌桶限流器

硬限制调度频率，防止调度器"疯跑"打垮下游：

```yaml
loadChecker:
  limiter:
    rateLimit: 10.0           # 每秒允许 10 次调度
    burstSize: 20             # 突发容量 20
    waitDuration: 5s          # 限流时等待 5 秒
```

### 5.3 数据库负载检查

通过 PromQL 查询 MySQL 的平均查询延迟，自适应感知 DB 压力：

```yaml
loadChecker:
  database:
    threshold: 100ms          # DB 响应超过 100ms 触发退避
    timeWindow: 5m            # 查询窗口 5 分钟
    backoffDuration: 10s      # 退避 10 秒
```

### 5.4 集群负载均衡检查

检查当前调度节点的调度耗时是否显著高于集群平均值：

```yaml
loadChecker:
  cluster:
    thresholdRatio: 1.2       # 超过均值 20% 触发降频
    timeWindow: 5m            # 查询窗口 5 分钟
    slowdownMultiplier: 2.0   # 降频乘数，sleep 时间翻倍
    minBackoffDuration: 5s    # 最小退避 5 秒
```

### 5.5 组合检查器（AND/OR）

```go
// AND 模式：全部检查器都通过才允许继续调度
composite := loadchecker.NewCompositeChecker(
    loadchecker.ModeAND,
    limiterChecker,
    databaseChecker,
    clusterChecker,
)

// OR 模式：任一检查器通过即允许
composite := loadchecker.NewCompositeChecker(
    loadchecker.ModeOR,
    checker1,
    checker2,
)
```

## 6. 分片任务

### 6.1 Range 分片

将数据按固定步长平均划分，适合等量数据处理：

```
总数据量: totalNums × step
分片数 = totalNums

  分片 0: [0, step)
  分片 1: [step, 2×step)
  分片 2: [2×step, 3×step)
  ...
  分片 N-1: [(N-1)×step, N×step)
```

**参数：**

| 参数 | 含义 | 示例 |
|------|------|------|
| `totalNums` | 分片数量 | `"10"` |
| `step` | 每片步长 | `"1000"` |

### 6.2 Weighted Dynamic Range 分片

根据执行节点权重动态分配数据范围，高性能节点处理更多数据：

```
total_tasks = 100000
节点 A (weight=3): 处理 [0, 30000)
节点 B (weight=2): 处理 [30000, 50000)  ← 实际可能有舍入
节点 C (weight=5): 处理 [50000, 100000) ← 最后一个节点兜底
```

**参数：**

| 参数 | 含义 | 来源 |
|------|------|------|
| `total_tasks` | 业务总数据量 | 通过 `Prepare` RPC 向执行器查询 |
| 节点权重 | 各节点处理能力系数 | 服务注册时携带的 Weight 字段 |

### 6.3 分片任务创建示例


## 7. DAG 工作流引擎

### 7.1 DSL 语法规范

平台使用基于 ANTLR4 的自定义 DSL 语言来定义 DAG 工作流。任务名由字母、数字和下划线组成。

**操作符：**

| 操作符 | 名称 | 语义 | 示例 |
|--------|------|------|------|
| `->` | 顺序执行 | 前一个完成后才执行下一个 | `A -> B -> C` |
| `&&` | 并行执行 (AND) | 所有任务并行启动，全部成功才继续 | `A && B && C` |
| `\|\|` | 选择执行 (OR) | 所有任务并行启动，任一成功即继续 | `A \|\| B` |
| `? :` | 条件分支 | 前置成功执行 `?` 分支，否则 `:` 分支 | `A ? B : C` |
| `()` | 分组 | 改变优先级 | `(A && B) -> C` |

### 7.2 节点类型与语义

| 节点类型 | 检查前驱条件 | 计算后继 |
|---------|------------|---------|
| **Single** | 前置任务必须成功 | 直接执行后继 |
| **And** | 所有前置任务必须成功 | 全部前置完成后执行后继 |
| **Or** | 任一前置任务成功即可 | 第一个成功的前置触发后继 |
| **Condition** | 前置必须成功 | 成功 → `?` 分支；失败 → `:` 分支 |

### 7.3 DAG 工作流示例

#### 示例 1：数据 ETL 管线

```
# 并行采集 → 清洗 → 并行转换 → 聚合 → 条件分支
(order_sync && user_sync && product_sync)
-> data_cleaning
-> (order_transform && user_transform)
-> data_aggregate
-> quality_check
? generate_report
: alert_notification
```

**执行流程：**

```
order_sync ──┐
user_sync ───┤ AND 全部成功
product_sync ┘
      │
      ▼
data_cleaning
      │
      ▼
order_transform ──┐
user_transform ───┘ AND 全部成功
      │
      ▼
data_aggregate
      │
      ▼
quality_check
      │
  成功? ──┬──→ generate_report
         └──→ alert_notification
```

#### 示例 2：简单顺序流

```
extract_data -> transform_data -> load_data
```

#### 示例 3：带备用路径

```
primary_api || backup_api -> process_result
```

`primary_api` 和 `backup_api` 同时启动，任一成功即执行 `process_result`。

#### 示例 4：任务创建

```go
plan := dao.Plan{
    Name:     "etl_pipeline",
    CronExpr: "0 0 2 * * ?",   // 每天凌晨 2 点
    ExecExpr: "(order_sync && user_sync) -> data_cleaning -> data_aggregate",
    Status:   domain.TaskStatusActive.String(),
    Version:  1,
    NextTime: time.Now().Add(3 * time.Second).UnixMilli(),
}
```

### 7.4 ANTLR4 解析流程

```
DSL 字符串
    │
    ▼
┌────────────────────────────┐
│ TaskOrchestrationLexer.g4 │ ← 词法分析

             ▼

             ▼

             ▼

```

### 7.5 事件驱动执行

DAG 节点的推进由 Kafka 事件驱动：

```

    ▼
```

## 8. 四重补偿机制

### 8.1 重试补偿器

| 属性 | 值 |
|------|------|
| **触发条件** | `status = FAILED_RETRYABLE` 且 `next_retry_time <= now` |
| **处理策略** | 指数退避重试，排除上次失败的执行节点 |
| **最大重试** | 可配置，默认 3 次 |
| **防空转** | MinDuration 兜底 sleep |

**配置：**

```yaml
compensator:
  retry:
    maxRetryCount: 3
    batchSize: 100
    minDuration: 1s
```

**重试退避策略：**

```
第 1 次重试: initialInterval (30s)
第 2 次重试: initialInterval × 2 (60s)
第 3 次重试: min(initialInterval × 4, maxInterval) (120s, 不超过 maxInterval)
```

**示例 — 任务配置重试：**

```go
RetryConfig: &domain.RetryConfig{
    MaxRetries:      3,
    InitialInterval: 30000,   // 30 秒
    MaxInterval:     300000,  // 5 分钟
}
```

### 8.2 重调度补偿器

| 属性 | 值 |
|------|------|
| **触发条件** | `status = FAILED_RESCHEDULED` |
| **处理策略** | 重新创建执行记录，指定到原执行节点继续执行 |
| **使用场景** | 执行器主动请求重调度（负载过高、需要迁移） |

**与重试的区别：**
- **重试**：排除故障节点，随机选其他节点
- **重调度**：由执行节点主动请求，可以携带恢复参数（如 offset、progress）

**执行器主动请求重调度示例：**

```go
// 执行器在上报状态时携带重调度标记
state := &executorv1.ExecutionState{
    Id:                eid,
    Status:            executorv1.ExecutionStatus_FAILED_RESCHEDULABLE,
    RequestReschedule: true,
    RescheduledParams: map[string]string{
        "offset":   "50000",   // 已处理到的偏移量
        "progress": "50",      // 完成进度
    },
}
```

### 8.3 中断补偿器

| 属性 | 值 |
|------|------|
| **触发条件** | `status = RUNNING` 且超过 `MaxExecutionSeconds` |
| **处理策略** | 向执行节点发送 gRPC `Interrupt` 请求 |
| **响应处理** | 根据执行器返回的状态更新数据库 |

**中断流程：**

```
中断补偿器
    │
    ▼ 查询超时执行记录
    │
    ▼ gRPC InterruptRequest(eid)
    │
    ▼ 执行器取消 context → goroutine 退出
    │
    ▼ 执行器返回 InterruptResponse
    │  (success=true, execution_state=FAILED_RESCHEDULABLE)
    │
    ▼ 更新数据库状态
```

### 8.4 分片聚合补偿器

| 属性 | 值 |
|------|------|
| **触发条件** | 分片父任务 `status = RUNNING` |
| **处理策略** | 检查所有子任务状态，聚合决定父任务最终结果 |
| **完成条件** | 全部子任务成功 → 父 SUCCESS；任一失败 → 父 FAILED |

**配置：**

```yaml
compensator:
  sharding:
    batchSize: 100
    minDuration: 1s
```

## 9. gRPC 服务协议

### 9.1 ExecutorService（调度器 → 执行器）

执行器需要实现此接口来接收调度器的调用：

```protobuf
service ExecutorService {
  // 执行一个任务（或分片）
  rpc Execute(ExecuteRequest) returns (ExecuteResponse);
  // 中断一个正在执行的任务
  rpc Interrupt(InterruptRequest) returns (InterruptResponse);
  // 查询一个任务的执行状态（轮询模式）
  rpc Query(QueryRequest) returns (QueryResponse);
  // 查询业务方任务总数量（分片任务用）
  rpc Prepare(PrepareRequest) returns (PrepareResponse);
}
```

**Execute 请求/响应：**

```protobuf
message ExecuteRequest {
  int64 eid = 1;                    // 执行记录 ID
  int64 task_id = 2;                // 任务 ID
  string task_name = 3;             // 任务名称
  map<string, string> params = 4;   // 业务参数 + 调度参数
  // params 示例:
  //   "start": "0", "end": "1000"         -- 分片范围
  //   "batch_size": "500"                 -- 业务参数
  //   "max_execution_seconds": "3600"     -- 超时时间
}

message ExecuteResponse {
  ExecutionState execution_state = 1;  // 执行状态快照
}
```

**Interrupt 请求/响应：**

```protobuf
message InterruptRequest {
  int64 eid = 1;   // 要中断的执行记录 ID
}

message InterruptResponse {
  bool success = 1;                    // 是否成功中断
  ExecutionState execution_state = 3;  // 中断后的状态
}
```

**Prepare 请求/响应（分片用）：**

```protobuf
message PrepareRequest {
  int64 eid = 1;
  int64 task_id = 2;
  string task_name = 3;
  map<string, string> params = 4;
}

message PrepareResponse {
  map<string, string> params = 1;  // 返回: {"total": "10000"}
}
```

**执行状态枚举：**

```protobuf
enum ExecutionStatus {
  UNKNOWN = 0;             // 未知状态
  RUNNING = 1;             // 正在执行
  FAILED_RETRYABLE = 2;   // 可重试失败
  FAILED_RESCHEDULABLE = 3; // 可重调度失败
  FAILED = 4;              // 不可恢复失败
  SUCCESS = 5;             // 执行成功
}
```

### 9.2 ReporterService（执行器 → 调度器）

执行器通过此接口向调度器上报执行状态：

```protobuf
service ReporterService {
  // 单个状态上报
  rpc Report(ReportRequest) returns (ReportResponse);
  // 批量状态上报
  rpc BatchReport(BatchReportRequest) returns (BatchReportResponse);
}

message ReportRequest {
  ExecutionState execution_state = 1;
}

message BatchReportRequest {
  repeated ReportRequest reports = 1;
}
```

### 9.3 状态上报双链路

```
                  ┌──────────────┐
                  │   执行器      │
                  └──────┬───────┘
                    ┌────┴────┐
                    │         │
                    ▼         ▼
              ┌─────────┐ ┌─────────┐
              │  gRPC   │ │  Kafka  │
              │ Report  │ │ Message │
              │ (同步)   │ │ (异步)  │
              └────┬────┘ └────┬────┘
                   │           │
                   ▼           ▼
              ┌────────────────────┐
              │     调度器          │
              │  ReporterServer    │
              │  + KafkaConsumer   │
              └────────────────────┘
```

- **gRPC 直接上报**：低延迟，适合关键状态变更（SUCCESS/FAILED）
- **Kafka 异步上报**：高吞吐，适合进度更新（RunningProgress）
- **双链路互补**：任一链路可用即可保证状态最终一致

## 10. 服务发现与路由

### 10.1 etcd 注册中心

```
┌───────────┐    Register     ┌──────┐    Watch     ┌───────────┐
│ Executor  │ ──────────────→ │ etcd │ ←─────────── │ Scheduler │
│ (启动注册) │    Lease(60s)  │      │   gRPC       │ (Resolver) │
└───────────┘                 └──────┘              └───────────┘
```

**注册示例：**

```go
// 执行器启动时注册到 etcd
reg.Register(ctx, registry.ServiceInstance{
    Name:    "orderExportService",     // 服务名，对应 GrpcConfig.ServiceName
    Address: "192.168.1.100:8880",     // 执行器地址
    ID:      "node-01",               // 节点 ID
    Weight:  3,                        // 节点权重（分片用）
})
```

**特性：**
- **Lease 机制**：60 秒自动续约，宕机后自动注销
- **Watch 机制**：服务变更实时推送给调度器的 Resolver
- **秒级感知**：节点上下线秒级生效

### 10.2 自定义 gRPC 负载均衡器

平台实现了自定义的 gRPC Balancer，支持多种路由策略：

```go
// 负载均衡器支持的路由模式
type RoutingPicker struct {
    connections []balancer.SubConn
}

func (p *RoutingPicker) Pick(info balancer.PickInfo) (balancer.PickResult, error) {
    ctx := info.Ctx

    // 1. 指定节点路由
    if nodeID := balancer.SpecificNodeIDFromCtx(ctx); nodeID != "" {
        return pickSpecificNode(nodeID)
    }

    // 2. 排除节点路由（故障转移）
    if excludeIDs := balancer.ExcludeNodeIDsFromCtx(ctx); len(excludeIDs) > 0 {
        return pickExcludingNodes(excludeIDs)
    }

    // 3. 默认轮询
    return pickRoundRobin()
}
```

### 10.3 路由策略

| 策略 | 场景 | 实现 |
|------|------|------|
| **指定节点** | 智能调度选定节点 | `balancer.WithSpecificNodeID(ctx, nodeID)` |
| **排除节点** | 重试时排除故障节点 | `balancer.WithExcludeNodeIDs(ctx, []string{failedNodeID})` |
| **轮询** | 默认 fallback | Round-Robin 在所有可用节点间轮询 |

## 11. 分库分表

### 11.1 分片策略

```
            ┌──────────────────────────────────┐
            │         分库分表路由               │
            │                                  │
            │  taskID % 2 → 选库 (task_0/task_1)│
            │  taskID % 2 → 选表 (task_0/task_1)│
            │                                  │
            │  execID % 2 → 选库               │

```

**分片规格：**

| 表 | 分库数 | 分表数/库 | 总表数 |

### 11.2 Snowflake ID 生成器

自定义 Snowflake 变体，ID 中内嵌分片路由信息：

```
┌─────────┬──────────┬──────────┬────────────┐
│  符号位  │  时间戳   │ 机器ID   │  序列号     │
│  1 bit  │  41 bits │ 10 bits  │  12 bits   │
└─────────┴──────────┴──────────┴────────────┘
```

- **全局唯一**：41 位时间戳 + 10 位机器 ID + 12 位序列号
- **分片路由**：`ID % 分片数` 直接路由，无需额外查询
- **趋势递增**：基于时间戳，写入 B+ 树性能好

## 13. 执行器开发指南

### 13.1 实现 ExecutorService 接口

每个业务方需要实现以下 gRPC 接口来接入调度平台：

```go
type ExecutorServiceServer interface {
    // 必须实现：接收并执行任务
    Execute(ctx context.Context, req *ExecuteRequest) (*ExecuteResponse, error)

    // 必须实现：处理中断请求
    Interrupt(ctx context.Context, req *InterruptRequest) (*InterruptResponse, error)

    // 可选：支持轮询模式的状态查询
    Query(ctx context.Context, req *QueryRequest) (*QueryResponse, error)

    // 可选：分片任务的准备阶段（返回业务数据总量）
    Prepare(ctx context.Context, req *PrepareRequest) (*PrepareResponse, error)
}
```

### 13.2 长任务执行器示例

以下是一个完整的长时间运行任务执行器的实现示例：

```go
package main

import (
    "context"
    "fmt"
    "strconv"
    "time"

    executorv1 "gitee.com/flycash/distributed_task_platform/api/proto/gen/executor/v1"
    reporterv1 "gitee.com/flycash/distributed_task_platform/api/proto/gen/reporter/v1"
    "github.com/ecodeclub/ekit/syncx"
)

// LongRunningExecutor 长任务执行器
type LongRunningExecutor struct {
    executorv1.UnimplementedExecutorServiceServer
    states  *syncx.Map[int64, executorv1.ExecutionState]
    cancels *syncx.Map[int64, context.CancelFunc]
    client  reporterv1.ReporterServiceClient
}

func (e *LongRunningExecutor) Execute(ctx context.Context, req *executorv1.ExecuteRequest) (*executorv1.ExecuteResponse, error) {
    eid := req.GetEid()

    // 幂等检查：如果已在执行，直接返回当前状态
    if st, ok := e.states.Load(eid); ok {
        return &executorv1.ExecuteResponse{ExecutionState: &st}, nil
    }

    // 初始化执行状态
    st := executorv1.ExecutionState{
        Id:       eid,
        TaskId:   req.GetTaskId(),
        TaskName: req.GetTaskName(),
        Status:   executorv1.ExecutionStatus_RUNNING,
    }
    e.states.Store(eid, st)

    // 解析参数
    start, _ := strconv.Atoi(req.GetParams()["start"])
    end, _ := strconv.Atoi(req.GetParams()["end"])

    // 启动异步执行
    runCtx, cancel := context.WithCancel(context.Background())
    e.cancels.Store(eid, cancel)
    go e.runTask(runCtx, eid, start, end)

    return &executorv1.ExecuteResponse{ExecutionState: &st}, nil
}

func (e *LongRunningExecutor) runTask(ctx context.Context, eid int64, start, end int) {
    total := end
    progress := start

    for progress < total {
        select {
        case <-ctx.Done():
            // 被中断，上报 FAILED_RESCHEDULABLE 状态
            e.reportState(eid, executorv1.ExecutionStatus_FAILED_RESCHEDULABLE, int32(progress*100/total))
            return
        default:
            progress++
            time.Sleep(100 * time.Millisecond) // 模拟业务处理

            // 每 30 秒上报一次进度
            if progress%300 == 0 {
                e.reportState(eid, executorv1.ExecutionStatus_RUNNING, int32(progress*100/total))
            }
        }
    }

    // 执行完成
    e.reportState(eid, executorv1.ExecutionStatus_SUCCESS, 100)
}

func (e *LongRunningExecutor) Interrupt(ctx context.Context, req *executorv1.InterruptRequest) (*executorv1.InterruptResponse, error) {
    eid := req.GetEid()
    if cancel, ok := e.cancels.Load(eid); ok {
        cancel() // 取消执行上下文
        cur, _ := e.states.Load(eid)
        return &executorv1.InterruptResponse{Success: true, ExecutionState: &cur}, nil
    }
    return &executorv1.InterruptResponse{Success: false}, nil
}

func (e *LongRunningExecutor) Query(ctx context.Context, req *executorv1.QueryRequest) (*executorv1.QueryResponse, error) {
    if st, ok := e.states.Load(req.GetEid()); ok {
        return &executorv1.QueryResponse{ExecutionState: &st}, nil
    }
    return &executorv1.QueryResponse{
        ExecutionState: &executorv1.ExecutionState{Id: req.GetEid(), Status: executorv1.ExecutionStatus_UNKNOWN},
    }, nil
}

func (e *LongRunningExecutor) Prepare(ctx context.Context, req *executorv1.PrepareRequest) (*executorv1.PrepareResponse, error) {
    // 返回业务数据总量（分片任务用）
    return &executorv1.PrepareResponse{
        Params: map[string]string{"total": "10000"},
    }, nil
}

func (e *LongRunningExecutor) reportState(eid int64, status executorv1.ExecutionStatus, progress int32) {
    st := executorv1.ExecutionState{
        Id:              eid,
        Status:          status,
        RunningProgress: progress,
    }
    e.states.Store(eid, st)
    e.client.Report(context.Background(), &reporterv1.ReportRequest{ExecutionState: &st})
}
```

```go
// 分片任务执行器：每个分片处理一个数据范围
```
