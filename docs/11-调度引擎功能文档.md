## 1. 调度引擎总览

```
Scheduler (V1/V2)
    │
    ├── scheduleLoop / scheduleOneLoop
    │       ├── TaskService.SchedulableTasks()    ← 拉取可调度任务
    │       ├── Picker.Pick(task)                  ← 智能选节点
    │       ├── Runner.Run(ctx, task)               ← 执行调度
    │       └── LoadChecker.Check(ctx)              ← 负载检查
    │
    └── renewLoop
            └── TaskAcquirer.Renew(nodeID)          ← 续约抢占
```

## 2. Scheduler V1（单机全量）

**包路径**：`internal/service/scheduler/scheduler.go`

V1 是单机全量扫描模式——所有 Scheduler 节点扫描同一张 tasks 表，通过 CAS 竞争抢占。

### 生命周期

实现 `ego/server.Server` 接口：

| 方法 | 说明 |
|------|------|
| `Start()` | 启动 `scheduleLoop` + `renewLoop` 两个 goroutine |
| `Stop()` | 取消 context |
| `GracefulStop(ctx)` | 取消 context + `wg.Wait()` 等待 goroutine 退出 |

### scheduleLoop 流程

```
for {
    1. 检查 ctx 是否取消
    2. 开始计时（Prometheus）
    3. ctx+超时 调用 TaskService.SchedulableTasks(preemptedTimeoutMs, batchSize)
    4. 如果无任务 → sleep(scheduleInterval) → continue
    5. 遍历任务：
       a. Picker.Pick(task) → 选最优节点 → ctx 注入 SpecificNodeID
       b. Runner.Run(ctx, task) → 抢占+创建执行记录+异步调用
    6. LoadChecker.Check(ctx) → 如果负载高 → sleep → continue
}
```

### renewLoop 流程

```
ticker(renewInterval=5s):
    TaskAcquirer.Renew(nodeID)  // 批量续约本节点持有的所有任务
```

## 3. Scheduler V2（分片分布式）

**包路径**：`internal/service/scheduler/schedulerv2.go`

V2 通过 `ShardingLoopJob` + 分布式锁实现分片调度，每个分片同一时刻只有一个节点处理。

### 与 V1 的区别

| 维度 | V1 | V2 |
|------|-----|-----|
| 数据扫描 | 全表扫描 | 按分片扫描（从 context 获取 Dst） |
| 并发控制 | CAS 竞争 | 分布式锁 + 信号量 |
| 扩展性 | 节点越多竞争越大 | 线性扩展，零竞争 |
| 额外依赖 | 无 | dlock（Redis/etcd）+ ShardingStrategy |

### 启动方式

```go
func (s *SchedulerV2) Start() error {
    go loopjob.NewShardingLoopJob(s.dclient, "scheduleKey", s.scheduleOneLoop, s.taskStr, s.sem)
    go s.renewLoop()
    return nil
}
```

`scheduleOneLoop` 与 V1 的 `scheduleLoop` 单次循环逻辑一致。

## 4. Runner 调度链

**包路径**：`internal/service/runner/`

```
Runner 接口
├── Dispatcher          ← 按 TaskType 路由
│   ├── NormalTaskRunner ← 处理 normal + sharding
│   └── PlanTaskRunner   ← 处理 plan（DAG）
```

### Runner 接口

```go
type Runner interface {
    Run(ctx context.Context, task domain.Task) error
    Retry(ctx context.Context, execution domain.TaskExecution) error
    Reschedule(ctx context.Context, execution domain.TaskExecution) error
}
```

| 方法 | 触发者 | 说明 |
|------|--------|------|
| `Run` | Scheduler | 首次调度执行 |
| `Retry` | RetryCompensator | 可重试失败的重新执行 |
| `Reschedule` | RescheduleCompensator | 重调度到其他节点 |

## 5. NormalTaskRunner

**包路径**：`internal/service/runner/normal_task_runner.go`

### Run 流程（普通任务）

```
1. acquireTask(task)  ← CAS 乐观锁抢占
2. 判断 ShardingRule:
   a. nil → handleNormalTask:
      - Create TaskExecution(PREPARE)
      - go func { invoker.Run(ctx, exec) → UpdateState }
   b. 有值 → handleShardingTask:
      - CreateShardingChildren → 1个父(RUNNING) + N个子(PREPARE)
      - 每个子 go func { invoker.Run(WithSpecificNodeID) → UpdateState }
```

### Retry 流程

```
1. 非分片任务 → 重新 acquireTask
2. go func {
     invoker.Run(WithExcludedNodeID(上次失败节点)) → UpdateState
   }
```

### Reschedule 流程

与 Retry 类似，但使用 `WithSpecificNodeID`（指定到原节点重新执行）。

### Context 路由注入

| 方法 | 注入内容 | 消费者 |
|------|---------|--------|
| `WithSpecificNodeIDContext` | 指定节点 ID | RoutingRoundRobin Balancer |
| `WithExcludedNodeIDContext` | 排除节点 ID | RoutingRoundRobin Balancer |

## 6. PlanTaskRunner（DAG 执行器）

**包路径**：`internal/service/runner/plan_task_runner.go`

嵌入 `NormalTaskRunner`，复用抢占+执行逻辑。

### Run 流程

```
1. acquireTask(plan任务)
2. Create PlanExecution(RUNNING)
3. GetPlan(planID) → 构建 DAG 图
4. 找到 RootTask（无前驱节点）
5. 每个根节点 → go p.run(ctx, task)
```

### NextStep 流程（事件驱动）

```
1. GetPlan(planID) → 获取 DAG 图
2. GetTask(taskName) → 找到当前节点
3. NextStep() → 获取后继节点
4. 如果无后继 → Produce(Plan完成事件)
5. 如果有后继 → 逐个 CheckPre():
   - 前驱全部满足 → go p.run(ctx, nextTask)
```

### run 方法（带重试循环）

```
for retries := 0; retries < 10; retries++ {
    err := NormalTaskRunner.Run(ctx, task)
    if err == nil → return
    if err == ErrTaskPreemptFailed → return (被别人抢了)
    sleep(500ms) → 继续重试
}
```

## 7. TaskAcquirer（抢占器）

**包路径**：`internal/service/acquirer/`

| 方法 | 说明 |
|------|------|
| `Acquire(ctx, taskID, version, nodeID)` | CAS 抢占：`UPDATE SET status=PREEMPTED, version+1 WHERE version=?` |
| `Release(ctx, taskID, nodeID)` | 释放抢占：`UPDATE SET status=ACTIVE WHERE schedule_node_id=?` |
| `Renew(ctx, nodeID)` | 批量续约：`UPDATE SET utime=now, version+1 WHERE schedule_node_id=?` |

## 8. Invoker（远程调用）

**包路径**：`internal/service/invoker/`

```
Dispatcher (Invoker 接口)
├── GrpcInvoker   ← execution.Task.GrpcConfig != nil
├── HttpInvoker   ← execution.Task.HTTPConfig != nil (TODO)
└── LocalInvoker  ← 默认（本地函数调用，测试用）
```

### Invoker 接口

```go
type Invoker interface {
    Run(ctx context.Context, execution domain.TaskExecution) (domain.ExecutionState, error)
    Prepare(ctx context.Context, execution domain.TaskExecution) (map[string]string, error)
}
```

- `Run`：发送 `Execute` gRPC 请求，返回 ExecutionState
- `Prepare`：分片任务的预查询（获取业务总数等），返回参数 map

## 9. Picker（节点选择器）

**包路径**：`internal/service/picker/`

### ExecutorNodePicker 接口

```go
type ExecutorNodePicker interface {
    Pick(ctx context.Context, task domain.Task) (nodeID string, err error)
}
```

### PrometheusPicker 实现

根据 `task.SchedulingStrategy` 构造 PromQL 查询：

| 策略 | PromQL |
|------|--------|
| CPU_PRIORITY | `topk(N, avg_over_time(executor_cpu_idle_percent{job="..."}[30s]) AND ON(instance) up == 1)` |
| MEMORY_PRIORITY | `topk(N, avg_over_time(executor_memory_available_bytes{job="..."}[30s]) AND ON(instance) up == 1)` |

从 TopN 结果中随机选一个 → 避免羊群效应。

## 10. 配置参数

```yaml
scheduler:
  batchTimeout: 3s          # 拉取任务超时
  batchSize: 100             # 每次拉取任务数
  preemptedTimeout: 10m      # 抢占超时（超过此时间视为可重新调度）
  scheduleInterval: 10s      # 无任务时休眠间隔
  renewInterval: 5s          # 续约间隔

intelligentScheduling:
  jobName: "executors"       # Prometheus job 名称
  topNCandidates: 3          # TopN 候选数
  timeWindow: 30s            # 指标查询窗口
  queryTimeout: 5s           # Prometheus 查询超时
```
