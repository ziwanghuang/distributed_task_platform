### 1.2 Scheduler vs Executor 职责分离

| 维度 | Scheduler（本项目） | Executor（外部服务） |
|------|---------------------|---------------------|
| 职责 | 任务发现、抢占、调度、状态管理、补偿 | 任务的实际执行逻辑 |
| 通信 | 作为 gRPC Client 向 Executor 发起调用 | 作为 gRPC Server 接收调用，作为 Client 上报状态 |
| 数据 | 拥有 tasks / task_executions 表的写权限 | 无直接数据库访问 |
| 扩展 | 水平扩展调度能力 | 水平扩展执行能力 |

**为什么这样设计：**
- **关注点分离**：调度逻辑（何时执行、在哪执行、失败怎么办）和业务逻辑（怎么执行）是正交的。分离后 Executor 可以用任何语言实现，只需遵循 gRPC proto 协议。
- **独立扩缩容**：调度节点（CPU 轻量、I/O 密集）和执行节点（CPU/内存密集）的资源画像不同，分离后可以独立调整副本数。
- **故障隔离**：Executor 的 OOM/Panic 不会影响调度器的正常运行。

## 2. 核心领域模型

### 2.1 Task（任务定义）

```go
// internal/domain/task.go
type Task struct {
    ID              int64
    Name            string
    Cfg             TaskCfg        // 执行配置（gRPC/HTTP/Local）
    Type            TaskType       // normal | plan
    CronExpr        string         // cron 表达式
    Status          TaskStatus     // ACTIVE | PREEMPTED | INACTIVE
    NextTime        int64          // 下次调度时间（毫秒时间戳）
    Version         int64          // 乐观锁版本号
    ScheduleNodeID  string         // 当前持有该任务的调度节点 UUID
    // ... 更多字段
}
```

#### 设计决策：为什么用毫秒时间戳而不是 `time.Time`？

项目中 `NextTime`、`CTime`、`UTime` 全部使用 `int64` 毫秒时间戳，而非 Go 的 `time.Time` 或 MySQL 的 `DATETIME`。

**原因：**
1. **跨时区一致性**：毫秒时间戳是 UTC 绝对值，不受数据库连接时区设置影响
2. **索引友好**：`WHERE next_time <= ?` 在 `BIGINT` 索引上比 `DATETIME` 比较更快（整数比较 vs 字符串解析）
3. **精度控制**：毫秒精度满足调度场景需求，同时避免了 MySQL `DATETIME` 的秒级精度限制

**🔴 不合理之处**：代码中大量 `time.Now().UnixMilli()` 散布在各处（DAO、Service、Compensator），没有统一的时间工具函数。一方面增加了测试难度（无法 mock 时间），另一方面如果要切换到微秒精度需要改动大量代码。**建议**：引入 `pkg/clock` 接口，统一时间获取入口。

#### 设计决策：三态状态模型 (ACTIVE / PREEMPTED / INACTIVE)

```
           创建                     调度器抢占
INACTIVE ──────► ACTIVE ──────────────► PREEMPTED
                  ▲                        │
                  │    执行完毕/释放         │
                  └────────────────────────┘
```

- **ACTIVE**：任务可被调度，处于等待队列中
- **PREEMPTED**：已被某个 Scheduler 节点抢占，正在处理中
- **INACTIVE**：已停用，不参与调度

**为什么只有 3 个状态：**
很多调度系统会设计 `WAITING → SCHEDULING → EXECUTING → COMPLETED` 等复杂状态。本项目将"执行"状态下沉到 `TaskExecution` 表，Task 本身只管"能不能被调度"这一个维度。这是一个**优雅的职责分离**——Task 是"定义"，TaskExecution 是"实例"。

### 2.2 TaskExecution（执行实例）

```go
// internal/domain/task_execution.go
type TaskExecution struct {
    ID               int64
    TaskID           int64
    Status           ExecStatus  // PREPARE → RUNNING → SUCCESS/FAILED/FAILED_RETRYABLE/FAILED_RESCHEDULED
    ShardingParentID *int64      // nil=普通任务, 0=分片父任务, >0=分片子任务
    RetryCount       int         // 已重试次数
    NextRetryTime    int64       // 下次重试时间
    Deadline         int64       // 超时截止时间
    // ... 从 Task 冗余的完整字段
}
```

#### 设计决策：六态执行状态机

```
                  ┌────────────────────────────────────┐
                  │                                    │
                  ▼                                    │
PREPARE ──► RUNNING ──► SUCCESS                        │
                │                                      │
                ├──► FAILED (终态)                      │
                │                                      │
                ├──► FAILED_RETRYABLE ─── 重试 ──► RUNNING
                │
                └──► FAILED_RESCHEDULED ─ 重调度 ─► (新 Execution)
```

**关键设计点：`FAILED_RETRYABLE` vs `FAILED_RESCHEDULED` 的分离**

这是本项目最精妙的设计之一：

| 维度 | FAILED_RETRYABLE | FAILED_RESCHEDULED |
|------|-------------------|--------------------|
| 语义 | 同一执行记录上重试 | 创建新执行记录，可能换节点 |
| 节点选择 | 排除上次失败的节点 | 可以指定新的目标节点 |
| Execution | 复用原 Execution，`retry_count++` | 创建新 Execution |
| 适用场景 | 临时性失败（网络抖动、超时） | 节点级故障（OOM、磁盘满） |

**为什么这样设计：**
- **重试**（Retryable）：问题是暂时的，换个时间再试就行，但要避开刚才失败的节点（通过 `WithExcludedNodeID`）
- **重调度**（Rescheduled）：问题出在执行节点本身，需要重新走一遍完整的调度流程（可能选择资源更充足的节点）

**🔴 不合理之处**：`FAILED_RETRYABLE` 的重试是在原 Execution 上 `retry_count++`，但如果重试过程中 Scheduler 节点自身挂了，补偿器 `RetryCompensator` 会重新捡起来继续重试。问题在于，补偿器找到的条件是 `status = FAILED_RETRYABLE AND next_retry_time <= now`，但没有检查 Execution 对应的 Task 当前是否被其他节点持有。**极端情况下可能出现两个 Scheduler 同时重试同一个 Execution**。建议在重试时增加 Execution 级别的乐观锁或者在查询条件中加入 `schedule_node_id` 过滤。

#### 设计决策：执行数据冗余（反范式）

`TaskExecution` 表冗余了 Task 的几乎所有字段：`name`、`cron_expr`、`grpc_config`、`http_config`、`schedule_strategy`、`sharding_rule` 等。

```go
// repository/task_execution.go - Create 方法中的映射
entity := dao.TaskExecution{
    TaskId:          execution.TaskID,
    Name:            execution.Name,            // 冗余
    CronExpr:        execution.CronExpr,        // 冗余
    GrpcConfig:      execution.GrpcConfig,      // 冗余
    HttpConfig:      execution.HttpConfig,       // 冗余
    // ... 十几个冗余字段
}
```

**为什么这样设计：**
1. **历史快照**：执行记录需要保留"执行那一刻的任务配置"。如果用户在执行期间修改了 Task 的 gRPC 地址，重试时应该用旧地址还是新地址？冗余保证了一致性。
2. **查询性能**：查询执行历史不需要 JOIN Task 表，单表即可满足所有展示需求。
3. **解耦**：补偿器重试时只需要读 TaskExecution，不依赖 Task 表的当前状态。

**🟡 权衡**：冗余带来了存储开销和数据一致性维护成本。如果 Task 配置频繁变更且执行记录量大（百万级），存储成本会显著增加。但在任务调度场景中，配置变更频率远低于执行频率，这个权衡是合理的。

### 2.3 ShardingParentID 的指针语义

```go
type TaskExecution struct {
    ShardingParentID *int64  // 注意是指针类型
}
```

三种语义：
- `nil`（`NULL`）：普通非分片任务
- `0`：分片父执行记录
- `> 0`：分片子执行记录，值为父 ID

**为什么用指针而不是 `int64 + 特殊值 -1`：**
- 利用 SQL 的 `NULL` 语义：`WHERE sharding_parent_id IS NULL` 天然区分普通任务
- 避免 magic number：用 `-1` 表示"无"在 Go 中需要额外的 `const` 和文档，而 `nil` 语义直观
- 数据库索引优化：MySQL 对 `IS NULL` 和 `IS NOT NULL` 有专门的索引优化路径

## 3. 调度引擎设计

### 3.1 调度主循环

```go
// service/scheduler/scheduler.go
func (s *Scheduler) scheduleLoop(ctx context.Context) {
    for {
        // 1. 批量获取可调度任务
        tasks := s.repo.FindSchedulableTasks(ctx, now, s.batchSize) // 默认 100

        // 2. 逐个执行
        for _, task := range tasks {
            s.runner.Run(ctx, task)
        }

        // 3. 负载检查 — 决定是否需要降速
        s.loadChecker.Check(ctx)  // 可能 sleep

        // 4. 无任务时休眠
        if len(tasks) == 0 {
            time.Sleep(s.scheduleInterval)  // 默认 10s
        }
    }
}
```

#### FindSchedulableTasks 的 SQL 设计

```sql
SELECT * FROM tasks
WHERE next_time <= ?                                    -- 到期了
  AND (status = 'ACTIVE'                                -- 未被抢占
    OR (status = 'PREEMPTED' AND utime <= ?))           -- 或者被抢占但已超时（10min）
ORDER BY next_time ASC
LIMIT 100
```

**为什么用 `utime` 超时而不是心跳表：**
- 简单直接：不需要额外的心跳表和对应的清理逻辑
- 与续约机制天然配合：`renewLoop` 每 5s 更新 `utime`，如果 10min 内没有续约，说明持有节点已死
- 减少表数量：用一个字段实现了"持有者存活检测"功能

**🟡 潜在问题**：`utime` 的超时阈值（10min）是硬编码在配置中的。如果续约间隔（5s）因 GC 暂停或网络延迟偶尔超过 10min，会导致任务被误认为可重调度。实际上 `10min / 5s = 120 次`续约机会，容错空间足够大。但建议将超时阈值设为续约间隔的**至少 3 倍**（当前是 120 倍，非常保守）。

### 3.2 任务抢占（乐观锁 CAS）

```go
// dao/task.go - Acquire
func (d *Task) Acquire(ctx context.Context, task domain.Task) (domain.Task, error) {
    return d.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
        // Step 1: CAS 更新
        result := tx.Model(&Task{}).
            Where("id = ? AND version = ?", task.ID, task.Version).
            Updates(map[string]any{
                "status":           domain.TaskStatusPreempted,
                "version":          task.Version + 1,
                "schedule_node_id": nodeID,
                "utime":           now(),
            })

        if result.RowsAffected == 0 {
            return ErrTaskVersionConflict  // 被别人抢走了
        }

        // Step 2: 重新查询最新数据（确保后续逻辑用的是最新版本）
        return tx.Where("id = ?", task.ID).First(&updatedTask).Error
    })
}
```

**为什么选择乐观锁而不是分布式锁（Redis/etcd）：**

| 维度 | 乐观锁 (version CAS) | 分布式锁 (Redis/etcd) |
|------|----------------------|-----------------------|
| 依赖 | 只依赖 MySQL | 需要额外组件 |
| 性能 | 冲突少时极高（单次 UPDATE） | 需要额外网络往返（SETNX + DEL） |
| 死锁风险 | 无（CAS 天然无锁） | 有（需要设置过期时间、看门狗续期） |
| 冲突处理 | 返回 RowsAffected=0，调用方决定 | 阻塞等待或立即失败 |
| 适用场景 | 冲突率低（任务数 >> 节点数） | 冲突率高（热点资源） |

在任务调度场景中，假设有 10000 个待调度任务和 3 个 Scheduler 节点，同时抢同一个任务的概率很低（每个节点从不同偏移量开始处理），乐观锁是最优选择。

**🟡 设计细节**：Acquire 放在事务中是为了保证"CAS + 重新查询"的原子性。如果两步分开，可能在 CAS 成功后、查询前被另一个事务修改（虽然概率极低，但事务保证了正确性）。

### 3.3 续约机制

```go
// service/scheduler/scheduler.go
func (s *Scheduler) renewLoop(ctx context.Context) {
    ticker := time.NewTicker(s.renewInterval)  // 默认 5s
    for {
        select {
        case <-ticker.C:
            s.dao.Renew(ctx, s.nodeID)
        case <-ctx.Done():
            return
        }
    }
}
```

```go
// dao/task.go - Renew
func (d *Task) Renew(ctx context.Context, nodeID string) error {
    return d.db.Model(&Task{}).
        Where("schedule_node_id = ? AND status = ?", nodeID, domain.TaskStatusPreempted).
        Updates(map[string]any{
            "utime":   now(),
            "version": gorm.Expr("version + 1"),
        }).Error
}
```

**为什么续约时也更新 version：**
- 防止"续约成功但同时有另一个节点在抢占"的竞态
- version 自增保证了即使两个节点同时对同一个 Task 操作，只有一个能成功

**🔴 不合理之处**：`Renew` 是批量操作（`WHERE schedule_node_id = ?`），不管有 1 个还是 1000 个任务，都是一条 SQL。这在任务数量少时很高效，但如果一个节点持有大量任务（比如 500+），这条 UPDATE 会触发大量行锁，可能影响其他事务。**建议**：当持有任务数超过阈值时，分批续约（每批 100 个）。

### 3.4 V1 vs V2 双版本架构

项目中每个核心组件都有 V1 和 V2 两个版本：

| 组件 | V1 | V2 |
|------|-----|-----|
| Scheduler | 简单循环 | ShardingLoopJob + 分布式锁 |
| RetryCompensator | 简单循环 | ShardingLoopJob + 分布式锁 |
| RescheduleCompensator | 同上 | 同上 |
| InterruptCompensator | 同上 | 同上 |
| ShardingCompensator | 同上 | 同上 |

```go
// V1: 所有节点扫描全表
func (c *RetryCompensator) Run(ctx context.Context) {
    for {
        executions := c.repo.FindRetryableExecutions(ctx, now, limit)
        for _, exec := range executions {
            c.runner.Retry(ctx, exec)
        }
    }
}

// V2: 分布式锁 + 分表
func (c *RetryCompensatorV2) Run(ctx context.Context) {
    c.shardingJob.Run(ctx, func(ctx context.Context, tableSuffix string) {
        // 只扫描自己锁住的分表
        executions := c.repo.FindRetryableExecutionsFromShard(ctx, tableSuffix, now, limit)
        // ...
    })
}
```

**为什么保留两个版本：**
1. **渐进式演进**：V1 适合单机或少量节点的简单场景，V2 适合多实例高并发场景
2. **配置切换**：通过配置文件选择使用哪个版本，无需代码变更
3. **面试价值**：展示了从简单到复杂的演进思路，体现系统设计能力

**🟡 代码问题**：V1 和 V2 的业务逻辑高度重复（80%+ 相同），违反了 DRY 原则。更好的做法是抽取公共的业务逻辑到一个 `retryBizLogic` 函数，V1 和 V2 只是不同的"调度框架"调用同一套业务逻辑。

## 4. 任务执行流程

### 4.1 Runner 调度链（责任链模式）

```
Scheduler.scheduleLoop()
    └── DispatcherRunner.Run(ctx, task)
            ├── task.Type == "normal" → NormalTaskRunner.Run(ctx, task)
            └── task.Type == "plan"  → PlanTaskRunner.Run(ctx, task)
```

**Runner Dispatcher 的路由逻辑：**

```go
// service/runner/dispatch_runner.go
func (d *Dispatcher) Run(ctx context.Context, task domain.Task) error {
    runner, ok := d.runners[task.Type]
    if !ok {
        return fmt.Errorf("unsupported task type: %s", task.Type)
    }
    return runner.Run(ctx, task)
}
```

**为什么用 Map 路由而不是 if-else：**
- **开闭原则**：新增任务类型只需注册 `runners["new_type"] = newRunner`，不修改已有代码
- **可测试性**：可以为每种 Runner 独立写单元测试
- **配置化**：未来可以从配置文件动态加载 Runner 映射

### 4.2 NormalTaskRunner 核心流程

```go
// service/runner/normal_task_runner.go
func (r *NormalTaskRunner) Run(ctx context.Context, task domain.Task) error {
    // Step 1: 乐观锁抢占
    task, err := r.acquirer.Acquire(ctx, task)
    if err == ErrVersionConflict {
        return nil  // 被其他节点抢走，静默跳过
    }

    // Step 2: 判断是否为分片任务
    if task.Cfg.ShardingRule != nil {
        return r.handleShardingTask(ctx, task)
    }
    return r.handleNormalTask(ctx, task)
}

func (r *NormalTaskRunner) handleNormalTask(ctx context.Context, task domain.Task) error {
    // Step 3: 创建执行记录（PREPARE 状态）
    execution, err := r.executionService.CreateExecution(ctx, task)

    // Step 4: 异步调用 Executor
    go func() {
        err := r.invoker.Invoke(ctx, execution)
        if err != nil {
            // 调用失败，更新为 FAILED
            r.executionService.UpdateState(ctx, execution.ID, FAILED)
        }
    }()

    return nil
}
```

#### Invoker 调度链

```
Invoker.Invoke(ctx, execution)
    ├── execution.GrpcConfig != nil → GrpcInvoker
    │       └── executorClient.Execute(ctx, &ExecuteRequest{...})
    ├── execution.HttpConfig != nil → HttpInvoker (TODO: 未实现)
    └── default → LocalInvoker (本地函数调用)
```

**为什么 Invoke 是异步的（go func）：**
- 调度器的核心循环不应该被单个任务的执行阻塞
- 一个慢任务不会影响其他任务的调度时机
- 缺点是错误处理变得更复杂（需要通过状态回写而非返回值传递错误）

**🔴 不合理之处**：异步 `go func()` 没有做并发控制。如果 `FindSchedulableTasks` 返回 100 个任务，会瞬间启动 100 个 goroutine 并发调用 Executor。虽然 Go 的 goroutine 很轻量，但下游 gRPC 连接可能成为瓶颈。**建议**：使用 `semaphore.Weighted` 或 worker pool 限制并发 Invoke 数量。

### 4.3 ExecutionService 状态机（核心中的核心）

`ExecutionService.UpdateState()` 是整个系统最关键的方法，处理所有状态转换：

```go
// service/task/execution_service.go - UpdateState（简化版）
func (s *ExecutionService) UpdateState(ctx context.Context, report domain.Report) error {
    switch report.Status {
    case RUNNING:
        if report.Progress > 0 {
            return s.repo.UpdateProgress(ctx, report)  // 进度上报
        }
        return s.repo.SetRunningState(ctx, report)     // PREPARE → RUNNING

    case FAILED_RETRYABLE:
        // 计算下次重试时间（指数退避）
        nextRetryTime := s.calculateRetryTime(execution)
        execution.RetryCount++
        execution.NextRetryTime = nextRetryTime
        return s.repo.UpdateRetryableState(ctx, execution)

    case FAILED_RESCHEDULED:
        // 合并重调度参数（可能指定新节点）
        execution.ScheduleParams = merge(execution.ScheduleParams, report.RescheduleParams)
        return s.repo.UpdateRescheduledState(ctx, execution)

    case SUCCESS, FAILED:
        // 终态处理
        err := s.repo.UpdateTerminalState(ctx, report)
        if err != nil { return err }

        // 释放 Task（状态回到 ACTIVE）
        s.taskRepo.Release(ctx, execution.TaskID)

        // 更新下次调度时间
        s.taskRepo.UpdateNextTime(ctx, execution.TaskID, nextTime)

        // 发送完成事件（触发 Plan 工作流推进）
        s.completeProducer.Send(ctx, CompleteEvent{
            TaskID:      execution.TaskID,
            ExecutionID: execution.ID,
            Status:      report.Status,
        })
        return nil
    }
}
```

#### 指数退避重试策略

```go
// 计算下次重试时间
func (s *ExecutionService) calculateRetryTime(exec domain.TaskExecution) int64 {
    // 基础间隔 * 2^(重试次数)
    interval := s.baseRetryInterval * time.Duration(math.Pow(2, float64(exec.RetryCount)))
    if interval > s.maxRetryInterval {
        interval = s.maxRetryInterval
    }
    return time.Now().Add(interval).UnixMilli()
}
```

**为什么用指数退避而不是固定间隔：**
- 避免"风暴效应"：如果下游服务故障，固定间隔会导致所有重试在同一时刻涌入
- 给下游恢复时间：1s → 2s → 4s → 8s → 16s → 32s → 60s(max)
- 行业标准：AWS、GCP 的重试策略都采用指数退避

**🟡 改进建议**：当前没有加 jitter（随机抖动）。在多实例场景下，如果多个 Execution 同时失败，它们的重试时间会高度同步（都在 2^n 秒后）。建议加入 `± 20%` 的随机抖动：`interval * (0.8 + rand.Float64() * 0.4)`。

### 4.4 Deadline 机制

```go
// dao/task_execution.go - Create
func (d *TaskExecution) Create(ctx context.Context, exec TaskExecution) error {
    exec.Deadline = time.Now().UnixMilli() + exec.MaxExecutionSeconds * 1000
    return d.db.Create(&exec).Error
}

// dao/task_execution.go - SetRunningState
func (d *TaskExecution) SetRunningState(ctx context.Context, id int64) error {
    // 重新查询并重新计算 deadline（从 RUNNING 状态开始计时）
    var exec TaskExecution
    d.db.Where("id = ?", id).First(&exec)
    exec.Deadline = time.Now().UnixMilli() + exec.MaxExecutionSeconds * 1000
    return d.db.Save(&exec).Error
}
```

**为什么在 SetRunningState 时重新计算 Deadline：**
- `Create` 时设置的 Deadline 包含了 `PREPARE → RUNNING` 的等待时间
- 实际执行可能在创建后数秒甚至数分钟才开始（Executor 可能有队列）
- 从 RUNNING 开始重新计时更公平

**🔴 不合理之处**：`SetRunningState` 中先 `First` 查询再 `Save` 更新，不是原子操作。在高并发下可能出现：查询时 Deadline 为 A，Save 时状态已被 InterruptCompensator 改为 FAILED（因为旧 Deadline 超时）。**建议**：使用单条 `UPDATE ... SET deadline = NOW() + max_seconds WHERE id = ? AND status = PREPARE` 保证原子性。

## 5. 补偿机制设计

### 5.1 四大补偿器总览

```
┌─────────────────────────────────────────────────────────────┐
│                    Compensator 架构                          │
│                                                              │
│  ┌──────────────────┐     ┌──────────────────────┐          │
│  │ RetryCompensator │     │ RescheduleCompensator│          │
│  │ 重试补偿          │     │ 重调度补偿            │          │
│  │                  │     │                      │          │
│  │ 条件:            │     │ 条件:                 │          │
│  │ status=RETRYABLE │     │ status=RESCHEDULED    │          │
│  │ next_retry<=now  │     │ (非分片父任务)         │          │
│  │                  │     │                      │          │
│  │ 动作: Runner.    │     │ 动作: Runner.         │          │
│  │       Retry()    │     │       Reschedule()   │          │

│                                                              │
│  ┌──────────────────┐     ┌──────────────────────┐          │
│  │InterruptCompensator│   │ShardingCompensator   │          │

│  │                  │     │                      │          │

│  │                  │     │                      │          │
│  │ 动作: gRPC       │     │ 动作: 聚合子任务状态  │          │
│  │ Interrupt()      │     │       更新父任务       │          │
│  └──────────────────┘     └──────────────────────┘          │
└─────────────────────────────────────────────────────────────┘
```

### 5.2 补偿器的通用模式

每个补偿器都遵循相同的模式：

```go
func (c *XxxCompensator) Run(ctx context.Context) {
    for {
        startTime := time.Now()

        // 1. 查找需要补偿的记录
        items := c.repo.FindXxx(ctx, limit)

        // 2. 逐个处理
        for _, item := range items {
            c.process(ctx, item)
        }

        // 3. 防旋转保护
        duration := time.Since(startTime)
        if duration < c.minLoopDuration {
            time.Sleep(c.minLoopDuration - duration)
        }
    }
}
```

#### 防旋转保护（Anti-Spin）

**为什么需要 `minLoopDuration`：**

如果没有需要补偿的记录，循环体几乎瞬间完成（只有一次 SQL 查询），会导致：
1. CPU 空转：无限快速循环消耗 CPU
2. 数据库压力：每毫秒一次 SELECT 查询

`minLoopDuration`（默认 1s）保证每次循环**至少**花费 1 秒，无论业务逻辑多快。

**🟡 改进建议**：当前的"固定最小间隔"策略比较简单。更好的做法是使用**自适应退避**：
- 有数据：立即下一轮（高吞吐）
- 无数据：逐步增加休眠时间（1s → 2s → 4s → 8s → 最大 30s）
- 重新发现数据：立即重置为 1s

### 5.3 InterruptCompensator（超时中断）

```go
func (c *InterruptCompensator) process(ctx context.Context, exec domain.TaskExecution) {
    // 1. 发送 gRPC Interrupt 到 Executor
    _, err := c.executorClient.Interrupt(
        runner.WithSpecificNodeID(ctx, exec.ScheduleNodeID),
        &executorv1.InterruptRequest{
            ExecutionId: exec.ID,
            TaskId:      exec.TaskID,
        },
    )

    // 2. 更新执行状态为 FAILED
    if err == nil {
        c.executionService.UpdateState(ctx, domain.Report{
            ExecutionID: exec.ID,
            Status:      domain.ExecStatusFailed,
        })
    }
}
```

**为什么需要主动 Interrupt 而不是等 Executor 自己超时：**
- Executor 可能没有超时机制（不是所有业务逻辑都有 deadline 感知）
- 资源释放：及早通知 Executor 释放 CPU/内存/连接等资源
- 快速恢复：不用等 Executor 的超时（可能设置得很长或没设置）

**🔴 不合理之处**：如果 Interrupt gRPC 调用本身超时或失败（比如 Executor 已经宕机），当前逻辑只是 log error 然后跳过。下次循环会再次尝试 Interrupt 同一个 Execution——但此时 Executor 可能已经恢复并完成了任务。**建议**：增加重试次数上限（比如 3 次），超过后直接标记为 FAILED 而不再尝试 Interrupt。

### 5.4 ShardingCompensator（分片聚合）

```go
func (c *ShardingCompensator) process(ctx context.Context, parentExec domain.TaskExecution) {
    // 1. 查询所有子任务
    children := c.repo.FindByShardingParentID(ctx, parentExec.ID)

    // 2. 聚合状态
    allDone := true
    anyFailed := false
    for _, child := range children {
        if !child.Status.IsTerminal() {
            allDone = false
            break
        }
        if child.Status == FAILED {
            anyFailed = true
        }
    }

    // 3. 更新父任务状态
    if allDone {
        if anyFailed {
            c.executionService.UpdateState(ctx, parentExec.ID, FAILED)
        } else {
            c.executionService.UpdateState(ctx, parentExec.ID, SUCCESS)
        }
        // 4. 释放 Task
        c.taskRepo.Release(ctx, parentExec.TaskID)
    }
}
```

**为什么分片聚合由补偿器做而不是实时回调：**
- 子任务可能分布在不同 Executor 节点，完成时间不同
- 如果每个子任务完成时都查一次"所有兄弟是否完成"，会产生大量并发查询
- 补偿器定期扫描更简单，也天然解决了"子任务汇报丢失"的问题

## 6. 工作流/Plan 引擎

### 6.1 DSL 设计

Plan 使用 ANTLR4 定义的自定义 DSL：

```
// 示例：A → B+C → D（B和C并行，都完成后执行D）
A -> {B, C} -> D

// 示例：条件分支
A -> [B: condition1, C: condition2] -> D
```

```antlr
// internal/service/task/plan/parser/Plan.g4 (ANTLR4 语法)
plan    : step ('->' step)* ;
step    : taskID                           // 单任务
        | '{' taskID (',' taskID)* '}'     // 并行任务组（AND语义）
        | '[' condExpr (',' condExpr)* ']' // 条件分支组（OR语义）
        ;
condExpr: taskID (':' condition)? ;
```

**为什么用自定义 DSL 而不是 JSON/YAML：**
- **可读性**：`A -> {B, C} -> D` 比等价的 JSON DAG 描述直观 10 倍
- **安全性**：DSL 是受限语言，不可能注入恶意代码（不像嵌入式 Lua/JS）
- **编译期检查**：ANTLR4 生成的 Parser 在解析时就能发现语法错误

**🟡 权衡**：自定义 DSL 的代价是学习成本和维护成本。ANTLR4 生成的 Parser 代码是自动生成的（`.g4 → .go`），不太好调试。如果 DSL 语法需要演进（比如添加循环、变量），维护成本会显著上升。

### 6.2 Plan DAG 构建

```go
// service/task/plan.go
func (s *PlanService) GetPlan(ctx context.Context, planTaskID int64) (*domain.Plan, error) {
    // 1. 获取 Plan 任务定义
    planTask, _ := s.taskRepo.FindByID(ctx, planTaskID)

    // 2. 解析 DSL 为 DAG 节点列表
    planTasks := s.parser.Parse(planTask.Cfg.ExecExpr)

    // 3. 并发获取每个节点对应的实际 Task 定义
    g, ctx := errgroup.WithContext(ctx)
    for i, pt := range planTasks {
        i, pt := i, pt
        g.Go(func() error {
            task, err := s.taskRepo.FindByID(ctx, pt.TaskID)
            planTasks[i].Task = task
            return err
        })
    }
    g.Wait()

    // 4. 解析前驱/后继关系
    s.resolveDependencies(planTasks)

    return &domain.Plan{Tasks: planTasks}, nil
}
```

**为什么并发获取 Task 定义（errgroup）：**
- 一个 Plan 可能包含 10+ 个 Task，串行查询需要 10 次 DB 往返
- 并发查询在理想情况下只需要 1 次 DB 往返的时间
- `errgroup` 提供了优雅的错误处理和 context 取消传播

### 6.3 Plan 执行与推进

```go
// service/runner/plan_task_runner.go
func (r *PlanTaskRunner) Run(ctx context.Context, planTask domain.Task) error {
    // 1. 抢占 Plan 任务
    planTask, _ = r.acquirer.Acquire(ctx, planTask)

    // 2. 解析 DAG
    plan, _ := r.planService.GetPlan(ctx, planTask.ID)

    // 3. 找到根节点（无前驱的节点）
    roots := plan.FindRootTasks()

    // 4. 启动根节点
    for _, root := range roots {
        r.normalRunner.Run(ctx, root.Task)
    }
    return nil
}

// 完成事件触发推进
func (r *PlanTaskRunner) NextStep(ctx context.Context, completedTaskID int64, planID int64) error {
    plan, _ := r.planService.GetPlan(ctx, planID)

    // 找到已完成任务的后继
    nextTasks := plan.FindNextTasks(completedTaskID)

    for _, next := range nextTasks {
        // 检查前驱是否全部满足
        if next.CheckPre() {
            r.normalRunner.Run(ctx, next.Task)
        }
    }
    return nil
}
```

#### CheckPre 的 AND/OR 语义

```go
// domain/plantask.go
func (pt *PlanTask) CheckPre() bool {
    switch pt.PreType {
    case AND:
        // 所有前驱都必须成功
        for _, pre := range pt.PreTasks {
            if pre.Status != SUCCESS { return false }
        }
        return true
    case OR:
        // 任一前驱成功即可
        for _, pre := range pt.PreTasks {
            if pre.Status == SUCCESS { return true }
        }
        return false
    case Condition:
        // 根据条件表达式判断
        return evaluateCondition(pt.Condition, pt.PreTasks)
    }
}
```

**🔴 不合理之处**：Plan 的 NextStep 通过 Kafka 完成事件触发，但没有处理"事件丢失"的场景。如果 Kafka 消费者 commit offset 后、处理前崩溃，这个完成事件就丢了，Plan 会永远卡住。**建议**：增加一个 `PlanProgressCompensator`，定期扫描状态为 RUNNING 的 Plan，检查是否有"前驱已完成但后继未启动"的情况。

## 7. 分片任务设计

### 7.1 分片执行模型

```
Task (sharding_rule: {type: "range", step: 100, total: 1000})
    │
    ├── TaskExecution (parent, sharding_parent_id = 0)
    │       ├── TaskExecution (child 1, sharding_parent_id = parent.ID, range: [0, 100))
    │       ├── TaskExecution (child 2, sharding_parent_id = parent.ID, range: [100, 200))
    │       ├── ...
    │       └── TaskExecution (child 10, sharding_parent_id = parent.ID, range: [900, 1000))
```

### 7.2 两种分片策略

#### Range 分片（固定步长）

```go
// domain/task.go
type RangeShardingRule struct {
    Step  int `json:"step"`   // 每片大小
    Total int `json:"total"`  // 总数据量
}

func (r *RangeShardingRule) Shards() []ShardRange {
    var shards []ShardRange
    for i := 0; i < r.Total; i += r.Step {
        end := min(i + r.Step, r.Total)
        shards = append(shards, ShardRange{Start: i, End: end})
    }
    return shards
}
```

**适用场景**：数据量固定、每片处理时间相近（如按 ID 范围扫描数据库）

#### Weighted Dynamic Range（加权动态分片）

```go
type WeightedDynamicRangeShardingRule struct {
    Total int `json:"total"`
}

func (r *WeightedDynamicRangeShardingRule) Shards(executors []ExecutorNode) []ShardRange {
    // 根据每个 Executor 的权重（CPU/内存）分配不同大小的数据范围
    totalWeight := sum(executor.Weight for executor in executors)
    offset := 0
    for _, executor := range executors {
        size := r.Total * executor.Weight / totalWeight
        shards = append(shards, ShardRange{
            Start:    offset,
            End:      offset + size,
            NodeID:   executor.NodeID,
        })
        offset += size
    }
    return shards
}
```

**适用场景**：Executor 节点配置不同（8核 vs 32核），按能力分配负载

**为什么需要加权分片：**
- 均匀分片会导致"木桶效应"——最慢的节点决定整体完成时间
- 按权重分配让能力强的节点处理更多数据，整体完成时间更短
- 权重来自 Prometheus 监控的实时 CPU/内存数据

### 7.3 分片任务的执行流程

```go
// service/runner/normal_task_runner.go
func (r *NormalTaskRunner) handleShardingTask(ctx context.Context, task domain.Task) error {
    // 1. 创建父执行记录
    parentExec := r.executionService.CreateShardingParent(ctx, task)

    // 2. 计算分片
    shards := task.Cfg.ShardingRule.Shards()

    // 3. 为每个分片创建子执行记录并异步调用
    for _, shard := range shards {
        childExec := r.executionService.CreateShardingChild(ctx, task, parentExec.ID, shard)
        go func(exec domain.TaskExecution) {
            // 使用 WithSpecificNodeID 指定目标节点（加权分片）
            invokeCtx := runner.WithSpecificNodeID(ctx, shard.NodeID)
            r.invoker.Invoke(invokeCtx, exec)
        }(childExec)
    }

    return nil
}
```

**🔴 不合理之处**：分片子任务的创建和调用在同一个方法中串行 + 并行混合执行。如果创建第 5 个子执行记录时数据库报错，前 4 个已经在异步执行了，会导致"部分分片执行、部分未执行"的不一致状态。**建议**：
1. 先批量创建所有子执行记录（事务保证原子性）
2. 全部创建成功后再批量发起异步调用
3. 如果创建失败，回滚事务并标记父执行为 FAILED

## 8. 服务治理与高可用

### 8.1 服务发现（etcd）

```go
// pkg/grpc/registry/etcd/registry.go
type Registry struct {
    client  *clientv3.Client
    leaseID clientv3.LeaseID
}

func (r *Registry) Register(ctx context.Context, instance ServiceInstance) error {
    // 1. 创建 lease（TTL = 30s）
    lease, _ := r.client.Grant(ctx, 30)
    r.leaseID = lease.ID

    // 2. 注册 key: /services/{serviceName}/{instanceID}
    key := fmt.Sprintf("/services/%s/%s", instance.ServiceName, instance.ID)
    value, _ := json.Marshal(instance)
    r.client.Put(ctx, key, string(value), clientv3.WithLease(lease.ID))

    // 3. 自动续约
    ch, _ := r.client.KeepAlive(ctx, lease.ID)
    go func() {
        for range ch {
            // 续约成功，保持注册
        }
        // ch 关闭意味着续约失败，实例会在 TTL 后自动注销
    }()
    return nil
}
```

**为什么用 etcd 而不是 Consul/Nacos：**
- Go 生态原生支持（Kubernetes 底层就用 etcd）
- 强一致性（Raft）保证服务列表的准确性
- Watch 机制提供实时通知，不需要轮询

### 8.2 自定义 gRPC Resolver

```go
// pkg/grpc/resolver.go
type Resolver struct {
    registry Registry
    cc       resolver.ClientConn
}

func (r *Resolver) watch() {
    // Watch etcd 变更事件
    events := r.registry.Subscribe(ctx, serviceName)
    for event := range events {
        // 将 ServiceInstance 转换为 gRPC 地址
        var addrs []resolver.Address
        for _, instance := range event.Instances {
            addrs = append(addrs, resolver.Address{
                Addr: instance.Address,
                Attributes: attributes.New(
                    "nodeID", instance.ID,
                    "weight", instance.Weight,
                    "capacity", instance.Capacity,
                ),
            })
        }
        // 更新 gRPC 连接池
        r.cc.UpdateState(resolver.State{Addresses: addrs})
    }
}
```

**为什么自定义 Resolver 而不是用 `grpc-go` 内置的 `dns` resolver：**
- DNS 不支持实时变更通知（依赖 TTL 刷新）
- DNS 无法携带节点元信息（weight, capacity, nodeID）
- etcd Watch 实现秒级服务发现

### 8.3 自定义 gRPC Balancer（路由轮询）

```go
// pkg/grpc/balancer/routing_round_robin.go
type RoutingRoundRobinPicker struct {
    subConns []balancer.SubConn
    next     uint64  // 轮询计数器
}

func (p *RoutingRoundRobinPicker) Pick(info balancer.PickInfo) (balancer.PickResult, error) {
    ctx := info.Ctx

    // 模式 1: 指定节点
    if nodeID, ok := GetSpecificNodeID(ctx); ok {
        for _, sc := range p.subConns {
            if sc.NodeID == nodeID {
                return balancer.PickResult{SubConn: sc}, nil
            }
        }
        return balancer.PickResult{}, ErrNodeNotFound
    }

    // 模式 2: 排除节点
    if excludeID, ok := GetExcludedNodeID(ctx); ok {
        candidates := filter(p.subConns, func(sc) bool { return sc.NodeID != excludeID })
        idx := atomic.AddUint64(&p.next, 1)
        return balancer.PickResult{SubConn: candidates[idx % len(candidates)]}, nil
    }

    // 模式 3: 默认轮询
    idx := atomic.AddUint64(&p.next, 1)
    return balancer.PickResult{SubConn: p.subConns[idx % len(p.subConns)]}, nil
}
```

**为什么需要三种路由模式：**

| 模式 | 使用场景 | 示例 |
|------|---------|------|
| 指定节点 | 智能调度选中了特定 Executor | `prometheusPicker` 选出 CPU 空闲最高的节点 |
| 排除节点 | 重试时避开上次失败的节点 | `RetryCompensator` 重试任务 |
| 默认轮询 | 无特殊要求的普通调度 | 常规任务调度 |

**设计亮点**：通过 `context.Value` 传递路由指令，对业务代码完全透明。调用方只需要 `WithSpecificNodeID(ctx, id)`，不需要知道 Balancer 的实现细节。

### 8.4 智能节点选择（Prometheus Picker）

```go
// service/picker/prometheus_picker.go
func (p *PrometheusPicker) Pick(ctx context.Context, strategy ScheduleStrategy) (string, error) {
    var query string
    switch strategy {
    case CPU_PRIORITY:
        query = fmt.Sprintf(
            `topk(%d, avg_over_time(executor_cpu_idle_percent{job="%s"}[%s]) AND ON(instance) up{job="%s"} == 1)`,
            p.topN, p.jobName, p.window, p.jobName,
        )
    case MEMORY_PRIORITY:
        query = fmt.Sprintf(
            `topk(%d, avg_over_time(executor_memory_available_bytes{job="%s"}[%s]) AND ON(instance) up{job="%s"} == 1)`,
            p.topN, p.jobName, p.window, p.jobName,
        )
    }

    // 查询 Prometheus
    result, _ := p.client.Query(ctx, query)

    // 从 Top-N 中随机选一个
    candidates := parseResult(result)
    return candidates[rand.Intn(len(candidates))].NodeID, nil
}
```

**为什么从 Top-N 中随机选而不是选最优：**
- **避免羊群效应**：如果总是选最优节点，所有调度器都会把任务发到同一个节点，瞬间过载
- **负载均衡**：Top-3 随机选 ≈ "在最好的 3 个节点中均匀分布"
- **容错**：即使 Prometheus 数据有延迟（30s 窗口），Top-3 中任意一个都不会太差

**🟡 改进建议**：当前 Prometheus 查询是同步阻塞的。如果 Prometheus 响应慢（>1s），会阻塞整个调度循环。建议：
1. 异步预取：后台协程定期（10s）刷新 Top-N 候选列表，调度时直接读缓存
2. 降级策略：Prometheus 查询超时时，回退到默认轮询

### 8.5 三层负载检查器

```go
// pkg/loadchecker/composite.go
type CompositeChecker struct {
    checkers []LoadChecker
    strategy Strategy  // AND | OR
}

func (c *CompositeChecker) Check(ctx context.Context) {
    switch c.strategy {
    case AND:
        // 所有 checker 都建议降速才降速
        for _, checker := range c.checkers {
            if !checker.ShouldSlow() { return }
        }
        time.Sleep(c.slowDuration)
    case OR:
        // 任一 checker 建议降速就降速
        for _, checker := range c.checkers {
            if checker.ShouldSlow() {
                time.Sleep(c.slowDuration)
                return
            }
        }
    }
}
```

三层检查器：

| 层 | 实现 | 检查内容 | 配置 |
|----|------|---------|------|
| 1 | LimiterChecker | 令牌桶限流 | 10/s, burst 20 |
| 2 | DatabaseLoadChecker | GORM 查询延迟 | 阈值 100ms |
| 3 | ClusterLoadChecker | 当前节点 vs 集群平均调度耗时 | 比率阈值 1.2 |

**为什么需要三层而不是一层：**
- **令牌桶**：防止瞬时过载（硬限制）
- **数据库延迟**：感知 MySQL 压力（自适应）
- **集群对比**：感知当前节点是否异常（横向对比）

三层联合判断，避免单一指标的误判。

### 9.1 事件流拓扑

  └────────────┘     └───────────────────┘     └───────────────────┘

### 9.2 为什么用 Kafka 而不是直接 gRPC 回调

| 维度 | Kafka 异步 | gRPC 直接回调 |
|------|-----------|--------------|
| 解耦 | Executor 不需要知道 Scheduler 地址 | 需要维护 Scheduler 的服务发现 |
| 缓冲 | Kafka 天然削峰填谷 | 大量并发回调可能压垮 Scheduler |
| 可靠性 | 消息持久化，消费失败可重试 | 回调失败需要 Executor 自己重试 |
| 延迟 | 稍高（毫秒级 → 百毫秒级） | 更低（直接 RPC） |

在任务调度场景中，**可靠性优先于延迟**，Kafka 是正确的选择。

### 9.3 Report vs BatchReport 双通道

```go
// 单条上报（实时性高的场景）
func (s *GrpcServer) Report(ctx context.Context, req *reporterv1.ReportRequest) {
    event := domain.Report{
        ExecutionID: req.ExecutionId,
        Status:      domain.ExecStatus(req.Status),
        Progress:    req.Progress,
    }
    s.reportProducer.Send(ctx, event)
}

// 批量上报（吞吐量高的场景）
func (s *GrpcServer) BatchReport(ctx context.Context, req *reporterv1.BatchReportRequest) {
    events := make([]domain.Report, len(req.Reports))
    for i, r := range req.Reports {
        events[i] = domain.Report{...}
    }
    s.batchReportProducer.SendBatch(ctx, events)
}
```

**为什么需要两个通道：**
- **单条上报**：用于状态变更等关键事件（PREPARE → RUNNING），需要尽快处理
- **批量上报**：用于进度更新等高频低优先级事件（progress: 30% → 31%），合并减少 I/O

**🟡 改进建议**：当前 BatchReport 的消费者是逐条处理的（for 循环 UpdateState），没有利用批量写入优势。建议消费端也做批量处理：
```go
// 建议：批量更新进度
s.repo.BatchUpdateProgress(ctx, reports)
```

## 11. 性能优化与设计权衡

### 11.1 ShardingLoopJob 框架（V2 核心）

```go
// pkg/loopjob/shardingloopjob.go
type ShardingLoopJob struct {
    shardStrategy  sharding.Strategy   // 分表策略
    lockClient     distributedlock.Client
    resourceSem    *ResourceSemaphore  // 并发控制
}

func (j *ShardingLoopJob) Run(ctx context.Context, bizFunc BizFunc) {
    // 1. 获取所有分表
    shards := j.shardStrategy.Broadcast()  // ["_0", "_1", ..., "_31"]

    for _, shard := range shards {
        go func(tableSuffix string) {
            // 2. 尝试获取分布式锁
            lock, err := j.lockClient.Lock(ctx, "loopjob:" + tableSuffix)
            if err != nil { return }  // 被其他节点锁住了

            // 3. 获取资源信号量
            j.resourceSem.Acquire()

            // 4. 业务循环
            j.bizLoop(ctx, lock, tableSuffix, bizFunc)

            // 5. 释放资源
            j.resourceSem.Release()
            lock.Unlock(ctx)
        }(shard)
    }
}

func (j *ShardingLoopJob) bizLoop(ctx context.Context, lock Lock, shard string, bizFunc BizFunc) {
    for {
        // 执行业务逻辑
        bizFunc(ctx, shard)

        // 续约锁
        lock.Refresh(ctx)
    }
}
```

**为什么需要 ShardingLoopJob：**
- V1 方案：所有 Scheduler 节点扫描全表 → 竞争激烈，大量无效查询
- V2 方案：分表 + 分布式锁 → 每个分表只有一个节点处理，零竞争

**ResourceSemaphore 的作用：**
- 限制单节点同时处理的分表数量（避免一个节点抢到所有锁）
- 信号量值 = min(总分表数 / 节点数, 配置上限)

### 11.2 Snowflake ID 生成

```go
// pkg/id_generator/snowflake.go
type SnowflakeNode struct {
    bizID    int64  // 12 bit, 业务标识
    sequence int64  // 12 bit, 序列号 (atomic)
}

// 64-bit ID 结构:
// [12 bit bizID] [12 bit sequence] [41 bit timestamp]
func (n *SnowflakeNode) Generate() int64 {
    seq := atomic.AddInt64(&n.sequence, 1) & 0xFFF  // 12 bit mask
    ts := time.Now().UnixMilli() & 0x1FFFFFFFFFF     // 41 bit mask
    return (n.bizID << 53) | (seq << 41) | ts
}
```

**🔴 不合理之处**：这个 Snowflake 实现有一个严重问题——**sequence 和 timestamp 不是同步递增的**。标准 Snowflake 在同一毫秒内递增 sequence，毫秒切换时 sequence 归零。但这个实现中 sequence 是全局原子递增的，不与毫秒绑定。

后果：
1. **ID 不单调递增**：如果两次 Generate() 在同一毫秒，sequence 不同但 timestamp 相同，ID 的大小取决于 sequence 的高位
2. **sequence 溢出**：12 bit = 4096，超过后 mask 会回绕，可能产生重复 ID

**建议**：参考标准 Snowflake 算法，在同一毫秒内递增 sequence，毫秒切换时归零，sequence 溢出时等待下一毫秒。

### 11.3 gRPC Client Pool（懒加载 + 线程安全）

```go
// pkg/grpc/clients_v2.go
type ClientsV2[T any] struct {
    pool    syncx.Map[string, T]
    factory func(serviceKey string) (T, error)
}

func (c *ClientsV2[T]) GetOrCreate(serviceKey string) (T, error) {
    // 双重检查：先 Load（无锁快路径），再 LoadOrStore（加锁慢路径）
    if client, ok := c.pool.Load(serviceKey); ok {
        return client, nil
    }

    client, err := c.factory(serviceKey)
    if err != nil {
        return zero, err
    }

    actual, loaded := c.pool.LoadOrStore(serviceKey, client)
    if loaded {
        // 有人先创建了，关闭我们多创建的连接
        client.Close()
        return actual, nil
    }
    return client, nil
}
```

**为什么用 `syncx.Map` 的 `LoadOrStore` 而不是 `sync.Mutex`：**
- `LoadOrStore` 保证只有一个 goroutine 的值被存入，其他的自动获取已存在的值
- 无锁读路径（`Load`）在连接已建立后是零开销的
- 代价是可能多创建一个连接然后关闭，但创建是一次性的，长期运行后不再发生

### 12.4 架构级思考

**1. 数据库是否会成为瓶颈？**

当前所有操作（调度、抢占、续约、补偿）都以 MySQL 为中心。在百万级任务量下：
- `FindSchedulableTasks` 的 `WHERE next_time <= ? AND status = ?` 需要在 `(status, next_time)` 上建复合索引
- `Renew` 批量更新 + `Acquire` CAS 更新会产生行锁竞争
- 建议在配置中预设 `(status, next_time)` 的复合索引，或在 GORM AutoMigrate 中显式定义

**2. 是否需要 Redis 缓存层？**

项目中 Redis 的依赖已经声明（`config.yaml` 中有 Redis 配置），但代码中未使用。在以下场景中 Redis 可以显著提升性能：
- 任务抢占：用 Redis 分布式锁替代 MySQL 乐观锁（高冲突场景）
- 续约：用 Redis 的 `EXPIRE` 替代 MySQL 的 `utime` 更新
- 节点状态缓存：缓存 Prometheus Picker 的结果

**3. 监控盲区**

当前 Prometheus 监控覆盖了 GORM 操作和调度器循环耗时，但缺少：
- 补偿器处理速率和积压量
- Kafka 消费延迟
- 任务从创建到执行的端到端延迟
- 节点持有任务数量的分布
