## 第1章 系统架构总览

### 1.1 项目定位与设计目标

本项目是一个**生产级分布式任务调度平台**，对标业界的 XXL-Job、Elastic-Job 等方案，但在以下维度做了显著增强：

| 维度 | XXL-Job | Elastic-Job | 本项目 |
|------|---------|-------------|--------|
| 调度模型 | 中心化 Admin | 去中心化 ZK | 多调度节点 + 乐观锁抢占 |
| 节点选择 | 轮询/随机/一致性哈希 | 分片 | **Prometheus 实时指标 + 智能选择** |
| 工作流 | 子任务链 | 不支持 | **ANTLR4 DSL + DAG 引擎** |
| 负载保护 | 无 | 无 | **三层负载检查器** |
| 分片策略 | 静态分片 | 静态分片 | **权重动态分片 + 静态范围分片** |
| 补偿机制 | 手动重试 | 失败转移 | **四种自动补偿器** |
| 架构演进 | 单版本 | 单版本 | **V1/V2 双版本平滑演进** |

**核心设计目标：**

1. **高可用**：多调度节点无状态部署，任务级乐观锁抢占，任何节点宕机不影响任务执行
2. **高性能**：三层负载保护 + 批量操作 + 异步事件驱动，支撑万级任务并发
3. **可扩展**：Strategy 模式贯穿全局（调度策略、分片策略、重试策略、节点选择策略），新增策略零侵入
4. **可观测**：Prometheus 全链路埋点，从调度耗时到数据库响应时间全覆盖
5. **领域驱动**：清晰的四层架构，领域模型承载核心业务规则

### 1.3 核心模块职责

#### 1.3.1 调度引擎 (Scheduler)

**源码位置**: `internal/service/scheduler/`

调度引擎是系统的"心脏"，负责任务的发现、抢占和分发。系统提供了两个版本：

| 版本 | 类型 | 文件 | 核心差异 |
|------|------|------|---------|
| V1 | `Scheduler` | `scheduler.go` | 简单 for 循环调度，单节点独占 |
| V2 | `SchedulerV2` | `schedulerv2.go` | ShardingLoopJob + 分布式锁 + 资源信号量 |

**V1 调度循环（`scheduler.go:112-166`）：**

```
scheduleLoop():
    for {
        1. 检查 ctx 是否取消
        2. 开始计时（Prometheus）
        3. 批量获取可调度任务 (SchedulableTasks)
        4. 如果没有任务 → sleep(ScheduleInterval)
        5. 遍历任务 → runner.Run(newContext(task), task)
        6. 负载检查 → loadChecker.Check()
        7. 如果负载过高 → sleep(duration) + 记录指标
    }
```

**V2 调度循环（`schedulerv2.go:102-166`）：**

V2 的核心改进在于将 `scheduleLoop` 改为 `ShardingLoopJob` 驱动：

```go
// schedulerv2.go:102-111
func (s *SchedulerV2) Start() error {
    const scheduleKey = "scheduleKey"
    // 改成分库调度
    go loopjob.NewShardingLoopJob(s.dclient, scheduleKey, s.scheduleOneLoop, s.taskStr, s.sem)
    go s.renewLoop()
    return nil
}
```

`ShardingLoopJob` 做了三件事：
1. **分布式锁**：通过 `dlock.Client` 获取锁，保证同一分片只有一个节点处理
2. **分片策略**：通过 `sharding.ShardingStrategy` 确定当前节点处理哪些分片
3. **资源信号量**：通过 `ResourceSemaphore` 控制并发度

**为什么需要 V1 → V2 演进：**

V1 在单库场景下工作良好，但当数据量增长到需要分库分表时，多个调度节点同时扫描所有库表会造成大量无效竞争。V2 引入分片机制，每个调度节点只负责自己分片内的任务，大幅降低锁冲突。

#### 1.3.2 任务执行层 (Runner + Invoker)

**源码位置**: `internal/service/runner/`, `internal/service/invoker/`

执行层采用**双层分发**架构：

```
Runner 层（任务编排）          Invoker 层（远程调用）
┌──────────────┐              ┌──────────────┐
│  Dispatcher  │─── normal ──→│  Dispatcher  │─── grpc ──→ GRPCInvoker
│              │              │              │─── http ──→ HTTPInvoker
│              │─── plan ────→│              │─── local ─→ LocalInvoker
└──────────────┘              └──────────────┘
       │                              │
  NormalTaskRunner              优先级：GRPC > HTTP > Local
  PlanTaskRunner               （由任务配置决定）
```

**Runner 分发逻辑（`runner/dispather.go:14-43`）：**

```go
func (d *Dispatcher) Run(ctx context.Context, task domain.Task) error {
    r, ok := d.runners[task.Type.String()]
    if !ok {
        return errs.ErrUnknownTaskType
    }
    return r.Run(ctx, task)
}
```

**Invoker 分发逻辑（`invoker/dispatcher.go:20-54`）：**

```go
func (d *Dispatcher) Invoke(ctx context.Context, exec domain.TaskExecution) error {
    // 优先级排列
    if exec.Task.GrpcConfig != nil {
        return d.grpcInvoker.Invoke(ctx, exec)
    }
    if exec.Task.HTTPConfig != nil {
        return d.httpInvoker.Invoke(ctx, exec)
    }
    return d.localInvoker.Invoke(ctx, exec)
}
```

**设计亮点**：Runner 层负责"做什么"（任务编排逻辑），Invoker 层负责"怎么调"（通信协议）。这种分离使得新增任务类型（如 WorkflowTaskRunner）或新增调用方式（如 Dubbo）互不影响。

#### 1.3.3 补偿引擎 (Compensator)

**源码位置**: `internal/compensator/`

补偿引擎是系统可靠性的核心保障，通过四个独立的补偿器覆盖所有异常场景：

| 补偿器 | 触发条件 | 处理逻辑 | 源码文件 |
|--------|---------|---------|---------|
| RetryCompensator | `FAILED_RETRYABLE` + 到达重试时间 | 调用 `Runner.Retry()` | `retry.go` |
| RescheduleCompensator | `FAILED_RESCHEDULED` | 调用 `Runner.Reschedule()`，排除原节点 | `reschedule.go` |
| ShardingCompensator | 分片子任务全部完成 | 聚合子任务状态，更新父任务 | `sharding.go` |
| InterruptCompensator | `RUNNING` + 超过 deadline | gRPC 调用 `Interrupt` 中断执行 | `interrupt.go` |

每个补偿器都有 V1（简单循环）和 V2（`ShardingLoopJob`）两个版本，与调度器的演进保持一致。

#### 1.3.4 工作流引擎 (Plan/DSL)

**源码位置**: `internal/dsl/parser/`, `internal/service/task/plan.go`

工作流引擎通过自定义 DSL 描述任务间的依赖关系，由 ANTLR4 解析为 AST，再转化为可执行的 DAG：

```
DSL 文本
   │
   ▼
ANTLR4 Lexer/Parser
   │
   ▼
AST (抽象语法树)
   │
   ▼
TaskOrchestrationVisitor
   │
   ▼
TaskPlan (DAG 执行计划)
   │
   ▼
PlanTask 链表 (PreTask ← Task → NextTask)
```

**支持的节点类型：**

| 节点类型 | 语义 | CheckPre 逻辑 | NextNodes 逻辑 |
|---------|------|--------------|----------------|
| SimpleNode | 单任务 | - | 返回自身名称 |
| AndNode | 并行（全部成功） | 所有前置 Success | 返回所有子节点名称 |
| OrNode | 并行（任一成功） | 任一前置 Success | 返回所有子节点名称 |
| ConditionNode | 条件分支 | 任一前置 Success | 根据执行状态选择 Success/Failure 分支 |
| EndNode | 终止 | - | 返回空 |

#### 1.3.5 事件驱动层 (Event)

**源码位置**: `internal/event/`, `internal/grpc/`

系统使用 **Kafka** 实现异步事件驱动，有两类事件：

**1. 执行完成事件（Complete Event）**

```
Executor 执行完成
    │
    ▼
gRPC Report → ReporterServer
    │
    ▼
CompleteProducer → Kafka (complete_topic)
    │
    ▼
CompleteConsumer:
    ├── normal task → PlanTaskRunner.NextStep() → 触发下游任务
    └── plan task  → 更新 Plan 状态 + 释放任务
```

**2. 执行上报事件（Report Event）**

```
Executor 进度上报
    │
    ▼
gRPC Report/BatchReport → ReporterServer
    │
    ▼
ReportEventProducer → Kafka (execution_report / execution_batch_report)
    │
    ▼
ReportEventConsumer → ExecutionService.UpdateState()
```

**为什么用事件驱动而不是同步调用：**
- Report 是高频操作（进度上报），同步处理会阻塞 gRPC 线程
- Complete Event 驱动工作流的下一步，需要保证 at-least-once 语义
- 解耦调度器和执行器的生命周期

### 1.6 分层架构详解

系统采用经典的**四层架构**，严格遵循依赖方向：

```
┌─────────────────────────────┐
│     Service / Scheduler     │  ← 业务编排层
│  (调度逻辑、补偿逻辑、Runner)  │
├─────────────────────────────┤
│         Domain              │  ← 领域模型层（纯逻辑，无依赖）
│  (Task, TaskExecution, Plan) │
├─────────────────────────────┤
│       Repository            │  ← 仓储层（领域↔持久化转换）
│  (TaskRepository, ...)      │
├─────────────────────────────┤
│         DAO                 │  ← 数据访问层
│  (GORMTaskDAO, ...)         │
├─────────────────────────────┤
│     Infrastructure          │  ← 基础设施层
│  (MySQL, Redis, Kafka, etcd) │
└─────────────────────────────┘
```

**关键约束：**

1. **Domain 层零依赖外部包**：`internal/domain/` 中的结构体只依赖标准库和内部定义，不依赖 GORM、gRPC 等框架
2. **Repository 负责转换**：`internal/repository/task.go` 中的 `toDomain()` 和 `toEntity()` 方法负责 Domain ↔ DAO 的双向转换
3. **Service 层编排**：`internal/service/` 中的服务只依赖 Domain 和 Repository 接口，不直接操作数据库
4. **ioc/ 组装一切**：`ioc/` 包通过 Wire 框架完成所有依赖注入，是唯一知道所有实现类的地方

**为什么选择四层而不是三层（Controller-Service-DAO）：**

引入 Repository 层的核心价值在于**隔离领域模型和持久化模型**。以 `Task` 为例：

- **Domain 层的 `Task`**：使用 `map[string]string` 存储参数，`ShardingRule` 是完整的结构体
- **DAO 层的 `Task`**：使用 `sqlx.JSONColumn[map[string]string]` 存储参数，`ShardingRule` 被 JSON 序列化

如果没有 Repository 层，Domain 模型就会被 GORM 标签"污染"，失去领域模型的纯粹性。

#### 2.2.1 Task 状态机

Task 有三个状态，通过乐观锁 CAS 操作进行状态转移：

```
                    ┌──────────────────┐
                    │                  │
          Acquire   │    ACTIVE        │   Create
       ┌───────────→│  (可调度)         │←──────────
       │            │                  │
       │            └────────┬─────────┘
       │                     │
       │              Acquire│(CAS: version match)
       │                     │
       │            ┌────────▼─────────┐
       │            │                  │
       │            │   PREEMPTED      │
       │   Release  │  (已抢占)         │───── Renew (续约, 更新 utime)
       │←───────────│                  │         │
       │            └────────┬─────────┘         │
       │                     │                   │
       │                     │                   │
       │            ┌────────▼─────────┐         │
       │            │                  │         │
       └────────────│   INACTIVE       │         │
         Deactivate │  (已停用)         │◀────────┘
                    │                  │  超时未续约
                    └──────────────────┘  → 重新变为可调度
```

**关键设计决策：**

1. **PREEMPTED 而非 LOCKED**：使用"抢占"语义而非"锁定"语义，因为任务可能被长时间持有（分钟级），传统锁不适用
2. **版本号驱动 CAS**：每次操作都递增 version，`Acquire` 时必须 version 匹配才能成功，自然避免了并发冲突
3. **utime 作为心跳**：`Renew` 操作更新 `utime`，当 `utime` 超过阈值时，任务被视为"僵尸"可被重新抢占

#### 2.2.3 核心业务方法

**计算下次执行时间（`task.go:120-129`）：**

```go
func (t *Task) CalculateNextTime() (time.Time, error) {
    p := cron.NewParser(
        cron.Second | cron.Minute | cron.Hour |
        cron.Dom | cron.Month | cron.Dow | cron.Descriptor,
    )
    s, err := p.Parse(t.CronExpr)
    if err != nil {
        return time.Time{}, err
    }
    return s.Next(time.Now()), nil
}
```

支持 7 字段 Cron 表达式（含秒），使用 `robfig/cron/v3` 库解析。

**智能合并调度参数（`task.go:131-150`）：**

```go
func (t *Task) UpdateScheduleParams(params map[string]string) {
    if params == nil {
        return           // nil = 无操作
    }
    if len(params) == 0 {
        t.ScheduleParams = make(map[string]string) // 空 map = 重置
        return
    }
    if t.ScheduleParams == nil {
        t.ScheduleParams = make(map[string]string)
    }
    for k, v := range params {
        t.ScheduleParams[k] = v  // 非空 map = 合并
    }
}
```

**三种语义的精妙设计**：
- `nil` → 不修改（调用者无意修改）
- `空 map` → 清空（业务意图是重置）
- `非空 map` → 合并（增量更新）

这种设计避免了"传 nil 到底是不改还是清空"的歧义。

#### 2.3.1 执行状态机

```
                        创建执行记录
                            │
                            ▼
                    ┌───────────────┐
                    │   PREPARE     │
                    │  (已创建)      │
                    └───────┬───────┘
                            │ Invoker 开始执行
                            ▼
                    ┌───────────────┐
             ┌──────│   RUNNING     │──────┐
             │      │  (执行中)      │      │
             │      └───────┬───────┘      │
             │              │              │
     执行成功 │     执行失败  │     可重试失败│     需重调度
             │              │              │         │
             ▼              ▼              ▼         ▼
     ┌──────────┐  ┌──────────┐  ┌──────────────┐ ┌─────────────────┐
     │ SUCCESS  │  │  FAILED  │  │FAILED_RETRY- │ │FAILED_RESCHED-  │
     │ (成功)    │  │ (失败)    │  │ABLE(可重试)  │ │ULED(需重调度)    │
     │ 终态     │  │ 终态      │  └──────┬───────┘ └────────┬────────┘
     └──────────┘  └──────────┘         │                   │
                                        │ RetryCompensator  │ RescheduleComp.
                                        ▼                   ▼
                                  重新进入 RUNNING      重新创建执行记录
                                  (同一 Execution)     (新 Execution)
```

**关键设计点：**

1. **终态明确**：只有 `SUCCESS` 和 `FAILED` 是终态，其他状态都可能发生转移
2. **重试与重调度的区别**：
   - 重试（`FAILED_RETRYABLE`）：同一个 Execution 记录，递增 `RetryCount`，在同一节点再次执行
   - 重调度（`FAILED_RESCHEDULED`）：创建新的 Execution 记录，排除原节点，在不同节点执行
3. **ShardingParentID 三值语义**：
   - `nil`：普通任务（非分片）
   - `0`：分片父任务
   - `>0`：分片子任务，值为父任务 ID

#### 2.3.2 参数合并机制

**GRPCParams（`task_execution.go:122-143`）：**

```go
func (te *TaskExecution) GRPCParams() map[string]string {
    result := make(map[string]string)
    // 1. 业务参数（GrpcConfig.Params）
    // 2. 调度参数覆盖（ScheduleParams，优先级更高）
```

- 调度器可以在每次执行时覆盖特定参数（如分片的 start/end）
- 超时参数始终存在，执行器可以据此设置 deadline

#### 2.4.1 Range 分片策略

**源码**: `task.go:201-214`

将数据按固定步长分片：

```
参数: step=1000, totalNums=5
结果:
  Shard 0: start=0, end=1000
  Shard 1: start=1000, end=2000
  Shard 2: start=2000, end=3000
  Shard 3: start=3000, end=4000
  Shard 4: start=4000, end=5000
```

**适用场景**：数据分布均匀，各分片处理量相近。

#### 2.4.2 WeightedDynamicRange 加权动态分片策略

**源码**: `task.go:216-260`

根据执行节点的注册权重动态分配分片大小：

```
参数: total_tasks=10000
节点: [A(weight=3), B(weight=2), C(weight=5)]
总权重: 10

分配:
  Node A: start=0,    end=3000   (3/10 * 10000)
  Node B: start=3000, end=5000   (2/10 * 10000)
  Node C: start=5000, end=10000  (剩余全部，避免舍入丢失)
```

**核心代码逻辑**：

```go
// task.go:240-258
// 遍历 N-1 个节点
for i := 0; i < len(w.ExecutorNodeInstances)-1; i++ {
    share := float64(totalTasks) * (float64(w.ExecutorNodeInstances[i].Weight) / float64(totalWeight))
    step = int64(share)
    end = start + step
    scheduleParams = append(scheduleParams, map[string]string{
        "start": strconv.FormatInt(start, 10),
        "end":   strconv.FormatInt(end, 10),
    })
    start = end
}
// 最后一个节点获得所有剩余任务（避免舍入误差导致任务丢失）
scheduleParams = append(scheduleParams, map[string]string{
    "start": strconv.FormatInt(start, 10),
    "end":   strconv.FormatInt(totalTasks, 10),
})
```

**设计亮点**：
1. **最后一个节点兜底**：避免浮点数舍入导致任务遗漏
2. **权重来自 etcd 注册信息**：节点可以根据自身硬件能力注册不同权重
3. **运行时动态计算**：每次执行时重新计算分片，适应节点动态上下线

```go
func (p *PlanTask) CheckPre() bool {
    switch p.AstPlanNode.Pre.Type() {
    case parser.NodeTypeSingle, parser.NodeTypeAnd:
        return p.preAllSuccess()  // AND 语义：所有前驱成功
    case parser.NodeTypeCondition, parser.NodeTypeOr:
        return p.preOneSuccess()  // OR 语义：任一前驱成功
    default:
        return false
    }
}
```

**设计精妙之处**：`SetNext` 方法接收当前执行状态作为参数（通过 `newAstExecution`），这意味着 `ConditionNode` 可以根据执行结果（成功/失败）动态决定走哪个分支。

#### 3.2.1 索引设计分析

**核心查询索引 `idx_next_time_status_utime`**：

这是系统最高频的查询，在 `FindSchedulableTasks` 中使用：

```sql
-- dao/task.go:119-128
SELECT * FROM tasks
WHERE next_time <= ? AND (
    status = 'ACTIVE'
    OR (status = 'PREEMPTED' AND utime <= ?)
)
ORDER BY next_time ASC
LIMIT ?
```

索引 `(next_time, status, utime)` 的设计考量：
1. **next_time 在最左**：支持 `next_time <= ?` 的范围扫描
2. **status 在中间**：用于过滤 ACTIVE/PREEMPTED 状态
3. **utime 在最右**：用于判断 PREEMPTED 状态的超时

**续约索引 `idx_schedule_node_id_status`**：

```sql
-- dao/task.go:167-173
UPDATE tasks SET version = version + 1, utime = ?
WHERE schedule_node_id = ? AND status = 'PREEMPTED'
```

这个索引让续约操作只扫描当前节点持有的任务。

#### 3.2.3 乐观锁实现

**抢占操作（`dao/task.go:131-164`）：**

```go
func (g *GORMTaskDAO) Acquire(ctx context.Context, id, version int64, scheduleNodeID string) (*Task, error) {
    err := g.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
        result := tx.Model(&Task{}).
            Where("id = ? AND version = ?", id, version).
            Updates(map[string]any{
                "status":           StatusPreempted,
                "schedule_node_id": scheduleNodeID,
                "version":          gorm.Expr("version + 1"),
                "utime":            time.Now().UnixMilli(),
            })
        if result.RowsAffected == 0 {
            return errs.ErrTaskPreemptFailed  // CAS 失败
        }
        // 重新查询获取最新数据
        var task Task
        if err := tx.Where("id = ?", id).First(&task).Error; err != nil {
            return err
        }
        acquiredTask = &task
        return nil
    })
    return acquiredTask, nil
}
```

**为什么用事务包裹 UPDATE + SELECT**：

单纯的 `UPDATE ... WHERE version = ?` 可以保证 CAS 语义，但调用方需要知道更新后的完整 Task 数据（包括新的 version）。将 UPDATE 和 SELECT 放在同一事务中，保证读到的是刚刚更新的数据。

**为什么 preemptedTimeout 是 10 分钟**：
- 续约间隔是 5 秒
- 正常情况下 utime 每 5 秒更新一次
- 10 分钟 = 120 次续约机会，容忍短暂网络抖动
- 如果 10 分钟内都没有续约，节点大概率已经宕机

**为什么 scheduleInterval 是 10 秒**：
- 任务粒度通常是分钟级（Cron 最小 1 秒）
- 10 秒的检查间隔平衡了及时性和数据库压力
- 配合负载检查器，高负载时自动降频

#### 5.1.3 TaskExecutionRepository 的特殊设计

**源码**: `internal/repository/task_execution.go`

`TaskExecutionRepository` 相比 `TaskRepository` 有两个特殊之处：

**1. Create 时自动补全 Task 信息（第97-115行）**：

```go
func (r *taskExecutionRepository) Create(ctx context.Context, execution domain.TaskExecution) (domain.TaskExecution, error) {
    if execution.Task.ID == 0 {
        return domain.TaskExecution{}, errors.New("Task.ID不能为空")
    }
    // 获取完整的Task信息
    completeTask, err := r.taskRepo.GetByID(ctx, execution.Task.ID)
    if err != nil {
        return domain.TaskExecution{}, fmt.Errorf("获取Task信息失败: %w", err)
    }
    completeTask.PlanExecID = execution.Task.PlanExecID
    execution.Task = completeTask
    // ...
}
```

**为什么要这样做**：调用方只需要传入 `Task.ID`，Repository 自动查询并冗余完整的 Task 信息到 `task_executions` 表。这保证了执行记录的**自包含性**——即使原始 Task 被修改，历史执行记录仍保存当时的配置快照。

**2. CreateShardingChildren 批量创建（第125-155行）**：

```go
func (r *taskExecutionRepository) CreateShardingChildren(
    ctx context.Context,
    createdParent domain.TaskExecution,
    executorNodeIDs []string,
    scheduleParams []map[string]string,
) ([]domain.TaskExecution, error) {
    children := make([]domain.TaskExecution, 0, len(scheduleParams))
    for i := range scheduleParams {
        child := createdParent          // 值拷贝
        child.ID = 0                    // 清除 ID，让数据库自增
        child.Status = domain.TaskExecutionStatusPrepare
        parentID := createdParent.ID
        child.ShardingParentID = &parentID
        if i < len(executorNodeIDs) {
            child.ExecutorNodeID = executorNodeIDs[i]
        }
        child.MergeTaskScheduleParams(scheduleParams[i])
        children = append(children, child)
    }
    // 批量创建
    createdExecutions, err := r.dao.BatchCreate(ctx, ...)
    // ...
}
```

**设计精妙之处**：
- 值拷贝父任务 → 子任务继承所有配置
- 清除 ID → 数据库自增生成新 ID
- `MergeTaskScheduleParams` → 每个子任务有不同的分片参数（start/end）
- `executorNodeIDs` 可选 → WeightedDynamic 策略预分配节点，Range 策略不预分配

```go
}
```

**冗余设计的权衡**：

| 方案 | 优点 | 缺点 |
|------|------|------|
| JOIN 查询 | 无数据冗余 | 性能差，尤其补偿器批量查询 |
| 冗余快照 | 查询高效、数据自包含 | 存储空间增加 |
| 宽表方案 | 查询最简单 | 更新麻烦 |

本项目选择**冗余快照**方案，原因：
1. 补偿器需要频繁批量查询 Execution + Task 信息
2. Task 配置在执行期间不应该变化（快照语义）
3. 存储空间在现代数据库中不是瓶颈

## 第6章 ID 生成器设计

### 6.1 Snowflake 变种算法

**源码**: `pkg/id_generator/snowflake.go`

系统使用了 Snowflake 算法的变种，核心改动是**将 workerID 替换为 bizID（分片 ID）**：

```
标准 Snowflake:
┌─────────┬────────────┬────────────┬────────────┐
│ 1 bit   │  41 bits   │  10 bits   │  12 bits   │
│ (符号位) │ (时间戳)    │ (机器ID)   │ (序列号)    │
└─────────┴────────────┴────────────┴────────────┘

本项目变种:
┌────────────┬────────────┬────────────┐
│  12 bits   │  12 bits   │  41 bits   │
│ (bizID)    │ (序列号)    │ (时间戳)    │
└────────────┴────────────┴────────────┘
```

**位分配常量**：

```go
const (
    timestampBits = 41  // 时间戳：约69年
    bizIDBits     = 12  // 业务ID：0-4095
    sequenceBits  = 12  // 序列号：0-4095

    // 位移：timestamp在最低位
    timestampShift = 0
    sequenceShift  = timestampBits           // 41
    bizIDShift     = sequenceBits + timestampBits  // 53
)
```

### 6.2 ID 组装逻辑

```go
// snowflake.go:49-61
func (g *Generator) GenerateID(shardingID int64) int64 {
    timestamp := time.Now().UnixMilli() - epochMillis
    shardingValue := shardingID & shardingIDMask
    sequence := atomic.AddInt64(&g.sequence, 1) - 1

    id := (shardingValue)<<bizIDShift |
          (sequence&sequenceMask)<<sequenceShift |
          (timestamp & timestampMask)
    return id
}
```

**为什么把 bizID 放在最高位**：

在分库分表场景下，路由规则通常是 `id % shardCount`。如果 bizID 在最高位，那么：

```
ID = bizID << 53 | sequence << 41 | timestamp

id % 4096 ≈ timestamp 的低12位（接近随机分布）
```

但如果用 `id >> 53` 取 bizID，可以直接获得分片信息，无需额外查询。

**反解方法**：

```go
func ExtractShardingID(id int64) int64 {
    return (id >> bizIDShift) & shardingIDMask
}

func ExtractTimestamp(id int64) time.Time {
    timestamp := id & timestampMask
    return time.Unix(0, (timestamp+epochMillis)*int64(time.Millisecond))
}

func ExtractSequence(id int64) int64 {
    return (id >> sequenceShift) & sequenceMask
}
```

### 6.3 与标准 Snowflake 的对比

| 维度 | 标准 Snowflake | 本项目变种 |
|------|---------------|-----------|
| 总位数 | 64 bit | 65 bit（12+12+41） |
| 时间精度 | 毫秒 | 毫秒 |
| 时间范围 | ~69年 | ~69年 |
| 机器/分片 | 10 bit (1024) | 12 bit (4096) |
| 序列号 | 12 bit (4096/ms) | 12 bit (4096/ms) |
| 排序性 | 时间有序 | 分片内时间有序 |
| 分片友好 | 需额外处理 | 原生支持 |

### 6.4 线程安全

使用 `sync/atomic` 保证序列号的原子递增：

```go
sequence := atomic.AddInt64(&g.sequence, 1) - 1
```

注意：这里没有时钟回拨保护。在分布式场景下如果发生时钟回拨，可能生成重复 ID。生产环境建议增加时钟检测逻辑。

## 第7章 sqlx.JSONColumn 泛型实现

### 7.1 设计动机

MySQL 8.0 提供了原生 JSON 类型，但 Go 的 `database/sql` 接口不直接支持 JSON 类型的序列化/反序列化。GORM 虽然有 JSON 支持，但需要每个字段手动实现 `Scan/Value` 方法。

`sqlx.JSONColumn[T]` 使用 Go 1.18+ 泛型，**一次实现、处处复用**：

```go
type JSONColumn[T any] struct {
    Val   T
    Valid bool
}
```

### 7.2 接口实现

**driver.Valuer（写入数据库）**：

```go
func (j JSONColumn[T]) Value() (driver.Value, error) {
    if !j.Valid {
        return nil, nil  // NULL
    }
    res, err := json.Marshal(j.Val)
    return string(res), err  // JSON 字符串
}
```

**sql.Scanner（从数据库读取）**：

```go
func (j *JSONColumn[T]) Scan(src any) error {
    var bs []byte
    switch val := src.(type) {
    case nil:
        return nil  // 数据库 NULL，不修改
    case []byte:
        bs = val
    case string:
        bs = []byte(val)
    default:
        return fmt.Errorf("不支持的类型 %v", src)
    }
    if err := json.Unmarshal(bs, &j.Val); err != nil {
        return err
    }
    j.Valid = true
    return nil
}
```

### 7.3 Valid 字段的重要性

`Valid` 字段区分了两种状态：
- `Valid=false`：数据库中是 NULL（或字段不存在）
- `Valid=true, Val=零值`：数据库中是有效的 JSON（如 `{}`）

这与 `sql.NullString`、`sql.NullInt64` 的设计思路一致。

### 7.4 使用示例

```go
// 写入时
task := dao.Task{
    GrpcConfig: sqlx.JSONColumn[domain.GrpcConfig]{
        Val:   domain.GrpcConfig{ServiceName: "user-service"},
        Valid: true,
    },
}

// 读取时
if task.GrpcConfig.Valid {
    serviceName := task.GrpcConfig.Val.ServiceName
}
```

## 第8章 错误处理设计

### 8.1 Sentinel Error 模式

**源码**: `internal/errs/error.go`

系统定义了统一的哨兵错误（Sentinel Errors），用于跨层传递业务错误语义：

```go
var (
    // 任务抢占相关
    ErrTaskPreemptFailed            = errors.New("任务抢占失败")
    ErrTaskReleaseFailed            = errors.New("任务释放失败")
    ErrTaskRenewFailed              = errors.New("任务续约失败")
    ErrTaskUpdateNextTimeFailed     = errors.New("任务更新下次执行时间失败")
    ErrTaskUpdateScheduleParamsFailed = errors.New("任务更新调度参数失败")

    // 执行相关
    ErrUnknownTaskType              = errors.New("未知的任务类型")
    ErrInvalidTaskExecutionStatus   = errors.New("无效的任务执行状态")
    ErrTaskExecutionNotFound        = errors.New("任务执行记录不存在")

    // Plan 相关
    ErrInitPlanFailed               = errors.New("初始化执行计划失败")
    ErrPlanTaskNotFound             = errors.New("计划任务不存在")

    // 补偿相关
    ErrCompensateFailed             = errors.New("补偿失败")
    ErrInterruptFailed              = errors.New("中断任务失败")
)
```

### 8.2 错误传递策略

```
DAO 层     → 包装数据库错误      → fmt.Errorf("%w: ...: %w", sentinelErr, dbErr)
Repository → 透传 DAO 错误       → 不做额外包装
Service    → 检查 sentinel error → errors.Is(err, errs.ErrXxx)
Scheduler  → 记录日志 + 继续     → 不让单个任务失败影响整体调度
```

**关键原则**：
1. DAO 层使用 `%w` 包装错误，保留错误链
2. Service 层使用 `errors.Is` 判断业务错误类型
3. 调度循环 `for range tasks` 中单个任务失败只记日志，不中断循环
