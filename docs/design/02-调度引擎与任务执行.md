## 第5章 调度引擎核心流程

### 5.1 调度器生命周期

**源码**: `internal/service/scheduler/scheduler.go`

调度器实现了 EGO 框架的 `server.Server` 接口，具备完整的生命周期管理：

```go
var _ server.Server = &Scheduler{}

type Scheduler struct {
    nodeID             string
    runner             runner.Runner
    taskSvc            task.Service
    execSvc            task.ExecutionService
    acquirer           acquirer.TaskAcquirer
    grpcClients        *grpc.ClientsV2[executorv1.ExecutorServiceClient]
    config             Config
    loadChecker        loadchecker.LoadChecker
    metrics            *prometheus.SchedulerMetrics
    executorNodePicker picker.ExecutorNodePicker
    ctx                context.Context
    cancel             context.CancelFunc
    logger             *elog.Component
}
```

**生命周期方法：**

| 方法 | 触发时机 | 行为 |
|------|---------|------|
| `Init()` | 框架初始化 | 空操作（所有初始化在构造函数完成） |
| `Start()` | 框架启动 | 启动 `scheduleLoop` + `renewLoop` 两个 goroutine |
| `Stop()` | 框架停止 | 调用 `cancel()` 取消上下文 |
| `GracefulStop()` | 优雅停机 | 同 `Stop()`，通过 context 传播取消信号 |
| `Info()` | 健康检查 | 返回服务信息，`Healthy = ctx.Err() == nil` |

**为什么 Init 是空操作**：所有组件通过构造函数注入，到达 `Init()` 时已完全初始化。这符合"构造函数注入 > 方法注入"的 DI 最佳实践。

### 5.2 V1 调度循环详解

**源码**: `scheduler.go:112-166`

V1 调度循环是一个经典的无限循环模型：

```go
func (s *Scheduler) scheduleLoop() {
    for {
        // ========== 阶段1: 退出检查 ==========
        if s.ctx.Err() != nil {
            s.logger.Info("调度循环结束")
            return
        }

        // ========== 阶段2: 开始计时 ==========
        stopRecordExecutionTimeFunc := s.metrics.StartRecordExecutionTime()
        s.logger.Info("开始一次调度")

        // ========== 阶段3: 获取可调度任务 ==========
        scheduleCtx, cancelFunc := context.WithTimeout(s.ctx, s.config.BatchTimeout)
        tasks, err := s.taskSvc.SchedulableTasks(
            scheduleCtx,
            s.config.PreemptedTimeout.Milliseconds(),
            s.config.BatchSize,
        )
        cancelFunc()

        // ========== 阶段4: 空闲等待 ==========
        if len(tasks) == 0 {
            stopRecordExecutionTimeFunc()
            time.Sleep(s.config.ScheduleInterval)
            continue
        }

        // ========== 阶段5: 逐个调度 ==========
        successCount := 0
        for i := range tasks {
            err1 := s.runner.Run(s.newContext(tasks[i]), tasks[i])
            if err1 != nil {
                s.logger.Error("调度任务失败", ...)
            } else {
                successCount++
            }
        }

        // ========== 阶段6: 负载检查 ==========
        duration, ok := s.loadChecker.Check(
            loadchecker.WithExecutionTime(s.ctx, stopRecordExecutionTimeFunc()),
        )
        if !ok {
            s.metrics.RecordLoadCheckerSleep("composite", duration)
            time.Sleep(duration)
        }
    }
}
```

**各阶段的设计考量：**

**阶段1 - 退出检查**：放在循环顶部而非 `select` 中，因为调度循环不是事件驱动的，而是主动轮询的。

**阶段3 - 超时控制**：`context.WithTimeout` 防止数据库查询阻塞。`cancelFunc()` 在查询完成后立即调用，释放资源。

**阶段4 - 空闲等待**：无任务时 sleep `ScheduleInterval`（默认 10s），避免空转。注意先调用 `stopRecordExecutionTimeFunc()` 停止计时再 sleep。

**阶段5 - 逐个调度**：用 `for range` 而非 goroutine 并行，因为每个 `runner.Run` 内部已经是异步的（go func）。串行调度保证了控制流的简单性。

**阶段6 - 负载检查**：将本轮调度的总耗时传给 `loadChecker`，如果耗时异常（数据库慢、网络抖动），checker 返回退避时间。

### 5.3 V2 调度循环详解

**源码**: `internal/service/scheduler/schedulerv2.go`

V2 的核心改进是将简单 for 循环替换为 `ShardingLoopJob`：

```go
type SchedulerV2 struct {
    // ... 与 V1 相同的字段
    dclient dlock.Client            // 分布式锁客户端
    sem     loopjob.ResourceSemaphore // 资源信号量
    taskStr sharding.ShardingStrategy // 分片策略
}

func (s *SchedulerV2) Start() error {
    const scheduleKey = "scheduleKey"
    go loopjob.NewShardingLoopJob(
        s.dclient,     // 分布式锁
        scheduleKey,   // 锁的 key
        s.scheduleOneLoop, // 单次调度逻辑
        s.taskStr,     // 分片策略
        s.sem,         // 资源信号量
    )
    go s.renewLoop()
    return nil
}
```

**`scheduleOneLoop` 与 V1 `scheduleLoop` 的区别**：

| 维度 | V1 scheduleLoop | V2 scheduleOneLoop |
|------|-----------------|-------------------|
| 循环控制 | 自己管理 for 循环 | 由 ShardingLoopJob 驱动 |
| 分布式协调 | 无 | 分布式锁 + 分片策略 |
| 并发控制 | 无 | ResourceSemaphore |
| 函数签名 | `func()` | `func(ctx context.Context) error` |
| 退出条件 | `ctx.Err() != nil` | 返回 `ctx.Err()` |

**V2 适用场景**：当任务数据分布在多个数据库分片中，多个调度节点需要协调分工时。

### 5.4 智能节点选择

**源码**: `scheduler.go:168-182`, `schedulerv2.go:168-182`

每个任务调度前，都会通过 `ExecutorNodePicker` 选择最优执行节点：

```go
func (s *Scheduler) newContext(task domain.Task) context.Context {
    if nodeID, err := s.executorNodePicker.Pick(s.ctx, task); err == nil && nodeID != "" {
        s.logger.Info("智能调度选择节点成功",
            elog.String("selectedNodeID", nodeID),
            elog.Int64("taskID", task.ID))
        return balancer.WithSpecificNodeID(s.ctx, nodeID)
    } else {
        s.logger.Error("智能调度选择节点失败，使用默认调度", ...)
        return s.ctx  // fallback: 使用默认轮询
    }
}
```

**关键设计**：
1. 智能选择失败时**降级为轮询**，不影响任务执行
2. 选择结果通过 `context` 传递，`balancer.WithSpecificNodeID` 设置上下文中的目标节点
3. gRPC 负载均衡器（routing_round_robin）会读取这个 context 值，路由到指定节点

### 5.5 续约循环

**源码**: `scheduler.go:185-199`

```go
func (s *Scheduler) renewLoop() {
    ticker := time.NewTicker(s.config.RenewInterval)
    defer ticker.Stop()
    for {
        select {
        case <-s.ctx.Done():
            return
        case <-ticker.C:
            err := s.acquirer.Renew(s.ctx, s.nodeID)
            if err != nil {
                s.logger.Error("批量续约失败", elog.FieldErr(err))
            }
        }
    }
}
```

**续约机制的核心**：

1. 每 `RenewInterval`（默认 5s）执行一次批量续约
2. 续约操作更新所有当前节点持有的 PREEMPTED 任务的 `utime` 和 `version`
3. 如果续约失败（网络抖动），下次仍会重试
4. 如果连续失败超过 `PreemptedTimeout`（默认 10min），其他节点可以重新抢占

**续约 vs 心跳的区别**：
- 心跳：节点级别，证明"我还活着"
- 续约：任务级别，证明"我还在处理这个任务"

本系统使用任务级续约，精度更高——即使节点活着但处理某个任务的 goroutine 卡死了，其他节点也能在超时后接管。

### 5.6 调度器配置详解

```go
type Config struct {

    ScheduleInterval time.Duration  // 无任务时的睡眠间隔（10s）

}
```

**参数关系图**：

```
时间线 ──────────────────────────────────────────────────→

│← BatchTimeout(3s) →│
│ 查询数据库           │
                      │← 逐个调度 + 负载检查 →│
                                               │← ScheduleInterval(10s) →│
                                               │ 空闲等待（无任务时）       │

│← RenewInterval(5s) →│← RenewInterval(5s) →│← RenewInterval(5s) →│
│ 续约                 │ 续约                 │ 续约                 │

│←──────────────── PreemptedTimeout(10min) ──────────────────────→│
│ 超过此时间未续约 → 任务可被重新抢占                                │
```

### 6.2 乐观锁抢占流程

```
调度器发现任务 (FindSchedulableTasks)
    │
    ▼ 任务: {id=1, version=5, status=ACTIVE}
    │
    │  Acquire(id=1, version=5, nodeID="scheduler-1")
    ▼
┌─────────────────────────────────────────────┐
│ MySQL 事务                                   │
│                                              │
│ UPDATE tasks SET                             │
│   status = 'PREEMPTED',                      │
│   schedule_node_id = 'scheduler-1',          │
│   version = version + 1,                     │
│   utime = NOW()                              │
│ WHERE id = 1 AND version = 5                 │
│                                              │
│ 影响行数 = 0 → ErrTaskPreemptFailed          │
│ 影响行数 = 1 → 重新 SELECT 获取最新数据        │
└─────────────────────────────────────────────┘
    │
    ▼ 返回: {id=1, version=6, status=PREEMPTED, scheduleNodeID="scheduler-1"}
```

### 6.3 可调度任务查询

**源码**: `internal/repository/dao/task.go:113-129`

```sql
SELECT * FROM tasks
WHERE next_time <= NOW_MS
  AND (
    status = 'ACTIVE'
    OR (status = 'PREEMPTED' AND utime <= NOW_MS - preemptedTimeoutMs)
  )
ORDER BY next_time ASC
LIMIT batchSize
```

**两个条件的含义**：

| 条件 | 含义 | 场景 |
|------|------|------|
| `status = 'ACTIVE'` | 正常的可调度任务 | 到达执行时间的新任务 |
| `status = 'PREEMPTED' AND utime <= ?` | 超时的僵尸任务 | 持有节点宕机，续约停止 |

**ORDER BY next_time ASC**：优先调度最早应该执行的任务，保证时间公平性。

### 6.4 释放操作

**源码**: `dao/task.go:180-209`

```go
func (g *GORMTaskDAO) Release(ctx context.Context, id int64, scheduleNodeID string) (*Task, error) {
    err := g.db.WithContext(ctx).Transaction(func(tx *gorm.DB) error {
        result := tx.Model(&Task{}).
            Where("id = ? AND status = ? AND schedule_node_id = ?",
                id, StatusPreempted, scheduleNodeID).
            Updates(map[string]any{
                "status":           StatusActive,
                "schedule_node_id": gorm.Expr("NULL"),
                "version":          gorm.Expr("version + 1"),
                "utime":            time.Now().UnixMilli(),
            })
        if result.RowsAffected == 0 {
            return errs.ErrTaskReleaseFailed
        }
        // ...
    })
}
```

**三重条件保护**：
1. `id = ?`：确保操作目标正确
2. `status = 'PREEMPTED'`：只有被抢占的任务才能释放
3. `schedule_node_id = ?`：只有持有者才能释放

这三个条件共同保证了**只有合法持有者才能释放任务**。

### 6.5 续约操作

**源码**: `dao/task.go:166-178`

```go
func (g *GORMTaskDAO) Renew(ctx context.Context, scheduleNodeID string) error {
    result := g.db.WithContext(ctx).
        Model(&Task{}).
        Where("schedule_node_id = ? AND status = ?",
            scheduleNodeID, StatusPreempted).
        Updates(map[string]any{
            "version": gorm.Expr("version + 1"),
            "utime":   time.Now().UnixMilli(),
        })
    return result.Error
}
```

**批量续约**：一条 UPDATE 语句更新当前节点持有的所有任务，效率极高。

**version 递增的必要性**：续约不仅更新 `utime`，还递增 `version`。这保证了如果有其他节点同时尝试 `Acquire`（通过僵尸检测），CAS 会失败——因为 version 已经变了。

### 6.6 抢占机制的并发安全分析

```
时间线 ──────────────────────────────────────────→

节点A:  FindSchedulableTasks → 发现 task1(v=5)
节点B:  FindSchedulableTasks → 发现 task1(v=5)

节点A:  Acquire(id=1, v=5) → 成功! task1 变为 v=6
节点B:  Acquire(id=1, v=5) → 失败! v=5 不匹配(已经是v=6)

节点A:  正常执行任务
节点B:  记录日志，跳过此任务
```

**关键保证**：
1. **互斥性**：同一时刻只有一个节点能成功抢占
2. **可重入性**：同一节点可以再次抢占自己持有的任务（version 匹配）
3. **公平性**：`ORDER BY next_time ASC` 保证先到期的任务先被处理

## 第7章 Runner 执行层

### 7.1 Runner 接口设计

**源码**: `internal/service/runner/types.go`

```go
type Runner interface {
    Run(ctx context.Context, task domain.Task) error
    Retry(ctx context.Context, execution domain.TaskExecution) error
    Reschedule(ctx context.Context, execution domain.TaskExecution) error
}
```

三个方法覆盖了任务执行的三种场景：
- `Run`：首次执行
- `Retry`：重试执行（同一个 Execution，递增 RetryCount）
- `Reschedule`：重调度执行（创建新 Execution，可能换节点）

### 7.2 Dispatcher - Runner 分发

**源码**: `internal/service/runner/dispather.go`

```go
type Dispatcher struct {
    runners map[string]Runner
}

func NewDispatcher(normalRunner *NormalTaskRunner, planRunner *PlanTaskRunner) *Dispatcher {
    return &Dispatcher{
        runners: map[string]Runner{
            domain.NormalTaskType.String(): normalRunner,
            domain.PlanTaskType.String():   planRunner,
        },
    }
}

func (d *Dispatcher) Run(ctx context.Context, task domain.Task) error {
    r, ok := d.runners[task.Type.String()]
    if !ok {
        return errs.ErrUnknownTaskType
    }
    return r.Run(ctx, task)
}
```

**Strategy 模式**：通过 map 将任务类型映射到对应的 Runner 实现。新增任务类型只需注册新的 Runner，零侵入现有代码。

### 7.3 NormalTaskRunner - 普通任务执行

**源码**: `internal/service/runner/normal_task_runner.go`

#### 7.3.1 Run 方法 - 核心执行流程

```go
func (s *NormalTaskRunner) Run(ctx context.Context, task domain.Task) error {
    // Step 1: 抢占任务
    acquiredTask, err := s.acquireTask(ctx, task)
    if err != nil {
        return err
    }
    acquiredTask.PlanExecID = task.PlanExecID

    // Step 2: 根据分片规则分发
    if acquiredTask.ShardingRule == nil {
        return s.handleNormalTask(ctx, acquiredTask)
    } else {
        return s.handleShardingTask(ctx, acquiredTask)
    }
}
```

**设计决策**：在 Runner 层判断是否分片，而不是在 Dispatcher 中分发到不同 Runner。原因是普通任务和分片任务的区别仅在于"如何创建执行记录"，核心流程（抢占→创建→调用→上报）是一致的。

#### 7.3.2 handleNormalTask - 非分片任务

```go
func (s *NormalTaskRunner) handleNormalTask(ctx context.Context, task domain.Task) error {
    // 创建执行记录
    execution, err := s.execSvc.Create(ctx, domain.TaskExecution{
        Task:      task,
        StartTime: time.Now().UnixMilli(),
        Status:    domain.TaskExecutionStatusPrepare,
    })
    if err != nil {
        s.releaseTask(ctx, task)  // 创建失败 → 释放任务
        return err
    }

    // 异步执行
    go func() {
        state, err1 := s.invoker.Run(ctx, execution)
        if err1 != nil {
            return
        }
        s.execSvc.UpdateState(ctx, state)
    }()
    return nil
}
```

**关键时序**：

```
同步阶段（调度循环内）:
    Acquire → Create Execution → return nil

异步阶段（goroutine）:
    Invoker.Run → 等待执行器返回 → UpdateState
```

**为什么异步执行**：
1. `Invoker.Run` 是 gRPC 阻塞调用，可能持续数分钟
2. 调度循环需要继续处理其他任务
3. 错误处理在异步阶段完成，通过 `UpdateState` 更新状态

**错误恢复机制**：
- goroutine 内 `Invoker.Run` 失败：执行记录保持 PREPARE 状态，不会永远卡住——因为超时后 `InterruptCompensator` 会发现并处理
- `UpdateState` 失败：执行器的状态上报（Report）机制会重新同步状态

#### 7.3.3 handleShardingTask - 分片任务

```go
func (s *NormalTaskRunner) handleShardingTask(ctx context.Context, task domain.Task) error {
    // 创建父任务 + 所有子任务的执行记录
    executions, err := s.execSvc.CreateShardingChildren(ctx, domain.TaskExecution{
        Task:      task,
        StartTime: time.Now().UnixMilli(),
        Status:    domain.TaskExecutionStatusRunning,
    })
    if err != nil {
        s.releaseTask(ctx, task)
        return err
    }

    // 并行执行所有子任务
    for i := range executions {
        execution := executions[i]
        go func() {
            state, err1 := s.invoker.Run(
                s.WithSpecificNodeIDContext(ctx, execution.ExecutorNodeID),
                execution,
            )
            if err1 != nil {
                return
            }
            s.execSvc.UpdateState(ctx, state)
        }()
    }
    return nil
}
```

**与非分片任务的区别**：

| 维度 | 非分片任务 | 分片任务 |
|------|-----------|---------|
| 执行记录 | 1 个 | 1 个父 + N 个子 |
| 初始状态 | PREPARE | 父=RUNNING, 子=PREPARE |
| 执行方式 | 单个 goroutine | N 个 goroutine 并行 |
| 节点选择 | Picker 选择 | 每个子任务可指定节点 |
| 完成判断 | 单个状态 | ShardingCompensator 聚合 |

**`WithSpecificNodeIDContext`**：为每个子任务设置指定的执行节点（WeightedDynamic 策略预分配的节点），通过 context 传递给 gRPC balancer。

#### 7.3.4 Retry 方法

```go
func (s *NormalTaskRunner) Retry(ctx context.Context, execution domain.TaskExecution) error {
    // 分片子任务无需重新抢占（父任务锁仍被持有）
    if !s.isShardedTaskExecution(execution) {
        acquiredTask, err := s.acquireTask(ctx, execution.Task)
        if err != nil {
            return err
        }
        execution.Task = acquiredTask
    }

    go func() {
        // 排除原节点，避免重试到同一个故障节点
        state, err1 := s.invoker.Run(
            s.WithExcludedNodeIDContext(ctx, execution.ExecutorNodeID),
            execution,
        )
        if err1 != nil {
            return
        }
        s.execSvc.UpdateState(ctx, state)
    }()
    return nil
}
```

**`WithExcludedNodeIDContext` 的妙用**：重试时排除上次执行失败的节点。如果节点 A 执行失败，重试时 gRPC balancer 会跳过节点 A，选择其他节点。

**分片子任务的特殊处理**（`isShardedTaskExecution`）：

```go
func (s *NormalTaskRunner) isShardedTaskExecution(execution domain.TaskExecution) bool {
    return execution.ShardingParentID != nil && *execution.ShardingParentID > 0
}
```

分片子任务不需要重新抢占父 Task，因为父 Task 的锁会一直持有到所有子任务完成（由 ShardingCompensator 释放）。

#### 7.3.5 Reschedule 方法

```go
func (s *NormalTaskRunner) Reschedule(ctx context.Context, execution domain.TaskExecution) error {
    if !s.isShardedTaskExecution(execution) {
        acquiredTask, err := s.acquireTask(ctx, execution.Task)
        if err != nil {
            return err
        }
        execution.Task = acquiredTask
    }

    go func() {
        // 使用指定节点（如果有），否则使用默认路由
        state, err1 := s.invoker.Run(
            s.WithSpecificNodeIDContext(ctx, execution.ExecutorNodeID),
            execution,
        )
        // ...
    }()
    return nil
}
```

**Retry vs Reschedule**：

| 维度 | Retry | Reschedule |
|------|-------|-----------|
| Execution | 复用同一个 | 创建新的 |
| 节点选择 | 排除原节点 | 可指定新节点 |
| 触发状态 | FAILED_RETRYABLE | FAILED_RESCHEDULED |
| 次数限制 | RetryConfig.MaxRetries | 无限制 |
| 语义 | "再试一次" | "换个地方试" |

### 7.4 PlanTaskRunner - 计划任务执行

**源码**: `internal/service/runner/plan_task_runner.go`

#### 7.4.1 组合继承设计

```go
type PlanTaskRunner struct {
    planService task.PlanService
    *NormalTaskRunner  // 嵌入，复用 Acquire/Release/Invoke 等方法
}
```

通过 Go 的嵌入组合，`PlanTaskRunner` 复用了 `NormalTaskRunner` 的所有方法，只需要重写 `Run` 和添加 `NextStep`。

#### 7.4.2 Run 方法 - Plan 执行入口

```go
func (p *PlanTaskRunner) Run(ctx context.Context, task domain.Task) error {
    // Step 1: 抢占 Plan（整个工作流只抢占一次）
    ta, err := p.acquireTask(ctx, task)
    if err != nil {
        return err
    }

    // Step 2: 创建 Plan 的执行记录（RUNNING）
    exec, err := p.execSvc.Create(ctx, domain.TaskExecution{
        Task:      ta,
        StartTime: time.Now().UnixMilli(),
        Status:    domain.TaskExecutionStatusRunning,
    })
    if err != nil {
        p.releaseTask(ctx, ta)
        return err
    }

    // Step 3: 获取 Plan DAG
    plan, err := p.planService.GetPlan(ctx, task.ID)
    if err != nil {
        return err
    }

    // Step 4: 启动所有根任务
    rootTasks := plan.RootTask()
    for idx := range rootTasks {
        rootTask := rootTasks[idx]
        rootTask.PlanExecID = exec.ID
        go p.run(ctx, rootTask.Task)  // 并行启动
    }
    return nil
}
```

**Plan 执行模型**：

```
Plan.Run()
    │
    ├── 抢占 Plan Task
    ├── 创建 Plan Execution (RUNNING)
    ├── GetPlan() → 解析 DSL → 构建 DAG
    │
    ├── go run(rootTask1)  ──→ NormalTaskRunner.Run()
    ├── go run(rootTask2)  ──→ NormalTaskRunner.Run()
    └── go run(rootTask3)  ──→ NormalTaskRunner.Run()
           │
           ▼ 任务完成后
        CompleteEvent → Consumer → NextStep()
           │
           ├── 检查后继任务的前驱是否满足
           ├── go run(nextTask1)
           └── go run(nextTask2)
```

#### 7.4.3 NextStep - 工作流推进

```go
func (p *PlanTaskRunner) NextStep(ctx context.Context, task domain.Task) error {
    // 重新获取 Plan（包含最新的执行状态）
    plan, err := p.planService.GetPlan(ctx, task.PlanID)
    if err != nil {
        return err
    }

    // 找到当前完成的任务在 Plan 中的位置
    planTask, ok := plan.GetTask(task.Name)
    if !ok {
        return fmt.Errorf("当前任务%s 不属于plan%s", task.Name, plan.Name)
    }

    // 获取后继任务
    tasks := planTask.NextStep()

    if len(tasks) == 0 {
        // 没有后继 = Plan 执行完毕
        // 发送 Plan 完成事件
        return p.producer.Produce(ctx, event.Event{
            TaskID:  plan.ID,
            Type:    domain.PlanTaskType,
            ExecStatus: planTask.TaskExecution.Status,
        })
    }

    // 检查并启动后继任务
    for idx := range tasks {
        nextPlanTask := tasks[idx]
        nextPlanTask.PlanExecID = plan.Execution.ID
        if nextPlanTask.CheckPre() {
            go p.run(ctx, nextPlanTask.Task)
        }
    }
    return nil
}
```

**`CheckPre()` 的关键作用**：

```
AND 节点: A AND B → C
    A 完成 → NextStep → C.CheckPre() → A✓ B? → false (B未完成)
    B 完成 → NextStep → C.CheckPre() → A✓ B✓ → true → 启动 C

OR 节点: A OR B → C
    A 完成 → NextStep → C.CheckPre() → A✓ → true → 启动 C
    B 完成 → NextStep → C 已在执行 → 无额外操作
```

#### 7.4.4 run 方法 - 带重试的任务启动

```go
func (p *PlanTaskRunner) run(ctx context.Context, task domain.Task) {
    for {
        err := p.NormalTaskRunner.Run(ctx, task)
        if err != nil && !errors.Is(err, errs.ErrTaskPreemptFailed) {
            p.logger.Error("运行任务失败", elog.FieldErr(err))
            time.Sleep(defaultRetrySleepTime)  // 500ms
            continue
        }
        return
    }
}
```

**循环重试的原因**：Plan 中的子任务可能因为 version 冲突（其他节点先抢了）而失败。此时不应该放弃，而是等待短暂时间后重试抢占。只有当 `ErrTaskPreemptFailed`（正常竞争失败）时才退出。

**优先级**：gRPC > HTTP > Local

**为什么 gRPC 优先**：
1. gRPC 支持双向流，可以在执行过程中流式上报进度
2. gRPC 有原生的超时、重试、负载均衡支持
3. 本项目的 gRPC 负载均衡器支持定向路由（routing_round_robin）
4. HTTP 作为兼容方案，用于对接不支持 gRPC 的遗留系统

}

func (g *GRPCInvoker) Run(ctx context.Context, exec domain.TaskExecution) (domain.ExecutionState, error) {
    // 获取 gRPC 客户端（通过自定义 resolver + balancer）
    client := g.clients.Get(exec.Task.GrpcConfig.ServiceName)

    // 构建 gRPC 请求（ctx 携带的路由信息会在 balancer 层被读取）
    resp, err := client.Execute(ctx, &executorv1.ExecuteRequest{
        TaskId:       exec.Task.ID,
        ExecutionId:  exec.ID,
        TaskName:     exec.Task.Name,
        Params:       exec.GRPCParams(),  // 合并后的参数
    })
    if err != nil {
        return domain.ExecutionState{}, fmt.Errorf("gRPC调用执行器失败: %w", err)
    }

    return domain.ExecutionStateFromProto(resp.State), nil
}
```

**`clients.Get(serviceName)` 的路由链**：

```

    │

    │
    ▼
Custom Resolver (etcd → 解析服务实例)
    │
    ▼
routing_round_robin Balancer:
    1. 检查 ctx 中是否有 SpecificNodeID → 路由到指定节点
    2. 检查 ctx 中是否有 ExcludedNodeID → 排除指定节点

### 9.2 UpdateState - 状态机实现

**源码**: `execution_service.go:229-306`

`UpdateState` 是整个系统的**状态转移中枢**，所有执行状态变更都必须经过这个方法：

```go
func (s *executionService) UpdateState(ctx context.Context, state domain.ExecutionState) error {
    // 1. 查询当前执行记录
    execution, err := s.FindByID(ctx, state.ID)
    if err != nil {
        return errs.ErrExecutionNotFound
    }

    // 2. 终态检查：已完成的不允许再变更
    if execution.Status.IsTerminalStatus() {
        return errs.ErrInvalidTaskExecutionStatus
    }

    // 3. 状态分发
    switch {
    case state.Status.IsRunning():
        // RUNNING → 更新进度或设置开始时间
        if execution.Status.IsRunning() {
            return s.updateRunningProgress(ctx, state)
        }
        return s.setRunningState(ctx, state)

    case state.Status.IsFailedRetryable():
        // FAILED_RETRYABLE → 计算重试时间
        return s.updateRetryState(ctx, execution, state)

    case state.Status.IsFailedRescheduled():
        // FAILED_RESCHEDULED → 标记需要重调度
        if state.RequestReschedule {
            execution.MergeTaskScheduleParams(state.RescheduleParams)
        }
        return s.updateState(ctx, execution, state)

    case state.Status.IsTerminalStatus():
        // SUCCESS/FAILED → 更新状态 + 释放任务 + 更新下次执行时间
        s.updateState(ctx, execution, state)
        if !isShardedTask {
            s.releaseTask(ctx, execution.Task)
            s.taskSvc.UpdateNextTime(ctx, execution.Task.ID)
        }
        s.sendCompletedEvent(ctx, state, execution)
        return nil

    default:
        return errs.ErrInvalidTaskExecutionStatus
    }
}
```

### 9.3 重试状态计算

**源码**: `execution_service.go:334-376`

```go
func (s *executionService) updateRetryState(ctx context.Context, execution domain.TaskExecution, state domain.ExecutionState) error {
    // 使用重试策略计算下次重试时间
    retryStrategy, _ := retry.NewRetry(execution.Task.RetryConfig.ToRetryComponentConfig())
    duration, shouldRetry := retryStrategy.NextWithRetries(int32(execution.RetryCount + 1))

    if shouldRetry {
        // 还能重试：计算下次时间
        execution.NextRetryTime = time.Now().Add(duration).UnixMilli()
        execution.RetryCount++
    } else if !state.Status.IsTerminalStatus() {
        // 不能重试了：强制设为 FAILED
        state.Status = domain.TaskExecutionStatusFailed
    }

    // 更新数据库
    err := s.UpdateRetryResult(ctx, state.ID, execution.RetryCount, execution.NextRetryTime, ...)
    if !shouldRetry {
        return errs.ErrExecutionMaxRetriesExceeded
    }
    return nil
}
```

**重试时间计算示例（指数退避）**：

```
RetryConfig: maxRetries=3, initialInterval=1s, maxInterval=30s

第1次失败: retryCount=1, nextRetryTime=now+1s
第2次失败: retryCount=2, nextRetryTime=now+2s
第3次失败: retryCount=3, nextRetryTime=now+4s
第4次失败: shouldRetry=false → 强制 FAILED
```

### 9.4 CreateShardingChildren - 分片任务创建

**源码**: `execution_service.go:98-149`

这是分片任务最核心的创建流程：

```go
func (s *executionService) CreateShardingChildren(ctx context.Context, parent domain.TaskExecution) ([]domain.TaskExecution, error) {
    shardingRule := parent.Task.ShardingRule

    var executorNodeIDs []string

    // WeightedDynamic 需要额外的准备工作
    if shardingRule.Type.IsWeightedDynamicRange() {
        // 1. 调用执行器的 Prepare 接口获取动态参数
        params, err := s.invoker.Prepare(ctx, parent)
        maps.Copy(shardingRule.Params, params)

        // 2. 从 etcd 获取执行节点及权重
        serviceInstances, err := s.registry.ListServices(ctx, parent.Task.GrpcConfig.ServiceName)
        shardingRule.ExecutorNodeInstances = serviceInstances

        // 3. 按顺序提取节点 ID
        executorNodeIDs = slice.Map(serviceInstances, func(_, src) string {
            return src.ID
        })
    }

    // 4. 计算分片参数
    scheduleParams := shardingRule.ToScheduleParams()

    // 5. 创建父执行记录
    parent.Status = domain.TaskExecutionStatusRunning
    created, err := s.repo.CreateShardingParent(ctx, parent)

    // 6. 创建子执行记录
    return s.repo.CreateShardingChildren(ctx, created, executorNodeIDs, scheduleParams)
}
```

**WeightedDynamic 的完整流程**：

```
1. Prepare RPC → 执行器返回 {total_tasks: "50000"}
2. ListServices → etcd 返回 [{id: "exec-1", weight: 3}, {id: "exec-2", weight: 7}]
3. ToScheduleParams → [{start:0, end:15000}, {start:15000, end:50000}]
4. CreateShardingParent → 父记录 (id=100, status=RUNNING)
5. CreateShardingChildren → [
     {id=101, parentID=100, nodeID="exec-1", params={start:0, end:15000}},
     {id=102, parentID=100, nodeID="exec-2", params={start:15000, end:50000}}
   ]
```

### 9.5 HandleReports - 批量上报处理

```go
func (s *executionService) HandleReports(ctx context.Context, reports []*domain.Report) error {
    var err error
    processedCount := 0
    skippedCount := 0

    for i := range reports {
        err1 := s.UpdateState(ctx, reports[i].ExecutionState)
        if err1 != nil {
            skippedCount++
            err = multierr.Append(err, err1)
            continue
        }
        processedCount++
    }
    return err
}
```

**使用 `multierr.Append`**：不因单个 Report 失败而中断其他 Report 的处理，最终返回所有错误的聚合。
