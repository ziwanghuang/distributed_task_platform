## 第10章 补偿引擎总体设计

### 10.1 补偿引擎架构

补偿引擎是系统**可靠性的核心保障**，通过四个独立的补偿器覆盖所有异常场景。每个补偿器都是一个独立的后台循环任务：

```
┌─────────────────────────────────────────────────────────────┐
│                     补偿引擎 (Compensator)                   │
│                                                              │
│  ┌──────────────────┐      ┌──────────────────────┐         │
│  │ RetryCompensator │      │ RescheduleCompensator│         │
│  │ 重试补偿器        │      │ 重调度补偿器           │         │
│  │                  │      │                      │         │
│  │ 触发：           │      │ 触发：                │         │
│  │ FAILED_RETRYABLE │      │ FAILED_RESCHEDULED   │         │
│  │ + 到达重试时间    │      │                      │         │
│  │                  │      │ 动作：                │         │
│  │ 动作：           │      │ Runner.Reschedule()  │         │
│  │ Runner.Retry()   │      │ (排除原节点)          │         │
│  └──────────────────┘      └──────────────────────┘         │
│                                                              │
│  ┌──────────────────┐      ┌──────────────────────┐         │
│  │ShardingCompensator│     │InterruptCompensator  │         │
│  │ 分片聚合补偿器     │     │ 超时中断补偿器        │         │
│  │                  │      │                      │         │
│  │ 触发：           │      │ 触发：                │         │
│  │ 分片子任务全完成  │      │ RUNNING + 超过deadline│         │
│  │                  │      │                      │         │
│  │ 动作：           │      │ 动作：                │         │
│  │ 聚合子任务状态    │      │ gRPC Interrupt()     │         │
│  │ → 更新父任务     │      │ → 强制中断执行        │         │
│  │ → 释放Task锁     │      │                      │         │
│  └──────────────────┘      └──────────────────────┘         │
└─────────────────────────────────────────────────────────────┘
```

### 10.2 补偿器通用模式

所有补偿器遵循相同的模式——"无限循环 + 防空转"：

```go
func (r *XxxCompensator) Start(ctx context.Context) {
    for {
        select {
        case <-ctx.Done():
            return
        default:
            startTime := time.Now()

            err := r.doCompensation(ctx)
            if err != nil {
                r.logger.Error("补偿失败", elog.FieldErr(err))
            }

            // 防空转：确保最小等待时间
            elapsed := time.Since(startTime)
            if elapsed < r.config.MinDuration {
                select {
                case <-ctx.Done():
                    return
                case <-time.After(r.config.MinDuration - elapsed):
                }
            }
        }
    }
}
```

**防空转机制**：如果一轮补偿耗时不足 `MinDuration`（默认 1s），则等待到 `MinDuration` 再开始下一轮。这避免了"查询→无结果→立即查询→无结果"的空转 CPU 浪费。

### 10.3 V1 vs V2 补偿器

| 版本 | 循环方式 | 分布式协调 | 文件 |
|------|---------|-----------|------|
| V1 | 简单 for 循环 | 无 | `retry.go`, `reschedule.go`, `sharding.go`, `interrupt.go` |
| V2 | ShardingLoopJob | 分布式锁 + 分片 | `retryv2.go`, `reschedulev2.go`, `shardingv2.go`, `interruptv2.go` |

V2 版本将循环逻辑委托给 `ShardingLoopJob`，实现了：
- 多调度节点间的补偿任务分片
- 分布式锁保证同一批记录不被重复补偿
- 资源信号量控制并发度

## 第11章 四种补偿器详解

### 11.1 RetryCompensator - 重试补偿器

**源码**: `internal/compensator/retry.go`

**职责**：发现 `FAILED_RETRYABLE` 状态且到达重试时间的执行记录，触发重试。

```go
type RetryCompensator struct {
    runner  runner.Runner
    execSvc task.ExecutionService
    config  RetryConfig
    logger  *elog.Component
}

type RetryConfig struct {
    MaxRetryCount          int64         // 最大重试次数
    PrepareTimeoutWindowMs int64         // PREPARE 超时窗口
    BatchSize              int           // 批量大小
    MinDuration            time.Duration // 防空转间隔
}
```

**核心逻辑（`retry.go:74-104`）**：

```go
func (r *RetryCompensator) retry(ctx context.Context) error {
    // 1. 查找可重试的执行记录
    executions, err := r.execSvc.FindRetryableExecutions(ctx, r.config.BatchSize)
    if err != nil {
        return fmt.Errorf("查找可重试任务失败: %w", err)
    }
    if len(executions) == 0 {
        return nil
    }

    // 2. 逐个触发重试
    for i := range executions {
        err = r.runner.Retry(ctx, executions[i])
        if err != nil {
            r.logger.Error("重试任务失败", ...)
            continue  // 单个失败不影响其他
        }
    }
    return nil
}
```

**`FindRetryableExecutions` 的 SQL 条件**：

```sql
SELECT * FROM task_executions
WHERE status = 'FAILED_RETRYABLE'
  AND next_retry_time <= NOW_MS
  AND retry_count < max_retry_count
ORDER BY next_retry_time ASC
LIMIT batchSize
```

**重试流程图**：

```
任务执行失败 (FAILED_RETRYABLE)
    │
    ▼
ExecutionService.UpdateState()
    │
    ├── 计算 nextRetryTime (指数退避)
    ├── 递增 retryCount
    └── 更新数据库
    │
    ▼ (等待 nextRetryTime)
    │
RetryCompensator.retry()
    │
    ├── FindRetryableExecutions (next_retry_time <= now)
    └── Runner.Retry(execution)
           │
           ├── Acquire (非分片任务需要重新抢占)
           └── Invoker.Run (排除原节点)
                  │
                  ▼
           UpdateState (新的执行结果)
                  │
                  ├── 成功 → SUCCESS
                  ├── 可重试失败 → FAILED_RETRYABLE → 回到开头
                  └── 不可重试失败 → FAILED
```

### 11.2 RescheduleCompensator - 重调度补偿器

**源码**: `internal/compensator/reschedule.go`

**职责**：发现 `FAILED_RESCHEDULED` 状态的执行记录，触发重调度。

与重试的核心区别：
1. **创建新 Execution**：重调度会创建新的执行记录，而不是复用原来的
2. **指定/排除节点**：可以指定新的执行节点或排除故障节点
3. **无次数限制**：重调度没有固定的最大次数限制
4. **执行器主动请求**：通常由执行器通过 `RequestReschedule=true` 发起

```go
func (r *RescheduleCompensator) reschedule(ctx context.Context) error {
    executions, err := r.execSvc.FindReschedulableExecutions(ctx, r.config.BatchSize)
    if err != nil {
        return fmt.Errorf("查找可重调度任务失败: %w", err)
    }
    if len(executions) == 0 {
        return nil
    }
    for i := range executions {
        err = r.runner.Reschedule(ctx, executions[i])
        if err != nil {
            r.logger.Error("重调度任务失败", ...)
            continue
        }
    }
    return nil
}
```

**重调度 vs 重试的场景选择**：

| 场景 | 选择 | 原因 |
|------|------|------|
| 代码 bug 导致的异常 | 重试 | 可能是瞬时问题 |
| 网络超时 | 重试 | 换个时间可能恢复 |
| 节点资源不足 | 重调度 | 需要换节点 |
| 数据分布不均 | 重调度 | 换节点可能有不同的数据分布 |
| 执行器OOM | 重调度 | 该节点不适合此任务 |

### 11.3 ShardingCompensator - 分片聚合补偿器

**源码**: `internal/compensator/sharding.go`

**职责**：监控分片父任务的所有子任务状态，当所有子任务完成后，聚合结果并更新父任务状态。

这是四个补偿器中**最复杂的一个**，因为它需要处理多个子任务的聚合逻辑。

#### 11.3.1 分页遍历机制

```go
func (r *ShardingCompensator) Start(ctx context.Context) {
    offset := 0
    for {
        select {
        case <-ctx.Done():
            return
        default:
        }

        // 分页查找分片父任务
        executions, err := r.execSvc.FindShardingParents(ctx, offset, r.config.BatchSize)
        if err != nil {
            offset += r.config.BatchSize  // 跳过当前批次
            continue
        }

        if len(executions) == 0 {
            time.Sleep(r.config.MinDuration)
            offset = 0  // 重置偏移，从头开始
            continue
        }

        for i := range executions {
            r.handle(ctx, executions[i])
        }
        offset += len(executions)  // 移动到下一批
    }
}
```

**分页遍历而非一次性加载**：分片父任务可能很多，使用 `offset + batchSize` 分页遍历，每次只处理一批。到达末尾（`len(executions) == 0`）后重置 `offset` 从头开始。

#### 11.3.2 聚合逻辑

```go
func (r *ShardingCompensator) handle(ctx context.Context, parent domain.TaskExecution) error {
    // 1. 查找所有子任务
    children, err := r.execSvc.FindShardingChildren(ctx, parent.ID)
    if err != nil {
        return err
    }

    // 2. 边界处理：无子任务 = 创建异常
    if len(children) == 0 {
        defer r.releaseTask(ctx, parent.Task)
        r.execSvc.UpdateScheduleResult(ctx, parent.ID, domain.TaskExecutionStatusFailed, ...)
        r.taskSvc.UpdateNextTime(ctx, parent.Task.ID)
        return nil
    }

    // 3. 检查子任务是否全部完成
    anyFailed := false
    successCount := 0
    for i := range children {
        if !children[i].Status.IsSuccess() && !children[i].Status.IsFailed() {
            // 有子任务还在运行 → 等待下次补偿
            return nil
        }
        if children[i].Status.IsFailed() {
            anyFailed = true
        }
        if children[i].Status.IsSuccess() {
            successCount++
        }
    }

    // 4. 所有子任务都完成了 → 释放任务锁
    defer r.releaseTask(ctx, parent.Task)

    // 5. 聚合状态
    var finalStatus domain.TaskExecutionStatus
    if anyFailed {
        finalStatus = domain.TaskExecutionStatusFailed
    } else {
        finalStatus = domain.TaskExecutionStatusSuccess
    }

    // 6. 计算总进度
    progress := int32((successCount * 100) / len(children))

    // 7. 更新父任务状态 + 下次执行时间
    r.execSvc.UpdateScheduleResult(ctx, parent.ID, finalStatus, progress, ...)
    r.taskSvc.UpdateNextTime(ctx, parent.Task.ID)
    return nil
}
```

**聚合规则**：
- **全部成功** → 父任务 SUCCESS
- **任一失败** → 父任务 FAILED（保守策略）
- **有未完成** → 跳过，等下次补偿

**进度计算**：`progress = successCount * 100 / totalChildren`

```
5个子任务：3成功2失败 → progress = 60, status = FAILED
5个子任务：5成功0失败 → progress = 100, status = SUCCESS
5个子任务：3成功1失败1运行中 → 跳过（不聚合）
```

**`defer r.releaseTask`**：只有在聚合完成后才释放 Task 锁。这意味着在所有子任务完成之前，父 Task 一直被持有，不会被重新调度。

### 11.4 InterruptCompensator - 超时中断补偿器

**源码**: `internal/compensator/interrupt.go`

**职责**：发现超过 deadline 仍在 RUNNING 的执行记录，通过 gRPC 发送中断命令。

```go
type InterruptCompensator struct {
    execSvc     task.ExecutionService
    config      InterruptConfig
    logger      *elog.Component
    grpcClients *grpc.ClientsV2[executorv1.ExecutorServiceClient]
}
```

#### 11.4.1 中断流程

```go
func (t *InterruptCompensator) interruptTaskExecution(ctx context.Context, execution domain.TaskExecution) error {
    // 1. 检查 gRPC 配置
    if execution.Task.GrpcConfig == nil {
        return fmt.Errorf("未找到GRPC配置，无法执行中断任务")
    }

    // 2. 获取 gRPC 客户端
    client := t.grpcClients.Get(execution.Task.GrpcConfig.ServiceName)

    // 3. 发送中断请求
    resp, err := client.Interrupt(ctx, &executorv1.InterruptRequest{
        Eid: execution.ID,
    })
    if err != nil {
        return fmt.Errorf("发送中断请求失败：%w", err)
    }

    // 4. 检查中断结果
    if !resp.GetSuccess() {
        return errs.ErrInterruptTaskExecutionFailed
    }

    // 5. 更新执行状态
    return t.execSvc.UpdateState(ctx, domain.ExecutionStateFromProto(resp.GetExecutionState()))
}
```

**中断时序图**：

```
InterruptCompensator         执行器 (Executor)
       │                           │
       │ FindTimeoutExecutions()   │
       │ (deadline < now &&        │
       │  status = RUNNING)        │
       │                           │
       │── gRPC Interrupt(eid) ───→│
       │                           │── 停止任务执行
       │                           │── 清理资源
       │←─ InterruptResponse ──────│
       │   {success: true,         │
       │    state: {FAILED, ...}}  │
       │                           │
       │ UpdateState(FAILED)       │
       │                           │
```

**超时检测 SQL**：

```sql
SELECT * FROM task_executions
WHERE status = 'RUNNING'
  AND deadline < NOW_MS
LIMIT batchSize
```

**`deadline` 的计算**：在创建 TaskExecution 时，`deadline = startTime + task.MaxExecutionSeconds * 1000`。默认 `MaxExecutionSeconds = 86400`（24小时）。

### 11.5 补偿器协作关系

```
任务执行完成
    │
    ├── SUCCESS → 完成事件 → NextStep (Plan)
    │
    ├── FAILED → 终态，不再补偿
    │
    ├── FAILED_RETRYABLE ──→ RetryCompensator ──→ Runner.Retry()
    │                                              │
    │                                    ┌─────────┤
    │                                    │         │
    │                            成功 ←──┘    再次失败
    │                                         │
    │                              超过最大次数?
    │                              ├── 否 → FAILED_RETRYABLE (循环)
    │                              └── 是 → FAILED
    │
    ├── FAILED_RESCHEDULED ─→ RescheduleCompensator ─→ Runner.Reschedule()
    │
    └── RUNNING (超时) ─────→ InterruptCompensator ─→ gRPC Interrupt
                                                       │
                                                       ▼
                                                    FAILED
```

**分片任务的特殊流程**：

```
分片父任务 (RUNNING)
    │
    ├── 子任务1: SUCCESS
    ├── 子任务2: FAILED_RETRYABLE → RetryCompensator → ...
    ├── 子任务3: RUNNING → 等待
    │
    ▼ (所有子任务都到达终态后)
    │
ShardingCompensator
    │
    ├── 聚合状态 → SUCCESS/FAILED
    ├── 释放父 Task 锁
    └── 更新下次执行时间
```

## 第12章 DSL 工作流引擎

### 12.1 DSL 语法设计

**源码**: `internal/dsl/parser/`

系统使用 ANTLR4 定义了一套任务编排 DSL，支持以下操作符：

| 操作符 | 语法 | 含义 | 示例 |
|--------|------|------|------|
| 序列 | `→` 或 `->` | 顺序执行 | `A -> B -> C` |
| AND | `&` | 全部成功后继续 | `A & B -> C` |
| OR | `\|` | 任一成功后继续 | `A \| B -> C` |
| 条件 | `?` | 根据结果分支 | `A ? B : C` |
| 分组 | `{}` | 汇聚组 | `{A,B,C} -> D` |

### 12.2 ANTLR4 解析流程

**源码**: `internal/dsl/parser/plan.go`

```go
type AstPlan struct{}

func (a *AstPlan) Parse(expression string) (TaskPlan, error) {
    // 1. 词法分析
    input := antlr.NewInputStream(expression)
    lexer := parser.NewTaskOrchestrationParserLexer(input)

    // 2. 语法分析
    stream := antlr.NewCommonTokenStream(lexer, antlr.TokenDefaultChannel)
    p := parser.NewTaskOrchestrationParserParser(stream)

    // 3. AST 遍历
    visitor := NewTaskOrchestrationVisitor()
    result := visitor.Visit(p.Program())

    // 4. 返回执行计划
    planRes := result.(*planRes)
    if planRes.err != nil {
        return nil, planRes.err
    }
    return &astTaskPlan{
        root:  planRes.root,
        end:   planRes.end,
        tasks: planRes.tasks,
    }, nil
}
```

**解析流水线**：

```
DSL 文本: "A -> B & C -> D"
    │
    ▼ ANTLR4 Lexer
Token 流: [TASK_NAME("A"), ARROW, TASK_NAME("B"), AND, TASK_NAME("C"), ARROW, TASK_NAME("D")]
    │
    ▼ ANTLR4 Parser
AST:
    Program
    └── Expression
        └── SequenceExpression
            ├── BasicExpression (A)
            ├── BasicExpression (AndExpression: B & C)
            └── BasicExpression (D)
    │
    ▼ TaskOrchestrationVisitor
PlanNode 列表:
    [
      {Node: SimpleNode("A")},
      {Node: AndNode(["B", "C"])},
      {Node: SimpleNode("D")}
    ]
    │
    ▼ mergePlan()
DAG:
    A ──→ B ──→ D
         │
    A ──→ C ──→ D
    (B & C 必须都成功，D 才会执行)
```

### 12.3 TaskOrchestrationVisitor 核心逻辑

**源码**: `internal/dsl/parser/visitor.go`

Visitor 是 AST 遍历的核心，每个 `Visit*` 方法处理一种语法结构。

#### 12.3.1 VisitSequenceExpression - 序列表达式

```go
func (v *TaskOrchestrationVisitor) VisitSequenceExpression(ctx *parser.SequenceExpressionContext) interface{} {
    expressions := ctx.AllBasicExpression()
    var pre *PlanNode
    plan := make([]*PlanNode, 0, len(expressions))

    for idx := range expressions {
        expression := expressions[idx]
        ans := v.VisitBasicExpression(expression.(*parser.BasicExpressionContext))

        switch ta := ans.(type) {
        case *conditionTask:
            // 条件分支：Pre ? Success : Failure
            prePlanTask := &PlanNode{Node: ta.Pre, Next: ta.Next}
            conTask := &PlanNode{Node: ta.Next, Pre: ta.Pre}
            if pre != nil {
                prePlanTask.Pre = pre.Node
                pre.Next = prePlanTask
            }
            pre = conTask
            plan = append(plan, prePlanTask, conTask)

        case Node:
            // 普通节点（Simple/And/Or）
            t := &PlanNode{Node: ta}
            plan = append(plan, t)
            if pre != nil {
                t.Pre = pre.Node   // 设置前驱
                pre.Next = t.Node  // 设置后继
            }
            pre = t
        }
    }
    return plan
}
```

**序列表达式的链表构建**：遍历所有基础表达式，通过 `pre` 指针将它们链接成链表。

#### 12.3.2 VisitConditionalExpression - 条件分支

```go
func (v *TaskOrchestrationVisitor) VisitConditionalExpression(ctx *parser.ConditionalExpressionContext) any {
    if ctx.QUESTION() != nil && len(ctx.AllTASK_NAME()) > 0 {
        taskPre := NewSimpleTask(ctx.TASK_NAME(0).GetText())
        successTask := NewSimpleTask(ctx.TASK_NAME(1).GetText())
        failureTask := NewSimpleTask(ctx.TASK_NAME(2).GetText())
        return &conditionTask{
            Pre:  taskPre,
            Next: NewConditionTask(successTask, failureTask),
        }
    }
    return nil
}
```

**条件分支语义**：`A ? B : C` 表示如果 A 成功则执行 B，失败则执行 C。

#### 12.3.3 mergePlan - 全局 DAG 构建

```go
func (v *TaskOrchestrationVisitor) mergePlan(plans [][]*PlanNode) *planRes {
    // 1. 收集所有任务名
    taskMap := v.getAllNode(plans)

    // 2. 构建 AST 节点的 TreeMap
    treeMap, err := v.setAstNode(plans)

    // 3. 为每个任务设置前驱和后继
    roots := make([]PlanNode, 0)
    end := make([]PlanNode, 0)

    for name, task := range taskMap {
        prenames, nextnames := v.getPreAndNext(plans, name)
        pre, _ := v.getPreAndNextTask(treeMap, prenames)
        next, _ := v.getPreAndNextTask(treeMap, nextnames)

        task.Pre = pre
        task.Next = next
        taskMap[name] = task

        if len(prenames) == 0 {
            roots = append(roots, task)  // 无前驱 = 根节点
        }
        if len(nextnames) == 0 {
            end = append(end, task)      // 无后继 = 终止节点
        }
    }

    // 4. 设置终止节点
    v.setEndNode(taskMap, end)

    return &planRes{root: roots, end: end[0], tasks: ans}
}
```

**DAG 构建的核心步骤**：
1. 从多条表达式链中提取所有唯一任务名
2. 为每个任务找到它在所有链中的前驱和后继
3. 如果多个前驱/后继，使用 And/Or 节点包装
4. 无前驱的任务 = 根节点（DAG 入口）
5. 无后继的任务 = 终止节点（替换为 EndNode）

### 12.4 Node 类型系统

**源码**: `internal/dsl/parser/node.go`, `internal/dsl/parser/type.go`

```go
type Node interface {
    ChildNodes() []string          // 包含的任务名列表
    NextNodes(exec Execution) []string  // 根据执行结果确定下一步
    Type() NodeType                // 节点类型
}
```

#### 12.4.1 SimpleNode

```go
type SimpleNode struct {
    name string
}

func (s SimpleNode) ChildNodes() []string {
    return []string{s.name}
}

func (s SimpleNode) NextNodes(_ Execution) []string {
    return []string{s.name}
}
```

最基本的节点，代表单个任务。

#### 12.4.2 AndNode

```go
type AndNode struct {
    nodes []SimpleNode
}

func (a AndNode) ChildNodes() []string {
    // 返回所有子节点的名称
}

func (a AndNode) NextNodes(_ Execution) []string {
    // 返回所有子节点的名称（全部需要执行）
}
```

AND 语义：所有子节点都需要成功，后续才能执行。`CheckPre` 时使用 `preAllSuccess()`。

#### 12.4.3 OrNode

```go
type OrNode struct {
    nodes []SimpleNode
}

func (o OrNode) NextNodes(_ Execution) []string {
    // 返回所有子节点的名称（任一成功即可）
}
```

OR 语义：任一子节点成功，后续即可执行。`CheckPre` 时使用 `preOneSuccess()`。

#### 12.4.4 ConditionNode

```go
type ConditionNode struct {
    Success SimpleNode
    Failure SimpleNode
}

func (c ConditionNode) NextNodes(exec Execution) []string {
    if exec.Status == TaskExecutionStatusSuccess {
        return []string{c.Success.name}  // 成功走 Success 分支
    }
    return []string{c.Failure.name}      // 失败走 Failure 分支
}
```

条件节点是**唯一需要 Execution 参数的节点类型**——它根据前驱任务的执行结果决定走哪个分支。

#### 12.4.5 EndNode

```go
type EndNode struct {
    name string
}

func (e EndNode) NextNodes(_ Execution) []string {
    return nil  // 终止节点没有后继
}
```

### 12.5 DSL 示例解析

**示例1：简单序列**

```
A -> B -> C
```

```
DAG: A ──→ B ──→ C(EndNode)
Root: [A]
```

**示例2：并行分支**

```
A & B -> C
```

```
DAG:
  A ──┐
      ├──→ C(EndNode)  [CheckPre: preAllSuccess]
  B ──┘
Root: [A, B]
```

**示例3：条件分支**

```
A ? B : C
```

```
DAG:
       ┌── B(EndNode)  [A 成功时]
  A ──→│
       └── C(EndNode)  [A 失败时]
Root: [A]
```

**示例4：复杂工作流**

```
A -> B & C -> D -> E ? F : G
```

```
DAG:
  A ──→ B ──┐
             ├──→ D ──→ E ──→ F(EndNode)  [E成功]
  A ──→ C ──┘              ──→ G(EndNode)  [E失败]
Root: [A]
```

### 12.6 PlanService - Plan 组装服务

**源码**: `internal/service/task/plan.go`

```go
type PlanService interface {
    GetPlan(ctx context.Context, planID int64) (*domain.Plan, error)
}

func (s *planService) GetPlan(ctx context.Context, planID int64) (*domain.Plan, error) {
    // 1. 获取 Plan 的 Task 定义
    planTask, err := s.taskRepo.GetByID(ctx, planID)

    // 2. 获取 Plan 中所有子任务
    tasks, err := s.taskRepo.FindByPlanID(ctx, planID)

    // 3. 并发获取每个任务的最新执行状态
    executions := make(map[int64]domain.TaskExecution)
    g, gCtx := errgroup.WithContext(ctx)
    for _, t := range tasks {
        task := t
        g.Go(func() error {
            exec, err := s.execRepo.FindExecutionByTaskIDAndPlanExecID(gCtx, task.ID, planExecID)
            // ...
            executions[task.ID] = exec
            return nil
        })
    }
    g.Wait()

    // 4. 解析 DSL → TaskPlan
    execPlan, err := s.parser.Parse(planTask.ExecExpr)

    // 5. 组装 Plan 对象
    return s.taskToPlan(planTask, tasks, executions, execPlan)
}
```

**`taskToPlan` 的核心**：将 TaskPlan (AST) + 任务列表 + 执行状态组装为完整的 Plan 对象。

```go
func (s *planService) taskToPlan(...) (*domain.Plan, error) {
    // 构建 taskName → PlanTask 的映射
    taskMap := make(map[string]*domain.PlanTask)
    for _, t := range tasks {
        pt := &domain.PlanTask{
            Task:          t,
            TaskExecution: executions[t.ID],
            AstPlanNode:   execPlan.GetPlanNode(t.Name),
        }
        taskMap[t.Name] = pt
    }

    // 设置前驱和后继关系
    for _, pt := range taskMap {
        pt.SetPre(taskMap)   // 根据 AST 设置前驱
        pt.SetNext(taskMap)  // 根据 AST + 执行状态设置后继
    }

    // 找出根节点
    roots := execPlan.RootPlanNodes()
    // ...
}
```

### 13.3 完成事件消费者

**源码**: `internal/event/complete/consumer.go`

```go
type Consumer struct {
    planRunner  *runner.PlanTaskRunner
    execSvc     task.ExecutionService
    taskSvc     task.Service
    acquirer    acquirer.TaskAcquirer
}

func (c *Consumer) Consume(ctx context.Context, event event.Event) error {
    switch event.Type {
    case domain.NormalTaskType:
        // 普通任务完成 → 如果属于 Plan，推进下一步
        if event.PlanID > 0 {
            task := domain.Task{
                ID:     event.TaskID,
                Name:   event.Name,
                PlanID: event.PlanID,
            }
            return c.planRunner.NextStep(ctx, task)
        }
        return nil

    case domain.PlanTaskType:
        // Plan 完成 → 更新 Plan 执行状态 + 释放
        exec, _ := c.execSvc.FindByID(ctx, event.ExecID)
        if event.ExecStatus.IsSuccess() {
            c.execSvc.UpdateScheduleResult(ctx, exec.ID, domain.TaskExecutionStatusSuccess, ...)
        } else {
            c.execSvc.UpdateScheduleResult(ctx, exec.ID, domain.TaskExecutionStatusFailed, ...)
        }
        // 释放 Plan 任务
        c.acquirer.Release(ctx, event.TaskID, event.ScheduleNodeID)
        // 更新下次执行时间
        c.taskSvc.UpdateNextTime(ctx, event.TaskID)
        return nil
    }
    return nil
}
```

**事件驱动的完整流程**：

```
1. 任务 B 执行完成 (SUCCESS)
2. ExecutionService.UpdateState → sendCompletedEvent
3. CompleteProducer → Kafka (complete_topic)
4. CompleteConsumer.Consume:
   a. event.Type == NormalTaskType && event.PlanID > 0
   b. PlanTaskRunner.NextStep(task_B)
   c. 获取 Plan DAG
   d. 找到 B 的后继任务 C
   e. C.CheckPre() → 检查所有前驱
   f. 如果满足 → go run(C)
```

### 13.4 执行上报事件

**源码**: `internal/event/reportevt/`

除了完成事件，系统还有执行上报事件用于异步处理 Report：

```
执行器上报进度
    │
    ▼
gRPC ReporterServer.Report()
    │
    ▼
ReportEventProducer → Kafka (execution_report)
    │
    ▼
ReportEventConsumer → ExecutionService.UpdateState()
```

**为什么要异步化 Report**：
1. 执行器可能高频上报进度（如每秒更新百分比）
2. 同步处理会阻塞 gRPC 线程
3. Kafka 提供了天然的削峰填谷能力
4. 消费者可以批量处理 Report，减少数据库操作

### 13.5 gRPC ReporterServer

**源码**: `internal/grpc/server.go`

```go
type ReporterServer struct {
    executorv1.UnimplementedReporterServiceServer
    producer event.ReportEventProducer
}

func (s *ReporterServer) Report(ctx context.Context, req *executorv1.ReportRequest) (*executorv1.ReportResponse, error) {
    state := domain.ExecutionStateFromProto(req.GetExecutionState())
    report := &domain.Report{ExecutionState: state}
    err := s.producer.Produce(ctx, report)
    return &executorv1.ReportResponse{Success: err == nil}, nil
}

func (s *ReporterServer) BatchReport(ctx context.Context, req *executorv1.BatchReportRequest) (*executorv1.BatchReportResponse, error) {
    reports := make([]*domain.Report, 0, len(req.GetReports()))
    for _, r := range req.GetReports() {
        reports = append(reports, &domain.Report{
            ExecutionState: domain.ExecutionStateFromProto(r.GetExecutionState()),
        })
    }
    err := s.producer.BatchProduce(ctx, reports)
    return &executorv1.BatchReportResponse{Success: err == nil}, nil
}
```

## 第14章 分片任务全流程

### 14.1 分片任务生命周期

```
1. 调度器发现分片任务
   FindSchedulableTasks → Task{ShardingRule != nil}
       │
2. Runner 抢占 + 分发
   NormalTaskRunner.Run → handleShardingTask
       │
3. 创建分片执行记录
   ExecutionService.CreateShardingChildren
   ├── WeightedDynamic: Prepare RPC + ListServices + 权重计算
   ├── Range: 固定步长计算
   ├── CreateShardingParent (status=RUNNING)
   └── CreateShardingChildren (status=PREPARE × N)
       │
4. 并行执行子任务
   for each child → go Invoker.Run(WithSpecificNodeIDContext)
       │
5. 执行器处理 + 上报
   Executor.Execute → Report/BatchReport → UpdateState
       │
6. 子任务完成（可能经过重试/重调度）
   SUCCESS / FAILED
       │
7. ShardingCompensator 聚合
   FindShardingParents → FindShardingChildren
   ├── 全部终态? → 聚合状态
   ├── 更新父任务状态
   ├── 释放 Task 锁
   └── 更新下次执行时间
```

### 14.2 Range vs WeightedDynamic 对比

| 维度 | Range | WeightedDynamic |
|------|-------|-----------------|
| 参数来源 | 任务定义时确定 | 运行时动态获取 |
| 分片数量 | `totalNums` | 执行节点数量 |
| 分片大小 | 固定 `step` | 按权重比例 |
| 节点分配 | 不预分配 | 按 etcd 权重预分配 |
| 适用场景 | 数据量已知且均匀 | 节点异构、数据量动态 |
| Prepare RPC | 不需要 | 需要（获取 total_tasks） |
| ListServices | 不需要 | 需要（获取节点权重） |

### 14.3 分片参数流转

```
Task.ShardingRule
  │ type: "weighted-dynamic-range"
  │ params: {}
  │
  ▼ Prepare RPC → {total_tasks: "100000"}
  │
  ▼ ListServices → [{id:"exec-1", weight:3}, {id:"exec-2", weight:7}]
  │
  ▼ ToScheduleParams()
  │   exec-1: {start:"0", end:"30000"}
  │   exec-2: {start:"30000", end:"100000"}
  │
  ▼ CreateShardingChildren
  │   child-1: {executorNodeID:"exec-1", scheduleParams:{start:"0", end:"30000"}}
  │   child-2: {executorNodeID:"exec-2", scheduleParams:{start:"30000", end:"100000"}}
  │
  ▼ Invoker.Run
  │   child-1 → exec-1: GRPCParams = {start:"0", end:"30000", max_execution_seconds:"86400"}
  │   child-2 → exec-2: GRPCParams = {start:"30000", end:"100000", max_execution_seconds:"86400"}
```
