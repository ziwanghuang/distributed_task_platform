## 第二十二章 Invoker 调用层与多协议支持

### 22.1 Invoker 接口设计

```go
// internal/service/invoker/types.go

type Invoker interface {
    Name() string
    // Run 执行任务，返回执行结果
    Run(ctx context.Context, execution domain.TaskExecution) (domain.ExecutionState, error)
    // Prepare 返回业务总数量（用于分片任务的准备阶段）
    Prepare(ctx context.Context, execution domain.TaskExecution) (map[string]string, error)
}
```

Invoker 是任务执行的最后一公里——它负责将调度平台的执行请求翻译成具体的远程调用或本地函数调用。

### 22.2 三种调用器实现

#### 22.2.1 GRPCInvoker —— 生产首选

```go
// internal/service/invoker/grpc.go

type GRPCInvoker struct {
    grpcClients *grpc.ClientsV2[executorv1.ExecutorServiceClient]
    logger      *elog.Component
}

func (r *GRPCInvoker) Run(ctx context.Context, exec domain.TaskExecution) (domain.ExecutionState, error) {
    // 通过泛型客户端池获取 gRPC 客户端
    client := r.grpcClients.Get(exec.Task.GrpcConfig.ServiceName)

    // 发送执行请求（ctx 中携带了路由信息 → Picker 会读取）
    resp, err := client.Execute(ctx, &executorv1.ExecuteRequest{
        Eid:      exec.ID,
        TaskId:   exec.Task.ID,
        TaskName: exec.Task.Name,
        Params:   exec.GRPCParams(),
    })
    if err != nil {
        return domain.ExecutionState{}, fmt.Errorf("发送gRPC请求失败: %w", err)
    }
    return domain.ExecutionStateFromProto(resp.GetExecutionState()), nil
}
```

**调用链路：**

```
GRPCInvoker.Run(ctx)
    │
    ▼
ClientsV2.Get(serviceName)
    │ (如果不存在则创建连接)
    │   → grpc.NewClient("executor:///service_name")
    │   → 注入 ResolverBuilder（etcd 服务发现）
    │   → 注入 routing_round_robin 负载均衡
    │
    ▼
client.Execute(ctx, request)
    │
    ▼
routingPicker.Pick(info)
    │
    ├── ctx 有 SpecificNodeID → 精确路由到指定节点
    ├── ctx 有 ExcludedNodeID → 排除节点后轮询
    └── 无路由提示 → 标准轮询
    │
    ▼
SubConn → TCP → Executor 节点
```

#### 22.2.2 HTTPInvoker —— HTTP 协议支持

```go
type HTTPInvoker struct {
    logger *elog.Component
    client *http.Client  // 30s 超时
}

func (i *HTTPInvoker) Run(ctx context.Context, exec domain.TaskExecution) (domain.ExecutionState, error) {
    requestData := map[string]any{
        "taskId":      exec.Task.ID,
        "taskName":    exec.Task.Name,
        "executionId": exec.ID,
        "params":      exec.Task.HTTPConfig.Params,
    }
    jsonBytes, _ := json.Marshal(requestData)

    req, _ := http.NewRequestWithContext(ctx, http.MethodPost,
        exec.Task.HTTPConfig.Endpoint, strings.NewReader(string(jsonBytes)))
    req.Header.Set("Content-Type", "application/json")

    resp, err := i.client.Do(req)
    // ... 处理响应
}
```

#### 22.2.3 LocalInvoker —— 本地函数调用

```go
type LocalInvoker struct {
    executeFuncs map[string]LocalExecuteFunc  // taskName → 执行函数
    prepareFuncs map[string]LocalPrepareFunc  // taskName → 准备函数
}

func (l *LocalInvoker) Run(ctx context.Context, execution domain.TaskExecution) (domain.ExecutionState, error) {
    fn, ok := l.executeFuncs[execution.Task.Name]
    if !ok {
        return domain.ExecutionState{}, fmt.Errorf("未注册方法：%s", execution.Task.Name)
    }
    return fn(ctx, execution)  // 直接调用本地注册的函数
}
```

**适用场景：** 开发调试、单元测试、Scheduler 与 Executor 同进程部署。

### 22.3 Invoker Dispatcher —— 策略路由

```go
// internal/service/invoker/dispatcher.go

type Dispatcher struct {
    http  *HTTPInvoker
    grpc  *GRPCInvoker
    local *LocalInvoker
}

func (r *Dispatcher) Run(ctx context.Context, execution domain.TaskExecution) (domain.ExecutionState, error) {
    switch {
    case execution.Task.GrpcConfig != nil:
        return r.grpc.Run(ctx, execution)       // 优先 gRPC
    case execution.Task.HTTPConfig != nil:
        return r.http.Run(ctx, execution)        // 其次 HTTP
    default:
        return r.local.Run(ctx, execution)       // 兜底 Local
    }
}
```

**优先级决策：gRPC > HTTP > Local**

任务在创建时通过配置（`GrpcConfig` 或 `HTTPConfig`）声明执行协议，Dispatcher 在运行时根据配置自动选择对应的调用器。

## 第二十三章 事件驱动架构

### 23.1 事件驱动全景

平台使用 Kafka 消息队列实现两类异步事件：

```
┌─────────────────────────────────────────────────────────────────┐
│                       事件驱动架构                                 │
│                                                                 │
│  ┌───────────────────────────────┐                              │
│  │ 1. Complete Event（任务完成）  │                              │
│  │                               │                              │
│  │ 触发场景：                      │                              │
│  │ • Plan 中的子任务完成          │                              │
│  │ • 普通任务完成                 │                              │
│  │                               │                              │
│  │ 生产者：NormalTaskRunner      │                              │
│  │        PlanTaskRunner         │                              │
│  │                               │                              │
│  │ 消费者：complete.Consumer     │                              │
│  │ • NormalTask with PlanID →    │                              │
│  │   触发 PlanTaskRunner.NextStep│                              │
│  │ • PlanTask → 更新执行状态     │                              │
│  │   + 更新下次执行时间           │                              │
│  │   + 释放任务锁                │                              │
│  └───────────────────────────────┘                              │
│                                                                 │
│  ┌───────────────────────────────┐                              │
│  │ 2. Report Event（执行上报）   │                              │
│  │                               │                              │
│  │ 触发场景：                      │                              │
│  │ • Executor 异步上报执行状态    │                              │
│  │                               │                              │
│  │ 生产者：ReportEventProducer   │                              │
│  │                               │                              │
│  │ 消费者：                       │                              │
│  │ • ReportEventConsumer（单条） │                              │
```

### 23.2 Complete Event —— 工作流驱动核心

#### 23.2.1 事件结构

```go
// internal/event/complete.go

type Event struct {
    PlanID         int64                      `json:"planId"`
    ExecID         int64                      `json:"execId"`
    TaskID         int64                      `json:"taskId"`
    Version        int64                      `json:"version"`
    ScheduleNodeID string                     `json:"scheduleNodeId"`
    Type           domain.TaskType            `json:"type"`
    ExecStatus     domain.TaskExecutionStatus `json:"execStatus"`
    Name           string                     `json:"name"`
}
```

#### 23.2.2 Consumer 路由逻辑

```go
func (c *Consumer) handle(ctx context.Context, evt event.Event) error {
    switch evt.Type {
    case domain.NormalTaskType:
        if evt.PlanID > 0 {
            return c.handlePlanTask(ctx, evt)  // Plan 中的子任务 → 触发下一步
        }
        return c.handleTask(ctx, evt)          // 普通任务 → 暂无操作
    case domain.PlanTaskType:
        return c.handlePlan(ctx, evt)          // Plan 自身完成 → 更新状态+释放
    }
}

// 触发 Plan 的下一步执行
func (c *Consumer) handlePlanTask(ctx context.Context, evt event.Event) error {
    return c.planRunner.NextStep(ctx, domain.Task{
        ID:     evt.TaskID,
        PlanID: evt.PlanID,
        Name:   evt.Name,
    })
}

// 处理 Plan 整体完成
func (c *Consumer) handlePlan(ctx context.Context, evt event.Event) error {
    // 1. 更新 Plan 的执行状态
    if evt.ExecStatus.IsSuccess() {
        err = c.execSvc.UpdateScheduleResult(ctx, evt.ExecID,
            domain.TaskExecutionStatusSuccess, 100, time.Now().UnixMilli(), nil, "")
    } else {
        err = c.execSvc.UpdateScheduleResult(ctx, evt.ExecID,
            domain.TaskExecutionStatusFailed, 0, time.Now().UnixMilli(), nil, "")
    }
    // 2. 更新下次执行时间
    c.taskSvc.UpdateNextTime(ctx, evt.TaskID)
    // 3. 释放 Plan 任务锁
    c.acquire.Release(ctx, evt.TaskID, evt.ScheduleNodeID)
}
```

**事件驱动工作流的执行流程：**

```
Plan 启动
    │
    ▼
PlanTaskRunner.Run()
    │ 抢占 Plan 任务
    │ 创建 Plan Execution
    │ 解析 DSL → 找到 Root Tasks
    │
    ▼
对每个 Root Task 启动 goroutine
    │
    ▼
NormalTaskRunner.Run() → Invoker.Run()
    │
    ▼
Executor 执行完成
    │
    ▼
ExecutionService.UpdateState()
    │ 状态为终态 → 发送 Complete Event
    │
    ▼
Kafka: Complete Event
    │
    ▼
complete.Consumer.handlePlanTask()
    │
    ▼
PlanTaskRunner.NextStep()
    │ 获取 Plan
    │ 获取后继节点
    │ 检查前驱条件 (CheckPre)
    │
    ├── 有后继 → 启动下一批任务
    └── 无后继 → 发送 Plan Complete Event → handlePlan() → 释放锁
```

### 23.3 Report Event —— 异步状态上报

```go
// 单条上报
type ReportEventConsumer struct {
    execSvc  task.ExecutionService
    consumer *mqx.Consumer
}

func (c *ReportEventConsumer) consumeExecutionReportEvent(ctx context.Context, message *mq.Message) error {
    report := &domain.Report{}
    json.Unmarshal(message.Value, report)

    if !report.ExecutionState.Status.IsValid() {
        return errs.ErrInvalidTaskExecutionStatus
    }

    return c.execSvc.HandleReports(ctx, []*domain.Report{report})
}

// 批量上报（吞吐量更高）
type BatchReportEventConsumer struct {
    svc      task.ExecutionService
    consumer *mqx.Consumer
}

func (c *BatchReportEventConsumer) consumeBatchExecutionReportEvent(ctx context.Context, message *mq.Message) error {
    batchReport := &domain.BatchReport{}
    json.Unmarshal(message.Value, batchReport)

    // 校验所有报告的状态
    for i := range batchReport.Reports {
        if !batchReport.Reports[i].ExecutionState.Status.IsValid() {
            return errs.ErrInvalidTaskExecutionStatus
        }
    }

    return c.svc.HandleReports(ctx, batchReport.Reports)
}
```

**为什么需要异步上报？**

```
同步上报：
  Executor → gRPC → Scheduler → DB
  问题：Executor 等待 Scheduler 处理完毕才能继续，增加延迟

异步上报（当前方案）：
  Executor → Kafka → Consumer → DB
  优势：
  1. Executor 发送到 Kafka 即返回，不阻塞执行器
  2. 支持批量消费，减少 DB 写入次数
  3. Kafka 天然的削峰填谷，应对突发状态上报
  4. 消费失败可重试，不丢失状态更新
```

## 第二十四章 设计模式全景

### 24.1 策略模式（Strategy Pattern）

策略模式是本项目中使用最广泛的设计模式，贯穿全局：

| 场景 | 接口 | 实现类 |
|------|------|--------|
| 负载均衡 | `balancer.Balancer` | `routingBalancer` |
| 节点选择 | `ExecutorNodePicker` | `Dispatcher`（内含 `prometheusPicker`） |
| 负载检查 | `LoadChecker` | `LimiterChecker`, `DatabaseLoadChecker`, `ClusterLoadChecker` |
| 任务调用 | `Invoker` | `GRPCInvoker`, `HTTPInvoker`, `LocalInvoker` |
| 任务运行 | `Runner` | `NormalTaskRunner`, `PlanTaskRunner` |
| 分片策略 | `ShardingRule` | `RangeShardingRule`, `WeightedDynamicRangeShardingRule` |
| 任务抢占 | `TaskAcquirer` | `MySQLTaskAcquirer` |
| 服务注册 | `Registry` | `etcd.Registry` |

### 24.2 组合模式（Composite Pattern）

```
LoadChecker (接口)
    ├── LimiterChecker        ← 叶子节点
    ├── DatabaseLoadChecker   ← 叶子节点
    ├── ClusterLoadChecker    ← 叶子节点
    └── CompositeChecker      ← 组合节点（包含多个 LoadChecker）
            ├── 策略 AND：所有子检查器通过才通过
            └── 策略 OR：任一子检查器通过即通过
```

组合模式使得单个检查器和多个检查器的组合对外表现一致，调用方无需关心内部结构。

### 24.3 分发器/门面模式（Dispatcher/Facade）

项目中有**三层 Dispatcher**，每层负责不同维度的路由决策：

```
Runner Dispatcher（类型路由）
    ├── NormalTaskType → NormalTaskRunner
    └── PlanTaskType   → PlanTaskRunner

Invoker Dispatcher（协议路由）
    ├── GrpcConfig ≠ nil → GRPCInvoker
    ├── HTTPConfig ≠ nil → HTTPInvoker
    └── default          → LocalInvoker

Picker Dispatcher（策略路由）
    ├── CPUPriority    → pickByMetric(executor_cpu_idle_percent)
    ├── MemoryPriority → pickByMetric(executor_memory_available_bytes)
    └── default        → pickByMetric(executor_cpu_idle_percent)
```

### 24.4 观察者模式（Observer Pattern）

事件驱动架构本质上是观察者模式的异步实现：

```
被观察者（事件源）                     观察者（事件消费者）
─────────────────                    ─────────────────
ExecutionService.UpdateState()   →   Complete Event → complete.Consumer
Executor 执行器                  →   Report Event   → ReportEventConsumer
etcd Watch 服务变更              →   Registry Event → executorResolver.watch()
```

### 24.5 模板方法模式

`ShardingLoopJob` 是一个经典的模板方法实现：

```
ShardingLoopJob 定义了"骨架"：
    1. 获取分片列表 (ShardingStrategy.Broadcast)
    2. 获取信号量 (ResourceSemaphore.Acquire)
    3. 获取分布式锁 (dlock.Lock)
    4. 执行业务 (biz func)  ← 由调用方注入
    5. 续约锁 (lock.Refresh)
    6. 释放锁和信号量

不同场景注入不同的 biz：
    • SchedulerV2: biz = scheduleOneLoop（调度逻辑）
    • CompensatorV2: biz = retryOneLoop / shardingOneLoop 等
```

### 24.6 Builder 模式

gRPC 负载均衡器的注册采用了 Builder 模式：

```go
// routingBalancerBuilder 实现 balancer.BalancerBuilder
type routingBalancerBuilder struct{}

func (b *routingBalancerBuilder) Build(cc balancer.ClientConn, _ balancer.BuildOptions) balancer.Balancer {
    return newRoutingBalancer(cc)
}

func (b *routingBalancerBuilder) Name() string {
    return RoutingRoundRobinName
}
```

### 24.7 泛型客户端池

```go
// 利用 Go 1.18+ 泛型实现类型安全的客户端池
type ClientsV2[T any] struct {
    clientMap syncx.Map[string, T]
    registry  registry.Registry
    timeout   time.Duration
    creator   func(conn *grpc.ClientConn) T  // 泛型工厂函数
}
```

这个设计使得同一个客户端池框架可以复用于不同的 gRPC 服务，只需传入不同的 `creator` 函数。

## 第二十六章 与业界方案对比分析

### 26.1 综合对比矩阵

| 维度 | XXL-Job | Elastic-Job | 本项目 |
|------|---------|-------------|--------|
| **语言** | Java | Java | Go |
| **架构模式** | 中心化 Admin | 去中心化（ZK 协调） | 多调度节点 + 乐观锁抢占 |
| **协调组件** | MySQL + 内存 | ZooKeeper | MySQL + etcd + Kafka |
| **任务抢占** | DB 行锁 / 调度中心分配 | ZK 临时节点 | **乐观锁 CAS（version 字段）** |
| **节点选择** | 轮询/随机/一致性哈希/LFU/LRU | 按分片分配 | **Prometheus 实时指标 + topK 随机** |
| **工作流** | 子任务链（父子依赖） | 不支持 | **ANTLR4 DSL → DAG 引擎** |
| **负载保护** | 无内置机制 | 无内置机制 | **三层负载检测（令牌桶+DB+集群）** |
| **分片策略** | 静态广播分片 | 静态均匀/自定义分片 | **静态范围 + 动态权重分片** |
| **补偿机制** | 失败重试（管理页面手动） | 失效转移（ZK 监控） | **四种自动补偿器（Retry/Reschedule/Sharding/Interrupt）** |
| **服务发现** | Admin 注册表 | ZK 临时节点 | **etcd Lease + Watch** |
| **可观测性** | 管理界面日志 | 控制台 | **Prometheus + 自定义 Metrics** |
| **架构演进** | 单版本 | 单版本 | **V1/V2 双版本（分库分表支持）** |
| **通信协议** | HTTP (RESTful) | 无（本地执行） | **gRPC（主）+ HTTP + Local** |
| **负载均衡** | 内置多种策略 | ZK 协调 | **自定义 routing_round_robin** |

### 26.2 核心差异化分析

#### 26.2.1 任务抢占机制对比

**XXL-Job：**
```
调度中心 → 选择执行器 → 直接发送执行请求
问题：调度中心是单点，虽然有集群模式但通过 DB 行锁保证一致性
```

**Elastic-Job：**
```
每个实例注册 ZK 临时节点 → Leader 选举 → Leader 分配分片
问题：ZK 强依赖 + Leader 切换延迟
```

**本项目：**
```sql
UPDATE tasks SET status = 'PREEMPTED', schedule_node_id = ?, version = version + 1
WHERE id = ? AND version = ?  -- CAS 乐观锁

优势：
1. 完全无锁竞争（乐观锁只在数据库层面竞争）
2. 无需 Leader 选举
3. 任何调度节点都可以抢占任何任务
4. 水平扩展只需增加调度节点
```

#### 26.2.2 节点选择机制对比

**XXL-Job：**
```
策略有限：轮询 → 随机 → 一致性哈希 → 最近最少使用 → 最不频繁使用
缺点：完全不考虑执行器的实际负载
```

**本项目：**
```
1. 通过 Prometheus 实时查询执行器的 CPU/内存指标
2. topK 选取资源最优的候选节点
3. 随机选择避免扎堆
4. 通过自定义 gRPC Balancer 实现精确路由

优势：
• 基于实际资源使用情况做决策
• 自适应：负载变化时自动调整选择
• 可扩展：新增指标只需增加一个 metricName
```

#### 26.2.3 工作流能力对比

**XXL-Job：**
```
子任务链：A → B → C（线性依赖）
限制：不支持并行分支、条件分支、聚合等复杂拓扑
```

**本项目：**
```
ANTLR4 DSL → DAG 引擎：
• Sequence：A → B → C（顺序执行）
• AND：A → (B, C) → D（并行，全部成功后继续）
• OR：A → (B, C) → D（并行，任一成功即继续）
• Condition：A → [success: B, failure: C]（条件分支）

实现机制：
• DSL 解析：ANTLR4 生成 AST → Visitor 遍历 → 构建 PlanTask DAG
• 执行控制：事件驱动（Kafka Complete Event）
• 前驱检查：PlanTask.CheckPre() 递归校验所有前驱是否完成
```

#### 26.2.4 补偿机制对比

**XXL-Job：**
```
手动重试：在管理界面点击"重试"
自动策略：仅支持失败重试（配置重试次数）
```

**Elastic-Job：**
```
失效转移：通过 ZK 监控节点下线 → 重新分配到其他节点
限制：不支持超时中断、分片聚合等
```

**本项目：**
```
四种自动补偿器，独立循环运行：

1. RetryCompensator
   • 指数退避重试：base * 2^retryCount
   • 查找 RetryWaiting 状态 + 已到重试时间的执行
   • 通过 Runner.Retry() 排除上次失败节点重试

2. RescheduleCompensator
   • 查找长时间 Running 但可能丢失的执行
   • 在不同节点上重新调度

3. ShardingCompensator
   • 分页遍历 Running 的父任务
   • 检查所有子分片是否完成
   • 聚合子分片状态 → 更新父任务 → 释放任务锁

4. InterruptCompensator
   • 查找超时的执行记录
   • 通过 gRPC Interrupt 接口中断执行器
   • 通过 WithSpecificNodeID 精确路由到执行节点
```

### 26.3 技术选型对比

| 组件 | XXL-Job | Elastic-Job | 本项目 | 选型理由 |
|------|---------|-------------|--------|---------|
| 调度协调 | MySQL | ZooKeeper | etcd | 轻量级、Watch 性能好、K8s 生态 |
| 消息队列 | - | - | Kafka | 削峰填谷、事件驱动工作流 |
| 通信协议 | HTTP | - | gRPC | 高性能、强类型、流式支持 |
| 指标系统 | - | - | Prometheus | 云原生标准、PromQL 强大 |
| 数据库 | MySQL | - | MySQL + GORM | ORM 生态好、插件机制 |
| 服务发现 | 内置 | ZK | etcd Registry | 自动注册/摘除、Lease 机制 |
| DSL 解析 | - | - | ANTLR4 | 工业级解析器生成、语法灵活 |

### 27.1 统一错误定义

```go
// internal/errs/error.go

var (
    // 任务生命周期错误
    ErrTaskPreemptFailed  = errors.New("任务抢占失败")

    // 执行记录错误
    ErrExecutionNotFound            = errors.New("执行记录不存在")
    ErrInvalidTaskExecutionStatus   = errors.New("执行记录状态非法")
    ErrInterruptTaskExecutionFailed = errors.New("中断任务执行失败")

    // 配置校验错误
    ErrInvalidTaskCronExpr       = errors.New("无效的cron表达式")
    ErrInvalidTaskShardingRule   = errors.New("分片规则非法")
    ErrTaskShardingRuleNotFound  = errors.New("分片规则未找到")

    // 状态更新错误
    ErrExecutionMaxRetriesExceeded   = errors.New("超过最大重试次数")
    ErrExecutionStateHandlerNotFound = errors.New("执行状态处理器未找到")

    // 资源限制
    ErrExceedLimit = errors.New("抢资源超出限制")
)
```

### 27.2 多层容错策略

```
Layer 1: 调度层容错
──────────────────
• 乐观锁抢占失败 → 跳过本次调度，其他节点会抢占
• 智能调度失败   → 降级为默认轮询
• Prometheus 查询超时 → 允许继续调度（降级策略）

Layer 2: 执行层容错
──────────────────
• gRPC 调用失败  → 返回错误，等待 RetryCompensator 重试
• 执行器超时     → InterruptCompensator 主动中断
• 节点下线       → RescheduleCompensator 重新调度

Layer 3: 基础设施容错
──────────────────
• etcd 不可用    → ResolverError() 不改变连接状态，等待恢复
• Kafka 发送失败 → 记录日志，后续补偿器兜底
• DB 负载过高    → DatabaseLoadChecker 降低调度频率
• 集群不均衡     → ClusterLoadChecker 节点级自适应

Layer 4: 分布式锁容错
──────────────────
• 锁获取失败     → 跳过本分表，继续处理其他分表
• 锁续约失败     → 退出 bizLoop，释放资源
• 锁释放失败     → 使用 Background Context 重试
                   → 最终等锁自动过期
```

## 第二十八章 核心设计亮点总结

### 28.1 面试回答框架

#### 亮点一：乐观锁任务抢占

**问题：** 如何保证分布式环境下任务不被重复调度？

**回答：**
> 使用数据库乐观锁（CAS）实现任务抢占。每个任务有 `version` 字段，调度器通过 `UPDATE ... WHERE id = ? AND version = ?` 原子性地抢占任务。只有一个节点能成功更新 version，其他节点的更新会返回 affected rows = 0。抢占成功后任务状态变为 PREEMPTED，并记录调度节点ID。同时有续约机制防止僵尸任务——调度节点定期 renew 更新 `updated_at`，如果调度节点宕机，续约停止，其他节点通过 `PREEMPTED + 超时` 条件重新发现并抢占该任务。

#### 亮点二：自定义 gRPC 负载均衡器

**问题：** 如何实现任务重试时排除失败节点？

**回答：**
> 自定义了 `routing_round_robin` gRPC 负载均衡器，支持三优先级路由：指定节点、排除节点、默认轮询。通过 Go 的 `context.WithValue` 在 RPC 调用链中透传路由信息。重试时使用 `WithExcludedNodeID` 排除上次失败的节点；智能调度时使用 `WithSpecificNodeID` 精确路由到 Prometheus 选出的最优节点。Balancer 内部维护了四层映射（Address→SubConn、SubConn→Address 反向映射、Address→NodeID、SubConn→NodeID 就绪集合），通过反向映射实现 O(1) 的连接状态查找。

#### 亮点三：三层负载检测

**问题：** 如何防止调度器过载？

**回答：**
> 设计了三层负载检测体系，通过 CompositeChecker（组合模式，AND 策略）串联：
> 1. **LimiterChecker**：令牌桶硬限流，限制调度频率上限
> 2. **DatabaseLoadChecker**：通过 Prometheus 查询 GORM 插件采集的数据库响应时间，超过阈值则退避
> 3. **ClusterLoadChecker**：将当前节点调度耗时与集群平均值对比，超过 1.2 倍则降频
>
> 三层中任一层判定负载过高，调度器就会暂停对应时长。这实现了从"本机限流"到"数据库感知"到"集群均衡"的渐进式负载保护。

#### 亮点四：ANTLR4 DSL 工作流引擎

**问题：** 如何实现复杂任务编排？

**回答：**
> 设计了基于 ANTLR4 的 DSL 工作流引擎。用户编写类似 `AND(A, B) → C → OR(D, E)` 的 DSL 定义工作流。解析流程：ANTLR4 词法/语法分析 → AST → `TaskOrchestrationVisitor` 遍历 → `mergePlan` 构建 PlanTask DAG。支持四种节点类型：Simple（单任务）、AND（并行全成功）、OR（并行任一成功）、Condition（条件分支）。执行由事件驱动——每个子任务完成后发送 Kafka Complete Event，Consumer 调用 `NextStep` 检查后继节点的前驱条件，满足则启动下一批任务。

#### 亮点五：ShardingLoopJob 分库分表框架

**问题：** 分库分表场景下如何调度和补偿？

**回答：**
> 抽象了 `ShardingLoopJob` 通用框架，封装了"遍历分片 → 获取信号量 → 抢分布式锁 → 执行业务 → 续约锁 → 释放资源"的完整流程。`ResourceSemaphore` 控制单节点同时处理的分表数量上限，防止单节点承担过多分表。分布式锁确保同一分表同一时刻只有一个节点处理。业务逻辑通过 `biz func(ctx context.Context) error` 注入，SchedulerV2 和所有 V2 补偿器都复用这个框架。

#### 亮点六：Prometheus 智能调度

**问题：** 如何选择最优执行节点？

**回答：**
> 通过 Prometheus PromQL 实时查询执行器节点的资源指标（CPU 空闲率或可用内存）。查询语句 `topk(3, avg_over_time(metric[1m]) AND ON(instance) up{job="executors"} == 1)` 一次性完成三件事：过滤在线节点、时间窗口平滑、选取 Top-K。然后从候选中随机选一个，避免多个调度器同时选择同一最优节点导致扎堆。选出的节点 ID 通过 `WithSpecificNodeID` 注入 context，经过自定义 Balancer 精确路由到该节点的 gRPC 连接。
