# 06-任务暂停 / 恢复 / 取消详细设计

## 1. 文档结论先说

这次不走“把所有语义都塞进 `tasks.status`”的路线。

更稳的方案是三层拆开：

1. **Task** 继续表达“这个任务当前还能不能被调度”
2. **TaskExecution** 表达“这一次执行实例当前处于什么运行状态”
3. **TaskControlOperation** 单独表达“用户发起了什么控制动作，以及这个动作是否已送达/已确认”

核心结论：

- `INACTIVE` 只能表示“停用任务定义”，**不能直接拿来当暂停/取消**
- `pause/resume/cancel` 要新增一条 **控制链路**，而不是只在调度主链里加几个 if
- `resume` 不需要单独造一套执行模型，**直接复用现有 `FAILED_RESCHEDULED + RescheduleParams` 的恢复思路**，但把状态语义升级为正式的 `PAUSED`
- `cancel` 是**执行实例终态**，不是任务定义终态；默认取消当前运行，不等于删除或永久停用任务
- `sharding` 可以在 V1 一起支持；`plan` 在 V1 先支持“冻结推进 + 取消当前运行中节点”，完整 DAG 分支级 checkpoint-resume 放到 V2

一句话总结：

> **Task 管可调度性，Execution 管运行态，ControlOperation 管控制意图。**

这样设计，才不会把 `Acquire / Release / Renew / UpdateNextTime / NextStep / Compensator` 全部搞脏。

## 2. 背景与现状锚点

### 2.1 Task 当前只有三态

当前 `internal/domain/task.go`：

```go
const (
    TaskStatusActive    TaskStatus = "ACTIVE"
    TaskStatusPreempted TaskStatus = "PREEMPTED"
    TaskStatusInactive  TaskStatus = "INACTIVE"
)
```

当前 `tasks` 表查询可调度任务的条件来自 `internal/repository/dao/task.go`：

```go
Where("next_time <= ? AND (status = ? OR (status = ? AND utime <= ?))",
    now, StatusActive, StatusPreempted, now-preemptedTimeoutMs)
```

这说明：

- `Task` 这张表表达的是**定义层/调度层状态**
- 它不表达“这次执行是暂停中还是取消中”
- `INACTIVE` 只是“停止参与调度”，不是“暂停中的运行实例”

### 2.2 Execution 当前只有运行/失败/重试/重调度语义

当前 `internal/domain/task_execution.go`：

```go
const (
    TaskExecutionStatusPrepare           TaskExecutionStatus = "PREPARE"
    TaskExecutionStatusRunning           TaskExecutionStatus = "RUNNING"
    TaskExecutionStatusSuccess           TaskExecutionStatus = "SUCCESS"
    TaskExecutionStatusFailed            TaskExecutionStatus = "FAILED"
    TaskExecutionStatusFailedRetryable   TaskExecutionStatus = "FAILED_RETRYABLE"
    TaskExecutionStatusFailedRescheduled TaskExecutionStatus = "FAILED_RESCHEDULED"
)
```

当前没有：

- `PAUSED`
- `CANCELLED`
- `PAUSING`
- `CANCELLING`

也就是说，运行态里还没有显式的“用户控制语义”。

### 2.3 主调度链没有控制切入点

`internal/service/runner/normal_task_runner.go` 当前主链：

```go
acquiredTask, err := s.taskAcquirer.Acquire(ctx, task.ID, task.Version, s.nodeID)
execution, err := s.execSvc.Create(ctx, domain.TaskExecution{...})
go func() {
    state, runErr := s.invoker.Run(ctx, execution)
    if updateErr := s.execSvc.UpdateState(ctx, state); updateErr != nil {
        ...
    }
}()
```

当前模型天然支持：

- 正常执行
- 失败重试
- 失败重调度
- 超时中断

但不支持：

- 用户主动暂停
- 用户主动恢复
- 用户主动取消

### 2.4 终态处理现在默认会 Release + UpdateNextTime

`internal/service/task/execution_service.go` 当前终态逻辑：

```go
if !isShardedTask {
    s.releaseTask(ctx, execution.Task)
    if _, err3 := s.taskSvc.UpdateNextTime(ctx, execution.Task.ID); err3 != nil {
        ...
    }
}
s.sendCompletedEvent(ctx, state, execution)
```

这段逻辑的问题很直接：

- 如果用户想“暂停当前任务”，这里会直接把任务释放回可调度态
- 如果用户想“取消当前执行，但保留任务定义”，这里没有区分语义
- 如果用户想“暂停但不推进下一次 cron”，这里也没有钩子

所以 Pause / Resume / Cancel 不能只改 API，必须把**终态收口逻辑**一起改掉。

### 2.5 当前 Executor 协议只有 `Interrupt`，没有 Pause / Cancel

`api/proto/executor/v1/executor.proto` 当前只有：

```proto
service ExecutorService {
  rpc Execute(ExecuteRequest) returns (ExecuteResponse);
  rpc Interrupt(InterruptRequest) returns (InterruptResponse);
  rpc Query(QueryRequest) returns (QueryResponse);
  rpc Prepare(PrepareRequest) returns (PrepareResponse);
}
```

这意味着：

- 系统级“超时中断”是有原语的
- 但用户级“暂停”和“取消”没有独立原语
- 如果强行复用 `Interrupt`，Pause 和 Cancel 的业务语义会混在一起

不建议这么干。

## 3. 目标与范围

## 3.1 目标

本设计要支持四类核心场景：

1. **暂停未运行任务**
   - 任务未被抢占时，直接冻结未来调度
2. **暂停运行中任务**
   - 尝试让执行节点协作式暂停，并返回恢复所需 checkpoint
3. **恢复已暂停任务**
   - 若有 checkpoint，则从 checkpoint 继续
   - 若无 checkpoint，则恢复 future schedule
4. **取消运行中任务**
   - 中止本次执行，清理状态，决定未来是否继续参与调度

## 3.2 V1 范围

V1 只做最值钱、最贴近现有实现的范围：

- `normal task`：全支持
- `sharding task`：支持，以父任务为入口，控制运行中的子任务
- `plan task`：支持“冻结后续推进 + 取消运行中节点”，**不做 DAG 分支级 checkpoint-resume**

## 3.3 非目标

V1 不做：

- UI 页面细节
- 审批流
- 多租户权限模型
- DAG 任意节点级 checkpoint 恢复
- 跨版本 executor 兼容层

## 4. 核心设计决策

## 4.1 不建议把 `PAUSING/CANCELLING/CANCELLED` 全塞进 `tasks.status`

原因很简单。

`tasks.status` 现在承担的是“调度可见状态”：

- `FindSchedulableTasks`
- `Acquire`
- `Release`
- `Renew`
- `UpdateNextTime`

都依赖它。

如果把这些控制中的瞬时态也塞进去，会立刻污染：

- 调度筛选条件
- 锁释放逻辑
- 僵尸任务接管逻辑
- 补偿器过滤条件
- Plan 完成后的 Release 语义

所以这里做一个更干净的拆分。

## 4.2 Task 只表达“可不可调度”，Execution 才表达“这次执行是什么状态”

建议调整为：

### Task 状态

- `ACTIVE`：可调度
- `PREEMPTED`：已被某个 scheduler 抢占
- `PAUSED`：冻结未来调度
- `INACTIVE`：人工停用/下线

### Task 额外字段

新增：

- `desired_status`：期望的下一个 Task 状态
  - `NULL`
  - `ACTIVE`
  - `PAUSED`
  - `INACTIVE`

它的作用是：

- 用户在运行中发起 Pause / Disable 时，不会立即改 `status`
- 但终态收口时，可以根据 `desired_status` 决定最终落点

这是整个方案里最关键的字段之一。

## 4.3 ControlOperation 单独建模，专门承载用户控制意图

新增控制操作表，而不是只在 `tasks`/`task_executions` 上硬写字段。

因为暂停/取消天生需要：

- 请求幂等
- 审计
- 重试
- ACK 关联
- 超时补发
- 失败原因记录

这些都不该压在 `task_executions` 一行里。

## 4.4 Resume 不单独发 RPC，直接复用现有恢复链路

当前系统已经有：

- `FAILED_RESCHEDULED`
- `RequestReschedule`
- `RescheduleParams`
- `runner.Reschedule()`

所以 Resume 最稳的做法不是新造 `Resume RPC`，而是：

- Pause 时由 executor 产出 checkpoint 参数
- Resume 时由 scheduler 重新驱动一次执行
- 恢复参数继续走 `schedule_params` / `reschedule_params`

这能最大化复用现有架构。

## 4.5 Cancel 是执行实例终态，不是任务定义终态

默认定义：

- `cancel` = 取消当前执行实例
- 不等于删除任务定义
- 不等于永久停用任务定义

所以默认 Cancel 后：

- `execution.status -> CANCELLED`
- `task.status -> ACTIVE` 或按 `desired_status` 落到 `PAUSED/INACTIVE`

这样才不会和 `INACTIVE` 重复。

## 5. 领域模型与表结构设计

## 5.1 tasks 表改动

### 5.1.1 状态枚举扩展

当前：

- `ACTIVE`
- `PREEMPTED`
- `INACTIVE`

调整为：

- `ACTIVE`
- `PREEMPTED`
- `PAUSED`
- `INACTIVE`

### 5.1.2 新增字段

建议新增：

```sql
ALTER TABLE tasks
ADD COLUMN desired_status ENUM('ACTIVE', 'PAUSED', 'INACTIVE') NULL COMMENT '用户期望的落点状态';
```

可选增强字段：

```sql
ALTER TABLE tasks
ADD COLUMN last_control_op_id BIGINT NULL COMMENT '最近一次控制操作ID';
```

### 5.1.3 字段语义

- `status`：当前真实调度状态
- `desired_status`：当前运行结束后的目标状态

例如：

- 任务运行中，用户点暂停：
  - `status = PREEMPTED`
  - `desired_status = PAUSED`
- 任务 Pause ACK 收到后：
  - `status = PAUSED`
  - `desired_status = NULL`

## 5.2 task_executions 表改动

### 5.2.1 状态枚举扩展

当前：

- `PREPARE`
- `RUNNING`
- `FAILED_RETRYABLE`
- `FAILED_RESCHEDULED`
- `FAILED`
- `SUCCESS`

调整为：

- `PREPARE`
- `RUNNING`
- `PAUSED`
- `CANCELLED`
- `FAILED_RETRYABLE`
- `FAILED_RESCHEDULED`
- `FAILED`
- `SUCCESS`

### 5.2.2 新增字段

建议新增：

```sql
ALTER TABLE task_executions
ADD COLUMN last_control_op_id BIGINT NULL COMMENT '最后一次控制操作ID',
ADD COLUMN terminal_reason VARCHAR(64) NOT NULL DEFAULT '' COMMENT 'SUCCESS/FAILED/CANCELLED 等终结原因';
```

说明：

- `PAUSED` 不是终态，但它是一个“冻结态”
- `CANCELLED` 是终态，必须并入 `IsTerminalStatus()`
- checkpoint 继续复用已有的 `task_schedule_params` 快照，不单独再建一张 snapshot 表

## 5.3 新增 `task_control_operations` 表

```sql
CREATE TABLE task_control_operations (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  request_id VARCHAR(64) NOT NULL COMMENT '幂等请求ID',
  task_id BIGINT NOT NULL,
  execution_id BIGINT NOT NULL DEFAULT 0,
  op_type ENUM('PAUSE', 'RESUME', 'CANCEL') NOT NULL,
  op_status ENUM('PENDING', 'SENT', 'ACKED', 'TIMEOUT', 'FAILED') NOT NULL,
  schedule_node_id VARCHAR(255) NOT NULL DEFAULT '',
  executor_node_id VARCHAR(255) NOT NULL DEFAULT '',
  reason VARCHAR(255) NOT NULL DEFAULT '',
  checkpoint_params JSON NULL,
  retry_count BIGINT NOT NULL DEFAULT 0,
  next_retry_time BIGINT NOT NULL DEFAULT 0,
  ctime BIGINT NOT NULL,
  utime BIGINT NOT NULL,
  UNIQUE KEY uniq_request_id (request_id),
  KEY idx_task_op_status (task_id, op_status),
  KEY idx_execution_op_status (execution_id, op_status)
);
```

### 5.3.1 这张表解决什么问题

它解决五个问题：

1. 控制请求幂等
2. ACK 关联
3. 控制事件重发
4. 超时补偿
5. 审计与排障

## 5.4 新增 `task_control_outbox` 表

```sql
CREATE TABLE task_control_outbox (
  id BIGINT PRIMARY KEY AUTO_INCREMENT,
  op_id BIGINT NOT NULL,
  topic VARCHAR(128) NOT NULL,
  payload JSON NOT NULL,
  status ENUM('PENDING', 'SENT', 'FAILED') NOT NULL,
  retry_count BIGINT NOT NULL DEFAULT 0,
  next_retry_time BIGINT NOT NULL DEFAULT 0,
  ctime BIGINT NOT NULL,
  utime BIGINT NOT NULL,
  KEY idx_status_next_retry (status, next_retry_time)
);
```

用途：

- 避免 API 请求成功但控制消息没发出去
- 控制动作与消息投递一起落盘
- 继续沿用你这个项目已有的“补偿器 + 最终一致性”风格

## 6. 状态机设计

## 6.1 Task 状态机

```text
                +-----------+
         Create  |  ACTIVE   |
      +--------->|           |<----------------+
      |          +-----+-----+                 |
      |                | Acquire              |
      |                v                      |
      |          +-----+-----+                |
      |          | PREEMPTED |                |
      |          |           |                |
      |          +--+-----+--+                |
      |             |     |                   |
      |  desired=   |     | Release +         |
      |  PAUSED     |     | desired=NULL      |
      |             |     +-------------------+
      |             |
      |             | Pause ACK / natural end + desired=PAUSED
      |             v
      |          +--+-----+
      |          | PAUSED  |
      |          +--+-----+
      |             |
      |             | Resume
      |             v
      +-------------+

      +----------------+
      |   INACTIVE     |
      +----------------+
```

关键点：

- `PREEMPTED` 期间收到 Pause，不直接跳到 `PAUSED`
- 先写 `desired_status = PAUSED`
- 真正进入 `PAUSED` 要么靠 Pause ACK，要么靠当前运行自然结束后的收口逻辑

## 6.2 Execution 状态机

```text
PREPARE -> RUNNING -> SUCCESS
                  -> FAILED
                  -> FAILED_RETRYABLE
                  -> FAILED_RESCHEDULED
                  -> PAUSED
                  -> CANCELLED

PAUSED -- Resume --> RUNNING
```

### 6.2.1 终态定义

终态应改为：

- `SUCCESS`
- `FAILED`
- `CANCELLED`

`PAUSED` 不是终态，它是“可恢复的冻结态”。

### 6.2.2 必须同步修改的领域方法

`internal/domain/task_execution.go` 里至少要改：

- `IsValid()`：加入 `PAUSED/CANCELLED`
- `IsTerminalStatus()`：加入 `CANCELLED`
- `SetRunningState` 允许从 `PAUSED -> RUNNING`

不改这三处，Pause / Resume 根本跑不起来。

## 6.3 ControlOperation 状态机

```text
PENDING -> SENT -> ACKED
             |       |
             |       +-- 完成
             v
          TIMEOUT -> 重试 -> SENT
             |
             v
           FAILED
```

## 7. API 与 RPC 设计

## 7.1 管理面 API

### 7.1.1 暂停任务

```http
POST /api/tasks/{id}/pause
Content-Type: application/json

{
  "requestId": "pause-task-1001-20260411-01",
  "reason": "人工暂停，等待下游窗口恢复",
  "mode": "COOPERATIVE"
}
```

返回：

```json
{
  "taskId": 1001,
  "taskStatus": "PREEMPTED",
  "desiredStatus": "PAUSED",
  "controlOpStatus": "PENDING"
}
```

### 7.1.2 恢复任务

```http
POST /api/tasks/{id}/resume
Content-Type: application/json

{
  "requestId": "resume-task-1001-20260411-01",
  "reason": "窗口恢复，继续执行"
}
```

### 7.1.3 取消任务当前执行

```http
POST /api/tasks/{id}/cancel
Content-Type: application/json

{
  "requestId": "cancel-task-1001-20260411-01",
  "reason": "人工终止本次执行",
  "disableAfterCancel": false,
  "force": false
}
```

说明：

- `disableAfterCancel=false`：取消当前执行，但任务定义保留
- `disableAfterCancel=true`：取消后任务进入 `INACTIVE`

## 7.2 Executor RPC 扩展

建议扩展 `executor.proto`：

```proto
enum ExecutionStatus {
  UNKNOWN = 0;
  RUNNING = 1;
  FAILED_RETRYABLE = 2;
  FAILED_RESCHEDULABLE = 3;
  FAILED = 4;
  SUCCESS = 5;
  PAUSED = 6;
  CANCELLED = 7;
}

message PauseRequest {
  int64 eid = 1;
  string request_id = 2;
  string reason = 3;
}

message PauseResponse {
  bool success = 1;
  ExecutionState execution_state = 2;
}

message CancelRequest {
  int64 eid = 1;
  string request_id = 2;
  string reason = 3;
  bool force = 4;
}

message CancelResponse {
  bool success = 1;
  ExecutionState execution_state = 2;
}

service ExecutorService {
  rpc Execute(ExecuteRequest) returns (ExecuteResponse);
  rpc Interrupt(InterruptRequest) returns (InterruptResponse);
  rpc Pause(PauseRequest) returns (PauseResponse);
  rpc Cancel(CancelRequest) returns (CancelResponse);
  rpc Query(QueryRequest) returns (QueryResponse);
  rpc Prepare(PrepareRequest) returns (PrepareResponse);
}
```

## 7.3 为什么 Resume 不发 RPC

因为当前系统已有：

- `ExecutionState.RequestReschedule`
- `ExecutionState.RescheduleParams`
- `runner.Reschedule()`

Resume 本质上就是：

- 让某个已冻结 execution 从 checkpoint 重新进入 `RUNNING`

这件事从 scheduler 端发起更合适，不需要 executor 再提供一个 Resume RPC。

## 8. 关键流程设计

## 8.1 暂停未运行任务

场景：

- `task.status = ACTIVE`
- 当前没有运行中的 execution

事务：

```text
Tx1:
1. 校验 task.status = ACTIVE
2. task.status = PAUSED
3. task.desired_status = NULL
4. 插入 task_control_operation(op_type=PAUSE, op_status=ACKED)
```

结果：

- 任务未来不再被调度
- 不需要发 RPC
- 不需要 outbox

## 8.2 暂停运行中任务

场景：

- `task.status = PREEMPTED`
- 存在 `execution.status in (PREPARE, RUNNING)`

### 8.2.1 请求创建事务

```text
Tx1: PauseTask(taskID, requestID)
1. SELECT task + latest execution FOR UPDATE
2. 校验 task.status = PREEMPTED
3. 幂等检查：request_id 是否已存在
4. task.desired_status = PAUSED
5. task.last_control_op_id = newOpID
6. 插入 task_control_operations(PAUSE, PENDING)
7. 插入 task_control_outbox(control.pause)
```

### 8.2.2 Relay 投递

`TaskControlRelay` 扫描 `task_control_outbox.status = PENDING`：

1. 找到对应 execution
2. 调用 `ExecutorService.Pause(eid, request_id)`
3. 成功后 outbox -> `SENT`
4. operation -> `SENT`

### 8.2.3 Pause ACK 收口事务

executor 返回：

```json
{
  "status": "PAUSED",
  "runningProgress": 67,
  "rescheduledParams": {
    "offset": "10000",
    "progress": "67"
  }
}
```

收口事务：

```text
Tx2: HandlePauseAck(opID, executionState)
1. execution.status -> PAUSED
2. execution.running_progress -> 67
3. execution.task_schedule_params -> checkpoint params
4. execution.last_control_op_id = opID
5. task.status = PAUSED
6. task.desired_status = NULL
7. task.schedule_node_id = NULL
8. task.version = version + 1
9. operation.op_status = ACKED
10. 写事件 task.paused（可选，用于审计/通知）
```

### 8.2.4 关键点

Pause ACK 后：

- **不能**再调用 `UpdateNextTime()`
- **不能**发 `complete event`
- **必须**释放 task 抢占，否则任务会一直卡在 PREEMPTED

## 8.3 恢复已暂停任务

分两类。

### 8.3.1 只有定义层暂停，没有 paused execution

场景：

- `task.status = PAUSED`
- 最新 execution 不存在，或不是 `PAUSED`

事务：

```text
Tx1: ResumeTask(taskID)
1. task.status = ACTIVE
2. task.desired_status = NULL
3. 重新计算 next_time = cron.Next(now)
4. operation(op_type=RESUME, op_status=ACKED)
```

### 8.3.2 有 paused execution，需要从 checkpoint 继续

这是 V1 的主场景。

流程：

```text
Tx1: ResumeTask(taskID)
1. 查 task.status = PAUSED
2. 查 latest execution.status = PAUSED
3. task.status = ACTIVE
4. task.desired_status = NULL
5. 插入 task_control_operations(RESUME, ACKED)
```

随后直接复用现有 `runner.Reschedule()` 思路：

- 取 paused execution
- 用 execution 上保存的 `task_schedule_params` 继续
- 重新抢占 task
- 再次调用 `invoker.Run()`

这要求改一处关键逻辑：

当前 `SetRunningState()` 只允许从：

- `PREPARE`
- `FAILED_RETRYABLE`
- `FAILED_RESCHEDULED`

进入 `RUNNING`

恢复后必须允许：

- `PAUSED -> RUNNING`

## 8.4 取消运行中任务

### 8.4.1 请求创建事务

```text
Tx1: CancelTask(taskID, requestID)
1. SELECT task + latest execution FOR UPDATE
2. 校验 task.status = PREEMPTED
3. 若 disableAfterCancel=true，则 task.desired_status = INACTIVE
   否则 task.desired_status = ACTIVE
4. 插入 task_control_operations(CANCEL, PENDING)
5. 插入 task_control_outbox(control.cancel)
```

### 8.4.2 Relay 调用 executor.Cancel

- 调用 `ExecutorService.Cancel(eid, request_id, force)`
- `force=false` 先走协作式取消
- 若多次超时失败，再降级到 `Interrupt`

### 8.4.3 Cancel ACK 收口事务

```text
Tx2: HandleCancelAck(opID, executionState)
1. execution.status -> CANCELLED
2. execution.end_time = now
3. execution.terminal_reason = 'USER_CANCELLED'
4. operation.op_status = ACKED
5. 根据 task.desired_status 决定任务最终状态：
   - ACTIVE  -> release + UpdateNextTime(now)
   - INACTIVE -> release + 不更新 next_time
   - PAUSED -> release + 不更新 next_time
6. 如是 plan/sharding，写完成/聚合事件
```

### 8.4.4 为什么 Cancel 默认还要 UpdateNextTime

因为取消的是“本次执行实例”，不是把 cron 定义删掉。

如果取消后任务还保持启用，就要重新计算下次 cron 时间。

否则：

- 旧的 `next_time` 已经过去
- 任务一释放就会立刻再次被调度
- 语义就乱了

## 9. Sharding 与 Plan 的特殊处理

## 9.1 Sharding 任务

### 9.1.1 控制入口仍然是父任务

现有分片模型：

- 父 execution：`sharding_parent_id = 0`
- 子 execution：`sharding_parent_id = parentID`

V1 设计：

- Pause / Cancel 入口只对父任务开放
- 真正执行控制的是所有 **RUNNING/PREPARE** 子 execution

### 9.1.2 Pause 规则

- 所有子任务都 ACK `PAUSED` 后
  - 父 execution -> `PAUSED`
  - task -> `PAUSED`
- 只要还有子任务未 ACK
  - 父 execution 维持 `RUNNING`
  - `TaskControlCompensator` 重试控制请求

### 9.1.3 Cancel 规则

- 所有子任务都进入 `CANCELLED/SUCCESS/FAILED` 后再收口父任务
- 父任务最终：
  - 只要存在 `CANCELLED` 且没有业务要求继续，则父 execution -> `CANCELLED`
  - 再依据 `desired_status` 更新 task

### 9.1.4 需要修改的补偿器

`internal/compensator/sharding.go` / `shardingv2.go` 当前只认：

- `SUCCESS`
- `FAILED`

要改为：

- `PAUSED` 视为“已冻结，不能再等它自然完成”
- `CANCELLED` 视为终态

否则父任务永远聚不拢。

## 9.2 Plan 任务

Plan 最麻烦，因为它不是一条 execution，而是一张 DAG。

### 9.2.1 V1 边界

V1 不做“任意 DAG 分支 checkpoint-resume”。

V1 只做两件事：

1. **Pause**：冻结后续 `NextStep` 推进
2. **Cancel**：停止当前运行中的节点，并把 Plan 执行收敛到失败/取消

### 9.2.2 Pause 的最小实现

当 `plan task` 收到 Pause：

- `task.desired_status = PAUSED`
- `complete.Consumer.handlePlanTask()` 在触发 `NextStep()` 前，先检查 plan task 当前状态
- 若 plan task 已 `PAUSED` 或 `desired_status=PAUSED`
  - **不再推进后继节点**

这样就先把 DAG 冻住。

### 9.2.3 Cancel 的最小实现

- 取消当前运行中的 root/leaf execution
- 阻断 `NextStep()`
- 让 `Plan.Execution` 最终落到 `CANCELLED` 或 `FAILED`

### 9.2.4 为什么 V1 不直接做 Plan Resume

因为当前 `PlanService.GetPlan()` 只是根据：

- task 定义
- 最新 Plan execution
- 各 task 最新 execution

重建 DAG 图。

它没有一套独立的“分支级 checkpoint 模型”。

硬做只会把设计写得很花，代码一落地就烂。

所以这里收敛：

- 先把 Pause / Cancel / Freeze-NextStep 跑通
- Plan Resume 的分支级恢复放 V2

这是更稳的路线。

## 10. 一致性与补偿设计

## 10.1 为什么不用 2PC

不需要，也不该用。

这个场景是典型的：

- MySQL 任务元数据
- executor 本地运行状态
- gRPC 控制调用
- 可能还要带 MQ/outbox

跨 DB + RPC + executor 本地内存状态，2PC 根本不好使。

正确做法就是：

- **本地事务落控制请求**
- **outbox 保证控制消息最终可达**
- **ACK 驱动状态收口**
- **补偿器重试超时控制请求**

## 10.2 幂等规则

### 10.2.1 API 幂等

`request_id` 全局唯一：

- 同一个 `request_id` 重复调用
- 直接返回已存在的 `task_control_operation`

### 10.2.2 Executor 幂等

同一个 `eid + request_id`：

- Pause 重复请求：返回当前已暂停状态
- Cancel 重复请求：返回当前已取消状态

### 10.2.3 ACK 幂等

`HandlePauseAck / HandleCancelAck` 必须满足：

- operation 已 `ACKED` 时重复 ACK 直接忽略
- execution 已是目标状态时直接返回成功

## 10.3 ControlCompensator

新增：`TaskControlCompensator`

职责：

- 扫描 `task_control_operations` 中 `PENDING/SENT/TIMEOUT`
- 按 `next_retry_time` 重发 Pause / Cancel RPC
- 超过阈值后：
  - Pause：降级为“等待当前执行自然结束后停到 PAUSED”
  - Cancel：升级为 `Interrupt` 强制中断

## 10.4 与现有补偿器的协同

### RetryCompensator / RescheduleCompensator

应过滤掉：

- `PAUSED`
- `CANCELLED`

否则：

- 被用户暂停的任务可能又被系统自动恢复
- 被用户取消的任务可能又被系统自动重试

### InterruptCompensator

继续保留，但角色变为：

- 系统级超时中断
- Cancel 超时后的最后兜底

### ShardingCompensator

要把：

- `PAUSED`
- `CANCELLED`

纳入聚合判断。

## 11. 对当前代码必须顺手修的两个问题

## 11.1 `FAILED_RESCHEDULED` 没被 `IsValid()` 认进去

当前 `internal/domain/task_execution.go`：

```go
func (t TaskExecutionStatus) IsValid() bool {
    switch t {
    case TaskExecutionStatusPrepare,
        TaskExecutionStatusRunning,
        TaskExecutionStatusSuccess,
        TaskExecutionStatusFailed,
        TaskExecutionStatusFailedRetryable:
        return true
    default:
        return false
    }
}
```

这里居然没把 `FAILED_RESCHEDULED` 算进去。

而当前 MQ / gRPC 上报链路会先校验 `IsValid()`。

这意味着：

- executor 上报 `FAILED_RESCHEDULED`
- consumer 可能直接拒绝

这本身就是一个前置 bug。

Pause / Resume 方案如果继续复用 reschedule checkpoint 思路，这里必须先修。

## 11.2 Release 不能再写死用本地 `nodeID`

当前 `execution_service.go`：

```go
func (s *executionService) releaseTask(ctx context.Context, task domain.Task) {
    if err := s.taskAcquirer.Release(ctx, task.ID, s.nodeID); err != nil {
        ...
    }
}
```

这个写法有隐患。

更稳的做法应该是：

```go
task.ScheduleNodeID
```

因为真正持锁的是 task 上记录的 `schedule_node_id`，不是“当前处理这条报告/ACK 的节点”。

Pause / Cancel 控制链做出来以后，这个问题会被放大，必须一起修。

至少补下面 10 组：

1. `ACTIVE` 任务暂停，直接进入 `PAUSED`
2. `RUNNING` 任务暂停，executor 返回 checkpoint，execution -> `PAUSED`

7. Pause 请求重复提交，保证幂等
8. Cancel ACK 重复上报，保证幂等
9. 分片任务 Pause，父子任务最终都能收口
10. Plan Pause 后，`NextStep` 不再继续推进

额外建议再补两组故障用例：

11. Pause RPC 超时，`TaskControlCompensator` 能重发
12. Cancel RPC 超时，最终降级到 `Interrupt`

## 14. 推荐落地顺序

不要一把梭。

建议按这个顺序落：

### Phase 1：最小闭环

1. 扩 Task / Execution 状态枚举
2. 修 `IsValid()` / `IsTerminalStatus()`
3. 引入 `desired_status`
4. 跑通 `ACTIVE -> PAUSED -> ACTIVE`

### Phase 2：运行中 Pause / Cancel

5. 新增 `task_control_operations`
6. 新增 `Pause/Cancel RPC`
7. 跑通运行中任务的 Pause ACK / Cancel ACK

### Phase 3：恢复与补偿

8. Resume 复用 `Reschedule` 路径
9. 新增 `TaskControlCompensator`
10. 修 Sharding / Plan 联动

## 15. 最后结论

如果只看“功能名”，任务暂停 / 恢复 / 取消像个普通后台能力。

但就这个项目来说，它其实是个很值钱的技术点，因为它会把这些东西一次性串起来：

- Task 与 Execution 的职责边界
- 用户控制语义与系统补偿语义的分离
- 本地事务 + outbox
- 幂等请求
- checkpoint 恢复
- Release / Renew / Acquire 的一致性
- 分片父子聚合
- DAG 冻结推进
- 为什么不用 2PC

所以这块一旦设计扎实，`distributed_task_platform` 的“调度内核成熟度”会明显上一个档。

这不是锦上添花，是核心亮点。
