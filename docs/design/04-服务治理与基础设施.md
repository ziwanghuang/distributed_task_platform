## 第十五章 自定义 gRPC 负载均衡器

### 15.1 设计背景与需求分析

#### 15.1.1 gRPC 原生负载均衡的局限性

gRPC 原生提供的负载均衡策略（如 `pick_first`、`round_robin`）虽然适用于通用的 RPC 调用场景，但在分布式任务调度平台中面临以下局限：

1. **无法精确路由到指定节点**：任务重试（Retry）和重新调度（Reschedule）场景需要将任务分别发送到"相同节点"或"不同节点"执行
2. **无法排除特定节点**：当某个节点执行任务失败后进行重新调度时，需要排除该节点
3. **无法感知业务级别的节点ID**：原生负载均衡器只感知 IP:Port，无法关联到业务层面的"执行器节点ID"

#### 15.1.2 设计方案概述

平台设计了名为 `routing_round_robin` 的自定义 gRPC 负载均衡器，实现了三优先级的路由选择策略：

```
优先级1：指定节点路由（Specific）→ 通过 context 传递指定 nodeID
优先级2：排除节点路由（Excluded）→ 通过 context 传递排除 nodeID
优先级3：标准轮询路由（RoundRobin）→ 无任何路由提示时的默认策略
```

### 15.3 Context API 设计

#### 15.3.1 types.go 源码解析

```go
package balancer

import "context"

// RoutingRoundRobinName 是路由式轮询负载均衡器的名称
const RoutingRoundRobinName = "routing_round_robin"

// contextKey 自定义类型，避免与其他包的 context key 冲突
type contextKey string

const ExcludedNodeIDContextKey contextKey = "excluded_node_id"
const SpecificNodeIDContextKey contextKey = "specific_node_id"

// WithExcludedNodeID 在 context 中设置要排除的节点 ID
func WithExcludedNodeID(ctx context.Context, nodeID string) context.Context {
    if nodeID == "" {
        return ctx  // 空字符串不写入，避免误触发排除逻辑
    }
    return context.WithValue(ctx, ExcludedNodeIDContextKey, nodeID)
}

// GetExcludeNode 安全地从 context 中读取排除节点ID
func GetExcludeNode(ctx context.Context) (string, bool) {
    nodeID, ok := ctx.Value(ExcludedNodeIDContextKey).(string)
    return nodeID, ok && nodeID != ""  // 双重校验：类型断言成功 + 非空字符串
}

// WithSpecificNodeID / GetSpecificNodeID 同理
```

#### 15.3.2 设计要点

| 设计决策 | 原因 |
|---------|------|
| 使用自定义 `contextKey` 类型而非 `string` | 避免与其他包的 context key 产生命名冲突 |
| 空字符串不写入 context | 防止无意义的空值触发路由逻辑 |
| `Get*` 函数返回 `(string, bool)` | 遵循 Go 惯例，调用方可通过第二个返回值判断是否存在 |
| 双重校验（类型断言 + 非空检查） | 即使 context 中存储了空字符串，也视为无效 |

#### 15.3.3 使用场景示例

```go
// 场景1：智能调度 - 指定 Prometheus 选出的最优节点
func (s *Scheduler) newContext(task domain.Task) context.Context {
    if nodeID, err := s.executorNodePicker.Pick(s.ctx, task); err == nil && nodeID != "" {
        return balancer.WithSpecificNodeID(s.ctx, nodeID)  // → 优先级1
    }
    return s.ctx  // → 优先级3（默认轮询）
}

// 场景2：重新调度 - 排除上次失败的节点
func (r *NormalTaskRunner) Reschedule(ctx context.Context, task domain.Task) error {
    // 排除上次执行失败的节点
    ctx = balancer.WithExcludedNodeID(ctx, task.LastExecutedNodeID)  // → 优先级2
    return r.dispatch(ctx, task)
}
```

### 15.5 Balancer 核心实现

#### 15.5.1 数据结构设计

`routingBalancer` 维护了 **四层映射关系**，这是整个负载均衡器的核心数据结构：

```go
type routingBalancer struct {
    cc balancer.ClientConn     // 向 gRPC 框架报告状态
    mu sync.RWMutex            // 保护所有映射的并发安全

    // 层1：正向映射 - 从解析器地址到 SubConn
    subConnMap map[resolver.Address]balancer.SubConn

    // 层2：反向映射 - 从 SubConn 到解析器地址（用于状态回调的O(1)查找）
    scToAddrMap map[balancer.SubConn]resolver.Address

    // 层3：节点映射 - 从解析器地址到业务节点ID
    nodeIDMap map[resolver.Address]string

    // 层4：就绪集合 - 只包含 READY 状态的连接（Picker 的唯一数据源）
    readySCs map[balancer.SubConn]string
}
```

**为什么需要反向映射 `scToAddrMap`？**

```
场景：gRPC 框架调用 UpdateSubConnState(sc, state)
     只传入 SubConn，不传入 Address

没有反向映射 → 需要遍历 subConnMap 找到对应的 Address：O(n)
有了反向映射 → 直接 scToAddrMap[sc] 查找：O(1)

在高并发场景下，连接状态变化频繁，O(1) 查找性能优势显著。
```

#### 15.5.2 UpdateClientConnState —— 地址列表变更处理

当服务发现（Resolver）推送新的地址列表时，gRPC 框架会调用此方法：

```go
func (b *routingBalancer) UpdateClientConnState(state balancer.ClientConnState) error {
    b.mu.Lock()
    defer b.mu.Unlock()

    // Step 1：构建新地址集合（用于快速查找）
    newAddrs := make(map[resolver.Address]struct{})
    for _, addr := range state.ResolverState.Addresses {
        newAddrs[addr] = struct{}{}
    }

    // Step 2：删除已下线的连接（存在于旧列表但不在新列表中）
    for addr, sc := range b.subConnMap {
        if _, ok := newAddrs[addr]; !ok {
            sc.Shutdown()                 // 关闭物理连接
            delete(b.subConnMap, addr)    // 清理正向映射
            delete(b.scToAddrMap, sc)     // 清理反向映射
            delete(b.nodeIDMap, addr)     // 清理节点映射
            // readySCs 会在 UpdateSubConnState(Shutdown) 中自动清理
        }
    }

    // Step 3：为新上线的地址创建连接
    for addr := range newAddrs {
        if _, ok := b.subConnMap[addr]; ok {
            continue  // 已存在，跳过
        }
        sc, err := b.cc.NewSubConn([]resolver.Address{addr}, balancer.NewSubConnOptions{})
        if err != nil {
            continue
        }
        b.subConnMap[addr] = sc
        b.scToAddrMap[sc] = addr
        b.nodeIDMap[addr] = b.extractNodeID(addr)  // 从 Attributes 提取节点ID
        sc.Connect()  // 异步发起连接 → 后续触发 UpdateSubConnState
    }

    // 注意：此处不调用 updatePicker，因为连接状态尚未确定
    // Picker 更新完全由 UpdateSubConnState 驱动
    return nil
}
```

**关键设计决策：不在 UpdateClientConnState 中更新 Picker**

```
为什么？

当新连接创建后，它的初始状态是 CONNECTING（正在建立 TCP 连接），
而不是 READY（连接就绪可用）。

如果此时就更新 Picker，那么 Picker 中会包含尚未就绪的连接，
客户端请求可能会被分配到这些连接上，导致请求失败。

正确做法：等待 gRPC 框架调用 UpdateSubConnState(sc, READY) 后，
再更新 Picker，确保 Picker 中的每一个连接都是可用的。
```

#### 15.5.3 UpdateSubConnState —— 连接状态变更处理

```go
func (b *routingBalancer) UpdateSubConnState(sc balancer.SubConn, state balancer.SubConnState) {
    b.mu.Lock()
    defer b.mu.Unlock()

    // 使用反向映射 O(1) 查找
    addr, ok := b.scToAddrMap[sc]
    if !ok {
        return  // 连接可能已被 UpdateClientConnState 移除
    }

    switch state.ConnectivityState {
    case connectivity.Ready:
        // 连接就绪 → 加入可用集合
        b.readySCs[sc] = b.nodeIDMap[addr]

    case connectivity.Idle, connectivity.Connecting, connectivity.TransientFailure:
        // 连接不可用 → 从可用集合移除
        delete(b.readySCs, sc)

    case connectivity.Shutdown:
        // 连接已关闭 → 彻底清理所有映射
        delete(b.subConnMap, addr)
        delete(b.scToAddrMap, sc)
        delete(b.nodeIDMap, addr)
        delete(b.readySCs, sc)
    }

    // 每次连接状态变化都重建 Picker
    b.updatePicker()
}
```

**连接状态转换图：**

```
                    ┌─────────┐
           ┌───────│  IDLE   │───────┐
           │       └────┬────┘       │
           │            │ Connect()  │
           │            ▼            │
           │    ┌─────────────┐      │
           │    │ CONNECTING  │      │
           │    └──────┬──────┘      │
           │           │             │
      成功 │           │ 失败        │
           │           ▼             │
           │  ┌──────────────────┐   │
           │  │ TRANSIENT_FAILURE│───┘  (自动重连)
           │  └──────────────────┘
           │
           ▼
      ┌─────────┐
      │  READY  │ ←── 只有这个状态的连接才会进入 readySCs
      └────┬────┘
           │
           ▼
      ┌──────────┐
      │ SHUTDOWN │ ←── 彻底清理
      └──────────┘
```

#### 15.5.4 extractNodeID —— 节点ID提取机制

```go
func (b *routingBalancer) extractNodeID(addr resolver.Address) string {
    if addr.Attributes != nil {
        if nodeIDVal := addr.Attributes.Value("nodeID"); nodeIDVal != nil {
            if nodeIDStr, ok := nodeIDVal.(string); ok {
                return nodeIDStr
            }
        }
    }
    // 兜底策略：使用网络地址作为节点标识
    return addr.Addr
}
```

这个方法连接了**服务发现层**和**负载均衡层**。Resolver 在解析服务实例时，将业务节点ID注入到 `resolver.Address.Attributes` 中：

### 15.6 Picker 核心实现

#### 15.6.1 三优先级路由算法

```go
func (p *routingPicker) Pick(info balancer.PickInfo) (balancer.PickResult, error) {
    if len(p.subConns) == 0 {
        return balancer.PickResult{}, balancer.ErrNoSubConnAvailable
    }

    // ═══════════════════════════════════════════════════════
    // 优先级1：指定节点路由（Specific）
    // 使用场景：智能调度选择了最优节点后，强制路由到该节点
    // ═══════════════════════════════════════════════════════
    specificNodeID, hasSpecific := GetSpecificNodeID(info.Ctx)
    if hasSpecific {
        for i := range p.nodeIDs {
            if p.nodeIDs[i] == specificNodeID {
                return balancer.PickResult{SubConn: p.subConns[i], Done: nil}, nil
            }
        }
        // 指定的节点不可用 → 返回错误（而非降级轮询）
        return balancer.PickResult{}, status.Errorf(codes.Unavailable,
            "指定的节点不可用: %s", specificNodeID)
    }

    // ═══════════════════════════════════════════════════════
    // 优先级2：排除节点路由（Excluded）
    // 使用场景：任务重新调度时，排除上次失败的节点
    // ═══════════════════════════════════════════════════════
    excludeNodeID, hasExclude := GetExcludeNode(info.Ctx)
    if !hasExclude || len(p.subConns) == 1 {
        return p.pickRoundRobin(), nil
    }

    // 构建排除后的候选列表
    candidateIndexes := make([]int, 0, len(p.subConns))
    for i, nodeID := range p.nodeIDs {
        if nodeID != excludeNodeID {
            candidateIndexes = append(candidateIndexes, i)
        }
    }

    if len(candidateIndexes) == 0 {
        // 所有节点都被排除 → 降级为普通轮询（容错）
        if len(p.subConns) > 0 {
            return p.pickRoundRobin(), nil
        }
        return balancer.PickResult{}, status.Errorf(codes.Unavailable,
            "所有可用节点都被排除，排除节点: %s", excludeNodeID)
    }

    // 在候选列表中进行轮询
    next := atomic.AddUint32(&p.next, 1)
    selectedIdx := int(next-1) % len(candidateIndexes)
    return balancer.PickResult{SubConn: p.subConns[candidateIndexes[selectedIdx]]}, nil
}
```

#### 15.6.2 原子轮询实现

```go
func (p *routingPicker) pickRoundRobin() balancer.PickResult {
    next := atomic.AddUint32(&p.next, 1)  // CAS原子递增
    idx := int(next-1) % len(p.subConns)  // next-1：因为 AddUint32 返回新值
    return balancer.PickResult{
        SubConn: p.subConns[idx],
        Done:    func(_ balancer.DoneInfo) {},  // 可用于请求完成后的回调
    }
}
```

**为什么使用 `atomic.AddUint32` 而非 `sync.Mutex`？**

```
Picker.Pick() 在每次 RPC 调用时都会被调用，是极高频操作。

Mutex: 每次 Pick 都需要 Lock/Unlock，在高并发下产生锁竞争
Atomic: 使用 CPU 级别的 CAS 指令，无锁实现，吞吐量更高

代价：uint32 在 4,294,967,295 次调用后会回绕（overflow），
但取模运算使得回绕后的行为与正常情况一致，不影响正确性。
```

#### 15.6.3 容错设计分析

| 异常场景 | 处理策略 | 设计原因 |
|---------|---------|---------|
| 指定节点不可用 | 返回错误 | 智能调度明确选择了该节点，不应偷偷降级 |
| 排除后无可用节点 | 降级为普通轮询 | 有总比没有好，至少能执行任务 |
| 连接列表为空 | 返回 ErrNoSubConnAvailable | gRPC 框架会等待连接恢复后重试 |
| 只有一个连接且需排除 | 使用该唯一连接 | 单连接场景下排除无意义 |

### 15.7 与标准 gRPC 负载均衡器的对比

| 特性 | `round_robin` | `pick_first` | `routing_round_robin` |
|------|--------------|-------------|----------------------|
| 路由策略 | 纯轮询 | 始终选第一个 | 指定/排除/轮询三优先级 |
| 业务节点感知 | ❌ | ❌ | ✅ |
| Context 路由 | ❌ | ❌ | ✅ |
| 连接管理 | gRPC 内置 | gRPC 内置 | 自定义四层映射 |
| 适用场景 | 通用 RPC | 简单场景 | 任务调度精确路由 |

### 16.2 Registry 接口设计

```go
// pkg/grpc/registry/types.go

type Registry interface {
    // Register 注册服务实例（带有丰富的元数据）
    Register(ctx context.Context, si ServiceInstance) error
    // UnRegister 注销服务实例
    UnRegister(ctx context.Context, si ServiceInstance) error

    ListServices(ctx context.Context, name string) ([]ServiceInstance, error)
```

#### 16.2.1 ServiceInstance 扩展字段设计

`ServiceInstance` 不仅包含基础的服务地址信息，还携带了**动态权重分片**所需的元数据：

```go
// 这些字段用于 WeightedDynamicRangeShardingRule
Weight       int64   // 节点权重 → 决定分片分配比例
InitCapacity int64   // 初始容量 → 分片大小的基准值
```

```go
type Registry struct {
```

#### 16.3.2 服务注册 —— 基于 Lease 的生命周期管理

```go
func (r *Registry) Register(ctx context.Context, si registry.ServiceInstance) error {

    if err != nil {
        return err

    return err
}

}
```

**Lease 自动续约机制：**

```
┌──────────────┐
│  Executor 启动 │
│  创建 Session  │──────▶ etcd 创建 Lease（TTL=15s）
└──────┬───────┘
       │
       ▼
┌──────────────┐        ┌──────────────┐
│ Register()   │──Put──▶│ etcd Key     │
│ 绑定 Lease   │        │ /services/   │
```

#### 16.3.3 服务发现 —— Watch 机制

```go
func (r *Registry) Subscribe(name string) <-chan registry.Event {
    ctx, cancel := context.WithCancel(context.Background())
    ctx = clientv3.WithRequireLeader(ctx)  // 只从 Leader 读取，保证数据一致性
    r.mutex.Lock()
    r.watchCancel = append(r.watchCancel, cancel)
    r.mutex.Unlock()

    ch := r.client.Watch(ctx, r.serviceKey(name), clientv3.WithPrefix())
    res := make(chan registry.Event)
    go func() {
        for {
            select {
            case resp := <-ch:
                if resp.Canceled { return }
                if resp.Err() != nil { continue }
                for _, event := range resp.Events {
                    res <- registry.Event{
                        Type: typesMap[event.Type],  // PUT → Add, DELETE → Delete
                    }
                }
            case <-ctx.Done():
                return
            }
        }
    }()
    return res
}
```

```go
}
```

### 17.2 Prometheus Picker —— 基于实时指标的节点选择

#### 17.2.1 核心算法：topK + 随机选择

```go
// internal/service/picker/prometheus_picker.go

func (p *prometheusPicker) pickByMetric(ctx context.Context, metricName string) (string, error) {
    queryCtx, cancel := context.WithTimeout(ctx, p.config.QueryTimeout)
    defer cancel()

    // 构建 PromQL 查询
    // 示例（CPU优先）：
    //   topk(3, avg_over_time(executor_cpu_idle_percent[1m])
    //         AND ON(instance) up{job="executors"} == 1)
    query := fmt.Sprintf(
        `topk(%d, avg_over_time(%s[%s]) AND ON(instance) up{job=%q} == 1)`,
        p.config.TopNCandidates,     // 3
        metricName,                  // executor_cpu_idle_percent
        p.config.TimeWindow.String(), // 1m
        p.config.JobName,            // executors
    )

    result, _, err := p.promClient.Query(queryCtx, query, time.Now())
    if err != nil {
        return "", fmt.Errorf("查询 %s 指标失败: %w", metricName, err)
    }

    nodeIDs, err := p.parseQueryResult(result)
    if len(nodeIDs) == 0 {
        return "", fmt.Errorf("未找到可用的执行节点")
    }

    // 从 Top-K 中随机选择一个（避免所有调度器都选择同一个"最优"节点）
    selectedNodeID := nodeIDs[rand.Intn(len(nodeIDs))]
    return selectedNodeID, nil
}
```

**PromQL 查询逐层分析：**

```
topk(3,                                    ← 选取指标值最大的 3 个节点
    avg_over_time(                         ← 对时间窗口内的数据取平均（平滑抖动）
        executor_cpu_idle_percent[1m]      ← CPU 空闲率指标，1分钟窗口
    )
    AND ON(instance)                       ← 按 instance 标签关联
    up{job="executors"} == 1               ← 只选择在线的执行器节点
)
```

**为什么是 topK + 随机 而不是直接选最优？**

```
场景：3 个调度器节点同时调度

如果都选 top-1（最优节点）：
  → 所有任务都发往同一个节点 → 该节点过载 → 性能恶化

topK(3) + 随机选择：
  → 3 个调度器各自从 3 个候选中随机选 → 负载分散
  → 既选择了"较好"的节点，又避免了"扎堆"
  → 实现了"资源感知"和"负载均衡"的平衡
```

### 18.1 LoadChecker 接口

### 18.2 三层检查器详解

#### 18.2.1 第一层：LimiterChecker —— 令牌桶限流

```go
// pkg/loadchecker/limiter.go

type LimiterChecker struct {
    limiter      *rate.Limiter   // golang.org/x/time/rate
    waitDuration time.Duration
}

func (c *LimiterChecker) Check(_ context.Context) (sleepDuration time.Duration, shouldSchedule bool) {
    reservation := c.limiter.Reserve()
    if reservation.OK() {
        return 0, true  // 有令牌，继续调度
    }

    // 令牌桶已空
    sleepDuration = reservation.Delay()  // 精确计算下一个令牌的等待时间
    reservation.Cancel()                 // 取消预留（不消耗令牌）

    if c.waitDuration > 0 && sleepDuration < c.waitDuration {
        sleepDuration = c.waitDuration   // 最小等待时间保护
    }
    return sleepDuration, false
}
```

**配置示例：**

```yaml
limiter:
  rateLimit: 10.0     # 每秒 10 次调度
  burstSize: 20       # 突发容量 20
  waitDuration: 100ms  # 最小等待时间
```

**作用：** 硬性限制调度频率上限，无论系统负载如何，都不会超过配置的 QPS。

#### 18.2.2 第二层：DatabaseLoadChecker —— 数据库负载感知

```go
// pkg/loadchecker/database.go

type DatabaseLoadChecker struct {
    promClient      v1.API
    threshold       time.Duration  // 数据库响应时间阈值
    timeWindow      time.Duration

        `]) / rate(gorm_query_duration_seconds_count[` + c.timeWindow.String() + `]))`

    // ... 解析结果
}
```

#### 18.2.3 第三层：ClusterLoadChecker —— 集群级别负载均衡

```go
// pkg/loadchecker/cluster.go

type ClusterLoadChecker struct {
    nodeID             string
    promClient         v1.API
    thresholdRatio     float64       // 1.2 表示超过集群平均 20%
    timeWindow         time.Duration
    slowdownMultiplier float64       // 降频乘数
    minBackoffDuration time.Duration
}

func (c *ClusterLoadChecker) Check(ctx context.Context) (sleepDuration time.Duration, shouldSchedule bool) {
    currentDuration := GetExecutionTime(ctx)  // 从 context 获取本次执行时间
    if currentDuration == 0 {
        return 0, true  // 无执行数据，允许调度
    }

    // 查询集群所有节点的平均调度循环耗时
    avgDuration, err := c.queryClusterAverageTime(ctx)
    if err != nil || avgDuration == 0 {
        return 0, true  // 降级策略
    }

    // 计算当前节点的性能比值
    ratio := float64(currentDuration) / float64(avgDuration)

    if ratio > c.thresholdRatio {
        // 当前节点性能低于集群平均 → 需要降频
        backoffDuration := time.Duration(float64(currentDuration) * c.slowdownMultiplier)
        if backoffDuration < c.minBackoffDuration {
            backoffDuration = c.minBackoffDuration
        }
        return backoffDuration, false
    }
    return 0, true
}
```

**集群负载均衡的核心思想：**

```
集群平均调度循环耗时 = avg(所有节点的 scheduler_loop_duration_seconds)

当前节点耗时 / 集群平均 > 1.2 (threshold)
    → 说明当前节点比集群平均慢 20%
    → 可能原因：CPU 负载高、IO 繁忙、网络延迟
    → 操作：降低调度频率，给当前节点"喘息"的机会
    → 同时其他节点会承担更多调度，实现集群级别的自适应负载均衡
```

### 18.3 CompositeChecker —— 组合模式

```go
// pkg/loadchecker/composite.go

type CompositeStrategy int

const (
    StrategyAND CompositeStrategy = iota  // 所有检查器都通过
    StrategyOR                             // 任一检查器通过
)

type CompositeChecker struct {
    checkers []LoadChecker
    strategy CompositeStrategy
}
```

#### 18.3.1 AND 策略

```go
func (c *CompositeChecker) checkAND(ctx context.Context) (sleepDuration time.Duration, shouldSchedule bool) {
    var maxDuration time.Duration
    for _, checker := range c.checkers {
        duration, ok := checker.Check(ctx)
        if !ok {
            if duration > maxDuration {
                maxDuration = duration  // 取最大睡眠时间
            }
        }
    }
    return maxDuration, maxDuration == 0  // 全部通过才调度
}
```

**AND 语义：** 所有检查器都说"可以"才能调度。只要有一个说"不行"，就需要等待。等待时间取所有"不行"中的最大值。

#### 18.3.2 OR 策略

```go
func (c *CompositeChecker) checkOR(ctx context.Context) (sleepDuration time.Duration, shouldSchedule bool) {
    minDuration := time.Hour
    allFailed := true
    for _, checker := range c.checkers {
        duration, ok := checker.Check(ctx)
        if ok {
            return 0, true  // 有一个通过就可以调度
        }
        allFailed = false
        if duration < minDuration {
            minDuration = duration
        }
    }
    if allFailed && minDuration == time.Hour {
        minDuration = time.Minute  // 兜底
    }
    return minDuration, false
}
```

## 第十九章 ShardingLoopJob 分库分表循环调度框架

### 19.1 设计背景

在 V2 架构中，数据库采用分库分表方案。调度器、补偿器等需要遍历所有分库分表执行业务逻辑。`ShardingLoopJob` 是一个通用框架，封装了以下关键问题：

1. **分布式锁互斥**：确保同一个分表在同一时刻只有一个节点处理
2. **资源信号量控制**：限制单节点同时处理的分表数量
3. **锁续约与释放**：自动续约 + 异常时安全释放
4. **优雅退出**：支持 context 取消

### 19.2 核心架构

```
ShardingLoopJob
    │
    ├── ShardingStrategy.Broadcast()  → 获取所有分库分表列表
    │       [db_0/tab_0, db_0/tab_1, db_1/tab_0, db_1/tab_1, ...]
    │
    ├── ResourceSemaphore.Acquire()   → 信号量控制并发数
    │
    ├── dlock.Client.NewLock()        → 为每个分表创建分布式锁
    │       key = "baseKey:db_0:tab_0"
    │
    ├── lock.Lock()                   → 尝试获取锁
    │
    └── go tableLoop(ctx, lock)       → 获取锁成功后，启动 goroutine 处理
            │
            ├── bizLoop(ctx, lock)    → 循环执行业务
            │       ├── biz(bizCtx)   → 执行实际业务逻辑
            │       └── lock.Refresh() → 续约分布式锁
            │
            ├── lock.Unlock()         → 释放锁（即使 ctx 取消也尝试释放）
            │
            └── resourceSemaphore.Release() → 释放信号量
```

### 19.3 Run 方法 —— 主循环

```go
func (l *ShardingLoopJob) Run(ctx context.Context) {
    for {
        // 遍历所有分库分表组合
        for _, dst := range l.shardingStrategy.Broadcast() {
            // Step 1: 检查资源信号量
            err := l.resourceSemaphore.Acquire(ctx)
            if err != nil {
                // 信号量已满 → 当前节点处理的分表数达到上限
                time.Sleep(l.retryInterval)
                continue
            }

            // Step 2: 创建分布式锁（每个分表一把锁）
            key := l.generateKey(dst.DB, dst.Table)
            lock, err := l.dclient.NewLock(ctx, key, l.retryInterval)
            if err != nil {
                l.resourceSemaphore.Release(ctx)  // 失败时释放信号量
                continue
            }

            // Step 3: 尝试获取锁（非阻塞，超时后继续）
            lockCtx, cancel := context.WithTimeout(ctx, l.defaultTimeout)
            err = lock.Lock(lockCtx)
            cancel()
            if err != nil {
                l.resourceSemaphore.Release(ctx)  // 锁被其他节点持有
                continue
            }

            // Step 4: 获取锁成功，启动独立 goroutine 处理该分表
            go l.tableLoop(sharding.CtxWithDst(ctx, dst), lock)
        }
    }
}
```

### 19.4 tableLoop —— 单分表处理循环

```go
func (l *ShardingLoopJob) tableLoop(ctx context.Context, lock dlock.Lock) {
    defer func() {
        _ = l.resourceSemaphore.Release(ctx)  // 无论如何都释放信号量
    }()

    // 执行业务循环
    err := l.bizLoop(ctx, lock)

    // 释放分布式锁
    // 关键：使用 Background Context，因为原始 ctx 可能已被取消
    unCtx, cancel := context.WithTimeout(context.Background(), l.defaultTimeout)
    unErr := lock.Unlock(unCtx)
    cancel()

    // 判断退出原因
    err = ctx.Err()
    switch {
    case errors.Is(err, context.Canceled), errors.Is(err, context.DeadlineExceeded):
        return  // 正常退出（如系统关闭）
    default:
        time.Sleep(l.retryInterval)  // 异常退出，等待后可被重新抢占
    }
}
```

**为什么 Unlock 使用 `context.Background()` 而非原始 ctx？**

```
场景：ctx 因系统关闭而被取消

如果使用 ctx 调用 Unlock：
    ctx.Done() → Unlock 立即返回 → 锁无法释放
    → 其他节点必须等锁过期才能接管（可能需要几十秒）

使用 context.Background()：
    即使原始 ctx 已取消，Unlock 仍有 defaultTimeout 秒来完成
    → 锁可以被快速释放
    → 其他节点可以立即接管
```

### 19.5 bizLoop —— 业务执行与锁续约

```go
func (l *ShardingLoopJob) bizLoop(ctx context.Context, lock dlock.Lock) error {
    const bizTimeout = 50 * time.Second
    for {
        // 1. 执行业务（带超时保护）
        bizCtx, cancel := context.WithTimeout(ctx, bizTimeout)
        err := l.biz(bizCtx)
        cancel()

        // 2. 检查是否需要退出
        if ctx.Err() != nil {
            return ctx.Err()
        }

        // 3. 续约分布式锁
        refCtx, cancel := context.WithTimeout(ctx, l.defaultTimeout)
        err = lock.Refresh(refCtx)
        cancel()
        if err != nil {
            return fmt.Errorf("分布式锁续约失败 %w", err)
        }
        // 4. 继续下一次循环
    }
}
```

**时间约束关系：**

```
分布式锁 TTL（retryInterval = 60s）
    > bizTimeout（50s）
        > 单次业务执行时间

这确保了：
1. 业务执行期间锁不会过期
2. 每次业务执行后都有时间续约
3. 如果续约失败，说明锁已经丢失，必须退出
```

### 19.6 ResourceSemaphore —— 资源信号量

```go
// pkg/loopjob/resource.go

type ResourceSemaphore interface {
    Acquire(ctx context.Context) error
    Release(ctx context.Context) error
}

type MaxCntResourceSemaphore struct {
    maxCount int      // 最大并发数
    curCount int      // 当前持有数
    mu       *sync.RWMutex
}

func (r *MaxCntResourceSemaphore) Acquire(context.Context) error {
    r.mu.Lock()
    defer r.mu.Unlock()
    if r.curCount >= r.maxCount {
        return errs.ErrExceedLimit  // 达到上限
    }
    r.curCount++
    return nil
}

func (r *MaxCntResourceSemaphore) Release(context.Context) error {
    r.mu.Lock()
    defer r.mu.Unlock()
    r.curCount--
    return nil
}

// 支持运行时动态调整最大并发数
func (r *MaxCntResourceSemaphore) UpdateMaxCount(maxCount int) {
    r.mu.Lock()
    defer r.mu.Unlock()
    r.maxCount = maxCount
}
```

**为什么不用 Go 标准库的 `semaphore.Weighted`？**

```
1. MaxCntResourceSemaphore 支持 UpdateMaxCount() 动态调整
   → 可以根据运行时负载动态增减并发数
   → 标准库的 Weighted 在创建后不支持调整

2. Acquire 是非阻塞的
   → 达到上限直接返回错误，由调用方决定如何处理
   → 标准库的 Acquire 是阻塞的

3. 与 ShardingLoopJob 的错误处理模型更匹配
```

### 19.7 ShardingStrategy —— 分片路由

```go
// pkg/sharding/strategy.go

type ShardingStrategy struct {
    dbPrefix      string  // "db"
    tablePrefix   string  // "task"
    tableSharding int64   // 表分片数
    dbSharding    int64   // 库分片数
}

// Shard 计算给定 ID 应落入哪个分片
func (s ShardingStrategy) Shard(id int64) Dst {
    dbHash := id % s.dbSharding
    tabHash := (id / s.dbSharding) % s.tableSharding
    return Dst{
        Table: fmt.Sprintf("%s_%d", s.tablePrefix, tabHash),  // task_3
        DB:    fmt.Sprintf("%s_%d", s.dbPrefix, dbHash),      // db_1
    }
}

// Broadcast 返回所有分片的完整列表
func (s ShardingStrategy) Broadcast() []Dst {
    ans := make([]Dst, 0, s.tableSharding*s.dbSharding)
    for i := 0; i < int(s.dbSharding); i++ {
        for j := 0; j < int(s.tableSharding); j++ {
            ans = append(ans, Dst{
                Table: fmt.Sprintf("%s_%d", s.tablePrefix, j),
                DB:    fmt.Sprintf("%s_%d", s.dbPrefix, i),
            })
        }
    }
    return ans
}
```

**分片上下文传递：**

```go
// 将分片目标注入到 context 中
func CtxWithDst(ctx context.Context, dst Dst) context.Context {
    return context.WithValue(ctx, dstKey{}, dst)
}

// 业务层从 context 中获取当前处理的分片
func DstFromCtx(ctx context.Context) (Dst, bool) {
    val := ctx.Value(dstKey{})
    res, ok := val.(Dst)
    return res, ok
}

// 使用示例（在 SchedulerV2 中）：
go l.tableLoop(sharding.CtxWithDst(ctx, dst), lock)
// → 业务函数中可以通过 DstFromCtx(ctx) 知道当前处理的是哪个分表
```

#### 20.1.2 回调工厂模式

```go
// before 是一个高阶函数（工厂函数），返回一个 gorm 回调
func (p *GormMetricsPlugin) before(operation string) func(*gorm.DB) {
    return func(db *gorm.DB) {
        // 在 GORM 的上下文中存储开始时间和操作类型
        db.Set(callbackStartTimeKey, time.Now())
        db.Set(callbackOperationKey, operation)
    }
}

// after 是通用的结束回调，所有操作复用
func (p *GormMetricsPlugin) after(db *gorm.DB) {
    opVal, _ := db.Get(callbackOperationKey)
    operation, ok := opVal.(string)
    if !ok { return }
    p.recordMetrics(db, operation)
}

func (p *GormMetricsPlugin) recordMetrics(db *gorm.DB, operation string) {
    startTimeVal, exists := db.Get(callbackStartTimeKey)
    if !exists { return }
    startTime := startTimeVal.(time.Time)

    duration := time.Since(startTime)
    table := db.Statement.Table
    if table == "" { table = "unknown" }

    // 记录到 Prometheus Histogram
    p.queryDuration.WithLabelValues(operation, table).Observe(duration.Seconds())
}
```

```go
func (m *SchedulerMetrics) StartRecordExecutionTime() func() time.Duration {
    start := time.Now()
    return func() time.Duration {
        duration := time.Since(start)
        m.loopDuration.WithLabelValues(m.nodeID).Observe(duration.Seconds())
        return duration  // 返回值会传递给 ClusterLoadChecker
    }

    if duration > 0 {
        m.sleepDuration.WithLabelValues(m.nodeID, checkerType).Observe(duration.Seconds())
    }
}
```

## 本文档总结

本文档深入分析了分布式任务调度平台的服务治理与基础设施层，核心亮点包括：

### 1. 自定义 gRPC 负载均衡器（routing_round_robin）
- 三优先级路由：Specific → Excluded → RoundRobin
- 四层映射 + 反向映射实现 O(1) 状态查找
- 无锁原子轮询（atomic.AddUint32）
- 与 Context API 结合实现透明的业务级路由

### 2. 服务注册与发现
- etcd Lease + Session 实现自动注册/摘除
- Watch 机制实现实时服务列表更新
- 自定义 Resolver 桥接 Registry 和 gRPC Balancer
- ServiceInstance 携带分片元数据

### 3. 智能节点选择
- Prometheus PromQL 实时查询节点资源指标
- topK + 随机选择平衡"选优"和"均衡"
- Dispatcher 门面模式支持 CPU/内存优先策略
- 优雅降级：失败时回退为默认轮询

### 4. 多层负载检测
- 令牌桶（硬限流）+ 数据库负载 + 集群均衡
- CompositeChecker 组合模式支持 AND/OR 策略
- 三层协同实现精细化的自适应调度控制

### 5. ShardingLoopJob 框架
- 分布式锁 + 资源信号量 + 分片策略三位一体
- 锁续约、异常释放、优雅退出的完整生命周期
- 通过 Context 传递分片信息，解耦框架与业务

### 6. Prometheus 可观测性
- GORM 插件自动采集数据库操作耗时
- SchedulerMetrics 驱动 ClusterLoadChecker 自适应
- 形成"采集 → 存储 → 查询 → 决策"的数据闭环
