# 分布式任务调度平台 — 项目概览

> 企业级大数据任务编排与调度中台，支持离线计算、实时计算、数据同步等全场景任务调度

---

## 一、项目定位

### 1.1 项目背景

在大数据平台建设过程中，面临以下核心挑战：

| 痛点 | 具体表现 |
|------|---------|
| **任务种类繁多** | Spark 离线计算、Flink 实时计算、DataX 数据同步、Hive SQL 批处理、ML 训练任务 |
| **依赖关系复杂** | 数据清洗→转换→聚合→报表，多源同步完成后才能启动下游，需 DAG 工作流 |
| **资源调度不合理** | 高峰期堆积、低峰期闲置，无法根据集群负载智能调度，资源利用率 < 40% |
| **任务失败率高** | 网络抖动、节点宕机导致失败，缺乏自动重试和故障转移，运维成本高 |

### 1.2 解决方案

设计并实现 **企业级大数据任务调度中台**，核心能力一览：

| 能力 | 说明 | 业务价值 |
|-----|------|---------|
| **统一调度** | 支持 Spark / Flink / DataX / Hive / Python / Shell 等多种计算引擎 | 降低运维复杂度 |
| **智能编排** | 基于 ANTLR4 的 DAG 工作流引擎，支持复杂依赖 | 开发效率提升 10 倍 |
| **负载感知** | 基于 Prometheus 的智能调度算法（CPU×0.5 + Mem×0.3 + Tasks×0.2） | 资源利用率提升 45% |
| **高可用** | 四重补偿机制（重试/重调度/中断/分片），任务成功率 99.9% | 年减少业务损失 600 万+ |
| **海量存储** | 分库分表 8 库 16 表，支持千万级任务 | 日均 500 万+ 任务 |

## 二、技术架构

### 2.1 技术栈

| 类别 | 技术选型 | 选型理由 |
|------|---------|---------|
| **编程语言** | Go 1.21+ | 协程模型天然适合高并发，单二进制部署，内存占用低 |
| **通信协议** | gRPC + Protobuf | 高性能二进制协议，支持流式传输 |
| **服务治理** | etcd（服务发现） | Raft 协议保证强一致性，Watch 机制秒级感知 |
| **消息队列** | Kafka | 高吞吐持久化，支持消息回溯 |
| **数据存储** | MySQL（分库分表） + Redis | 关系型存储 + 分布式锁/缓存 |
| **监控** | Prometheus + Grafana + OpenTelemetry | 100+ 核心指标，全链路追踪 |
| **DSL 引擎** | ANTLR4 | 声明式工作流编排语言 |
| **依赖注入** | Wire | 编译时生成，无运行时反射开销 |

### 2.3 数据流

```
任务创建 → 调度器抢占 → 智能选择执行节点 → 分发到执行器 
→ 执行结果上报(Kafka) → 状态更新 → 事件通知 → 触发下游任务
```

## 三、多计算引擎支持

| 计算引擎 | 任务类型 | 调用方式 | 典型应用场景 |
|---------|---------|---------|-------------|
| **Spark** | 离线计算 | gRPC/HTTP | ETL、数据清洗、特征工程 |
| **Flink** | 实时计算 | gRPC/HTTP | 实时分析、流式处理 |
| **DataX** | 数据同步 | HTTP | 数据库同步、文件同步 |
| **Hive** | 批处理 | JDBC | 数据仓库查询、报表生成 |
| **Python** | 脚本任务 | gRPC/HTTP | 机器学习、数据分析 |
| **Shell** | 脚本任务 | gRPC/HTTP | 系统运维、文件处理 |

### 统一执行器适配器

```go
// 统一的大数据任务适配器接口
type BigDataAdapter interface {
    Name() string
    Submit(ctx context.Context, config *BigDataConfig) (applicationID string, err error)
    Query(ctx context.Context, applicationID string) (ExecutionState, error)
    Cancel(ctx context.Context, applicationID string) error
    GetLogs(ctx context.Context, applicationID string, lines int) (string, error)
}
```

新增计算引擎只需实现 `BigDataAdapter` 接口，通过 `BigDataInvoker` 统一调度，无需修改核心代码。

## 四、典型应用场景

### 4.1 数据仓库 ETL

```yaml
# 分层数据处理 — 每日凌晨自动执行
workflow:
  name: "daily_etl_workflow"
  schedule: "0 2 * * *"
  dsl: |
    (order_sync && user_sync && product_sync)  # ODS层并行采集
    -> data_cleaning                            # DWD层清洗
    -> (order_transform && user_transform)      # DWD层转换
    -> data_aggregate                           # DWS层聚合
    -> data_quality_check                       # 质量校验
    ? (daily_report && weekly_report)           # ADS层报表
    : alert_notification                        # 异常告警
```

**业务价值**：数据处理时间从 8h→4h，报表自动化，错误率从 5%→0.1%

### 4.2 实时数据分析

```yaml
# 实时用户行为分析
workflow:
  name: "realtime_user_behavior"
  type: "long_running"
  tasks:
    - name: "flink_streaming_job"
      type: "flink"
      params:
        parallelism: 8
        checkpoint_interval: "60s"
  health_check:
    interval: "30s"
    failure_threshold: 3
  restart_strategy:
    type: "fixed_delay"
    attempts: 3
```

**业务价值**：推荐响应时间秒级→毫秒级，风控准确率提升 20%

### 4.3 机器学习训练

```yaml
# 每日模型训练与部署
dsl: |
  feature_engineering -> model_training -> model_evaluation
  ? model_deployment
  : alert_notification

# 支持分片任务 — 5 个模型并行训练
model_training:
  type: "sharding"
  sharding_rule:
    total_shards: 5
  params:
    models: ["lr", "gbdt", "xgboost", "lightgbm", "catboost"]
```

**业务价值**：训练时间从 8h→2h，支持多模型并行训练和自动择优部署
