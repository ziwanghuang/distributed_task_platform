## 1. gRPC 通信协议

### ExecutorService（Scheduler → Executor）

```protobuf
service ExecutorService {
    rpc Execute(ExecuteRequest) returns (ExecuteResponse);    // 执行任务
    rpc Interrupt(InterruptRequest) returns (InterruptResponse); // 中断任务
    rpc Query(QueryRequest) returns (QueryResponse);          // 查询状态
    rpc Prepare(PrepareRequest) returns (PrepareResponse);    // 分片预查询
}
```

| 方法 | 触发者 | 说明 |
|------|--------|------|
| Execute | NormalTaskRunner / Retry / Reschedule | 下发任务到执行器 |
| Interrupt | InterruptCompensator | 中断超时任务 |
| Query | (预留) | 轮询执行状态 |
| Prepare | CreateShardingChildren | 查询业务总数（分片用） |

### ReporterService（Executor → Scheduler）

```protobuf
service ReporterService {
    rpc Report(ReportRequest) returns (ReportResponse);           // 单条上报
    rpc BatchReport(BatchReportRequest) returns (BatchReportResponse); // 批量上报
}
```

- **Report**：状态变更等关键事件 → Kafka `execution_report` topic
- **BatchReport**：进度更新等高频事件 → Kafka `execution_batch_report` topic

## 2. etcd 服务发现

**包路径**：`pkg/grpc/registry/etcd/`

### 注册流程

```
1. 创建 lease（TTL = 30s）
2. PUT /services/{serviceName}/{instanceID} → JSON(ServiceInstance)
3. KeepAlive(leaseID) → 自动续约
```

### ServiceInstance

```go
type ServiceInstance struct {
    ID          string  // 实例唯一标识
    ServiceName string  // 服务名
    Address     string  // gRPC 地址
    Weight      int64   // 节点权重（用于加权分片）
    // ...
}
```

### 故障自动注销

- lease TTL = 30s
- KeepAlive 失败 → lease 过期 → key 自动删除 → 60s 内完成故障注销

## 3. 自定义 gRPC Resolver

**包路径**：`pkg/grpc/resolver.go`

### 工作原理

```
etcd Watch(/services/{serviceName}/)
    │
    ▼
Resolver.watch() → 监听变更事件
    │
    ▼
将 ServiceInstance 转为 resolver.Address（携带 nodeID、weight 等 Attributes）
    │
    ▼
cc.UpdateState(resolver.State{Addresses: addrs})
    │
    ▼
gRPC 连接池自动更新
```

**优势**：相比 DNS Resolver，支持实时变更通知（Watch）和节点元信息（weight, nodeID）。

## 4. RoutingRoundRobin 负载均衡器

**包路径**：`pkg/grpc/balancer/`

### 三种路由模式

| 模式 | 触发条件 | 逻辑 |
|------|---------|------|
| 指定节点 | `ctx` 中有 `SpecificNodeID` | 精确匹配指定节点 |
| 排除节点 | `ctx` 中有 `ExcludedNodeID` | 过滤后轮询 |
| 默认轮询 | 无特殊指令 | `atomic.AddUint64` 轮询 |

### Context 辅助函数

```go
balancer.WithSpecificNodeID(ctx, nodeID)   // 设置指定节点
balancer.WithExcludedNodeID(ctx, nodeID)   // 设置排除节点
```

**使用场景**：

- Scheduler → `WithSpecificNodeID`（Picker 选出最优节点）
- RetryCompensator → `WithExcludedNodeID`（排除上次失败节点）
- ShardingTask → `WithSpecificNodeID`（分片亲和性路由）

## 5. gRPC Client Pool

**包路径**：`pkg/grpc/clients_v2.go`

### ClientsV2 泛型连接池

```go
type ClientsV2[T any] struct {
    pool    syncx.Map[string, T]              // service → client 映射
    factory func(serviceKey string) (T, error) // 连接创建工厂
}
```

### GetOrCreate 流程

```
1. pool.Load(serviceKey) → 有缓存直接返回（无锁快路径）
2. factory(serviceKey) → 创建新连接
3. pool.LoadOrStore → 原子存入（防止并发重复创建）
4. 如果已被别人存入 → 关闭多余连接，返回已有连接
```

## 6. 三级 LoadChecker

**包路径**：`pkg/loadchecker/`

### 接口

```go
type LoadChecker interface {
    Check(ctx context.Context) (sleepDuration time.Duration, shouldSchedule bool)
}
```

### 三级检查器

| 级别 | 实现 | 检查内容 | 说明 |
|------|------|---------|------|
| 1 | LimiterChecker | 令牌桶限流 | 硬限制 QPS 上限 |
| 2 | DatabaseLoadChecker | DB 查询延迟 | 自适应感知 MySQL 压力 |
| 3 | ClusterLoadChecker | 节点 vs 集群调度耗时 | 横向对比发现异常节点 |

### CompositeChecker（组合检查器）

```go
type CompositeChecker struct {
    checkers []LoadChecker
    strategy CompositeStrategy  // AND / OR
}
```

- **AND 策略**（默认）：所有检查器都通过才允许调度
  - 任一不通过 → 返回最大 sleepDuration
- **OR 策略**：任一检查器通过即允许调度
  - 全不通过 → 返回最小 sleepDuration

## 7. 配置参数

```yaml
loadChecker:
  limiter:
    rateLimit: 10.0           # 令牌桶 QPS
    burstSize: 20             # 突发容量
    waitDuration: 5s          # 限流时等待时间
  database:
    threshold: 100ms          # DB 响应时间阈值
    timeWindow: 5m            # 查询时间窗口
    backoffDuration: 10s      # 负载过高退避时间
  cluster:
    thresholdRatio: 1.2       # 超过均值 20% 触发降频
    timeWindow: 5m            # 查询时间窗口
    slowdownMultiplier: 2.0   # 降频乘数
    minBackoffDuration: 5s    # 最小退避时间
```
