# 架构优化路线

> 本文梳理平台的未来优化方向，按投入产出比划分为 P0~P3 四个优先级，并附大数据场景适配方案。

---

## 一、优化全景图

```
P0（立即实施，1~3 天）
  ├── 数据库连接池调优
  ├── 批量操作优化（消除 N+1）
  ├── 熔断器（gobreaker）
  └── 统一错误处理

P1（近期实施，3~7 天）
  ├── 本地缓存（Ristretto + Redis 两级）
  ├── 异步写入队列（AsyncWriter）
  ├── 增强监控告警（OpenTelemetry + Prometheus 规则）
  └── 完善单元测试（testify/mock + testcontainers）

P2（中期规划，1~2 周）
  ├── 配置中心（动态热更新）
  ├── API 网关（Kong/APISIX）
  ├── 插件化架构（TaskPlugin 接口）
  └── 代码生成工具（GORM Gen）

P3（长期规划，1+ 月）
  ├── 服务网格（Istio）
  ├── 多租户支持
  └── 大数据任务适配（Yarn/Flink/Spark/DataX）
```

### 2.2 批量操作优化

消除 N+1 查询问题：

```go
// ❌ 原始方式
for _, exec := range executions {
    task, _ := repo.FindTaskByID(ctx, exec.TaskID)
}

// ✅ 优化后
func (r *TaskRepository) FindByIDs(ctx context.Context, ids []int64) (map[int64]*domain.Task, error) {
    var tasks []*domain.Task
    err := r.db.WithContext(ctx).Where("id IN ?", ids).Find(&tasks).Error
    taskMap := make(map[int64]*domain.Task, len(tasks))
    for _, t := range tasks {
        taskMap[t.ID] = t
    }
    return taskMap, err
}
```

**收益**：DB 查询次数降低 90%，批量操作性能提升 10×。

### 2.3 熔断器

执行器节点故障时，调度器不应持续重试打垮下游。

```go
type CircuitBreakerInvoker struct {
    breakers sync.Map
    invoker  invoker.Invoker
}

func (cbi *CircuitBreakerInvoker) Invoke(ctx context.Context, req *InvokeRequest) error {
    cb, _ := cbi.breakers.LoadOrStore(req.ExecutorAddr, gobreaker.NewCircuitBreaker(gobreaker.Settings{
        Name:        req.ExecutorAddr,
        MaxRequests: 3,
        Interval:    10 * time.Second,
        Timeout:     60 * time.Second,
        ReadyToTrip: func(counts gobreaker.Counts) bool {
            return counts.Requests >= 3 && float64(counts.TotalFailures)/float64(counts.Requests) >= 0.6
        },
    }))

    _, err := cb.(*gobreaker.CircuitBreaker).Execute(func() (interface{}, error) {
        return nil, cbi.invoker.Invoke(ctx, req)
    })
    return err
}
```

**收益**：自动熔断故障节点，避免雪崩，可用性提升至 99.99%。

## 三、P1 — 近期实施

### 3.1 多级缓存架构

```
请求 → L1 本地缓存（Ristretto）→ L2 Redis → L3 数据库
         命中率 90%+              命中率 95%+     最终兜底
```

```go
type CacheManager struct {
    local *ristretto.Cache
    redis *redis.Client
}

func (cm *CacheManager) GetTask(ctx context.Context, taskID int64) (*domain.Task, error) {
    // L1 本地缓存
    if val, found := cm.local.Get(taskID); found {
        return val.(*domain.Task), nil
    }
    // L2 Redis
    val, err := cm.redis.Get(ctx, fmt.Sprintf("task:%d", taskID)).Result()
    if err == nil {
        task := &domain.Task{}
        json.Unmarshal([]byte(val), task)
        cm.local.Set(taskID, task, 1)
        return task, nil
    }
    // L3 数据库
    task, err := cm.repo.FindByID(ctx, taskID)
    if err != nil {
        return nil, err
    }
    // 回填
    data, _ := json.Marshal(task)
    cm.redis.Set(ctx, fmt.Sprintf("task:%d", taskID), data, 10*time.Minute)
    cm.local.Set(taskID, task, 1)
    return task, nil
}
```

**收益**：Redis 访问 -80%，DB 查询 -90%，热点读延迟 < 1ms。

### 3.2 异步写入队列

```go
type AsyncWriter struct {
    queue chan *domain.TaskExecution
    repo  repository.TaskExecutionRepository
}

func NewAsyncWriter(repo repository.TaskExecutionRepository) *AsyncWriter {
    aw := &AsyncWriter{queue: make(chan *domain.TaskExecution, 10000), repo: repo}
    for i := 0; i < 10; i++ {
        go aw.batchWrite()
    }
    return aw
}

func (aw *AsyncWriter) batchWrite() {
    ticker := time.NewTicker(100 * time.Millisecond)
    batch := make([]*domain.TaskExecution, 0, 100)
    for {
        select {
        case exec := <-aw.queue:
            batch = append(batch, exec)
            if len(batch) >= 100 {
                aw.repo.BatchUpdate(context.Background(), batch)
                batch = batch[:0]
            }
        case <-ticker.C:
            if len(batch) > 0 {
                aw.repo.BatchUpdate(context.Background(), batch)
                batch = batch[:0]
            }
        }
    }
}
```

**收益**：吞吐量提升 5~10×，P99 延迟降低 80%。

func TenantMiddleware(next http.Handler) http.Handler {
    return http.HandlerFunc(func(w http.ResponseWriter, r *http.Request) {
        tenantID := r.Header.Get("X-Tenant-ID")
        if tenantID == "" {
            http.Error(w, "Missing tenant ID", http.StatusBadRequest)

        next.ServeHTTP(w, r.WithContext(ctx))

### 5.3 大数据任务适配方案

#### 设计思路

采用 **适配器模式**，在现有 Invoker 体系上新增 `BigDataInvoker`，通过不同的 Adapter 实现对接各大数据引擎。

```
┌────────────┐
│ Dispatcher  │
└─────┬──────┘
      │
      ├── GRPCInvoker
      ├── HTTPInvoker
      ├── LocalInvoker
      └── BigDataInvoker
            ├── YarnAdapter   → Yarn REST API
            ├── FlinkAdapter  → Flink REST API
            ├── SparkAdapter  → Spark REST API
            └── DataXAdapter  → DataX CLI
```

#### 核心接口

```go
// BigDataAdapter 统一适配器接口
type BigDataAdapter interface {
    Name() string
    Submit(ctx context.Context, config *domain.BigDataConfig) (applicationID string, err error)
    Query(ctx context.Context, applicationID string) (domain.ExecutionState, error)
    Cancel(ctx context.Context, applicationID string) error
    GetLogs(ctx context.Context, applicationID string, lines int) (string, error)
}
```

#### 领域模型扩展

```go
type BigDataConfig struct {
    TaskType     BigDataTaskType   `json:"taskType"`     // yarn/flink/spark/datax
    ClusterURL   string            `json:"clusterUrl"`
    MainClass    string            `json:"mainClass"`
    JarPath      string            `json:"jarPath"`
    Args         []string          `json:"args"`
    Conf         map[string]string `json:"conf"`
    Resources    *ResourceConfig   `json:"resources"`
}

type ResourceConfig struct {
    DriverMemory   string `json:"driverMemory"`
    DriverCores    int    `json:"driverCores"`
    ExecutorMemory string `json:"executorMemory"`
    ExecutorCores  int    `json:"executorCores"`
    NumExecutors   int    `json:"numExecutors"`
    Queue          string `json:"queue"`
}
```

#### BigDataInvoker 实现

```go
type BigDataInvoker struct {
    adapters map[domain.BigDataTaskType]bigdata.BigDataAdapter
}

func (b *BigDataInvoker) Run(ctx context.Context, execution domain.TaskExecution) (domain.ExecutionState, error) {
    config := execution.Task.BigDataConfig
    adapter, ok := b.adapters[config.TaskType]
    if !ok {
        return domain.ExecutionState{}, fmt.Errorf("unsupported: %s", config.TaskType)
    }

    applicationID, err := adapter.Submit(ctx, config)
    if err != nil {
        return domain.ExecutionState{Status: domain.TaskExecutionStatusFailedRetryable}, err
    }

    state, _ := adapter.Query(ctx, applicationID)
    state.ID = execution.ID
    state.TaskID = execution.Task.ID
    state.ExecutorNodeID = applicationID
    return state, nil
}
```

#### 各 Adapter 对接方式

| Adapter | 通信方式 | 核心 API | 状态查询 |
|---------|---------|---------|---------|
| YarnAdapter | HTTP | `POST /ws/v1/cluster/apps` | `GET /ws/v1/cluster/apps/{id}` |
| FlinkAdapter | HTTP | `POST /jars/{id}/run` | `GET /jobs/{id}` |
| SparkAdapter | HTTP | `POST /v1/submissions/create` | `GET /v1/submissions/status/{id}` |
| DataXAdapter | CLI | `datax.py {config.json}` | 进程退出码 |

#### 状态映射

| 大数据引擎状态 | 平台领域状态 |
|-------------|------------|
| SUBMITTED / ACCEPTED | `PREPARE` |
| RUNNING | `RUNNING` |
| FINISHED / SUCCEEDED | `SUCCESS` |
| FAILED | `FAILED_RETRYABLE` |
| KILLED | `FAILED_RESCHEDULED` |

**收益**：统一管理 Spark/Flink/DataX 等大数据任务，新增引擎只需实现 Adapter 接口。

## 六、优先级总结

| 级别 | 内容 | 预计工期 | 核心收益 |
|------|------|---------|---------|
| **P0** | 连接池调优 / 批量优化 / 熔断器 / 统一错误 | 1~3 天 | 性能 +20~30%，可用性 99.99% |
| **P1** | 多级缓存 / 异步写入 / 增强监控 / 完善测试 | 3~7 天 | 吞吐量 5~10×，DB 查询 -90% |
| **P2** | 配置中心 / API 网关 / 插件架构 / 代码生成 | 1~2 周 | 运维效率 +3×，扩展性 |
| **P3** | 服务网格 / 多租户 / 大数据适配 | 1+ 月 | 云原生能力，SaaS 化 |

> 建议优先实施 P0 + P1，投入产出比最高，能快速见效。P2/P3 根据业务需求和团队资源逐步推进。
