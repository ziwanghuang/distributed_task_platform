## 1. 概述

### 1.1 项目背景

本项目是一个基于 Go 语言的分布式任务调度平台，采用 ego 微服务框架，核心技术栈：

| 组件 | 技术选型 | 用途 |
|------|---------|------|
| 微服务框架 | ego | 应用容器、配置管理、日志 |
| RPC 通信 | gRPC + Protobuf | 任务下发、执行状态查询 |
| 消息队列 | Kafka | 异步状态上报、事件驱动 |
| 关系数据库 | MySQL (分库分表) | 任务元数据、执行记录持久化 |
| 服务注册 | etcd | 执行节点注册与发现 |
| 缓存/锁 | Redis | 分布式锁、资源信号量 |
| 监控 | Prometheus + Grafana | 指标采集、智能调度决策 |
| ID 生成 | Snowflake 变体 | 分布式唯一 ID（嵌入分片信息） |

### 1.2 系统架构概览

```
                    ┌──────────────────────────────┐
                    │       Scheduler (调度器)       │
                    │  ┌─────────┐  ┌────────────┐ │
                    │  │ V1 Loop │  │ V2 Sharding│ │
                    │  │ Job     │  │ LoopJob    │ │
                    │  └────┬────┘  └─────┬──────┘ │
                    │       │             │        │
                    │  ┌────▼─────────────▼──────┐ │
                    │  │   Task Acquirer (CAS)    │ │
                    │  └────────────┬────────────┘ │
                    │               │              │
                    │  ┌────────────▼────────────┐ │
                    │  │  Runner Dispatcher       │ │
                    │  │  ├─ NormalTaskRunner     │ │
                    │  │  └─ PlanTaskRunner (DAG) │ │
                    │  └────────────┬────────────┘ │
                    │               │              │
                    │  ┌────────────▼────────────┐ │
                    │  │  Invoker Dispatcher      │ │
                    │  │  ├─ gRPC Invoker        │ │
                    │  │  ├─ HTTP Invoker        │ │
                    │  │  └─ Local Invoker       │ │
                    │  └────────────┬────────────┘ │
                    └───────────────┼──────────────┘
                                    │
         ┌──────────────────────────┼──────────────────────┐
         │                          │                      │
┌────────▼────────┐  ┌─────────────▼──────┐  ┌────────────▼───────┐
│ Executor Node 1  │  │ Executor Node 2    │  │ Executor Node N    │
│ (gRPC Server)    │  │ (gRPC Server)      │  │ (gRPC Server)      │
└────────┬────────┘  └──────────┬─────────┘  └──────────┬─────────┘
         │                      │                       │
         └──────────────────────┼───────────────────────┘
                                │
                     ┌──────────▼──────────┐
                     │  Reporter Server     │
                     │  (gRPC + Kafka MQ)   │
                     └──────────┬──────────┘
                                │
               ┌────────────────┼────────────────┐
               │                │                │
      ┌────────▼──────┐ ┌──────▼─────┐ ┌────────▼────┐
      │    MySQL      │ │   Kafka    │ │   etcd      │
      │  db_0 / db_1  │ │            │ │             │
      └───────────────┘ └────────────┘ └─────────────┘
```

### 1.3 压测范围

1. **调度主链路**: Scheduler → Acquirer → Runner → Invoker → Executor → Reporter
2. **补偿链路**: Retry / Reschedule / Interrupt / Sharding Compensator
3. **DAG 工作流链路**: PlanService → PlanTaskRunner → CompleteEvent → 下一步调度
4. **分片链路**: 分片策略解析 → 子任务创建 → 并行执行 → 汇聚
5. **基础设施**: MySQL 分库分表、Kafka 吞吐、etcd Watch/Lease

### 1.4 术语定义

| 术语 | 说明 |
|------|------|
| TPS | Transactions Per Second，每秒事务数 |
| QPS | Queries Per Second，每秒查询数 |
| P50/P95/P99 | 延迟百分位数 |
| CAS | Compare-And-Swap，乐观锁机制 |
| DAG | Directed Acyclic Graph，有向无环图 |
| ShardingLoopJob | 基于分片的分布式循环任务框架 |
| Compensator | 补偿器，用于处理异常状态的任务 |

## 2. 压测目标与验收标准

### 2.1 性能目标

#### 2.1.1 调度吞吐量

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 单节点调度 TPS | ≥ 500 tasks/s | 从 SchedulableTask 查询到 Invoke 完成 |
| 集群调度 TPS (3节点) | ≥ 1200 tasks/s | 线性扩展比 ≥ 0.8 |
| 集群调度 TPS (5节点) | ≥ 1800 tasks/s | 线性扩展比 ≥ 0.72 |
| CAS 抢占成功率 | ≥ 30% (3节点竞争) | 乐观锁竞争下的有效吞吐 |

#### 2.1.2 延迟目标

| 指标 | P50 | P95 | P99 |
|------|-----|-----|-----|
| 任务调度延迟 (调度→下发) | ≤ 10ms | ≤ 50ms | ≤ 100ms |
| gRPC 任务执行调用 | ≤ 5ms | ≤ 20ms | ≤ 50ms |
| Kafka 异步上报延迟 | ≤ 20ms | ≤ 100ms | ≤ 200ms |
| DAG 步骤间延迟 | ≤ 50ms | ≤ 200ms | ≤ 500ms |
| CAS 抢占延迟 | ≤ 2ms | ≤ 10ms | ≤ 20ms |

#### 2.1.3 资源目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| CPU 使用率 | ≤ 70% | 调度节点稳态 |
| 内存使用 | ≤ 2GB | 单节点 RSS |
| Goroutine 数量 | ≤ 10000 | 无泄露 |
| MySQL 连接数 | ≤ 100/节点 | 连接池上限 |
| Kafka Consumer Lag | ≤ 1000 | 消费延迟 |

### 2.2 稳定性目标

| 指标 | 目标值 | 说明 |
|------|--------|------|
| 7×24h 连续运行 | 零 OOM、零 panic | 长时间稳定性 |
| 任务丢失率 | 0% | 所有已调度任务必须有终态 |
| 重复执行率 | ≤ 0.1% | CAS 防重机制有效性 |
| 故障恢复时间 | ≤ 30s | 节点宕机后任务自动补偿 |

### 2.3 验收标准矩阵

| 等级 | 条件 | 判定 |
|------|------|------|
| ✅ 通过 | 所有指标达标，无 P0/P1 问题 | 可上线 |
| ⚠️ 有条件通过 | 核心指标达标，存在 P2 问题 | 修复后上线 |
| ❌ 不通过 | 存在 P0/P1 问题或核心指标未达标 | 需优化后重测 |

#### 3.3.3 Mock Executor 实现

```go
// bench/mock_executor.go
package main

import (
    "context"
    "math/rand"
    "time"

    executorv1 "distributed_task_platform/api/gen/executor/v1"
    "google.golang.org/grpc"
)

type MockExecutorServer struct {
    executorv1.UnimplementedExecutorServiceServer
    minLatency time.Duration
    maxLatency time.Duration
    failRate   float64
}

func (s *MockExecutorServer) Execute(
    ctx context.Context,
    req *executorv1.ExecuteRequest,
) (*executorv1.ExecuteResponse, error) {
    // 模拟执行延迟
    latency := s.minLatency +
        time.Duration(rand.Int63n(int64(s.maxLatency-s.minLatency)))
    time.Sleep(latency)

    // 模拟失败率
    if rand.Float64() < s.failRate {
        return &executorv1.ExecuteResponse{
            State: &executorv1.ExecutionState{
                TaskId:      req.TaskId,
                ExecutionId: req.ExecutionId,
                Status:      executorv1.ExecutionStatus_EXECUTION_STATUS_FAILED,
            },
        }, nil
    }

    return &executorv1.ExecuteResponse{
        State: &executorv1.ExecutionState{
            TaskId:      req.TaskId,
            ExecutionId: req.ExecutionId,
            Status:      executorv1.ExecutionStatus_EXECUTION_STATUS_SUCCESS,
            Progress:    100,
        },
    }, nil
}

func (s *MockExecutorServer) Interrupt(
    ctx context.Context,
    req *executorv1.InterruptRequest,
) (*executorv1.InterruptResponse, error) {
    time.Sleep(10 * time.Millisecond)
    return &executorv1.InterruptResponse{Success: true}, nil
}

func main() {
    server := grpc.NewServer()
    executorv1.RegisterExecutorServiceServer(server, &MockExecutorServer{
        minLatency: 5 * time.Millisecond,
        maxLatency: 50 * time.Millisecond,
        failRate:   0.05,
    })
    // listen on :9090
}
```

## 5. 核心压测场景

### 5.1 调度器主循环压测 (Scheduler Loop)

#### 5.1.1 测试目标

验证调度器核心循环的吞吐能力和稳定性。调度器分为两个版本：

- **V1 (scheduler.go)**: 简单循环模式，单 goroutine 轮询 `SchedulableTasks`
- **V2 (schedulerv2.go)**: ShardingLoopJob 模式，基于分布式锁+分片的高可用调度

#### 5.1.2 V1 调度器压测

**场景 A: 基准吞吐量**

```
测试名称: scheduler_v1_baseline
前置条件:
  - 单节点部署
  - 10000 个 Cron 任务（每 5 秒触发一次）
  - 20 个 Mock Executor 节点
  - Mock Executor 响应延迟: 5-50ms
  - scheduleInterval: 1s (config.yaml 中的 scheduler.interval)

执行步骤:
  1. 启动 Scheduler 和所有 Executor
  2. 等待 warm-up 30s
  3. 持续运行 10 分钟
  4. 采集指标

监控指标:
  - scheduled_tasks_total (Prometheus counter)
  - schedule_duration_seconds (Prometheus histogram)
  - acquire_success_rate (CAS 成功率)
  - goroutine_count
  - cpu_usage / memory_usage

验收标准:
  - 调度 TPS ≥ 500/s
  - 调度延迟 P99 ≤ 100ms
  - CPU ≤ 70%
  - 无 goroutine 泄露
```

**场景 B: 高负载下的 LoadChecker 限流**

```
测试名称: scheduler_v1_load_check
前置条件:
  - 50000 个任务同时到达调度时间
  - ClusterLoadChecker + DatabaseLoadChecker 均启用
  - Prometheus 正常采集指标

执行步骤:
  1. 批量更新所有任务的 next_exec_time 为当前时间
  2. 观察调度器行为
  3. 记录 LoadChecker 的限流触发次数

监控指标:
  - load_check_triggered_total (按 checker 类型分)
  - schedule_skip_count (被跳过的调度批次)
  - actual_interval vs base_interval (动态调整比)
  - cluster_cpu_utilization (Prometheus PromQL 查询)
  - db_qps / db_active_connections

验收标准:
  - LoadChecker 正确触发限流
  - 限流后 interval 自动增大
  - 负载下降后 interval 自动恢复
  - 数据库不被打爆（连接数 ≤ 上限 80%）

代码参考 (pkg/loadchecker):
  - CompositeLoadChecker.Check() 聚合多个检查器
  - ClusterLoadChecker 通过 PromQL 查询集群 CPU
  - DatabaseLoadChecker 通过 PromQL 查询 DB QPS
  - Limiter 通过 rate.Limiter 做最终限流决策
```

**场景 C: Renew 心跳压测**

```
测试名称: scheduler_v1_renew_loop
前置条件:
  - 1000 个长时间运行的任务（Mock Executor 返回 RUNNING 状态）
  - renewInterval: 5s (config.yaml)

执行步骤:
  1. 调度 1000 个任务，Executor 持续返回 RUNNING
  2. 观察 renewLoop 行为
  3. 持续 30 分钟

监控指标:
  - renew_success_count / renew_fail_count
  - renew_duration_seconds
  - task_execution_version (应持续递增)

验收标准:
  - 所有 RUNNING 任务的 update_time 持续更新
  - Renew 失败率 ≤ 0.1%
  - Renew 不影响正常调度（无资源抢占）

代码参考 (internal/service/scheduler/scheduler.go):

  - 通过 UpdateUtime 更新时间戳防止被误判为超时
```

#### 5.1.3 V2 调度器压测 (ShardingLoopJob)

**场景 D: 分布式调度扩展性**

```
测试名称: scheduler_v2_scale_out
前置条件:
  - ShardingLoopJob 配置: shardCount=8
  - MaxCntResourceSemaphore: maxCount=3
  - Redis 分布式锁可用

阶梯测试:
  Step 1: 1 个调度节点
    - 该节点获得全部 8 个分片
    - 记录基准 TPS

  Step 2: 3 个调度节点
    - 分片自动分配 (约 3/3/2)
    - 记录集群 TPS
    - 计算扩展比 = 集群TPS / (基准TPS × 3)

  Step 3: 5 个调度节点
    - 分片自动分配 (约 2/2/2/1/1)
    - 记录集群 TPS
    - 计算扩展比

  Step 4: 8 个调度节点
    - 每节点 1 个分片
    - 记录集群 TPS（理论最大值）

监控指标:
  - shard_acquired_total (每个节点获取的分片数)
```

### B.4 分片查询优化

```sql
-- Snowflake ID 自带分片信息，查询时可直接路由
-- 对应代码: pkg/id_generator/snowflake.go

-- ID 结构: [unused 1bit][timestamp 41bit][bizID 12bit][sequence 12bit]
-- bizID 的低 2 bit 决定分库分表:
--   bit[0] → db_index (0 or 1)
--   bit[1] → table_index (0 or 1)

-- 通过 ID 快速定位分片
-- db_index = (id >> 12) & 1
-- table_index = (id >> 13) & 1
```

## 附录 D: 压测结果分析方法论

### D.1 性能瓶颈定位流程

```
Step 1: 宏观指标分析
  └─ TPS 未达标?
     ├─ CPU 瓶颈 → pprof CPU profile → 找到热点函数
     ├─ 内存瓶颈 → pprof heap profile → 找到大对象分配
     ├─ I/O 瓶颈 → fgprof wall-clock profile → 找到等待点
     ├─ 锁瓶颈 → pprof mutex/block profile → 找到竞争点
     └─ 外部依赖瓶颈 → 逐一排查 MySQL/Kafka/etcd/Redis

Step 2: 微观分析
  └─ 定位到具体函数后
     ├─ 是否可优化算法? (时间复杂度)
     ├─ 是否可减少内存分配? (对象池/sync.Pool)
     ├─ 是否可减少锁粒度? (读写锁/分段锁)
     ├─ 是否可批量操作? (批量 DB/批量 MQ)
     └─ 是否可异步化? (channel/goroutine)

Step 3: 优化验证
  └─ 修改后重跑 benchmark
     ├─ 对比 TPS 变化
     ├─ 对比延迟变化
     ├─ 确认无回归
     └─ 提交优化
```

### D.2 延迟分析方法

```
延迟分解 (一次完整的调度链路):

Total Latency = T_query + T_cas + T_run + T_invoke + T_report

T_query:  查询可调度任务 (MySQL SELECT)
  └─ 正常: 1-5ms
  └─ 异常: > 50ms → 检查索引/慢查询

T_cas:    CAS 抢占 (MySQL UPDATE)
  └─ 正常: 0.5-2ms
  └─ 异常: > 10ms → 检查行锁竞争

T_run:    Runner 处理 (内存操作)
  └─ 正常: < 0.1ms
  └─ 异常: > 1ms → 检查 DSL 解析

T_invoke: 任务下发 (gRPC call)
  └─ 正常: 5-20ms (含 Executor 处理)
  └─ 异常: > 100ms → 检查网络/Executor 负载

T_report: 状态上报 (Kafka produce)
  └─ 正常: 1-5ms (异步)
  └─ 异常: > 50ms → 检查 Kafka 负载/batch 配置
```

### D.3 容量规划公式

```
所需调度节点数 = ceil(目标 TPS / 单节点 TPS × 安全系数)

  安全系数 = 1.5 (预留 50% buffer)

  例: 目标 3000 TPS, 单节点 500 TPS
  → ceil(3000 / 500 × 1.5) = ceil(9) = 9 个调度节点

所需 Executor 节点数 = ceil(峰值并发任务数 / 单节点并发能力)

  单节点并发能力 = CPU 核数 × 并发系数 (I/O 密集型取 5-10)

  例: 峰值 5000 并发任务, 4C Executor, 系数 8
  → ceil(5000 / (4 × 8)) = ceil(156) ≈ 160 个 Executor

所需 MySQL 容量 = 数据增长量/天 × 保留天数

  每条执行记录 ≈ 1KB
  10000 TPS × 86400s × 1KB = 864GB/天
  保留 30 天 → 25.9TB
  分 4 个分片 → 每分片 ≈ 6.5TB
```

## 附录 E: 常见问题与排查指南

### E.1 调度延迟突增

```
现象: 调度延迟从 10ms 突增到 500ms+

排查步骤:
  1. 检查 MySQL 慢查询日志
     → SHOW GLOBAL STATUS LIKE 'Slow_queries';
     → 查看 slow.log 最近记录

  2. 检查 MySQL 连接池
     → SHOW PROCESSLIST;
     → 是否有大量 Sleep/Lock 状态连接

  3. 检查 GC 暂停
     → go tool pprof -http=:8080 /debug/pprof/heap
     → 看 runtime.GC 的调用频率

  4. 检查 Goroutine 阻塞
     → go tool pprof /debug/pprof/block
     → 找到阻塞最久的位置

  5. 检查网络延迟
     → ping 各中间件节点
     → 网络是否有抖动
```

### E.2 CAS 成功率过低

```
现象: CAS 成功率 < 10% (3节点竞争下期望 ~33%)

排查步骤:
  1. 检查是否存在慢事务
     → 长事务持有行锁导致其他 CAS 等待超时

  2. 检查 batchSize 是否过小
     → 每次只取 10 个任务, 3 节点竞争同一批
     → 增大 batchSize 可降低竞争

  3. 检查是否有热点任务
     → 某些任务被频繁调度, 成为竞争焦点
     → 考虑引入随机 offset

  4. 检查 MySQL 隔离级别
     → SHOW VARIABLES LIKE 'transaction_isolation';
     → RC 级别比 RR 对 CAS 更友好
```

### E.3 Kafka Consumer Lag 持续增长

```
现象: Consumer Lag 持续增长, 不收敛

排查步骤:
  1. 检查 Consumer 消费速度
     → 是否有慢消费者 (单条处理 > 100ms)
     → 是否 DB 写入成为瓶颈

  2. 检查 Partition 分配
     → kafka-consumer-groups.sh --describe --group scheduler-consumer
     → 是否分配均匀

  3. 检查 Consumer 数量
     → Consumer 数量应 ≤ Partition 数量
     → 超过则有 Consumer 空闲

  4. 增加消费并行度
     → 增加 Partition 数量
     → 增加 Consumer 实例
     → Consumer 内部多 goroutine 处理
```

### E.4 Goroutine 泄露

```
现象: Goroutine 数量持续增长

排查步骤:
  1. 获取 Goroutine dump
     → curl http://localhost:6060/debug/pprof/goroutine?debug=2 > goroutine.txt

  2. 分析堆栈
     → 找到数量异常多的相同堆栈
     → 通常是 channel 或 lock 阻塞

  3. 常见泄露场景:
     → Context 未正确取消
     → Channel 无人读取/写入
     → gRPC stream 未关闭
     → HTTP Body 未关闭

  4. 使用 goleak 自动检测
     → import "go.uber.org/goleak"
     → defer goleak.VerifyNone(t) // 在测试中
```

### E.5 分片数据倾斜

```
现象: 某个分片数据量远大于其他分片

排查步骤:
  1. 检查 Snowflake ID 分布
     → 统计每个分片的 ID 数量

  2. 检查 bizID 分配
     → bizID 是否均匀
     → 是否有热点 bizID

  3. 检查任务创建模式
     → 是否短时间大量创建同 bizID 任务
     → 导致 sequence 溢出重叠

  4. 临时缓解方案
     → 手动迁移热点数据
     → 调整 bizID 映射规则
```
