## 二、面试话术

### 2.1 开场介绍（30 秒电梯演讲）

> 我负责的这个项目是一个企业级分布式任务调度平台，对标开源的 DolphinScheduler。它的核心价值是为公司 50+ 业务线提供统一的任务编排能力，每天处理 500 万+ 任务，系统可用性达到 99.99%。
>
> 技术上，我们采用 Go + gRPC + etcd + Kafka 的技术栈，实现了四重补偿机制、智能调度算法、工作流 DSL 引擎等核心能力。
>
> 这个项目上线后，为公司每年节省成本 200 万+，任务成功率从 95% 提升到 99.9%。

### 2.2 技术深度展示

**选项 A：四重补偿机制**

> 传统调度系统只有简单的重试，但分布式环境下故障类型非常多样化。我们设计了四层补偿：
>
> - **重试补偿**：处理瞬时故障（网络抖动），指数退避，最多 3 次
> - **重调度补偿**：处理节点故障（执行器宕机），重新选择健康节点
> - **中断补偿**：处理超时任务（执行卡死），发送中断请求释放资源
> - **分片补偿**：处理分片任务的子任务聚合，全部成功才算成功
>
> 四层补偿相互配合，让任务成功率从 95% 提升到了 99.9%。

**选项 B：智能调度算法**

> 系统实时采集每个节点的 CPU、内存、任务数、网络延迟等指标，用多维度评分模型计算负载得分。
>
> 评分用 Sigmoid 函数做归一化，权重根据任务类型动态调整——CPU 密集型任务提高 CPU 权重，IO 密集型任务提高网络权重。
>
> 还引入了 TopN 随机选择，避免所有任务都涌向同一个"最优"节点。
>
> 效果：资源利用率从 55% 提升到 85%，服务器成本降低 35%。

**选项 C：分布式一致性**

> 分布式环境最大的挑战是如何保证任务不被重复调度。我们设计了多层防护：
>
> 1. **乐观锁**：CAS 操作 `UPDATE ... SET version=version+1 WHERE version=?`，只有一个调度器能抢占成功
> 2. **分布式锁**：Redis `SET NX EX`，补偿器场景防重复
> 3. **幂等设计**：唯一索引 + INSERT IGNORE，重复执行不影响结果
> 4. **状态机**：严格状态流转 ACTIVE→PREEMPTED→RUNNING→SUCCESS/FAILED
>
> 四层防护即使在极端情况下也能保证不重复。

### 2.3 难点攻克叙事

> 项目中最大的挑战是如何在千万级数据量下保证调度延迟 < 100ms。
>
> 初期单表数据量到 500 万后，查询性能急剧下降。我们分三步解决：
>
> 1. **分库分表**：设计了 8 库 16 表的分片策略，基于 taskID 哈希路由，单表数据量控制在 60 万以内
> 2. **多级缓存**：本地 Ristretto + Redis 两级缓存，L1 命中率 90%+，DB 查询量降低 90%
> 3. **异步 + 批量**：状态上报走 Kafka 异步 + 批量处理，DB 写入性能提升 80%
>
> 最终 P99 延迟从 500ms 降到了 100ms 以内。

### 2.4 技术选型话术

> **Q: 为什么选 Go 而不是 Java？**
> - 协程模型天然适合高并发调度场景，单个协程 2KB 内存，轻松创建数万个
> - 编译为单二进制文件，部署简单，512M 内存即可运行，Java 至少 4G
> - GC STW < 1ms，Java 可能数十 ms
>
> **Q: 为什么选 etcd 而不是 ZooKeeper？**
> - etcd 基于 Raft，读写性能更高
> - 原生 gRPC 接口，与我们的技术栈一致
> - K8s 也用 etcd，云原生生态更好
>
> **Q: 为什么选 gRPC 而不是 HTTP？**
> - Protobuf 二进制序列化，数据量比 JSON 小 3~5 倍
> - HTTP/2 多路复用，连接池复用效率高
> - 强类型接口定义，编译时检查，减少运行时错误

## 三、高频 Q&A

### 一、架构设计类

#### Q1: 请介绍这个系统的整体架构？

**三层架构**：调度层（Scheduler）、执行层（Executor）、存储层（MySQL/Redis/Kafka）

**核心组件**：
- 调度器集群：任务抢占、调度、续约
- 执行器集群：任务执行，动态上下线
- 补偿器集群：四种补偿器处理异常
- etcd：服务注册与发现
- Kafka：异步事件驱动

**数据流**：任务创建 → 调度器抢占（乐观锁） → 选择执行器（智能调度） → 分发执行（gRPC） → 结果上报（Kafka 批量） → 状态更新 → 触发后续任务

---

#### Q2: 为什么选择这套技术栈？

| 组件 | 选型 | 原因 | 对比方案 |
|-----|------|------|---------|
| 语言 | Go | 高并发（协程）、高性能（编译型）、部署简单 | Java（JVM 重）、Python（GIL） |
| RPC | gRPC | 高性能（Protobuf）、HTTP/2 多路复用 | HTTP（性能低）、Thrift（生态弱） |
| 注册中心 | etcd | 强一致性（Raft）、Watch 机制 | ZK（运维复杂）、Consul（CP 弱） |
| 消息队列 | Kafka | 高吞吐、持久化、消息回溯 | RabbitMQ（吞吐低）、Redis Streams |
| 存储 | MySQL | 成熟稳定、事务支持 | PostgreSQL、TiDB |
| 分布式锁 | Redis | 性能高、成熟 | etcd（延迟高）、ZK |

### 二、高可用类

#### Q3: 调度器宕机了怎么办？

**三道防线**：
1. **集群化部署**：多节点运行，任务抢占机制保证同一任务只被一个调度器处理
2. **任务续约机制**：调度器每 30 秒续约，宕机后续约失败，任务锁自动释放
3. **重调度补偿器**：扫描超时未续约的任务，自动重新调度到其他节点

**恢复时间**：续约超时 30s + 补偿扫描 30s + 重调度 5s ≈ 65s。优化版本通过 etcd Watch 实现主动通知，恢复时间 < 10s。

---

#### Q4: 如何保证任务不被重复调度？

**四层防护**：

```sql
-- 第一层：乐观锁（CAS 操作）
UPDATE task SET status='PREEMPTED', version=version+1
WHERE id=? AND version=? AND status='ACTIVE'
```

```go
// 第二层：分布式锁（补偿器场景）
redis.SetNX(ctx, "task:lock:123", nodeID, 60*time.Second)
```

```sql
-- 第三层：幂等设计（唯一索引）
CREATE UNIQUE INDEX idx_execution_id ON task_execution(execution_id);
```

```go
// 第四层：状态机（严格流转）
allowedTransitions = map[Status][]Status{
    ACTIVE:    {PREEMPTED},
    PREEMPTED: {RUNNING, ACTIVE},
    RUNNING:   {SUCCESS, FAILED_RETRYABLE, FAILED_RESCHEDULED},
}
```

---

#### Q5: 四种补偿器分别解决什么问题？

| 补偿器 | 处理状态 | 场景 | 策略 |
|-------|---------|------|------|
| 重试补偿器 | `FAILED_RETRYABLE` | 网络抖动、临时错误 | 指数退避重试，最多 3 次 |
| 重调度补偿器 | `FAILED_RESCHEDULED` | 执行器宕机 | 重新选择健康节点 |
| 中断补偿器 | `RUNNING`（超时） | 任务卡死 | 发送中断请求，更新为 FAILED |
| 分片补偿器 | 父任务 `RUNNING` | 子任务完成聚合 | 全成功→成功，任一失败→失败 |

**为什么需要多个**：职责分离，每个补偿器专注一类异常，互不影响，便于维护和扩展。

### 三、高性能类

#### Q6: 分库分表如何实现？

**策略**：8 库 16 表，每库 2 表。

```go
tableIndex := taskID % 16           // 确定表（0~15）
dbIndex    := (taskID / 16) % 8     // 确定库（0~7）
```

**路由**：通过 Context 传递分片信息，DAO 层从 Context 读取目标分片。

**查询优化**：
- 单任务查询：直接路由到目标分片
- 全表扫描：`ShardingLoopJob` 并发扫描所有分片，每个分片独立维护 offset

**追问：扩容怎么办？**
- 双写策略：新旧分片同时写入，逐步迁移
- 预留方案：虚拟分片数远大于物理分片，物理扩容只需迁移虚拟分片

#### Q7: 智能调度是如何实现的？

```go
// 多维度加权评分
score := cpuUsage * 0.5 + memUsage * 0.3 + taskCount * 0.2

// Sigmoid 归一化
cpuScore := 1 / (1 + math.Exp(10*(cpu-0.7)))
```

- 查询 Prometheus 获取实时指标（CPU/内存/任务数）
- TopN 随机选择：先选出得分最高的 N 个节点，再随机选一个（避免热点）
- 降级策略：Prometheus 不可用时降级为随机选择

**效果**：资源利用率从 55% 提升到 85%，避免单节点过载。

#### Q9: 批量上报的实现和权衡？

**触发策略**：数量满 100 条 **或** 时间满 5 秒，先到先触发。

**参数权衡**：
- 批量大小 100：太小效果不明显，太大延迟高
- 间隔 5s：太短频繁触发，太长状态更新不及时
- 关键状态（任务完成）立即上报，不等待批量窗口
- 批量失败时降级为单条上报

**性能**：100 条上报从 1000ms 降到 30ms，提升 33 倍。

### 四、工作流编排类

#### Q10: 工作流编排是怎么实现的？

**DSL 语法**（基于 ANTLR4）：
```
task1 -> task2 -> task3          // 顺序执行
task1 && task2 && task3          // 并行执行
task1 || task2 || task3          // 选择执行（任一成功即可）
task1 ? task2 : task3            // 条件分支
(task1 && task2) -> (task3 || task4)  // 组合
```

**实现流程**：DSL 字符串 → ANTLR4 词法分析 → 语法分析 → AST → 执行引擎

**为什么选 ANTLR4**：
- 语法定义简单（BNF 范式），自动生成 Lexer/Parser
- 支持错误恢复和友好提示
- Go 语言官方支持，社区活跃

### 五、Kafka 相关

#### Q11: Kafka 如何保证消息不丢失？

| 环节 | 配置 | 作用 |
|-----|------|------|
| 生产者 | `acks=all` | 等待所有副本确认 |
| Broker | `min.insync.replicas=2` | 至少 2 个副本写入 |
| 消费者 | 手动提交 offset | 处理成功后再提交 |
| 分区 | `partition key = taskID` | 同一任务消息有序 |
| 去重 | 基于 `execution_id` | 幂等消费 |

### 六、监控与可观测性

#### Q12: 如何实现全链路追踪？

```go
// OpenTelemetry 追踪示例
ctx, span := tracer.Start(ctx, "ScheduleTask",
    trace.WithAttributes(
        attribute.Int64("task.id", taskID),
        attribute.String("task.type", task.Type),
    ))
defer span.End()
```

每个环节（抢占→选节点→gRPC 调用→执行→上报）都创建子 Span，通过 Context 传播 TraceID，实现端到端追踪。配合 Jaeger UI 可视化调用链。
