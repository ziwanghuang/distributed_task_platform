## 1. 分库分表策略

**包路径**：`pkg/sharding/strategy.go`

### ShardingStrategy

```go
type ShardingStrategy struct {
    dbPrefix      string // 数据库名前缀（如 "task"）
    tablePrefix   string // 表名前缀（如 "task"）
    tableSharding int64  // 每个库的表数量
    dbSharding    int64  // 数据库数量
}
```

### 路由算法

```
dbHash  = id % dbSharding           → 数据库后缀
tabHash = (id / dbSharding) % tableSharding → 表后缀
```

**示例**（`dbSharding=2, tableSharding=2`）：

| ID | 库 | 表 |
|----|-----|-----|
| 0 | task_0 | task_0 |
| 1 | task_1 | task_0 |
| 2 | task_0 | task_1 |
| 3 | task_1 | task_1 |

### 核心方法

| 方法 | 说明 |
|------|------|
| `Shard(id int64) Dst` | 根据 ID 计算目标库表 |
| `Broadcast() []Dst` | 返回所有库表组合（笛卡尔积） |

### Dst（分片目标）

```go
type Dst struct {
    Table       string // 完整表名（如 "task_0"）
    DB          string // 完整库名（如 "task_0"）
    TableSuffix int64  // 表后缀编号
    DBSuffix    int64  // 库后缀编号
}
```

### Context 传递

```go
CtxWithDst(ctx, dst)     // ShardingLoopJob 写入
DstFromCtx(ctx) → Dst    // DAO 层读取
```

## 2. Snowflake ID 生成器

**包路径**：`pkg/id_generator/snowflake.go`

### ID 结构（64 bit）

```
[12 bit bizID] [12 bit sequence] [41 bit timestamp]
```

| 段 | 位数 | 范围 | 说明 |
|----|------|------|------|
| bizID | 12 | 0-4095 | 业务标识，嵌入分片路由信息 |
| sequence | 12 | 0-4095 | 序列号（atomic 递增） |
| timestamp | 41 | ~69年 | 毫秒时间戳 |

### 方法

| 方法 | 说明 |
|------|------|
| `Generate() int64` | 生成全局唯一 ID |
| `ExtractShardingID(id int64) int64` | 从 ID 中提取 bizID 用于分片路由 |

## 3. ShardingLoopJob 框架

**包路径**：`pkg/loopjob/shardingloopjob.go`

V2 补偿器的核心执行框架，将分片 + 分布式锁 + 业务循环封装为通用框架。

### 执行模型

```
Run(ctx) → 外层无限循环
    │
    └── 遍历所有分片 (Broadcast)
            │
            ├── Acquire 信号量（控制并发上限）
            ├── NewLock + Lock（竞争分布式锁）
            └── go tableLoop（独立 goroutine）
                    │
                    └── bizLoop（锁保护下持续执行）
                            ├── biz(ctx)       — 执行一次业务
                            └── lock.Refresh   — 续约锁
```

### 关键时序

- `bizTimeout = 50s` < `lockTTL = 1min` → 确保业务在锁过期前完成
- `defaultTimeout = 3s` → Lock/Unlock/Refresh 操作超时
- `retryInterval = 1min` → 获取锁失败重试间隔

### 锁 Key 格式

```
{baseKey}:{db}:{table}
```

例如：`retryKey:task_0:task_execution_0`

## 4. ResourceSemaphore 资源信号量

**包路径**：`pkg/loopjob/resource.go`

### 接口

```go
type ResourceSemaphore interface {
    Acquire(ctx context.Context) error   // 获取许可
    Release(ctx context.Context) error   // 释放许可
}
```

### MaxCntResourceSemaphore

基于 `sync.RWMutex` 的计数信号量：

```go
type MaxCntResourceSemaphore struct {
    maxCount int      // 最大并发数
    curCount int      // 当前已发出许可数
    mu       *sync.RWMutex
}
```

- `Acquire`：`curCount >= maxCount` → 返回 `ErrExceedLimit`
- `UpdateMaxCount`：支持运行时动态调整

## 5. 重试框架

**包路径**：`pkg/retry/`

### 工厂函数

```go
func NewRetry(cfg Config) (strategy.Strategy, error)
```

根据 `cfg.Type` 创建策略：

| Type | 策略 | 参数 |
|------|------|------|
| `"fixed"` | FixedIntervalRetryStrategy | `Interval` + `MaxRetries` |
| `"exponential"` | ExponentialBackoffRetryStrategy | `InitialInterval` + `MaxInterval` + `MaxRetries` |

### Strategy 接口

```go
type Strategy interface {
    NextWithRetries(retries int32) (time.Duration, bool)  // 无状态（从 DB 读重试次数）
    Next() (time.Duration, bool)                           // 有状态（内部计数）
    Report(err error) Strategy                             // 预留（自适应）
}
```

### 指数退避公式

```
interval = InitialInterval × 2^(retries-1)
if interval > MaxInterval → interval = MaxInterval
if retries > MaxRetries → return (0, false)
```

## 8. 事件系统

**包路径**：`internal/event/`

### Kafka Topic 拓扑

```
execution_report        → ReportEventConsumer      → HandleReports (单条)
execution_batch_report  → BatchReportEventConsumer  → HandleReports (批量)
task_complete           → CompleteConsumer          → NextStep / Plan完成
```

### CompleteProducer

```go
type CompleteProducer interface {
    Produce(ctx context.Context, evt Event) error
}
```

将 Event 序列化为 JSON 发送到 Kafka，下游 `CompleteConsumer` 根据 `Event.Type` 路由处理。
