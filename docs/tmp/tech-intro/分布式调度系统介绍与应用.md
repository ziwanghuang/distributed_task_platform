## 🎯 什么是分布式调度系统

**分布式调度系统**（Distributed Scheduling System）是一种在分布式环境中，协调和管理多个节点上任务执行的系统。它负责将任务合理地分配到不同的计算节点上执行，并确保任务按照预定的时间、顺序和依赖关系正确执行。

### 核心特征

1. **分布式架构**：多个调度节点协同工作，避免单点故障
2. **任务调度**：根据时间、事件或依赖关系触发任务执行
3. **负载均衡**：智能分配任务到不同的执行节点
4. **高可用性**：支持故障转移和任务重试
5. **可扩展性**：支持水平扩展，应对大规模任务调度

## 💡 分布式调度系统的核心技术

### 1. **调度策略**
```
时间调度：
- Cron表达式：0 0 2 * * ? （每天凌晨2点）
- 固定频率：每5分钟执行一次
- 固定延迟：上次执行完成后5分钟再执行

依赖调度：
- DAG工作流：任务之间的依赖关系
- 上游成功：上游任务成功后才执行
- 上游完成：上游任务完成（成功或失败）后执行

事件调度：
- 消息触发：接收到Kafka消息后执行
- API触发：通过API手动触发
- 文件触发：监控文件变化后执行
```

### 2. **负载均衡**
```
调度算法：
- 轮询（Round Robin）：依次分配到各节点
- 随机（Random）：随机选择节点
- 最少任务（Least Task）：选择任务最少的节点
- 一致性哈希：相同任务分配到相同节点
- 权重分配：根据节点性能分配
```

### 3. **高可用保障**
```
故障转移：
- 主备切换：Master故障后自动切换
- 任务重新调度：Worker故障后重新分配任务
- 心跳检测：定时检测节点存活

容错机制：
- 失败重试：自动重试N次
- 超时控制：超时自动终止
- 降级策略：系统过载时降级处理
```

## 🎯 核心结论

**推荐方案**：**Kafka延迟队列（或RocketMQ延迟消息）** ✅

**理由**：订单超时取消是典型的**延迟任务场景**，而非定时调度场景，使用消息队列的延迟机制更加高效、实时、可靠。

## 📊 两种方案对比

### 方案一：Kafka延迟队列（推荐）⭐⭐⭐⭐⭐

#### 架构设计
```
用户下单
   ↓
创建订单（状态：待支付）
   ↓
发送延迟消息到Kafka（延迟30分钟）
   ↓
[30分钟后]
   ↓
消费者接收消息
   ↓
检查订单状态
   ↓
├─ 已支付 → 忽略消息
└─ 未支付 → 取消订单 + 释放库存 + 发送通知
```

#### 技术实现（时间轮算法）
```go
// 订单创建时发送延迟消息
func CreateOrder(order *Order) error {
    // 1. 创建订单
    if err := orderRepo.Create(order); err != nil {
        return err
    }
    
    // 2. 发送延迟消息（30分钟后执行）
    msg := &DelayMessage{
        OrderID:   order.ID,
        DelayTime: 30 * time.Minute,
        Topic:     "order-timeout-check",
    }
    
    return kafkaProducer.SendDelayMessage(msg)
}

// 消费者处理超时检查
func HandleOrderTimeout(msg *Message) error {
    orderID := msg.OrderID
    
    // 1. 查询订单状态
    order, err := orderRepo.GetByID(orderID)
    if err != nil {
        return err
    }
    
    // 2. 如果已支付，直接返回
    if order.Status == OrderStatusPaid {
        log.Info("订单已支付，忽略超时检查", orderID)
        return nil
    }
    
    // 3. 使用分布式锁避免重复取消
    lock := redis.Lock("order:cancel:" + orderID)
    if !lock.TryLock() {
        return nil
    }
    defer lock.Unlock()
    
    // 4. 取消订单
    if err := CancelOrder(order); err != nil {
        return err
    }
    
    // 5. 释放库存
    if err := ReleaseInventory(order); err != nil {
        return err
    }
    
    // 6. 发送通知
    SendCancelNotification(order)
    
    return nil
}
```

#### 优势 ✅
1. **实时性高**：消息到期立即触发，延迟<100ms
2. **性能强**：支持百万级并发，QPS可达10万+
3. **可靠性高**：消息持久化，不会丢失
4. **扩展性好**：水平扩展消费者即可
5. **解耦性强**：订单服务和取消逻辑解耦
6. **成本低**：无需额外的调度系统

#### 劣势 ❌
1. **Kafka原生不支持延迟消息**：需要自己实现时间轮或使用RocketMQ
2. **时间精度有限**：通常只支持固定的延迟级别（如1s、5s、10s、30s、1m、2m、5m、10m、30m）
3. **不支持取消**：消息发出后难以取消（需要在消费端判断）

### 方案二：分布式调度系统 ⭐⭐⭐

#### 架构设计
```
用户下单
   ↓
创建订单（状态：待支付）
   ↓
创建调度任务（30分钟后执行）
   ↓
调度系统存储任务到数据库
   ↓
[调度器轮询或时间轮触发]
   ↓
[30分钟后]
   ↓
调度器触发任务执行
   ↓
Worker执行超时检查逻辑
   ↓
检查订单状态
   ↓
├─ 已支付 → 任务完成
└─ 未支付 → 取消订单 + 释放库存 + 发送通知
```

#### 技术实现
```go
// 订单创建时创建调度任务
func CreateOrder(order *Order) error {
    // 1. 创建订单
    if err := orderRepo.Create(order); err != nil {
        return err
    }
    
    // 2. 创建调度任务
    task := &ScheduleTask{
        TaskID:      uuid.New(),
        OrderID:     order.ID,
        ExecuteTime: time.Now().Add(30 * time.Minute),
        TaskType:    "order-timeout-check",
        Status:      TaskStatusPending,
    }
    
    return scheduleRepo.CreateTask(task)
}

// 调度器定时扫描待执行任务
func ScheduleLoop() {
    ticker := time.NewTicker(10 * time.Second)
    defer ticker.Stop()
    
    for range ticker.C {
        // 1. 查询待执行的任务
        tasks, err := scheduleRepo.GetPendingTasks(time.Now())
        if err != nil {
            log.Error("查询待执行任务失败", err)
            continue
        }
        
        // 2. 分发任务到Worker
        for _, task := range tasks {
            go ExecuteTask(task)
        }
    }
}

// Worker执行任务
func ExecuteTask(task *ScheduleTask) error {
    // 1. 查询订单状态
    order, err := orderRepo.GetByID(task.OrderID)
    if err != nil {
        return err
    }
    
    // 2. 如果已支付，直接返回
    if order.Status == OrderStatusPaid {
        return scheduleRepo.UpdateTaskStatus(task.TaskID, TaskStatusCompleted)
    }
    
    // 3. 取消订单
    if err := CancelOrder(order); err != nil {
        return err
    }
    
    // 4. 更新任务状态
    return scheduleRepo.UpdateTaskStatus(task.TaskID, TaskStatusCompleted)
}
```

#### 优势 ✅
1. **功能丰富**：支持复杂的调度策略（Cron、依赖调度等）
2. **可视化管理**：提供任务管理界面
3. **任务可取消**：可以在执行前取消任务
4. **统一管理**：所有定时任务统一管理
5. **审计日志**：完整的任务执行记录

#### 劣势 ❌
1. **实时性差**：通常有10秒-1分钟的轮询延迟
2. **性能较低**：需要频繁查询数据库，QPS受限
3. **资源消耗大**：需要额外的调度系统和数据库
4. **复杂度高**：引入额外的系统组件
5. **成本高**：需要维护调度系统

// 主流程：Kafka延迟消息
func CreateOrder(order *Order) error {
    // 1. 创建订单
    if err := orderRepo.Create(order); err != nil {
        return err
    }
    
    // 2. 发送延迟消息（主流程）
    msg := &DelayMessage{
        OrderID:   order.ID,
        DelayTime: 30 * time.Minute,
        Topic:     "order-timeout-check",
    }
    if err := kafkaProducer.SendDelayMessage(msg); err != nil {
        log.Error("发送延迟消息失败", err)
        // 不影响订单创建，由兜底机制处理
    }
    
    return nil
}

// 兜底机制：定时扫描超时订单
func CompensateTimeoutOrders() {
    ticker := time.NewTicker(10 * time.Minute)
    defer ticker.Stop()
    
    for range ticker.C {
        // 1. 查询超时未支付的订单
        // 创建时间 < 当前时间-30分钟 且 状态=待支付
        orders, err := orderRepo.GetTimeoutOrders(30 * time.Minute)
        if err != nil {
            log.Error("查询超时订单失败", err)
            continue
        }
        
        // 2. 批量取消订单
        for _, order := range orders {
            if err := CancelOrderWithLock(order); err != nil {
                log.Error("取消订单失败", order.ID, err)
            }
        }
        
        log.Info("兜底机制处理完成", len(orders))
    }
}

// 带分布式锁的取消订单
func CancelOrderWithLock(order *Order) error {
    // 1. 分布式锁避免重复取消
    lock := redis.Lock("order:cancel:" + order.ID)
    if !lock.TryLock() {
        return nil
    }
    defer lock.Unlock()
    
    // 2. 再次检查订单状态（双重检查）
    latestOrder, err := orderRepo.GetByID(order.ID)
    if err != nil {
        return err
    }
    
    if latestOrder.Status != OrderStatusPending {
        return nil // 订单状态已变更，无需取消
    }
    
    // 3. 取消订单
    return CancelOrder(latestOrder)
}

    // 计算任务应该放在哪个槽位

    tw.slots[slot].PushBack(task)
