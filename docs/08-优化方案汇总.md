## 1. TaskExecution 表索引缺失（性能 · P0）

### 现状分析

`Task` 表定义了 5 个索引（`uniq_idx_name`、`idx_schedule_node_id_status`、`idx_next_time_status_utime`、`idx_plan_id`、`idx_type`），覆盖完善。

但 **`TaskExecution` 表没有定义任何索引**（除主键外），所有 GORM struct tag 中完全没有 `index:` 声明。

### 问题定位

四大补偿器的核心查询全部在做**全表扫描**：

| 查询方法 | WHERE 条件 | 当前索引 | 影响 |
|---------|-----------|---------|------|
| `FindRetryableExecutions` | `status='FAILED_RETRYABLE' AND next_retry_time <= ?` | ❌ 无 | 重试补偿器每 1s 一次全表扫描 |
| `FindTimeoutExecutions` | `deadline <= ? AND status='RUNNING'` | ❌ 无 | 中断补偿器每 1s 一次全表扫描 |
| `FindReschedulableExecutions` | `status='FAILED_RESCHEDULED' AND sharding_parent_id ...` | ❌ 无 | 重调度补偿器全表扫描 |
| `FindShardingParents` | `sharding_parent_id=0 AND status='RUNNING'` ORDER BY `utime` | ❌ 无 | 分片补偿器全表扫描 |
| `FindShardingChildren` | `sharding_parent_id = ?` | ❌ 无 | 每个分片父任务触发一次 |
| `FindByTaskID` | `task_id = ?` | ❌ 无 | 查询历史执行记录 |
| `FindExecutionByPlanID` | `task_plan_exec_id = ?` | ❌ 无 | Plan 工作流查询 |

当 `task_executions` 表达到 10 万级数据量时，这些查询的耗时会从毫秒级退化到秒级。

### 具体方案

在 `TaskExecution` struct 中添加索引定义：

```go
type TaskExecution struct {
    ID int64 `gorm:"type:bigint;primaryKey;autoIncrement;"`

    // 为补偿器查询添加索引
    TaskID           int64  `gorm:"...; index:idx_task_id"`
    Status           string `gorm:"...; index:idx_status_next_retry_time,priority:1; index:idx_status_deadline,priority:1; index:idx_sharding_parent_status,priority:2"`
    NextRetryTime    int64  `gorm:"...; index:idx_status_next_retry_time,priority:2"`
    Deadline         int64  `gorm:"...; index:idx_status_deadline,priority:2"`
    ShardingParentID sql.NullInt64 `gorm:"...; index:idx_sharding_parent_status,priority:1; index:idx_sharding_parent_id"`
    TaskPlanExecID   int64  `gorm:"...; index:idx_task_plan_exec_id"`

    // ... 其他字段不变
}
```

**需要添加的 6 个索引**：

| 索引名 | 字段 | 服务于 |
|--------|------|--------|
| `idx_status_next_retry_time` | `(status, next_retry_time)` | `FindRetryableExecutions` |
| `idx_status_deadline` | `(status, deadline)` | `FindTimeoutExecutions` |
| `idx_sharding_parent_status` | `(sharding_parent_id, status)` | `FindShardingParents` |
| `idx_sharding_parent_id` | `(sharding_parent_id)` | `FindShardingChildren` |
| `idx_task_id` | `(task_id)` | `FindByTaskID` |
| `idx_task_plan_exec_id` | `(task_plan_exec_id)` | `FindExecutionByPlanID` |

### 预期收益

- 补偿器查询从全表扫描降至索引查找：**查询耗时降低 95%+**
- 在百万级执行记录下，查询耗时从秒级回到毫秒级
- 投入成本：约 10 行 struct tag 修改 + 一次数据库 migrate

## 2. goroutine 并发控制与 panic 防护（可靠性 · P0）

### 现状分析

通过源码审查发现：

1. **`normal_task_runner.go`** 中有 **4 处** `go func()` 无并发控制：
   - `handleNormalTask()`（第 101 行）：每个普通任务启动一个 goroutine
   - `handleShardingTask()`（第 141 行）：N 个分片 = N 个 goroutine
   - `Retry()`（第 194 行）：每次重试启动一个 goroutine
   - `Reschedule()`（第 236 行）：每次重调度启动一个 goroutine

2. **`ioc/app.go`** 的 `StartTasks()`：6 个后台 goroutine **无 recover**

3. **全项目 0 处 `recover()` 调用**，但有 **52+ 处 `panic()`**（含 2 处 `panic("implement me")`）

### 问题定位

- `FindSchedulableTasks` 返回 100 个任务 → 瞬间启动 100 个 goroutine → 下游 gRPC 连接池被打满
- 分片任务 10 个分片 → 10 个 goroutine → 叠加上面 100 个 → 可能同时 1000+ goroutine
- 任何一个 goroutine panic（例如 `HTTPInvoker.Prepare` 的 `panic("implement me")`）→ 整个进程崩溃

### 具体方案

**方案 A：Worker Pool 限制并发**

```go
type NormalTaskRunner struct {
    // ... 现有字段
    workerPool chan struct{}  // 并发控制信号量
}

func NewNormalTaskRunner(..., maxConcurrency int) *NormalTaskRunner {
    return &NormalTaskRunner{
        // ...
        workerPool: make(chan struct{}, maxConcurrency), // 默认 50
    }
}

func (s *NormalTaskRunner) handleNormalTask(ctx context.Context, task domain.Task) error {
    execution, err := s.execSvc.Create(ctx, ...)
    if err != nil { ... }

    // 获取令牌（阻塞等待或超时）
    select {
    case s.workerPool <- struct{}{}:
    case <-ctx.Done():
        return ctx.Err()
    }

    go func() {
        defer func() {
            <-s.workerPool  // 释放令牌
            if r := recover(); r != nil {
                s.logger.Error("goroutine panic recovered",
                    elog.Any("panic", r),
                    elog.String("stack", string(debug.Stack())))
            }
        }()
        state, err1 := s.invoker.Run(ctx, execution)
        // ...
    }()
    return nil
}
```

**方案 B：后台任务 Supervisor 模式**

```go
func (a *SchedulerApp) StartTasks(ctx context.Context) {
    for _, t := range a.Tasks {
        t := t
        go func() {
            for {
                func() {
                    defer func() {
                        if r := recover(); r != nil {
                            elog.Error("后台任务 panic，即将重启",
                                elog.Any("panic", r),
                                elog.String("stack", string(debug.Stack())))
                        }
                    }()
                    t.Start(ctx)
                }()

                select {
                case <-ctx.Done():
                    return
                case <-time.After(3 * time.Second):
                    // 3 秒后自动重启
                }
            }
        }()
    }
}
```

### 预期收益

- 消除进程级崩溃风险（panic → recover + 自动重启）
- 控制并发上限，保护下游 gRPC 连接池
- 投入成本：约 100 行代码改动

## 3. Snowflake ID 生成器重写（正确性 · P0）

### 现状分析

`pkg/id_generator/snowflake.go` 中的实现：

```go
func (n *SnowflakeNode) Generate() int64 {
    seq := atomic.AddInt64(&n.sequence, 1) & 0xFFF  // 12 bit mask
    ts := time.Now().UnixMilli() - epochMillis
    return (n.bizID << 53) | (seq << 41) | ts
}
```

### 问题定位

1. **sequence 不与毫秒绑定**：标准 Snowflake 在毫秒切换时归零 sequence，但这里 sequence 是全局递增的
2. **sequence 12 bit = 4096**：一旦计数超过 4096 次 mask 回绕，**同一毫秒内可能产生重复 ID**
3. **ID 不单调递增**：同一毫秒两次调用，timestamp 相同但 sequence 不同，排序结果取决于 sequence 的高位分布

### 具体方案

参考标准 Twitter Snowflake 算法重写：

```go
type SnowflakeNode struct {
    mu        sync.Mutex
    bizID     int64
    sequence  int64
    lastMilli int64
}

func (n *SnowflakeNode) Generate() int64 {
    n.mu.Lock()
    defer n.mu.Unlock()

    now := time.Now().UnixMilli() - epochMillis

    if now == n.lastMilli {
        n.sequence = (n.sequence + 1) & 0xFFF // 12 bit
        if n.sequence == 0 {
            // 同一毫秒 sequence 用尽，等待下一毫秒
            for now <= n.lastMilli {
                now = time.Now().UnixMilli() - epochMillis
            }
        }
    } else {
        n.sequence = 0
    }

    n.lastMilli = now
    return (n.bizID << 53) | (n.sequence << 41) | now
}
```

### 预期收益

- 消除 ID 重复风险
- ID 单调递增，利于数据库 B+ 树索引顺序写入
- 投入成本：约 30 行代码重写

## 4. 分片子任务创建原子性保证（正确性 · P0）

### 现状分析

`normal_task_runner.go` 的 `handleShardingTask()` 中，子任务创建和异步调用在同一个循环里：

```go
// 当前实现（简化）
executions, err := s.execSvc.CreateShardingChildren(ctx, ...)
// 如果 CreateShardingChildren 内部是逐个创建的，
// 创建第 5 个时失败，前 4 个已写入 DB

for i := range executions {
    go func() {
        s.invoker.Run(ctx, executions[i])
    }()
}
```

### 问题定位

如果 `CreateShardingChildren` 在批量创建过程中（例如第 5 个子任务）数据库报错：
- 前 4 个子执行记录已持久化
- 父执行记录状态为 RUNNING
- ShardingCompensator 发现父任务下只有 4 个子任务（预期 10 个）
- 子任务可能部分执行、部分缺失 → **数据不一致**

### 具体方案

```go
func (s *NormalTaskRunner) handleShardingTask(ctx context.Context, task domain.Task) error {
    // Step 1: 在事务中批量创建所有子执行记录（原子性）
    executions, err := s.execSvc.CreateShardingChildrenInTx(ctx, domain.TaskExecution{
        Task:      task,
        StartTime: time.Now().UnixMilli(),
        Status:    domain.TaskExecutionStatusRunning,
    })
    if err != nil {
        // 事务回滚，所有子记录和父记录都不会写入
        s.releaseTask(ctx, task)
        return err
    }

    // Step 2: 全部创建成功后，再批量发起异步调用
    for i := range executions {
        execution := executions[i]
        s.workerPool <- struct{}{} // 并发控制
        go func() {
            defer func() { <-s.workerPool }()
            state, _ := s.invoker.Run(
                s.WithSpecificNodeIDContext(ctx, execution.ExecutorNodeID),
                execution,
            )
            s.execSvc.UpdateState(ctx, state)
        }()
    }
    return nil
}
```

### 预期收益

- 保证分片子任务的创建要么全部成功、要么全部不存在（原子性）
- 消除"部分分片执行、部分缺失"的不一致状态

## 5. 补偿器自适应退避与并发安全（性能 · P1）

### 现状分析

四个补偿器都使用固定的 `minDuration`（1~2s）作为防空转间隔：

```go
// 当前：无论有没有数据，都等至少 1s
if elapsed < r.config.MinDuration {
    time.Sleep(r.config.MinDuration - elapsed)
}
```

同时，补偿器查找到的 Execution 被直接传给 `runner.Retry()` / `runner.Reschedule()`，**没有 Execution 级别的并发控制**。多个 Scheduler 节点可能同时对同一个 Execution 发起重试。

### 问题定位

1. **空转浪费**：无数据时固定 1s 间隔 → 每秒 1 次 SQL 查询 × 4 个补偿器 × N 个节点 = 大量无效 DB 负载
2. **有数据时不够快**：有积压时也要等 1s → 补偿吞吐上限 = batch_size / 1s
3. **多节点竞争**：V1 版本所有节点扫描全表，同一 Execution 可能被两个节点同时重试

### 具体方案

**自适应退避**：

```go
func (r *RetryCompensator) Start(ctx context.Context) {
    backoff := r.config.MinDuration // 初始 1s

    for {
        executions, err := r.execSvc.FindRetryableExecutions(ctx, r.config.BatchSize)
        if err != nil {
            backoff = min(backoff*2, 30*time.Second) // 错误时指数退避
            time.Sleep(backoff)
            continue
        }

        if len(executions) == 0 {
            backoff = min(backoff*2, 30*time.Second) // 无数据时逐步增大间隔
            time.Sleep(backoff)
            continue
        }

        backoff = r.config.MinDuration // 有数据时重置
        for i := range executions {
            r.runner.Retry(ctx, executions[i])
        }
    }
}
```

**Execution 级乐观锁**：

```sql
-- 补偿器查找时增加 CAS 占用
UPDATE task_executions
SET status = 'RETRYING', utime = ?
WHERE id = ? AND status = 'FAILED_RETRYABLE' AND next_retry_time <= ?
```

### 预期收益

- 空闲时 DB 查询从 1/s 降至 1/30s（降低 97%）
- 有积压时立即处理，吞吐量提升 3~5 倍
- 消除多节点重复重试风险

## 6. SetRunningState 竞态修复与 Deadline 原子更新（正确性 · P1）

### 现状分析

`dao/task_execution.go` 的 `SetRunningState()` 分两步操作：

```go
func (g *GORMTaskExecutionDAO) SetRunningState(ctx context.Context, id int64, ...) error {
    // Step 1: 先查询获取 MaxExecutionSeconds
    var execution TaskExecution
    g.db.Where("id = ?", id).First(&execution)

    // Step 2: 再更新
    newDeadline := now + execution.TaskMaxExecutionSeconds * 1000
    g.db.Where("id = ? AND (status = ? OR ...)").Updates(...)
}
```

### 问题定位

Step 1 和 Step 2 之间不是原子的：
1. 查询时 status = PREPARE
2. 在 Step 1 和 Step 2 之间，InterruptCompensator 发现该 Execution 旧 Deadline 已超时，标记为 FAILED
3. Step 2 的 WHERE 条件不匹配（status 已变为 FAILED）→ `RowsAffected = 0` → 返回错误
4. 但实际上 Executor 已经在正常执行了

### 具体方案

使用子查询在单条 SQL 中完成：

```go
func (g *GORMTaskExecutionDAO) SetRunningState(ctx context.Context, id int64, progress int32, executorNodeID string) error {
    now := time.Now().UnixMilli()

    result := g.db.WithContext(ctx).Exec(`
        UPDATE task_executions
        SET status = 'RUNNING',
            running_progress = ?,
            stime = ?,
            deadline = ? + task_max_execution_seconds * 1000,
            utime = ?,
            executor_node_id = ?
        WHERE id = ?
          AND status IN ('PREPARE', 'FAILED_RETRYABLE', 'FAILED_RESCHEDULED')
    `, progress, now, now, now, executorNodeID, id)

    if result.RowsAffected == 0 {
        return errs.ErrSetExecutionStateRunningFailed
    }
    return result.Error
}
```

### 预期收益

- 消除查询-更新之间的竞态窗口
- 将 2 次 DB 往返减少为 1 次
- 避免 InterruptCompensator 与 SetRunningState 的冲突

## 7. Plan 工作流进度补偿器（服务治理 · P1）

### 现状分析

Plan 的工作流推进依赖 Kafka 完成事件：

```
任务完成 → Kafka CompleteEvent → Consumer → PlanTaskRunner.NextStep()
```

### 问题定位

如果 Kafka Consumer commit offset 后、处理 `NextStep()` 前崩溃：
- 完成事件丢失
- Plan 的后继任务永远不会被启动
- Plan 永远卡在 RUNNING 状态

### 具体方案

新增 `PlanProgressCompensator`：

```go
type PlanProgressCompensator struct {
    planSvc     task.PlanService
    execSvc     task.ExecutionService
    planRunner  runner.PlanTaskRunner
    config      PlanCompensatorConfig
}

func (c *PlanProgressCompensator) Start(ctx context.Context) {
    for {
        // 查找状态为 RUNNING 的 Plan Execution
        planExecs := c.execSvc.FindRunningPlanExecutions(ctx, c.config.BatchSize)

        for _, planExec := range planExecs {
            plan := c.planSvc.GetPlan(ctx, planExec.TaskID)

            for _, task := range plan.Tasks {
                // 检查：前驱已全部完成，但当前任务未启动
                if task.CheckPre() && !task.HasExecution() {
                    c.planRunner.LaunchTask(ctx, task)
                }
            }
        }

        time.Sleep(c.config.Interval) // 默认 30s
    }
}
```

### 预期收益

- 消除 Kafka 事件丢失导致的 Plan 死锁
- 保证工作流最终一致性推进
- 补偿间隔 30s，对正常流程无影响

## 8. 引入 pkg/clock 时间抽象层（可测试性 · P1）

### 现状分析

全项目有 **90+ 处** `time.Now()` 调用，分布在 25+ 个 Go 源文件中：

| 文件 | 调用次数 |
|------|---------|
| `dao/task_execution.go` | 10 |
| `dao/sharding_task_execution.go` | 10 |
| `dao/task.go` | 7 |
| `dao/sharding_task.go` | 6 |
| `service/task/execution_service.go` | 3 |
| `compensator/*.go` | 5 |
| 其他 | ~49 |

### 问题定位

1. **无法 mock 时间**：单元测试中无法控制"当前时间"，测试超时、Deadline、重试间隔等逻辑非常困难
2. **精度切换成本高**：如果要从毫秒切换到微秒，需要修改 90+ 处代码
3. **时间漂移**：同一个业务流程中多次调用 `time.Now()` 得到不同时间，可能导致数据不一致

### 具体方案

```go
// pkg/clock/clock.go
package clock

import "time"

type Clock interface {
    Now() time.Time
    NowMilli() int64
}

// RealClock 生产环境使用
type RealClock struct{}

func (RealClock) Now() time.Time     { return time.Now() }
func (RealClock) NowMilli() int64    { return time.Now().UnixMilli() }

// MockClock 测试环境使用
type MockClock struct {
    fixed time.Time
}

func NewMockClock(t time.Time) *MockClock { return &MockClock{fixed: t} }
func (m *MockClock) Now() time.Time      { return m.fixed }
func (m *MockClock) NowMilli() int64     { return m.fixed.UnixMilli() }
func (m *MockClock) Advance(d time.Duration) { m.fixed = m.fixed.Add(d) }
```

通过 Wire 注入：所有 DAO、Service、Compensator 依赖 `clock.Clock` 接口。

### 预期收益

- 测试中可精确控制时间流逝，覆盖 Deadline、重试、续约超时等边界场景
- 精度切换只需修改 `Clock` 实现
- 投入成本：约 200 行代码（接口 + 注入 + 替换 90 处调用）

## 9. 重试策略增加 Jitter + 熔断保护（服务治理 · P1）

### 现状分析

当前指数退避的重试时间计算：

```go
// execution_service.go
execution.NextRetryTime = time.Now().Add(duration).UnixMilli()
// duration = baseInterval * 2^retryCount
```

同时，Invoker 层没有熔断机制——即使某个 Executor 节点持续失败，调度器仍会不断向其发送任务。

### 问题定位

1. **重试风暴**：如果同时有 100 个 Execution 失败，它们的重试时间高度同步（都在 2^n 秒后），下一波重试又会同时到来
2. **无熔断**：Executor 节点 OOM 后，调度器继续发任务 → 节点更难恢复 → 恶性循环

### 具体方案

**Jitter 随机抖动**：

```go
func (s *ExecutionService) calculateRetryTime(exec domain.TaskExecution) int64 {
    baseInterval := s.baseRetryInterval * time.Duration(math.Pow(2, float64(exec.RetryCount)))
    if baseInterval > s.maxRetryInterval {
        baseInterval = s.maxRetryInterval
    }

    // 添加 ±25% 的 jitter
    jitter := time.Duration(float64(baseInterval) * (0.75 + rand.Float64()*0.5))
    return time.Now().Add(jitter).UnixMilli()
}
```

**Invoker 层熔断器**：

```go
type CircuitBreakerInvoker struct {
    breakers sync.Map  // map[serviceName]*gobreaker.CircuitBreaker
    invoker  invoker.Invoker
}

func (cbi *CircuitBreakerInvoker) Run(ctx context.Context, exec domain.TaskExecution) (domain.ExecutionState, error) {
    serviceName := exec.Task.GrpcConfig.ServiceName
    cb, _ := cbi.breakers.LoadOrStore(serviceName, gobreaker.NewCircuitBreaker(gobreaker.Settings{
        Name:        serviceName,
        MaxRequests: 3,                    // 半开状态探测请求数
        Interval:    10 * time.Second,     // 统计窗口
        Timeout:     60 * time.Second,     // 熔断持续时间
        ReadyToTrip: func(counts gobreaker.Counts) bool {
            return counts.Requests >= 5 && float64(counts.TotalFailures)/float64(counts.Requests) >= 0.6
        },
    }))

    result, err := cb.(*gobreaker.CircuitBreaker).Execute(func() (interface{}, error) {
        return cbi.invoker.Run(ctx, exec)
    })
    // ...
}
```

### 预期收益

- Jitter 使重试时间分散，避免"雷鸣群效应"
- 熔断器自动隔离故障节点，保护健康节点不受影响
- 故障恢复后自动探测恢复（半开 → 关闭）

## 10. Prometheus Picker 异步预取与降级（性能 · P1）

### 现状分析

`prometheusPicker.Pick()` 在调度主循环中**同步调用** Prometheus API：

```go
func (p *PrometheusPicker) Pick(ctx context.Context, strategy ScheduleStrategy) (string, error) {
    result, _ := p.client.Query(ctx, query)  // 同步阻塞
    candidates := parseResult(result)
    return candidates[rand.Intn(len(candidates))].NodeID, nil
}
```

### 问题定位

- Prometheus 响应正常约 50~200ms
- 一旦 Prometheus 负载高或网络抖动，响应可能 > 1s
- 调度主循环被阻塞：100 个任务 × 1s/任务 = 整个调度批次延迟 100s

### 具体方案

```go
type CachedPicker struct {
    picker      *PrometheusPicker
    cache       atomic.Value  // 存储 []CandidateNode
    refreshInterval time.Duration
}

func (c *CachedPicker) Start(ctx context.Context) {
    // 后台协程定期刷新候选列表
    ticker := time.NewTicker(c.refreshInterval)  // 默认 10s
    for {
        select {
        case <-ticker.C:
            candidates, err := c.picker.fetchCandidates(ctx)
            if err != nil {
                c.logger.Warn("Prometheus 查询失败，使用上次缓存", elog.FieldErr(err))
                continue  // 降级：使用上次成功的结果
            }
            c.cache.Store(candidates)
        case <-ctx.Done():
            return
        }
    }
}

func (c *CachedPicker) Pick(ctx context.Context, strategy ScheduleStrategy) (string, error) {
    candidates, ok := c.cache.Load().([]CandidateNode)
    if !ok || len(candidates) == 0 {
        return "", nil  // 降级到默认轮询
    }
    return candidates[rand.Intn(len(candidates))].NodeID, nil
}
```

### 预期收益

- 调度主循环的 Pick 操作从 50~200ms 降至 **< 1μs**（纯内存读取）
- Prometheus 故障时自动降级到上次缓存 → 默认轮询
- 不影响智能调度的准确性（10s 刷新，数据延迟可接受）

## 11. 批量续约分批执行（性能 · P2）

### 现状分析

```go
// dao/task.go - Renew
func (g *GORMTaskDAO) Renew(ctx context.Context, scheduleNodeID string) error {
    g.db.Where("schedule_node_id = ? AND status = ?", scheduleNodeID, StatusPreempted).
        Updates(map[string]any{"version": gorm.Expr("version + 1"), "utime": ...})
}
```

单条 SQL 批量更新所有该节点持有的任务。

### 问题定位

如果一个节点持有 500 个任务，这条 UPDATE 会：
1. 锁住 500 行
2. 行锁持有时间 = 500 行的写入时间
3. 其他事务对这些行的 Acquire、Release 操作被阻塞

### 具体方案

```go
func (g *GORMTaskDAO) Renew(ctx context.Context, scheduleNodeID string) error {
    const batchSize = 100

    var ids []int64
    g.db.Model(&Task{}).
        Where("schedule_node_id = ? AND status = ?", scheduleNodeID, StatusPreempted).
        Pluck("id", &ids)

    for i := 0; i < len(ids); i += batchSize {
        end := min(i+batchSize, len(ids))
        batch := ids[i:end]

        g.db.Model(&Task{}).
            Where("id IN ?", batch).
            Updates(map[string]any{
                "version": gorm.Expr("version + 1"),
                "utime":   time.Now().UnixMilli(),
            })
    }
    return nil
}
```

### 预期收益

- 行锁持有时间从 "500 行一次" 降至 "100 行一次"
- 其他事务的等待时间缩短 80%
- 续约总耗时增加约 5ms（5 次 × 1ms），完全可接受

## 13. 优雅关闭与 Goroutine 生命周期管理（架构 · P2）

### 现状分析

Scheduler 的 `Stop()` 方法仅取消 context，没有等待所有 goroutine 退出，也没有释放持有的任务：

```go
func (s *Scheduler) Stop() error {
    s.cancel()  // 取消 context
    return nil  // 不等待 goroutine 退出
}
```

### 问题定位

- 进程退出时，该节点持有的所有任务仍然是 PREEMPTED 状态
- 需要等到续约超时（10 分钟）后，其他节点才能重新调度
- 在此期间，这些任务处于"锁定"状态，无法被执行

### 具体方案

```go
func (s *Scheduler) Stop() error {
    s.cancel()

    // 1. 等待所有活跃的 goroutine 完成（超时 30s）
    done := make(chan struct{})
    go func() {
        s.wg.Wait()
        close(done)
    }()

    select {
    case <-done:
    case <-time.After(30 * time.Second):
        s.logger.Warn("部分 goroutine 未在 30s 内退出")
    }

    // 2. 释放所有持有的任务
    ctx, cancel := context.WithTimeout(context.Background(), 10*time.Second)
    defer cancel()

    if err := s.dao.ReleaseAllByNode(ctx, s.nodeID); err != nil {
        s.logger.Error("释放节点任务失败", elog.FieldErr(err))
    } else {
        s.logger.Info("已释放所有持有的任务", elog.String("nodeID", s.nodeID))
    }

    return nil
}
```

### 预期收益

- 滚动更新时任务不会被锁 10 分钟
- 节点退出后 < 10s 内任务即可被其他节点调度
- 减少滚动部署期间的任务调度空窗期

func (c *BatchReportConsumer) consume(ctx context.Context, reports []domain.Report) error {
    // 按类型分组
    progressUpdates := make([]domain.Report, 0)
    stateChanges := make([]domain.Report, 0)

    for _, r := range reports {
        if r.Status == domain.ExecStatusRunning && r.Progress > 0 {
            progressUpdates = append(progressUpdates, r)
        } else {
            stateChanges = append(stateChanges, r)
        }
    }

    // 进度更新：批量写入
    if len(progressUpdates) > 0 {
        c.execRepo.BatchUpdateProgress(ctx, progressUpdates)
    }

    // 状态变更：逐条处理（涉及状态机逻辑，不能简单批量）

    }
    return nil
}

## 18. 数据库层引入多级缓存（性能 · P3）

### 现状分析

项目中 Redis 依赖已声明（`config.yaml` 有 Redis 配置），但代码中完全未使用。所有操作（调度、抢占、续约、补偿）都以 MySQL 为中心。

### 问题定位

在百万级任务量下：
- `FindSchedulableTasks` 每 10s 扫描一次
- 4 个补偿器各自 1s 扫描一次
- Renew 每 5s 批量更新
- **单节点 DB QPS ≈ 10 + 4 + 1 = 15/s（核心 SQL）**
- 3 个节点 → 45/s（核心 SQL）
- 加上 Acquire/Release/UpdateState → 可能达到 100+ QPS

当前场景还可以承受，但任务量翻 10 倍后 MySQL 会成为瓶颈。

### 具体方案

```
请求 → L1 本地缓存（Ristretto）→ L2 Redis → L3 MySQL
       命中率 90%+               命中率 95%+    最终兜底
```

适用场景（只读 or 低频变更）：
- `GetByID(taskID)` → Task 配置变更频率极低
- `FindByPlanID(planID)` → Plan 结构不变
- Prometheus Picker 的候选列表 → 已在方案 10 中通过 `atomic.Value` 缓存

**不适用**：`FindSchedulableTasks`、`Acquire`、`Renew` 等写操作和实时性要求高的查询。

### 预期收益

- Task 配置类查询 DB 命中从 100% 降至 5%
- Plan 解析的 N 次 `FindByID` 从 DB 往返变为内存读取
- 为百万级任务量提供扩展空间
