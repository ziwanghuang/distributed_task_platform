## 一、整体架构设计（15 题）

### Q1 ⭐ 请用一两句话概括这个项目解决什么问题？
**考察点**: 项目定位

**详细解答**:

这是一个企业级分布式任务调度平台，对标 Apache DolphinScheduler，用 Go 从零实现。核心解决分布式环境下三个问题：**任务什么时候跑**（Cron 调度 + 下次执行时间计算）、**在哪个节点跑**（Prometheus 指标驱动的智能节点选择）、**失败了怎么办**（4 种补偿器 × V1/V2 双版本）。

支持三种任务类型：离线批处理任务、实时流处理任务、数据同步任务。任务可以通过 gRPC/HTTP/本地函数三种方式执行，并支持工作流编排（ANTLR4 DSL → DAG）和分片并行执行。

### Q2 ⭐⭐ 项目的整体分层架构是怎样的？为什么选择 DDD 分层？
**考察点**: 架构思维

**详细解答**:

采用 DDD 四层架构，从上到下：

1. **domain 层**（`internal/domain/`）— 纯领域模型，定义 Task、TaskExecution、Plan、PlanTask 等核心实体，以及状态枚举（TaskStatus、TaskExecutionStatus）、分片规则（ShardingRule）、调度参数计算等业务规则。**零外部依赖**，不引入任何框架/DB 包。
2. **service 层**（`internal/service/`）— 业务编排层，包含 Scheduler（调度引擎）、Runner（执行分发）、Invoker（远程调用）、Picker（节点选择）、Acquirer（任务抢占）、Compensator（补偿器）。
3. **repository 层**（`internal/repository/`）— 仓储抽象，定义 TaskRepository/TaskExecutionRepository 接口，屏蔽持久化细节。
4. **dao 层**（`internal/repository/dao/`）— 数据访问实现，GORM 操作 MySQL。

选择 DDD 的原因：
- 调度平台业务规则复杂——补偿策略选择、分片规则计算（Range/WeightedDynamicRange）、DAG 前置条件检查、Cron 表达式解析等，这些逻辑应该沉淀在 domain 层
- 避免 service 层膨胀成"上帝类"——比如 `Task.CalculateNextTime()` 放在 domain 而不是 service
- 便于单元测试——domain 层无外部依赖，可以直接测试业务规则

### Q3 ⭐⭐ 核心模块有哪些？它们之间的调用关系是怎样的？
**考察点**: 模块划分能力

**详细解答**:

六大核心模块及其调用链：

```
Scheduler（调度引擎）
  ├── scheduleLoop: 定时触发
  │   ├── taskSvc.SchedulableTasks() → 从 DB 拉取到期任务
  │   ├── executorNodePicker.Pick() → 向 Prometheus 查询选节点
  │   ├── runner.Run() → 分发执行
  │   │   ├── acquirer.Acquire() → 乐观锁抢占任务
  │   │   ├── execSvc.Create() → 创建 TaskExecution 记录
  │   │   └── invoker.Run() → 发起 gRPC/HTTP/Local 调用
  │   └── loadChecker.Check() → 三层负载检查决定是否降速
  └── renewLoop: 定时续约
      └── acquirer.Renew() → 更新 utime 维持抢占
```

补偿器独立运行，形成"正向调度 + 逆向补偿"双通道：
- **RetryCompensator** → `execSvc.FindRetryableExecutions()` → `runner.Retry()`
- **RescheduleCompensator** → `execSvc.FindReschedulableExecutions()` → `runner.Reschedule()`
- **InterruptCompensator** → `execSvc.FindTimeoutExecutions()` → `grpcClient.Interrupt()`
- **ShardingCompensator** → `execSvc.FindShardingParents()` → 聚合子任务 → `execSvc.UpdateScheduleResult()`

### Q4 ⭐⭐ 为什么选择 Go 而不是 Java 来实现调度平台？
**考察点**: 技术选型思考

**详细解答**:

四个核心原因：

1. **并发模型天然适配**：调度平台需要大量并发——每个任务执行是独立的 goroutine，续约、补偿器各自独立循环。Go 的 goroutine（~2KB 初始栈）比 Java 线程（~1MB 栈）轻量 500 倍，单机轻松支撑数万并发任务。源码中 `handleShardingTask` 直接 `go func()` 为每个子分片启动 goroutine，Java 需要线程池管理。

2. **gRPC 原生支持**：项目的核心通信协议是 gRPC（ExecutorService、ReporterService），Go 是 gRPC 的一等公民，proto 编译、拦截器链、自定义 Balancer/Resolver 都有原生支持。项目中自定义了 `routingBalancer` + `routingPicker`，Go 的 `balancer.Balancer` 接口设计清晰。

3. **部署简单**：编译为单二进制文件，无需 JVM。配合 ego 框架的配置管理 + 生命周期管理，一个 binary + 一个 config.yaml 即可部署。

4. **资源占用低**：调度器作为"控制面"不需要大量内存，Go 程序典型内存占用 50-200MB，Java 的 Spring Boot 应用通常 500MB+。在 K8s 环境中，更低的资源占用意味着更高的部署密度。

### Q6 ⭐⭐⭐ Scheduler 实现了 `ego.server.Server` 接口，这样做有什么好处？
**考察点**: 框架集成思维

**详细解答**:

源码中 `var _ server.Server = &Scheduler{}` 明确实现了 ego 的 Server 接口，需要实现 `Start()`、`Stop()`、`GracefulStop(ctx)`、`Info()`、`Init()` 等方法。

好处：
1. **生命周期统一管理**：ego 框架会在启动时调用 `Start()`，收到 SIGTERM/SIGINT 时调用 `GracefulStop()`。调度器不需要自己处理信号，框架帮你搞定。
2. **健康检查集成**：`Info()` 方法返回 `ServiceInfo`，其中 `Healthy = s.ctx.Err() == nil`，框架可以暴露健康检查端点，K8s 的 liveness/readiness 探针直接对接。
3. **指标采集**：ego 框架自动为 Server 组件注入 Prometheus 指标（启动时间、状态等）。
4. **统一日志**：使用 `elog.DefaultLogger.With(elog.FieldComponentName("SchedulerV2"))` 创建带组件名的 logger，日志中自动带上 "SchedulerV2" 标识。

具体实现：`Start()` 中启动两个 goroutine（`go s.scheduleLoop()` + `go s.renewLoop()`），`GracefulStop()` 通过 `s.cancel()` 取消 context 通知两个循环退出。

### Q7 ⭐⭐ 项目中的 IoC 层（`ioc/` 包）起什么作用？为什么单独抽出来？
**考察点**: 依赖管理

**详细解答**:

`ioc/` 包包含 19 个文件，每个文件一个 `Init` 函数，负责组件的初始化和依赖装配：

- `ioc/scheduler.go` — `InitScheduler()` 装配 Scheduler，注入 Runner、TaskService、Acquirer、Picker、LoadChecker 等全部依赖
- `ioc/runner.go` — `InitRunner()` 装配 NormalTaskRunner → PlanRunner → Dispatcher
- `ioc/compensator.go` — 4 个补偿器的 Init 函数
- `ioc/db.go`/`etcd.go`/`mq.go` — 基础设施初始化

单独抽出来的好处：
1. **避免 main 函数膨胀**：main 只需调用 `ioc.InitXxx()` 系列函数
2. **初始化顺序集中管控**：DB 先于 DAO，DAO 先于 Repository，Repository 先于 Service——这个顺序在 ioc 层一目了然
3. **便于替换实现**：测试时可以写 `ioc_test/` 注入 Mock 实现
4. **循环依赖检测**：编译期就能发现——如果 A 依赖 B、B 又依赖 A，Go 编译器直接报错

### Q8 ⭐⭐ 请画出从"创建任务"到"任务执行完成"的完整链路。
**考察点**: 全链路理解

**详细解答**:

```
用户创建任务
    │  API 层接收请求
    ▼
TaskService.Create()
    │  写入 tasks 表，status=ACTIVE，计算 next_time
    ▼
Scheduler.scheduleLoop() 定时触发（如每 3 秒）
    │  SELECT * FROM tasks WHERE next_time<=NOW() AND (status='ACTIVE' OR (status='PREEMPTED' AND utime<=NOW()-timeout))
    ▼
发现可调度任务
    │
    ├── executorNodePicker.Pick() → PromQL 查 Prometheus → 返回 TopN 候选 → 随机选 1 个
    │   如果 Pick 成功 → balancer.WithSpecificNodeID(ctx, nodeID) 写入 context
    │   如果 Pick 失败 → 使用原始 ctx（fallback 到 Round-Robin）
    ▼
runner.Run(ctx, task)
    │
    ├── acquirer.Acquire(taskID, version, nodeID)
    │   UPDATE tasks SET status='PREEMPTED', version=version+1 WHERE id=? AND version=?
    │   如果 RowsAffected==0 → 抢占失败，跳过
    │
    ├── execSvc.Create() → 创建 TaskExecution 记录，status=PREPARE
    │
    └── go func() {  // 异步执行
            invoker.Run(ctx, execution)
            │  GRPCInvoker → grpcClient.Execute(ctx, &ExecuteRequest{...})
            │  routingPicker 从 ctx 读取 SpecificNodeID → 精准路由到目标节点
            │
            执行节点执行任务...
            │
            执行节点通过 ReporterService.Report() 上报状态
            │  → Kafka ReportEventProducer 异步处理
            │  → execSvc.UpdateState() 更新 TaskExecution 状态
        }

并行：Scheduler.renewLoop() 每 N 秒
    └── acquirer.Renew(nodeID) → UPDATE tasks SET utime=NOW() WHERE schedule_node_id=? AND status='PREEMPTED'

任务完成后：
    ├── 普通任务：更新 TaskExecution.status=SUCCESS/FAILED
    └── Plan 任务：发送 Kafka CompleteEvent → PlanTaskRunner.NextStep() → 推进 DAG 下一步
```

### Q9 ⭐⭐ 项目的配置管理是怎么做的？有哪些关键配置项？
**考察点**: 工程化能力

**详细解答**:

使用 ego 框架的 YAML 配置体系（`config/config.yaml`），关键配置分六大块：

1. **调度器参数**：`batchTimeout`（单次调度超时）、`batchSize`（批量拉取数量）、`scheduleInterval`（调度间隔）、`renewInterval`（续约间隔）、`preemptedTimeout`（PREEMPTED 状态超时时间）
2. **智能调度**：`topNCandidates`（TopN 候选数）、`timeWindow`（指标查询时间窗口如 5m）、`jobName`（Prometheus job 名）、`queryTimeout`（查询超时）
3. **负载检查**：三层配置——LimiterChecker（rate/burst）、DatabaseLoadChecker（指标名/阈值/查询 range）、ClusterLoadChecker（CPU/Memory 阈值）
4. **补偿器**：4 个补偿器各自的 `batchSize`、`minDuration`、`maxRetryCount`、`prepareTimeoutWindowMs`
5. **中间件**：MySQL DSN、etcd 地址、Kafka broker + topic 配置、Prometheus 地址
6. **分库分表**：数据库数量、表数量

### Q11 ⭐⭐⭐ 如果要新增一种任务类型（比如 Spark 任务），需要改哪些地方？
**考察点**: 扩展性理解

**详细解答**:

得益于策略模式，改动只需 3 处：

1. **新增 Invoker 实现**：创建 `invoker/spark.go`，实现 `Invoker` 接口的 `Name()`/`Run()`/`Prepare()` 方法。`Run()` 中通过 Spark REST API 提交作业。
2. **注册到 Dispatcher**：在 `invoker/dispatcher.go` 的路由 map 中加一行 `"SPARK": sparkInvoker`。
3. **配置扩展**：在 `domain/task.go` 中新增 `SparkConfig` 结构体（集群地址、jar 路径等），在 `Task` 结构体中新增字段。

**不需要改的**：调度引擎（Scheduler）、Runner、Picker、LoadChecker、Compensator —— 它们只关心接口，不关心具体实现。这就是策略模式的价值。

### Q12 ⭐ 项目的 proto 文件定义了哪些服务？
**考察点**: RPC 接口设计

**详细解答**:

两个核心 gRPC 服务定义在 `api/proto/` 下：

**ExecutorService**（执行节点暴露的服务，调度器调用）：
- `Execute(ExecuteRequest)` → 执行任务，传入 eid（execution ID）、name、params
- `Interrupt(InterruptRequest)` → 中断执行中的任务
- `Query(QueryRequest)` → 查询任务执行状态
- `Prepare(PrepareRequest)` → 预检查（执行前验证环境是否就绪）

**ReporterService**（调度器暴露的服务，执行节点回调）：
- `Report(ReportRequest)` → 单条状态上报
- `BatchReport(BatchReportRequest)` → 批量上报

核心消息：`ExecutionState` 包含 eid、status（枚举：RUNNING/SUCCESS/FAILED/FAILED_RETRYABLE/FAILED_RESCHEDULABLE）、progress（0-100）、rescheduleParams、executorNodeId。

### Q13 ⭐⭐⭐ 项目如何处理优雅关闭？
**考察点**: 生产级能力

**详细解答**:

源码中 `GracefulStop` 的实现：

```go
func (s *Scheduler) GracefulStop(_ context.Context) error {
    s.cancel()  // 取消 context
    return nil
}
```

当 `cancel()` 被调用：
- `scheduleLoop` 中 `s.ctx.Err() != nil` 检查通过，循环退出
- `renewLoop` 中 `<-s.ctx.Done()` 触发，循环退出

**已知缺陷**：
1. **没有 drain 机制** — 正在执行中的 goroutine（`go func()` 调用 `invoker.Run`）不受 context 取消影响（它们用的是新 context），关闭时这些 goroutine 会"悬空"
2. **没有等待完成** — `GracefulStop` 调用 `cancel()` 后直接返回，不等待 scheduleLoop/renewLoop 实际退出
3. **已抢占任务不释放** — 关闭时 PREEMPTED 状态的任务不会被 Release，需要等 InterruptCompensator 检测到超时后回收

改进方案：用 `sync.WaitGroup` 等待所有 goroutine 退出，关闭前释放所有已抢占任务，增加 `shutdownTimeout` 配置。

### Q14 ⭐⭐⭐ 这个项目和 DolphinScheduler/XXL-Job 的核心差异在哪？
**考察点**: 方案对比能力

**详细解答**:

| 维度 | 本项目 | DolphinScheduler | XXL-Job |

### Q15 ⭐⭐ 项目的依赖注入是手动还是自动的？有什么利弊？
**考察点**: 工程化决策

**详细解答**:

**手动注入**。`ioc/` 包中 19 个 Init 函数显式构造依赖链。例如 `ioc/scheduler.go`：

```go
func InitScheduler(runner, taskSvc, execSvc, acquirer, grpcClients, config, loadChecker, metrics, picker) *Scheduler {
    return scheduler.NewScheduler(runner, taskSvc, ..., picker)
}
```

**优点**：
- 依赖关系在代码中一目了然，IDE 可以直接跳转
- 编译期发现缺失依赖（忘记传参直接编译报错）
- 不需要运行时反射，启动快
- 不需要额外学习 DI 框架

**缺点**：
- 大量样板代码（19 个文件，每个都是构造函数调用）
- 新增组件必须手动维护初始化链
- 依赖关系变化时需要修改多个 Init 函数

**改进建议**：Go 社区推荐 Google Wire（编译期代码生成），保留编译期检查又减少样板代码。只需写 Provider 和 Injector 声明，Wire 自动生成初始化代码。

## 二、调度引擎（12 题）

### Q16 ⭐⭐ Scheduler 的核心循环（scheduleLoop）是怎么工作的？
**考察点**: 核心机制

**详细解答**:

源码 `scheduler.go` 第 112-166 行，`scheduleLoop` 是一个 `for` 无限循环：

```go
func (s *Scheduler) scheduleLoop() {
    for {
        if s.ctx.Err() != nil { return }  // 退出检查
        
        // 1. 开始计时
        stopFunc := s.metrics.StartRecordExecutionTime()
        
        // 2. 从 DB 拉取到期任务
        tasks, err := s.taskSvc.SchedulableTasks(ctx, preemptedTimeout, batchSize)
        
        // 3. 如果没有任务，sleep 后继续
        if len(tasks) == 0 {
            time.Sleep(s.config.ScheduleInterval)
            continue
        }
        
        // 4. 遍历任务，逐个调度
        for i := range tasks {
            err := s.runner.Run(s.newContext(tasks[i]), tasks[i])
        }
        
        // 5. 负载检查——调度完一批后检查系统负载
        duration, ok := s.loadChecker.Check(ctx)
        if !ok {
            time.Sleep(duration)  // 负载过高，主动降速
        }
    }
}
```

关键设计点：
- **没有用 Ticker**：scheduleLoop 用的是"执行完一批 → 检查负载 → 如果没任务 sleep"的模式，而不是固定间隔触发。这样在任务密集时可以连续调度不间断。
- **负载检查在调度后**：先调度完一批，再检查负载。如果负载过高则 sleep 降速，实现了"反馈式调节"。
- **BatchSize 控制**：单次最多拉 `batchSize` 个任务，防止一次处理太多。

### Q17 ⭐⭐⭐ `newContext` 方法做了什么？为什么叫"智能调度"？
**考察点**: 调度算法

**详细解答**:

源码 `scheduler.go` 第 168-182 行：

```go
func (s *Scheduler) newContext(task domain.Task) context.Context {
    if nodeID, err := s.executorNodePicker.Pick(s.ctx, task); err == nil && nodeID != "" {
        return balancer.WithSpecificNodeID(s.ctx, nodeID)
    } else {
        return s.ctx  // fallback 到默认（Round-Robin）
    }
}
```

`Pick` 的内部实现（`prometheusPicker.pickByMetric`）：
1. 构造 PromQL：`topk(N, avg_over_time(cpu_usage[5m]) AND ON(instance) up{job="executor"} == 1)`
2. 调用 Prometheus HTTP API 执行查询
3. 解析结果提取 `node_id` 标签
4. 从 TopN 候选中 `rand.Intn(len(nodeIDs))` 随机选一个

之所以叫"智能"，因为它是**数据驱动**的决策，不是简单的轮询/随机：
- 基于**实时指标**（最近 5 分钟 CPU/Memory 均值）
- **过滤掉已下线节点**（`up == 1`）
- **TopN + 随机**避免热点（不是直接选最优，而是从较优的 N 个中随机）
- 支持**策略切换**（CPU_PRIORITY / MEMORY_PRIORITY），根据任务特性选择关注 CPU 还是 Memory

### Q18 ⭐⭐ renewLoop 的作用是什么？如果不续约会怎样？
**考察点**: 续约机制

**详细解答**:

源码 `scheduler.go` 第 185-198 行：

```go
func (s *Scheduler) renewLoop() {
    ticker := time.NewTicker(s.config.RenewInterval)
    for {
        select {
        case <-s.ctx.Done(): return
        case <-ticker.C:
            s.acquirer.Renew(s.ctx, s.nodeID)
        }
    }
}
```

`Renew` 的 SQL（`dao/task.go` 第 167 行）：
```sql
UPDATE tasks SET version=version+1, utime=NOW() WHERE schedule_node_id=? AND status='PREEMPTED'
```

本质是定期刷新 `utime`，告诉系统"我还活着，这些任务我还在处理"。

1. `utime` 停止更新
2. `InterruptCompensator.FindTimeoutExecutions()` 查询：`WHERE status='RUNNING' AND deadline < NOW()`，发现超时任务
3. 发送 `grpcClient.Interrupt()` 请求中断执行节点上的任务
4. 更新 TaskExecution 状态

同时，`FindSchedulableTasks` 中有僵尸任务检测：`WHERE status='PREEMPTED' AND utime <= NOW()-preemptedTimeoutMs`，超时未续约的 PREEMPTED 任务会被重新拉出来调度。

### Q19 ⭐⭐⭐ 任务抢占是如何实现的？为什么用乐观锁而不是分布式锁？
**考察点**: 并发控制

**详细解答**:

源码 `dao/task.go` 第 131-164 行，`Acquire` 在事务中执行：

```go
result := tx.Model(&Task{}).
    Where("id = ? AND version = ?", id, version).
    Updates(map[string]any{
        "status":           "PREEMPTED",
        "schedule_node_id": scheduleNodeID,
        "version":          gorm.Expr("version + 1"),
        "utime":            time.Now().UnixMilli(),
    })
if result.RowsAffected == 0 {
    return errs.ErrTaskPreemptFailed  // CAS 失败
}
```

等价 SQL：`UPDATE tasks SET status='PREEMPTED', schedule_node_id=?, version=version+1, utime=? WHERE id=? AND version=?`

选择乐观锁而非分布式锁的原因：

1. **冲突概率可控**：调度场景是"N 个调度器竞争 M 个任务"（M >> N），大部分任务不会被多个调度器同时抢占。乐观锁在低冲突场景下性能远优于悲观锁。
2. **无额外组件依赖**：乐观锁只需 MySQL（已有），分布式锁需要 etcd/Redis。虽然项目已经有 etcd，但给每个任务创建一把锁的开销很大（假设 1000 个待调度任务 → 1000 把锁）。
3. **性能更好**：乐观锁是一次 UPDATE（含 WHERE version=?），成功的话一次 IO 完成。分布式锁需要 Lock → 操作 → Unlock 至少三次 IO。
4. **死锁风险低**：乐观锁 CAS 失败直接返回，不存在锁超时/死锁问题。分布式锁如果 Unlock 失败，需要等 TTL 自动释放。

注：分布式锁在项目中用于 `ShardingLoopJob`（补偿器 V2 的分片巡检），因为那里是"一个分片只能有一个巡检器"的排他场景，适合悲观锁。

### Q20 ⭐⭐ `SchedulableTasks` 是怎么查询可调度任务的？
**考察点**: 查询设计

**详细解答**:

源码 `dao/task.go` 第 113-129 行：

```go
err := g.db.WithContext(ctx).
    Where("next_time <= ? AND (status = ? OR (status = ? AND utime <= ?))",
        now, StatusActive, StatusPreempted, now-preemptedTimeoutMs).
    Order("next_time ASC").
    Limit(limit).
    Find(&tasks).Error
```

等价 SQL：
```sql
SELECT * FROM tasks 
WHERE next_time <= NOW_MS 
  AND (status = 'ACTIVE' OR (status = 'PREEMPTED' AND utime <= NOW_MS - timeout))
ORDER BY next_time ASC 
LIMIT ?
```

查询逻辑解读：
1. `next_time <= NOW_MS` — 到了执行时间的任务
2. `status = 'ACTIVE'` — 正常可调度的任务
3. **OR** `status = 'PREEMPTED' AND utime <= NOW_MS - timeout` — **僵尸任务检测**：PREEMPTED 状态但长时间未续约（utime 太旧），说明抢占它的调度器可能已经挂了

索引利用：Task 表有联合索引 `idx_next_time_status_utime(next_time, status, utime)`，这个查询可以利用该索引。

`ORDER BY next_time ASC` 保证最紧急的任务优先调度，`LIMIT` 控制单批数量。

### Q21 ⭐⭐ 如果调度器挂了，正在执行的任务怎么办？
**考察点**: 容错设计

**详细解答**:

多层保护机制：

**层 1：续约停止 → 僵尸任务回收**
调度器挂了 → `renewLoop` 停止 → `tasks.utime` 不再更新 → `FindSchedulableTasks` 的僵尸检测条件命中（`status='PREEMPTED' AND utime <= NOW()-timeout`）→ 其他调度器可以重新抢占这些任务。

**层 2：InterruptCompensator 中断超时执行**
执行中的 TaskExecution 的 `deadline` 字段标记了截止时间。如果执行超时，`InterruptCompensator` 检测到后通过 gRPC `Interrupt` 中断执行节点上的任务。

**层 3：RetryCompensator 重试失败任务**
如果任务最终状态是 `FAILED_RETRYABLE`，RetryCompensator 会按重试策略重新调度。

**层 4：RescheduleCompensator 换节点重试**
如果任务标记为 `FAILED_RESCHEDULED`，会排除上次失败的节点重新调度。

**已知风险**：
- 如果执行节点也挂了（不只是调度器挂了），gRPC Interrupt 请求无法送达，需要等 Interrupt 超时后再由其他补偿机制处理
- 没有告警通知——调度器挂了只能通过指标/日志发现

### Q22 ⭐⭐ 调度器如何避免重复调度同一个任务？
**考察点**: 幂等性

**详细解答**:

通过 **乐观锁 + 状态机** 二重保证：

**第一重：状态过滤**
`FindSchedulableTasks` 只查 `status='ACTIVE'`（和超时的 PREEMPTED）。任务被抢占后状态变为 `PREEMPTED`，下次查询时自然排除。

**第二重：乐观锁 CAS**
即使两个调度器同时查到同一个 ACTIVE 任务，`Acquire` 的 CAS 更新 `WHERE id=? AND version=?` 只有一个能成功（`RowsAffected=1`），另一个 `RowsAffected=0` 返回 `ErrTaskPreemptFailed`。

这两重机制的组合实现了幂等：
- 正常情况：状态过滤已经排除了已抢占的任务
- 边界情况（并发竞争）：CAS 保证只有一个成功

### Q23 ⭐⭐⭐ 调度器的 BatchSize 参数如何设置？设大了/设小了分别有什么影响？
**考察点**: 参数调优

**详细解答**:

`BatchSize`（源码中 `config.BatchSize`）控制 `SchedulableTasks` 的 `LIMIT` 值。

**设太大（如 1000）**：
- 单次循环处理时间过长 → 影响调度及时性（后面的任务要等前面的处理完）
- 内存峰值高 → 1000 个 Task 对象 + 1000 个 TaskExecution 对象同时在内存中
- MySQL 查询压力 → 大 LIMIT 可能导致临时文件排序
- 大量 goroutine 同时启动 → `handleShardingTask` 每个子分片一个 goroutine，如果 1000 个任务各有 10 个分片 = 10000 goroutine

**设太小（如 5）**：
- 调度吞吐量不足 → 每秒只能调度 5 / 调度间隔 个任务
- 高峰期任务积压 → ACTIVE 任务越堆越多

**合理值评估公式**：
`BatchSize = 单个任务调度耗时(ms) × 可接受延迟(ms) / (调度间隔(ms) × 调度器数量)`

例如：单任务调度 5ms，可接受 1s 延迟，调度间隔 0（紧凑模式），3 个调度器 → `BatchSize ≈ 200/3 ≈ 66`。实际推荐 50~200。

### Q24 ⭐⭐ 任务的 `next_time` 是怎么计算的？
**考察点**: 调度时间计算

**详细解答**:

源码 `domain/task.go` 第 120-129 行：

```go
func (t *Task) CalculateNextTime() (time.Time, error) {
    p := cron.NewParser(
        cron.Second | cron.Minute | cron.Hour | cron.Dom | cron.Month | cron.Dow | cron.Descriptor,
    )
    s, err := p.Parse(t.CronExpr)
    if err != nil {
        return time.Time{}, err
    }
    return s.Next(time.Now()), nil
}
```

使用 `robfig/cron/v3` 库解析 Cron 表达式。支持 6 位格式（含秒）：`秒 分 时 日 月 周`。

调用时机：任务被成功调度后，`taskSvc.UpdateNextTime` 调用此方法计算下次时间并写入 DB。这样下次 `scheduleLoop` 查询时，这个任务不会被再次拉出来（因为 `next_time` 已经指向未来）。

**已知问题**：`time.Now()` 硬编码在 domain 方法中，无法在测试中控制时间。应该注入 `Clock` 接口。

### Q25 ⭐⭐⭐ 如何实现调度器的高可用？当前方案有什么局限？
**考察点**: 高可用设计

**详细解答**:

**当前方案**：多个 Scheduler 实例并行运行，通过乐观锁竞争同一批任务。天然支持水平扩展——加机器就加调度能力。

**局限**：

1. **惊群效应**：假设 10 个调度器同时查出 100 个待调度任务，每个调度器都尝试抢占全部 100 个，但每个任务只有 1 个能成功 → 90% 的 CAS 操作浪费（9 次失败 / 1 次成功）。
2. **无任务分区**：所有调度器查的是同一张表，没有按 `task_id % N` 分配（如 Kafka Consumer Group 的分区分配策略）。
3. **无 Leader 选举**：没有主备切换机制，所有实例都是对等的。如果想实现"只有 Leader 调度"以避免竞争，需要引入 etcd 选举。

**改进方案**：
- 引入**分区调度**：每个调度器只负责 `task_id % N = myIndex` 的任务
- 或者使用 **etcd Leader 选举**：只有 Leader 执行调度，其他实例热备

### Q26 ⭐⭐ 为什么 scheduleLoop 和 renewLoop 要分成两个独立循环？
**考察点**: 设计思考

**详细解答**:

**职责不同**：
- `scheduleLoop`：发现新任务并分发，涉及 DB 查询 + Prometheus 查询 + gRPC 调用，单次可能耗时数百毫秒
- `renewLoop`：维持已抢占任务的"心跳"，只是一条简单的 UPDATE SQL，几毫秒完成

**频率不同**：
- `scheduleLoop` 根据负载动态调节频率（有任务时连续跑，无任务时 sleep `ScheduleInterval`）
- `renewLoop` 固定间隔（`RenewInterval`，如 10 秒），必须稳定执行

如果合在一起：
- 调度大批量任务时（比如 200 个），处理耗时可能 2 秒 → 续约被阻塞 2 秒 → 如果 `renewInterval` 是 10 秒，实际续约间隔变成 12 秒 → 可能触发超时检测
- renewLoop 不需要和调度逻辑耦合

**考察点**: 配置理解

**详细解答**:

### Q28 ⭐⭐ Runner 接口是如何设计的？为什么要区分 Normal 和 Plan？
**考察点**: 接口抽象

**详细解答**:

源码 `runner/types.go`：

```go
type Runner interface {
    Run(ctx context.Context, task domain.Task) error
    Retry(ctx context.Context, execution domain.TaskExecution) error
    Reschedule(ctx context.Context, execution domain.TaskExecution) error
}
```

三个方法对应三种调度场景：
- `Run` — 首次执行：抢占 → 创建 Execution → Invoker 调用
- `Retry` — 重试：已有 Execution 记录，排除上次节点后重新调用
- `Reschedule` — 换节点重调度：指定新节点后重新调用

区分 Normal 和 Plan 的原因：
- **NormalTaskRunner**：直接调用 Invoker 执行单个任务，逻辑简单
- **PlanTaskRunner**：需要解析 DAG 结构，找到根节点，启动多个 goroutine 并行执行，通过 Kafka 事件驱动 `NextStep` 推进

`PlanTaskRunner` 嵌入了 `*NormalTaskRunner`（Go 的组合继承），复用了抢占、执行、释放逻辑，只重写了 `Run` 方法增加 DAG 编排。

Dispatcher（`runner/dispather.go`）根据 `task.Type` 路由：`"normal" → NormalTaskRunner`，`"plan" → PlanTaskRunner`。

### Q29 ⭐⭐⭐ NormalTaskRunner 的 `handleShardingTask` 是如何实现分片任务的？
**考察点**: 分片执行

**详细解答**:

源码 `runner/normal_task_runner.go` 第 120-158 行：

```go
func (s *NormalTaskRunner) handleShardingTask(ctx context.Context, task domain.Task) error {
    // 1. 批量创建子任务执行记录
    executions, err := s.execSvc.CreateShardingChildren(ctx, domain.TaskExecution{
        Task:      task,
        StartTime: time.Now().UnixMilli(),
        Status:    domain.TaskExecutionStatusRunning,
    })
    
    // 2. 为每个子分片启动 goroutine
    for i := range executions {
        execution := executions[i]
        go func() {
            state, err1 := s.invoker.Run(
                s.WithSpecificNodeIDContext(ctx, execution.ExecutorNodeID),  // 指定节点
                execution,
            )
            if err1 != nil { return }
            s.execSvc.UpdateState(ctx, state)
        }()
    }
    return nil
}
```

流程：
1. `task.ShardingRule.ToScheduleParams()` 根据分片规则计算出 N 组参数（如 Range 分片：[0,100], [100,200], ...）
2. `CreateShardingChildren` 批量创建 N 个 TaskExecution 记录，每个携带不同的 `ScheduleParams`
3. 为每个子任务启动独立 goroutine，通过 `WithSpecificNodeIDContext` 指定目标节点

**已知问题**：
- goroutine 没有并发控制（无 semaphore/worker pool），100 个分片 = 100 个并发 goroutine
- 没有 `recover()` — 任何一个 goroutine panic 都会导致进程崩溃
- 子任务创建非原子 — 部分创建失败无法回滚

### Q30 ⭐⭐ Invoker 的三种实现（gRPC/HTTP/Local）分别适用什么场景？

**GRPCInvoker**（`invoker/grpc.go`）：

**HTTPInvoker**（`invoker/http.go`）：

- 适用：轻量级任务（数据清理、定时通知），避免网络开销
- 方便测试（无需启动执行节点）

### Q31 ⭐⭐⭐ gRPC Invoker 是如何找到目标执行节点的？
**考察点**: 服务发现 + 路由

**详细解答**:

完整调用链路：

```
Scheduler.newContext()
  │ picker.Pick() 选出 nodeID
  │ balancer.WithSpecificNodeID(ctx, nodeID) 写入 context
  ▼
NormalTaskRunner.handleNormalTask()
  │ invoker.Run(ctx, execution)
  ▼
GRPCInvoker.Run()
  │ client := s.grpcClients.Get(serviceName)
  │ client.Execute(ctx, request)  // ctx 中携带了 nodeID
  ▼
gRPC 框架调用 routingPicker.Pick(info)
  │ info.Ctx 中读取 SpecificNodeID
  │ 遍历 p.nodeIDs 找到匹配的 SubConn
  │ 返回 PickResult{SubConn: matchedConn}
  ▼
gRPC 通过该 SubConn 发送请求到目标节点
```

`routingPicker.Pick()` 的三级优先策略（源码 `balancer/routing_picker.go` 第 30-81 行）：
1. **Specific 模式**（最高优先级）：`GetSpecificNodeID(ctx)` → 精确匹配 nodeID → 找不到返回 `codes.Unavailable`
2. **Exclude 模式**（次优先级）：`GetExcludeNode(ctx)` → 排除指定节点 → 在剩余节点中轮询
3. **Round-Robin**（默认）：无路由策略时简单轮询

### Q32 ⭐⭐ 任务的状态机有哪些状态？状态流转是怎样的？
**考察点**: 状态设计

**详细解答**:

**Task 状态**（3 种）：
- `ACTIVE` — 可调度
- `PREEMPTED` — 已被调度器抢占
- `INACTIVE` — 停止执行（可重新激活）

**TaskExecution 状态**（7 种）：
- `PREPARE` — 已创建，等待执行
- `RUNNING` — 执行中
- `SUCCESS` — 成功（终态）
- `FAILED` — 失败不可重试（终态）
- `FAILED_RETRYABLE` — 失败可重试（RetryCompensator 处理）
- `FAILED_RESCHEDULED` — 失败需换节点（RescheduleCompensator 处理）

状态流转：
```
Task:    ACTIVE ──Acquire──► PREEMPTED ──Release──► ACTIVE
                                │
                           (执行完成后)
                                
Execution:  PREPARE ──执行开始──► RUNNING ──成功──► SUCCESS
                                    │
                                    ├──失败(可重试)──► FAILED_RETRYABLE ──补偿──► (重新执行)
                                    ├──失败(换节点)──► FAILED_RESCHEDULED ──补偿──► (重新执行)
                                    └──失败(终态)──► FAILED
```

关键的 `IsTerminalStatus()`：只有 `SUCCESS` 和 `FAILED` 是终态，`FAILED_RETRYABLE` 和 `FAILED_RESCHEDULED` 是中间态（会被补偿器继续处理）。

### Q33 ⭐⭐⭐ `WithExcludedNodeIDContext` 和 `WithSpecificNodeIDContext` 是怎么配合 gRPC 路由的？
**考察点**: 上下文传递

**详细解答**:

这是一套 **context 传值 → gRPC 拦截** 的机制：

**写入端**（Runner 层）：
```go
// 首次执行：指定节点
func (s *NormalTaskRunner) WithSpecificNodeIDContext(ctx, executorNodeID) context.Context {
    return balancer.WithSpecificNodeID(ctx, executorNodeID)
}

// 重试：排除上次失败的节点
func (s *NormalTaskRunner) WithExcludedNodeIDContext(ctx, executorNodeID) context.Context {
    return balancer.WithExcludedNodeID(ctx, executorNodeID)
}
```

**读取端**（Picker 层，`routing_picker.go`）：
```go
func (p *routingPicker) Pick(info balancer.PickInfo) {
    // 1. 检查 Specific（首次执行场景）
    if specificNodeID, ok := GetSpecificNodeID(info.Ctx); ok {
        for i := range p.nodeIDs {
            if p.nodeIDs[i] == specificNodeID { return p.subConns[i] }
        }
    }
    // 2. 检查 Exclude（重试场景）
    if excludeNodeID, ok := GetExcludeNode(info.Ctx); ok {
        // 过滤掉排除节点后轮询
    }
    // 3. 默认 Round-Robin
}
```

使用场景映射：
- **Run()**（首次执行）→ Scheduler 通过 Picker 选好节点 → WithSpecificNodeID → 精准路由
- **Retry()**（重试）→ 排除上次失败节点 → WithExcludedNodeID → 轮询剩余节点
- **Reschedule()**（重调度）→ 指定补偿器选择的新节点 → WithSpecificNodeID

### Q34 ⭐⭐ 任务执行结果是如何上报的？
**考察点**: 状态上报

**详细解答**:

两种上报方式：

**方式 1：同步返回**
`GRPCInvoker.Run()` 调用 `client.Execute()` 后，执行节点直接在 gRPC Response 中返回 `ExecutionState`。Runner 收到后调用 `execSvc.UpdateState()`。适用于短任务（秒级完成）。

**方式 2：异步上报**
执行节点在运行过程中通过 `ReporterService` 主动上报：
- `Report(ReportRequest)` — 单条上报，含 `ExecutionState`（eid、status、progress）
- `BatchReport(BatchReportRequest)` — 批量上报，高频场景减少 RPC 次数

上报链路：执行节点 → gRPC `ReporterService` → `ReportEventProducer` 写入 Kafka → `ReportEventConsumer` 消费 → `execSvc.UpdateState()` 写入 DB。

通过 Kafka 异步化的好处：
- 上报不阻塞执行（fire-and-forget）
- 削峰（大量任务同时完成时不会打爆调度器）
- 解耦（执行节点不需要知道调度器的具体地址）

### Q35 ⭐⭐⭐ `CreateShardingChildren` 的实现有什么问题？
**考察点**: 代码质量

**详细解答**:

该方法创建分片子任务的 TaskExecution 记录。在分库分表场景下（`ShardingTaskExecutionDAO.BatchCreate`），使用了 GORM 的 `DryRun` 模式生成 SQL 模板再手动执行。

**问题清单**：

1. **非原子操作**：子任务分散在不同分片（可能在不同 DB），没有分布式事务。如果创建到第 5 个分片时失败，前 4 个已经写入无法回滚 → 父任务有部分子任务，ShardingCompensator 会把无子任务的父任务标记为 FAILED。
2. **无幂等保护**：如果调度器在创建子任务中途重启，重试时会重复创建 → 一个父任务可能有双份子任务。
3. **DryRun 脆弱性**：依赖 GORM 的 `DryRun` 内部实现生成 SQL 字符串，GORM 版本升级可能改变 SQL 格式。
4. **源码中有 `panic("implement me")`**：`ShardingTaskExecutionDAO` 第 496 行，某些方法未实现就直接 panic。

### Q36 ⭐⭐ 任务执行超时是怎么检测的？
**考察点**: 超时处理

**详细解答**:

通过 `InterruptCompensator` **异步被动检测**：

1. 创建 TaskExecution 时设置 `Deadline = StartTime + Task.MaxExecutionSeconds * 1000`
2. `InterruptCompensator` 定时执行 `execSvc.FindTimeoutExecutions(ctx, batchSize)`
3. 查询条件：`WHERE status IN ('RUNNING', 'PREPARE') AND deadline < NOW()`
4. 对超时任务调用 `grpcClient.Interrupt()` 发送中断请求

**局限**：
- 检测是**被动轮询**（默认每 N 秒扫一次），不是主动超时 → 检测延迟 = 轮询间隔
- 没有在执行时设置 `context.WithTimeout` → 执行节点不会主动超时
- 如果 gRPC 连接已断开，Interrupt 请求无法送达

### Q37 ⭐⭐⭐ 如果让你设计一个更好的 timeout 机制，你会怎么做？
**考察点**: 设计能力

**详细解答**:

**三层超时防御**：

**层 1：主动超时（执行侧）**
创建 Execution 时计算 timeout，在 gRPC 请求的 metadata 中传递 `grpc-timeout`。执行节点用 `context.WithTimeout(ctx, timeout)` 创建超时 context，所有阻塞操作都受此 context 控制，超时后自动取消。

**层 2：心跳检测（调度侧）**
执行节点每 N 秒通过 `ReporterService.Report` 发送心跳（进度更新）。调度器端维护 `lastHeartbeat` 时间戳，如果超过 `2*heartbeatInterval` 没有收到心跳，判定为超时。

**层 3：补偿器兜底（现有机制）**
保留 `InterruptCompensator` 作为最后的安全网，但降低轮询频率（因为层 1、2 已经覆盖了大部分场景）。

**对比现有方案**：
- 层 1 是毫秒级的主动超时（现在没有）
- 层 2 是秒级的心跳检测（现在没有）
- 层 3 是分钟级的轮询补偿（现有方案）

## 四、补偿机制（12 题）

### Q38 ⭐⭐ 项目中有哪四种补偿器？各自解决什么问题？
**考察点**: 容错设计

**详细解答**:

| 补偿器 | 触发条件 | 处理逻辑 | 目标状态 |
|--------|---------|---------|---------|
| **RetryCompensator** | `FAILED_RETRYABLE` 且重试次数未耗尽 | 调用 `runner.Retry()`，排除上次节点重新执行 | → RUNNING → SUCCESS/FAILED |
| **RescheduleCompensator** | `FAILED_RESCHEDULED` | 调用 `runner.Reschedule()`，指定新节点重新执行 | → RUNNING → SUCCESS/FAILED |
| **InterruptCompensator** | `RUNNING/PREPARE` 且 `deadline < NOW()` | 通过 gRPC `Interrupt` 中断执行节点 | → FAILED/CANCELLED |
| **ShardingCompensator** | 分片父任务 `RUNNING`，子任务全部完成 | 聚合子任务结果，更新父任务终态 | → SUCCESS/FAILED |

### Q39 ⭐⭐⭐ 补偿器的 V1 和 V2 版本有什么区别？为什么要演进？
**考察点**: 演进思维

**详细解答**:

**V1**（如 `retry.go`）：单库版本，直接查询单个 MySQL 实例。循环结构简单：

```go
for {
    executions := execSvc.FindRetryableExecutions(ctx, batchSize)
    for _, exec := range executions {
        runner.Retry(ctx, exec)
    }
    time.Sleep(minDuration)
}
```

**V2**（如 `retryv2.go`）：分库分表版本，基于 `ShardingLoopJob` 模板方法：

```go
ShardingLoopJob.Run(ctx):
    for each (db, table) in shardingStrategy.Broadcast():
        semaphore.Acquire()          // 控制并发度
        lock := dclient.NewLock(key)  // 创建分布式锁
        lock.Lock()                   // 获取锁
        go tableLoop(ctx, lock):      // 异步处理该分片
            bizLoop(ctx, lock):       // 执行业务逻辑
                biz(ctx)              // V2 补偿器实现的具体逻辑
                lock.Refresh()        // 锁续约
```

**演进原因**：
- 数据量增长后单库成为瓶颈，需要分库分表
- 分库分表后需要遍历所有 (db, table) 组合执行巡检
- 多个补偿器实例不能重复处理同一个分片 → 需要分布式锁
- 不能所有分片同时巡检（资源消耗太大）→ 需要信号量控制并发

### Q40 ⭐⭐⭐ ShardingLoopJob 的模板方法是怎么设计的？
**考察点**: 设计模式实战

**详细解答**:

源码 `pkg/loopjob/shardingloopjob.go`，核心结构：

```go
type ShardingLoopJob struct {
    shardingStrategy  sharding.ShardingStrategy  // 分片策略
    dclient           dlock.Client               // 分布式锁客户端
    biz               func(ctx context.Context) error  // 业务函数（子类注入）
    resourceSemaphore ResourceSemaphore          // 并发控制信号量
}
```

骨架流程（`Run` 方法）：
1. **外层循环**：遍历所有分片 `shardingStrategy.Broadcast()` → 返回 `[]Dst{DB, Table}`
2. **信号量控制**：`resourceSemaphore.Acquire()` — 如果达到上限就 sleep 等待
3. **分布式锁**：`dclient.NewLock(key)` + `lock.Lock()` — 确保一个分片只被一个实例处理
4. **异步处理**：`go tableLoop(ctx, lock)` — 每个分片一个 goroutine

`bizLoop` 内部：
```go
for {
    biz(ctx)           // 执行一次业务（子类实现）
    if ctx.Err() { return }
    lock.Refresh()     // 锁续约
}
```

子类只需实现 `biz` 函数。例如 RetryCompensatorV2 的 biz 就是"从当前分片查询 FAILED_RETRYABLE 记录并重试"。

### Q41 ⭐⭐ RetryCompensator 的重试策略有哪些？
**考察点**: 重试机制

**详细解答**:

两种策略，通过 `retry.NewStrategy(config)` 工厂创建：

**FixedIntervalRetryStrategy**：
- `NextRetryTime = now + fixedInterval`
- 适用：临时故障（网络抖动），期望固定间隔后恢复

**ExponentialBackoffRetryStrategy**：
- `NextRetryTime = now + initialInterval × 2^retryCount`（不超过 maxInterval）
- 适用：下游过载，需要给予恢复时间

源码中 `RetryConfig.ToRetryComponentConfig()` 默认使用指数退避：
```go
func (r *RetryConfig) ToRetryComponentConfig() retry.Config {
    return retry.Config{
        Type: "exponential",
        ExponentialBackoff: &retry.ExponentialBackoffConfig{
            InitialInterval: time.Duration(r.InitialInterval) * time.Millisecond,
            MaxInterval:     time.Duration(r.MaxInterval) * time.Millisecond,
            MaxRetries:      r.MaxRetries,
        },
    }
}
```

### Q43 ⭐⭐⭐ ShardingCompensator 是如何聚合子任务结果的？
**考察点**: 分片聚合

**详细解答**:

源码 `compensator/sharding.go` 第 102-166 行，`handle` 方法：

```go
func (r *ShardingCompensator) handle(ctx, parent TaskExecution) error {
    // 1. 查所有子任务
    children := r.execSvc.FindShardingChildren(ctx, parent.ID)
    
    // 2. 边界：无子任务 → 父任务标记 FAILED
    if len(children) == 0 {
        r.execSvc.UpdateScheduleResult(ctx, parent.ID, FAILED, ...)
        return
    }
    
    // 3. 逐个检查子任务状态
    for _, child := range children {
        if !child.Status.IsSuccess() && !child.Status.IsFailed() {
            return nil  // 有子任务还在运行，等下次补偿
        }
        if child.Status.IsFailed() { anyFailed = true }
        if child.Status.IsSuccess() { successCount++ }
    }
    
    // 4. 所有子任务完成，确定父任务终态
    if anyFailed {
        finalStatus = FAILED
    } else {
        finalStatus = SUCCESS
    }
    progress = successCount * 100 / len(children)
    
    // 5. 更新父任务 + 释放 Task 锁 + 更新下次执行时间
    r.execSvc.UpdateScheduleResult(ctx, parent.ID, finalStatus, progress, ...)
    r.taskAcquirer.Release(ctx, parent.Task.ID, r.nodeID)
    r.taskSvc.UpdateNextTime(ctx, parent.Task.ID)
}
```

关键逻辑：
- 只有**所有子任务都到达终态**（SUCCESS 或 FAILED）才聚合
- 有任一子任务 FAILED → 父任务 FAILED
- 全部 SUCCESS → 父任务 SUCCESS
- 聚合完成后释放 Task 的 PREEMPTED 锁（因为分片任务的锁从创建到所有子任务完成一直持有）

### Q44 ⭐⭐ 补偿器的执行频率如何配置？
**考察点**: 参数调优

**详细解答**:

每个补偿器都有 `MinDuration` 配置（防空转最小等待时间）。以 RetryCompensator 为例：

```go
elapsed := time.Since(startTime)
if elapsed < r.config.MinDuration {
    time.Sleep(r.config.MinDuration - elapsed)
}
```

这意味着每轮补偿至少间隔 `MinDuration`。推荐值：
- **RetryCompensator**: 30s~60s（重试不需要太频繁）
- **RescheduleCompensator**: 30s~60s（同上）
- **InterruptCompensator**: 10s~30s（超时需要及时检测）
- **ShardingCompensator**: 5s~10s（子任务完成后需要尽快聚合）

过高频率：增加 DB 查询压力；过低频率：故障恢复延迟增大。

### Q45 ⭐⭐⭐ 如果补偿器自身执行失败了怎么办？
**考察点**: 深度思考

**详细解答**:

当前实现：补偿器失败后**记录日志 → 继续下一轮循环**。以 RetryCompensator 为例：

```go
err := r.retry(ctx)
if err != nil {
    r.logger.Error("重试失败", elog.FieldErr(err))
}
// 不 return，继续循环
```

**风险**：
1. **持久性故障无感知**：DB 挂了 → 每轮都失败 → 只有日志没有告警
2. **goroutine panic 致命**：补偿器运行在独立 goroutine 中（`ioc/app.go` 启动），没有 `recover()` → 一个 panic 就直接崩进程
3. **无健康检查**：外部无法知道补偿器是否在正常工作

**改进方案**：
- 加 `defer recover()` 保护所有 goroutine
- 连续失败 N 次后触发告警（Prometheus counter + AlertManager）
- 增加补偿器健康检查指标（last_success_time、error_count）
- 失败后指数退避，避免在 DB 故障时疯狂查询加重负担

### Q46 ⭐⭐ RescheduleCompensator 和 RetryCompensator 的区别是什么？
**考察点**: 概念区分

**详细解答**:

核心区别在于 **是否显式换节点**：

| 维度 | RetryCompensator | RescheduleCompensator |
|------|------------------|----------------------|
| 触发状态 | `FAILED_RETRYABLE` | `FAILED_RESCHEDULED` |
| 调用方法 | `runner.Retry()` | `runner.Reschedule()` |
| 节点选择 | `WithExcludedNodeIDContext` — 排除上次节点 | `WithSpecificNodeIDContext` — 指定新节点 |
| 适用场景 | 临时故障（OOM、网络抖动）| 节点级故障（磁盘满、配置错误、版本不兼容）|
| 重试次数 | 受 `MaxRetries` 限制 | 通常重试 1-2 次 |

Retry 是"再试一次，可能换节点也可能不换"（Exclude 策略不保证一定换），Reschedule 是"明确要求换个节点再来"。

---

### Q47 ⭐⭐⭐ 如果重试次数耗尽了怎么办？
**考察点**: 终态处理

**详细解答**:

重试策略的 `NextWithRetries` 返回三个值：`(nextTime, retries, shouldRetry)`。当 `shouldRetry=false`（重试次数达到 `MaxRetries`），补偿器将任务状态更新为 `FAILED`（终态），不再重试。

**当前缺陷**：任务默默进入 FAILED 终态，没有通知机制。

**改进方案**：
1. **Webhook 通知**：任务进入终态时发送 HTTP 回调到业务方配置的 URL
2. **Kafka 事件**：发送 `TaskFailedEvent` 到专门的 topic，下游服务消费后触发告警
3. **Prometheus 指标**：`task_final_failed_total` counter，配合 AlertManager 规则告警
4. **死信队列**：耗尽重试的任务放入 DLQ，人工介入处理

---

### Q48 ⭐⭐ 补偿器如何避免和调度器的并发冲突？
**考察点**: 并发安全

**详细解答**:

通过**状态机隔离**：

- 调度器只处理 `ACTIVE`（和超时 PREEMPTED）状态的 Task
- RetryCompensator 只处理 `FAILED_RETRYABLE` 的 Execution
- RescheduleCompensator 只处理 `FAILED_RESCHEDULED` 的 Execution
- InterruptCompensator 只处理超时的 `RUNNING/PREPARE` Execution
- ShardingCompensator 只处理分片父任务

它们操作的是**不同状态的不同表**，天然不冲突。

唯一的交汇点：补偿器将任务回滚到 `ACTIVE` 时（Retry/Reschedule 需要释放 Task 锁），用的是 CAS 更新（`WHERE status='PREEMPTED' AND schedule_node_id=?`），确保不会覆盖其他调度器的抢占。

### Q49 ⭐⭐⭐ 如果让你重新设计补偿机制，你会怎么做？
**考察点**: 设计能力

**详细解答**:

5 个改进方向：

1. **事件驱动替代轮询**：任务失败时主动发 Kafka 事件触发补偿（`TaskFailedEvent`），而不是定时扫表。优点：实时性好、减少 DB 压力。轮询作为兜底保留但降低频率。

2. **统一补偿框架**：抽象 `CompensationStrategy` 接口，4 种补偿器只是策略实现。框架负责锁管理、重试、告警等横切关注点。

3. **补偿链**：支持链式补偿：Retry(3次) → Reschedule(2次) → 降级执行 → 告警通知。当前 4 种补偿器是独立的，无法自动升级。

4. **幂等补偿**：每次补偿带 `compensationId`（= executionId + retryCount），避免补偿器重启后重复补偿。

5. **可观测性**：每种补偿器暴露 Prometheus 指标——触发次数、成功率、平均延迟、队列深度。配合 Grafana Dashboard 一目了然。

## 五、分库分表（10 题）

### Q50 ⭐⭐ 分库分表的策略是怎样的？
**考察点**: 分片设计

**详细解答**:

源码 `pkg/sharding/strategy.go`：

```go
type ShardingStrategy struct {
    DBPattern    string  // 如 "task_db_%d"
    TablePattern string  // 如 "tasks_%02d"
    DBCount      int     // 8
    TableCount   int     // 16
}

func (s *ShardingStrategy) Shard(shardingID int64) Dst {
    dbIdx := shardingID % int64(s.DBCount)      // 0-7
    tableIdx := shardingID % int64(s.TableCount) // 0-15
    return Dst{
        DB:    fmt.Sprintf(s.DBPattern, dbIdx),
        Table: fmt.Sprintf(s.TablePattern, tableIdx),
    }
}
```

8 库 × 16 表 = 128 个物理分片。shardingID 是分片键（业务维度 ID），同一 shardingID 的所有数据路由到同一分片。

`Broadcast()` 返回所有 (db, table) 组合，用于补偿器遍历全量分片。

选 2 的幂次方便位运算优化（虽然当前用的是取模）。

### Q51 ⭐⭐⭐ Snowflake ID 是怎么改造的？为什么要把 shardingID 编码进 ID？
**考察点**: ID 设计

**详细解答**:

源码 `pkg/id_generator/snowflake.go`：

```go
const (
    timestampBits = 41  // 时间戳
    bizIDBits     = 12  // shardingID
    sequenceBits  = 12  // 序列号
)

func (g *Generator) GenerateID(shardingID int64) int64 {
    timestamp := time.Now().UnixMilli() - epochMillis
    shardingValue := shardingID & shardingIDMask  // 取低 12 位
    sequence := atomic.AddInt64(&g.sequence, 1) - 1
    
    id := (shardingValue << bizIDShift) |      // 高位：shardingID
          (sequence & sequenceMask) << sequenceShift |  // 中位：序列号
          (timestamp & timestampMask)          // 低位：时间戳
    return id
}
```

布局：`[12位 shardingID][12位 sequence][41位 timestamp]` = 65 位

**编码 shardingID 的好处**：通过 `ExtractShardingID(id)` 从任意 ID 反推分片位置：`(id >> bizIDShift) & shardingIDMask`。这样跨系统传递 ID 时不需要额外查表就能路由到正确的分片。

### Q52 ⭐⭐⭐ 当前 Snowflake 实现有什么已知 bug？
**考察点**: 代码质量

**详细解答**:

三个严重问题：

1. **位溢出**：41+12+12=65 位超过 int64 的 63 位有效位（最高位是符号位）。当 shardingID 较大时，`shardingValue << 53` 会溢出到符号位，生成负数 ID。

2. **序列号不绑定毫秒**：`sequence` 是全局原子递增的，没有在每毫秒重置。这意味着序列号会持续增长直到溢出 12 位（4096），然后与其他毫秒的 ID 产生碰撞。正确做法应该是同一毫秒内递增，不同毫秒重置为 0。

3. **无时钟回拨保护**：NTP 同步可能导致系统时间回退，回退后生成的 timestamp 与之前重复，导致 ID 重复。应该检测时钟回拨并等待或报错。

### Q53 ⭐⭐ 跨分片查询是怎么实现的？
**考察点**: 分片挑战

**详细解答**:

以 `ShardingTaskDAO.FindSchedulableTasks` 为例，需要遍历所有分片查询可调度任务：

```
for each (db, table) in allShards:
    result := db.Query("SELECT * FROM {table} WHERE next_time<=? AND status='ACTIVE' LIMIT ?")
    allResults = append(allResults, result...)
```

**性能问题**：
- 串行遍历 128 个分片（8库×16表），假设每次查询 5ms → 总耗时 640ms
- 没有并行化（应该用 errgroup 并发查询）
- 没有全局排序合并（每个分片内有序，但合并后无序）
- 深分页问题（每个分片都要查 offset+limit 条数据）

**优化方向**：
- 并发查询所有分片（goroutine + errgroup）
- 归并排序合并结果
- 游标分页替代 offset 分页

### Q54 ⭐⭐⭐ `BatchCreate` 为什么用 DryRun？有什么风险？

### Q55 ⭐⭐ 分片键（shardingID）应该选什么字段？
**考察点**: 分片设计

**详细解答**:

好的分片键应满足三个条件：
1. **数据分布均匀**：避免热点分片
2. **查询命中单片**：大多数查询能定位到单个分片
3. **业务语义清晰**：便于理解和运维

本项目适合用 **租户ID/项目ID** 作为分片键：
- 同一租户的所有任务在同一分片 → 查询不跨分片
- 不同租户数据均匀分布

不适合用 **任务ID**：自增 ID 只能按范围分片，会导致最新数据集中在最后一个分片（热点）。

### Q56 ⭐⭐⭐ 分库分表后，补偿器怎么巡检所有分片？
**考察点**: 分片 + 补偿结合

**详细解答**:

假设 3 个补偿器实例并行运行，128 个分片：
1. 实例 A 遍历到 (db_0, table_00) → 尝试获取分布式锁 → 成功 → 处理该分片

### Q57 ⭐⭐ 如何保证 Task 和 TaskExecution 在同一个分片？
**考察点**: 数据一致性

**详细解答**:

通过相同的 shardingID 保证。Task 和 TaskExecution 都用 `task.BizID`（即 shardingID）做路由：

dst := shardingStrategy.Shard(task.BizID)  // Task 路由

### Q58 ⭐⭐⭐ 如果要从 8 库扩到 16 库，怎么办？
**考察点**: 扩容方案

**详细解答**:

当前 `shardingID % 8` 硬编码了分片数，不支持在线扩容。扩容选项：

1. **停服迁移**：停写 → 按新规则重新分配数据 → 切换配置 → 恢复。简单但影响业务。
2. **双写迁移**：新老分片同时写入 → 历史数据后台迁移 → 全部完成后切读 → 关闭老分片。不停服但复杂。
3. **一致性哈希**：替换取模为一致性哈希环，扩容时只需迁移环上相邻节点的数据（~1/N 的数据量）。
4. **中间件方案**：引入 ShardingSphere-Proxy 等中间件托管分片逻辑，扩容由中间件处理。

建议在设计初期就考虑一致性哈希替代取模。

### Q59 ⭐⭐⭐ 分库分表后的分页查询怎么做？
**考察点**: 分片难题

**详细解答**:

经典难题。假设要查第 3 页（每页 20 条，按创建时间排序）：

**暴力方案**：每个分片查 `LIMIT 60`（前 3 页），内存归并排序，取第 41-60 条。128 个分片 × 60 条 = 7680 条数据在内存中排序。**深分页时内存爆炸**。

**游标分页**：`WHERE id > lastId ORDER BY id LIMIT 20`。每个分片基于游标查询，归并排序取 20 条，记录最后一条的 ID 作为下次游标。性能稳定，不受页深影响。

**搜索引擎方案**：将数据同步到 Elasticsearch，分页查询走 ES。ES 的 `search_after` 天然支持高效深分页。

当前项目中补偿器用 offset 分页遍历**单个分片**（不是跨分片分页），所以没有这个问题。

## 六、工作流 / DAG 引擎（8 题）

### Q60 ⭐⭐ 工作流引擎是怎么实现的？
**考察点**: 技术选型

**详细解答**:

基于 ANTLR4 的 DSL 解析 + DAG 执行引擎：

1. **DSL 定义**：`internal/dsl/` 目录下的 `.g4` 文件定义了工作流表达式的语法规则
2. **编译**：ANTLR4 根据 .g4 文件自动生成 Go 的 Lexer/Parser 代码
3. **AST 转 DAG**：`PlanService.GetPlan()` 解析 DSL 表达式，生成 Node 节点树（SimpleNode/AndNode/OrNode/ConditionNode/EndNode）
4. **执行**：`PlanTaskRunner` 从 Root 节点开始，逐步推进 DAG

选 ANTLR4 的原因：成熟的 parser generator，语法定义清晰（BNF 范式），自动生成类型安全的 Go 代码，支持复杂的条件分支表达式。

---

### Q61 ⭐⭐ DAG 中有哪些节点类型？
**考察点**: DAG 设计

**详细解答**:

源码 `internal/dsl/parser/node.go` 定义了 5 种节点，都实现 `Node` 接口：

| 节点类型 | 含义 | `NextNodes(execution)` 行为 |
|---------|------|---------------------------|
| **SimpleNode** | 单任务节点 | 前置成功 → 返回自身 name |
| **AndNode** | 并行分支（所有任务同时启动）| 前置成功 → 返回所有子任务 name |
| **OrNode** | 可选分支 | 前置成功 → 返回所有子任务 name（任一完成即可推进下游）|
| **ConditionNode** | 条件分支 | 前置成功 → successTask；前置失败 → failureTask |
| **EndNode** | 终止节点 | 标记工作流完成 |

### Q62 ⭐⭐⭐ PlanTaskRunner 是怎么驱动 DAG 执行的？
**考察点**: DAG 调度

**详细解答**:

事件驱动模型。源码 `runner/plan_task_runner.go`：

**启动（`Run` 方法）**：
```go
func (p *PlanTaskRunner) Run(ctx, task) error {
    ta := p.acquireTask(ctx, task)              // 抢占 Plan 任务
    exec := p.execSvc.Create(ctx, TaskExecution{...})  // 创建 Execution
    plan := p.planService.GetPlan(ctx, task.ID)   // 解析 DAG
    rootTasks := plan.RootTask()                  // 找根节点
    for _, root := range rootTasks {
        go p.run(ctx, root.Task)                  // 并行启动根节点
    }
}
```

**推进（`NextStep` 方法）**：
```go
func (p *PlanTaskRunner) NextStep(ctx, task) error {
    plan := p.planService.GetPlan(ctx, task.PlanID)
    planTask := plan.GetTask(task.Name)
    nextTasks := planTask.NextStep()  // 获取后继节点
    
    if len(nextTasks) == 0 {
        // 没有后继 → 发送 Plan 结束事件
        p.producer.Produce(ctx, CompleteEvent{...})
        return
    }
    
    for _, next := range nextTasks {
        if next.CheckPre() {    // 检查前置条件是否满足
            go p.run(ctx, next.Task)  // 满足则启动
        }
    }
}
```

**驱动机制**：
- 任务完成 → Kafka `CompleteEvent` → `CompleteConsumer` 消费 → 调用 `NextStep`
- `NextStep` 检查后继节点的 `CheckPre()`（And 要求全部前置完成，Or 要求任一前置完成）
- 满足则启动后继节点 → 又完成 → 又触发 NextStep → 直到 EndNode

---

### Q63 ⭐⭐ DAG 中的 `CheckPre` 是怎么实现的？
**考察点**: 前置条件检查

**详细解答**:

`CheckPre()` 在 `domain/plantask.go` 中实现，检查当前节点的所有前置节点（`Pre` 字段）是否满足条件。

逻辑取决于节点类型：
- **AndNode 语义**：所有前置节点的 `TaskExecution.Status` 都是 `SUCCESS` 才返回 true
- **OrNode 语义**：任一前置节点的 `TaskExecution.Status` 是 `SUCCESS` 即返回 true
- **SimpleNode**：只有一个前置，检查它是否 SUCCESS

检查是通过查询 TaskExecution 表中前置节点的状态实现的。如果前置条件不满足，`NextStep` 不会启动该节点，等待下次事件触发再检查。

---

### Q64 ⭐⭐ 如果 DAG 中某个节点失败了，怎么处理？
**考察点**: 错误传播

**详细解答**:

当前行为：节点失败 → 状态变为 `FAILED_RETRYABLE` → RetryCompensator 重试。如果重试耗尽 → 状态变为 `FAILED`（终态）→ 下游节点的 `CheckPre()` 永远不满足 → **工作流"卡住"**。

**缺陷**：
1. 没有全局工作流超时机制
2. 不会自动取消下游节点
3. 不会发送工作流失败通知
4. 父 Plan 的 Execution 状态可能永远不会更新

**改进方案**：
- 增加 `PlanTimeout` 配置，超时后标记整个 Plan 失败并取消所有进行中的子任务
- 节点失败后判断 ConditionNode 的 failure 分支是否有处理
- 增加工作流级别的失败回调通知

---

### Q65 ⭐⭐⭐ DSL 表达式的语法是怎么定义的？
**考察点**: DSL 设计

**详细解答**:

ANTLR4 的 `.g4` 文件定义语法规则。节点类型直接映射到表达式语法：

- `A -> B -> C` — 串行：A 完成后执行 B，B 完成后执行 C（SimpleNode 链）
- `A -> (B, C) -> D` — 并行：A 后 B 和 C 同时启动（AndNode），都完成后执行 D
- `A -> (B | C) -> D` — 或分支：A 后 B 和 C 同时启动（OrNode），任一完成即执行 D
- `A -> [condition] ? B : C` — 条件分支：A 成功走 B，A 失败走 C（ConditionNode）

编译过程：DSL 字符串 → ANTLR4 Lexer 词法分析 → Parser 语法分析 → 生成 AST → `PlanService` 转换为 Node 节点链。

### Q66 ⭐⭐ PlanService 中的 `errgroup` 是用来做什么的？
**考察点**: 并发模式

**详细解答**:

`PlanService.GetPlan()` 加载 Plan 数据需要查询多个表。使用 `errgroup.Group` 实现并发查询：

```go
g, ctx := errgroup.WithContext(ctx)
var plan domain.Plan
var tasks []domain.Task
var executions []domain.TaskExecution

g.Go(func() error { plan, err = planRepo.GetByID(ctx, id); return err })
g.Go(func() error { tasks, err = taskRepo.FindByPlanID(ctx, id); return err })
g.Go(func() error { executions, err = execRepo.FindByPlanID(ctx, id); return err })

if err := g.Wait(); err != nil { return err }
// 组装 DAG 结构...
```

三个查询并发执行，总耗时 = max(三个查询时间) 而非 sum。errgroup 还提供了：任一失败自动取消其他 goroutine 的能力。

### Q67 ⭐⭐⭐ 如果 DAG 出现环路怎么办？
**考察点**: 深度思考

**详细解答**:

当前实现**没有环路检测**。如果定义了 `A -> B -> A` 的循环依赖，`NextStep` 会无限循环推进（A 完成 → 触发 B → B 完成 → 触发 A → ...）。

**检测方案**：在 DSL 编译阶段增加拓扑排序验证：
```go
func ValidateDAG(nodes []Node) error {
    // Kahn 算法：
    // 1. 计算每个节点的入度
    // 2. 将入度为 0 的节点加入队列
    // 3. BFS 遍历，每处理一个节点，将其后继的入度减 1
    // 4. 如果最终遍历的节点数 < 总节点数 → 存在环路
    if visitedCount < totalNodes {
        return fmt.Errorf("DAG contains cycle")
    }
}
```

应该在 Plan 创建时（而非运行时）做校验，拒绝保存有环的 DAG。

### Q71 ⭐⭐ 项目中的重试策略是怎么实现的？
**考察点**: 重试机制

**详细解答**:

`pkg/retry/` 包定义了 `Strategy` 接口和两种实现：

```go
type Strategy interface {
    NextWithRetries(retries int64) (nextTime time.Time, newRetries int64, shouldRetry bool)
    Next() time.Time
    Report(retries int64)
}
```

**FixedIntervalRetryStrategy**：`NextTime = now + fixedInterval`，不关心已重试次数，适合固定频率重试。

**ExponentialBackoffRetryStrategy**：`NextTime = now + initialInterval × 2^retries`（不超过 maxInterval），指数递增间隔，适合下游过载场景。

工厂函数根据配置创建：
```go
func NewStrategy(config Config) Strategy {
    switch config.Type {
    case "fixed":     return NewFixedInterval(config.FixedInterval)
    case "exponential": return NewExponentialBackoff(config.ExponentialBackoff)
    }
}
```

### Q72 ⭐⭐ 为什么自己实现 gRPC 负载均衡而不用 Envoy/Istio？
**考察点**: 技术选型

**详细解答**:

核心原因：调度场景需要**应用层路由**。

Envoy/Istio 是 L4/L7 代理，只能做通用负载均衡（轮询/加权/一致性哈希），无法感知业务上下文。但调度平台需要：

1. **指定节点执行**：Picker 选好了节点 A，gRPC 调用必须精准路由到 A（不能被 Envoy 轮询到 B）
2. **排除节点重试**：上次在节点 A 失败了，重试时要排除 A

这两个需求需要从 Go context 中读取路由策略，只有自定义 Picker 才能做到。Envoy 的 metadata routing 虽然也能实现类似功能，但需要：配置 xDS 路由规则 + 在请求 header 中传递路由信息 + Envoy 解析并执行，链路更长且引入了额外的运维复杂度。

---

### Q73 ⭐⭐ 如果 etcd 挂了，服务发现会怎样？
**考察点**: 容灾能力

**详细解答**:

分短暂和长期故障两种情况：

**短暂故障**（秒~分钟）：
- Watch 连接断开，但 gRPC Resolver **已缓存的地址列表仍可用**
- 现有 RPC 调用正常，因为 SubConn 已经建立
- 新注册的服务无法被发现，下线的服务不会被移除

**长期故障**：
- 旧实例下线后不会被 Watch 感知，但 SubConn 会检测到连接断开（TRANSIENT_FAILURE）
- 新实例无法注册，地址列表逐渐过期
- 如果调度器进程重启，Resolver Build 时无法从 etcd 拉取初始地址列表 → **服务地址全部丢失**

**改进**：增加本地文件缓存，在 `ListServices` 成功时写入本地文件，etcd 不可用时从文件恢复。

### Q74 ⭐⭐⭐ `ClientsV2` 和 `Clients` 的区别是什么？
**考察点**: 演进思路

**详细解答**:

**Clients V1**：使用 ego 框架内置的 `egrpc.Component`，通过 `syncx.Map` 缓存。简单但不支持自定义 Resolver 和 Balancer。

**ClientsV2**：使用原生 `grpc.NewClient` + 自定义 `ResolverBuilder` + `routingBalancer`：
```go
func (c *ClientsV2[T]) GetOrCreate(serviceName) T {
    conn := grpc.NewClient(
        "etcd:///"+serviceName,
        grpc.WithDefaultServiceConfig(`{"loadBalancingPolicy":"routing_balancer"}`),
    )
    return c.newClient(conn)  // 泛型创建 gRPC 客户端
}
```

V2 是为了实现"指定节点执行"和"排除节点重调度"两个核心需求。V1 的轮询策略无法满足。

### Q75 ⭐⭐ 如何实现服务的优雅上下线？
**考察点**: 生产级能力

**详细解答**:

**上线**：启动 → `Registry.Register()` → etcd Put 带 Lease → Watch 通知其他服务 → Resolver UpdateState → Balancer 创建新 SubConn → 自动纳入负载均衡

**下线**：收到 SIGTERM → ego 框架调用 `GracefulStop()` → 停止接收新 gRPC 请求 → 等待进行中的请求完成 → etcd Session Close → Lease 过期 → Key 自动删除 → Watch 通知移除

**当前问题**：调度器的 GracefulStop 只是 `cancel()` context，没有：① 等待进行中的 goroutine 完成 ② 释放已抢占的任务 ③ drain 续约循环

---

### Q76 ⭐⭐ 项目中有没有熔断/限流机制？
**考察点**: 服务治理

**详细解答**:

**限流有**：`LimiterChecker` 使用令牌桶限制调度频率。是调度器**自限流**（控制自己的调度速率），不是对外限流。

**熔断没有**：如果某个执行节点持续失败，调度器仍然会给它分配任务。没有节点级熔断机制。

**改进建议**：
1. gRPC 客户端添加熔断器（如 `sony/gobreaker`）
2. 维护节点健康评分（连续失败 N 次 → 标记不健康 → 暂时排除）
3. 在 Picker 层增加健康过滤（只从健康节点中选择）

### Q77 ⭐⭐ gRPC 连接是如何管理的？
**考察点**: 连接管理

**详细解答**:

`ClientsV2` 使用 `syncx.Map` 缓存 gRPC 连接，Key 是 serviceName，Value 是 `*grpc.ClientConn`。

```go
func (c *ClientsV2[T]) Get(serviceName string) T {
    val, ok := c.clients.Load(serviceName)
    if ok { return val.(T) }
    return c.GetOrCreate(serviceName)
}
```

`GetOrCreate` 保证同一服务只创建一个 ClientConn。gRPC 的 ClientConn 内部维护 HTTP/2 连接池（对每个目标地址维护多个底层 TCP 连接），所以一个 ClientConn 就够了，不需要自己实现连接池。

## 八、智能节点选择与负载控制（8 题）

### Q78 ⭐⭐⭐ Prometheus Picker 是怎么实现智能选节点的？
**考察点**: 核心亮点

**详细解答**:

源码 `picker/prometheus_picker.go` 的 `pickByMetric` 方法，三步走：

**Step 1：构造 PromQL**
```go
query := fmt.Sprintf(
    `topk(%d, avg_over_time(%s[%s]) AND ON(instance) up{job=%q} == 1)`,
    topN, metricName, timeWindow, jobName,
)
```
- `avg_over_time(cpu_usage[5m])`：5 分钟内 CPU 使用率均值（平滑毛刺）
- `AND ON(instance) up{job="executor"} == 1`：JOIN 过滤，只保留存活节点
- `topk(3, ...)`：取 CPU 空闲率最高的 3 个节点

**Step 2：执行查询**
```go
result, _, err := p.promClient.Query(queryCtx, query, time.Now())
```

**Step 3：随机选择**
```go
nodeIDs := parseQueryResult(result)  // 提取 node_id 标签
selectedNodeID := nodeIDs[rand.Intn(len(nodeIDs))]
```

---

### Q79 ⭐⭐ 为什么从 TopN 中随机选而不是直接选最优？
**考察点**: 负载均衡思维

**详细解答**:

避免"羊群效应"（thundering herd）。假设 10 个调度器都选最优节点 A → A 瞬间接收 10 个任务 → A 过载。

TopN + 随机实现了**"宏观最优 + 微观随机"**：保证选的都是负载较低的节点（宏观），但不会集中在同一个节点（微观）。

类似"Power of Two Choices"算法的思想——从少量随机候选中选最优，比纯轮询好得多，又避免了全局最优的热点问题。

---

### Q80 ⭐⭐ CompositeChecker 的 AND/OR 策略分别什么场景用？
**考察点**: 组合模式

**详细解答**:

源码 `pkg/loadchecker/composite.go`：

**AND 策略**（`checkAND`）：所有 Checker 通过才返回 `shouldSchedule=true`。当前配置使用 AND：`LimiterChecker AND DatabaseLoadChecker AND ClusterLoadChecker`。适用于严格控制场景——三层中任一层过载都应该暂停调度。

**OR 策略**（`checkOR`）：任一 Checker 通过即返回 `shouldSchedule=true`。适用于冗余检查场景——只要有一个指标正常就继续。

---

### Q81 ⭐⭐ DatabaseLoadChecker 是怎么判断数据库负载的？
**考察点**: 监控驱动

**详细解答**:

通过 Prometheus 查询 MySQL 的响应时间指标。如果 P99 响应时间超过阈值（如 200ms），判定数据库过载，返回 `shouldSchedule=false` 和建议的 sleep duration。

这是**背压机制**——上游（调度器）根据下游（MySQL）的承受能力调整发送速率。防止调度器在 DB 已经过载时继续写入大量 TaskExecution 记录导致雪崩。

---

### Q82 ⭐⭐ ClusterLoadChecker 和 DatabaseLoadChecker 的区别？
**考察点**: 多层负载控制

**详细解答**:

两者检查不同层面：
- **DatabaseLoadChecker**：检查**存储层**（MySQL）→ 响应时间指标
- **ClusterLoadChecker**：检查**计算层**（执行节点集群）→ CPU/Memory 聚合指标

形成"存储 + 计算"双保险。例如：MySQL 正常但执行节点全部 CPU 95% → ClusterLoadChecker 拦截。

---

### Q83 ⭐⭐ LimiterChecker 的令牌桶参数如何设置？
**考察点**: 限流配置

**详细解答**:

`LimiterChecker` 使用令牌桶算法，核心参数：
- `rate`：每秒产生的令牌数（调度 QPS 上限）
- `burst`：桶容量（允许的突发量）

设置建议：
- `rate` = 集群理论最大调度能力的 80%
- `burst` = rate × 2~3（允许短期突发，如补偿器批量触发重试）

---

### Q84 ⭐⭐⭐ PromQL 中 `avg_over_time` 和 `up` 过滤怎么配合的？
**考察点**: 监控查询

**详细解答**:

PromQL：`topk(N, avg_over_time(cpu_usage[5m]) AND ON(instance) up{job="executor"} == 1)`

拆解：
1. `up{job="executor"} == 1`：筛选存活的执行节点（up=1 表示 Prometheus 最近一次 scrape 成功）
2. `avg_over_time(cpu_usage[5m])`：计算每个节点 5 分钟内 CPU 使用率的均值（平滑掉瞬时尖峰）
3. `AND ON(instance)`：通过 `instance` 标签 JOIN 两个结果集，只保留同时存活且有指标的节点
4. `topk(N, ...)`：取 CPU 使用率最**低**的 N 个（topk 取最大值，但 CPU 空闲率越高越好）

关键：`up` 过滤防止选到已下线但 Prometheus 还保留历史数据的节点。

### Q85 ⭐⭐ 如果 Prometheus 查询失败，Picker 怎么处理？
**考察点**: 容错

**详细解答**:

源码 `scheduler.go` 的 `newContext`：
```go
if nodeID, err := s.executorNodePicker.Pick(s.ctx, task); err == nil && nodeID != "" {
    return balancer.WithSpecificNodeID(s.ctx, nodeID)
} else {
    return s.ctx  // fallback：使用原始 context
}
```

如果 Pick 失败（Prometheus 超时/不可用），返回不带路由策略的 context → `routingPicker` 进入 Round-Robin 模式 → 随机轮询所有可用节点。

这是一种**优雅降级**：智能调度不可用时回退到简单轮询，不影响核心调度流程。

## 九、事件驱动架构（5 题）

### Q86 ⭐⭐ Kafka 在项目中承担了哪些角色？
**考察点**: 消息驱动

**详细解答**:

两大角色：

1. **任务完成事件**（`CompleteProducer/Consumer`）：任务完成 → 发送 `Event{TaskID, ExecStatus}` → Consumer 调用 `PlanTaskRunner.NextStep()` 推进 DAG。

2. **状态上报事件**（`ReportEventProducer/Consumer`）：执行节点 → `ReporterService.Report()` → 写入 Kafka → Consumer 异步处理 → `execSvc.UpdateState()`。解耦上报和持久化，防止上报阻塞执行。

---

### Q87 ⭐⭐ 为什么用 Kafka 而不是直接 gRPC 回调？
**考察点**: 架构决策

**详细解答**:

四个原因：
1. **解耦**：执行节点不需要知道调度器地址
2. **削峰**：高并发完成事件被 Kafka 缓冲
3. **可靠性**：Kafka 持久化消息，调度器暂时不可用也不丢事件
4. **扇出**：一个事件可被多个消费者处理（状态更新、审计、告警）

### Q88 ⭐⭐ `mqx.Consumer` 的泛型封装是怎么做的？
**考察点**: Go 泛型

**详细解答**:

`pkg/mqx/consumer.go` 使用 Go 1.18+ 泛型：

```go
type Consumer[T any] struct {
    handler func(ctx context.Context, msg T) error
}

func (c *Consumer[T]) Consume(raw []byte) error {
    var msg T
    json.Unmarshal(raw, &msg)
    return c.handler(context.Background(), msg)
}
```

使用时只需：`mqx.NewConsumer[event.Event](handler)`，自动处理 JSON 反序列化。

---

### Q89 ⭐⭐⭐ 如果 Kafka 消费者处理失败，消息会丢吗？
**考察点**: 消息可靠性

**详细解答**:

取决于 offset 提交策略。当前实现使用手动提交（at-least-once）：处理成功 → 提交 offset。失败 → 不提交 → 下次重新消费。

**问题**：
1. **无死信队列**：反复失败的消息会阻塞后续消费（poison message）
2. **无幂等保护**：重复消费可能导致重复执行 NextStep
3. **无重试限制**：失败的消息会无限重试

**改进**：增加重试计数 → 超过 N 次放入 DLQ → 人工介入。消费逻辑增加幂等检查。

---

### Q90 ⭐⭐ 事件驱动和轮询各自适合什么场景？
**考察点**: 架构模式

**详细解答**:

**事件驱动**：适合实时响应 → 任务完成驱动 DAG、状态上报。优点：低延迟。缺点：需要消息中间件，故障恢复复杂。

**轮询**：适合定期巡检 → 补偿器扫描异常任务、调度器拉取待调度任务。优点：实现简单、不依赖 MQ。缺点：延迟 = 轮询间隔。

项目的组合：**"正向事件驱动 + 逆向轮询补偿"**。正常流程靠 Kafka 事件高效推进，异常情况靠补偿器定时扫表兜底。
