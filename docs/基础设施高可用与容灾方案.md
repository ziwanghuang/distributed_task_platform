# 基础设施高可用与容灾方案

## 概览

本项目依赖 5 个核心基础设施：**MySQL、Redis、Kafka、etcd、Prometheus**。本文档逐一分析每个组件的故障影响、现有容错机制的不足，并给出**可落地的兜底方案**和**减少故障概率的配置优化**。

### 故障影响速查表

| 组件 | 角色 | 故障影响 | 现有容错 | 风险等级 |
|------|------|----------|----------|----------|
| MySQL | 核心存储 | **全面停摆**：调度、抢占、续约、补偿全部不可用 | 仅启动时重试 | 🔴 P0 |
| Redis | 分布式锁 | V2 调度器/补偿器无法获取分片锁，调度停止 | 无 | 🟠 P1 |
| Kafka | 异步上报 | 执行结果异步上报失败，但 gRPC 直接上报仍可用 | 启动时重试 | 🟡 P2 |
| etcd | 服务发现 | 新 Executor 无法注册，调度器无法发现新节点 | lease 自动续期 | 🟠 P1 |
| Prometheus | 指标查询 | 智能调度退化为轮询 | ✅ 已有降级逻辑 | 🟢 P3 |

## 一、MySQL — 核心单点（P0）

### 1.1 故障影响分析

MySQL 是唯一的状态存储，挂了等于整个系统挂了：

| 受影响模块 | 代码位置 | 具体影响 |
|------------|----------|----------|
| 调度循环 | `scheduler.go:scheduleLoop()` → `SchedulableTasks()` | 查不到待调度任务，循环空转 |
| 任务抢占 | `dao/task.go:Acquire()` (CAS + version) | 抢不到任务，无法分发 |
| 心跳续约 | `scheduler.go:renewLoop()` → `Renew()` | 续约失败，任务被误判超时 |
| 状态更新 | `execution_service.go:UpdateState()` | 执行结果写不进去 |
| 4 个补偿器 | `retry.go / reschedule.go / interrupt.go / sharding.go` | 全部查询失败，补偿停止 |

### 1.2 现有机制

```go
// ioc/db.go — 仅启动时有重试
func WaitForDBSetup() {
    strategy := retry.NewExponentialBackoffRetryStrategy(
        time.Second, maxInterval, maxRetries, // 1s→10s, 最多10次
    )
    // ... 启动后 DAO 层零重试，错误直接返回
}
```

**问题**：启动后任何 MySQL 操作失败 = 直接报错，没有重试、没有熔断、没有连接池调优。

### 1.3 兜底方案

#### 方案 A：DAO 层重试装饰器

核心思路：不改现有 DAO 接口，通过装饰器模式包一层重试逻辑。

```go
// pkg/retry/dao_retry.go
package retry

import (
    "context"
    "time"
    "errors"

    "gorm.io/gorm"
)

// RetryConfig 重试配置
type RetryConfig struct {
    MaxRetries    int           // 最大重试次数
    InitialDelay  time.Duration // 初始延迟
    MaxDelay      time.Duration // 最大延迟
    RetryableErrs []error       // 可重试的错误类型
}

var DefaultRetryConfig = RetryConfig{
    MaxRetries:   3,
    InitialDelay: 100 * time.Millisecond,
    MaxDelay:     2 * time.Second,
}

// DoWithRetry 带指数退避的重试
func DoWithRetry[T any](ctx context.Context, cfg RetryConfig, fn func() (T, error)) (T, error) {
    var (
        result T
        err    error
        delay  = cfg.InitialDelay
    )
    for i := 0; i <= cfg.MaxRetries; i++ {
        result, err = fn()
        if err == nil {
            return result, nil
        }
        // 不可重试的错误直接返回
        if !isRetryable(err, cfg.RetryableErrs) {
            return result, err
        }
        // 最后一次不等了
        if i == cfg.MaxRetries {
            break
        }
        select {
        case <-ctx.Done():
            return result, ctx.Err()
        case <-time.After(delay):
        }
        delay = min(delay*2, cfg.MaxDelay)
    }
    return result, err
}

func isRetryable(err error, retryableErrs []error) bool {
    // 连接错误、超时都值得重试
    if errors.Is(err, gorm.ErrInvalidDB) {
        return true
    }
    for _, re := range retryableErrs {
        if errors.Is(err, re) {
            return true
        }
    }
    // 检查网络类错误（connection refused, timeout 等）
    errMsg := err.Error()
    retryKeywords := []string{
        "connection refused", "connection reset",
        "broken pipe", "i/o timeout", "EOF",
        "invalid connection", "bad connection",
    }
    for _, kw := range retryKeywords {
        if containsIgnoreCase(errMsg, kw) {
            return true
        }
    }
    return false
}
```

**使用示例**（以 `FindSchedulableTasks` 为例）：

```go
// internal/repository/dao/task.go
func (dao *GORMTaskDAO) FindSchedulableTasks(ctx context.Context, now time.Time, limit int) ([]Task, error) {
    return retry.DoWithRetry(ctx, retry.DefaultRetryConfig, func() ([]Task, error) {
        var tasks []Task
        err := dao.db.WithContext(ctx).
            Where("next_time <= ? AND status = ?", now.UnixMilli(), TaskStatusWaiting).
            Order("next_time ASC").Limit(limit).Find(&tasks).Error
        return tasks, err
    })
}
```

#### 方案 B：连接池调优

当前项目**没有显式配置连接池参数**，全靠框架默认值。加上这几行可以大幅减少 MySQL 连接问题：

```go
// ioc/db.go — 初始化 DB 后加上连接池配置
func initDB(db *gorm.DB) {
    sqlDB, err := db.DB()
    if err != nil {
        panic(err)
    }

    sqlDB.SetMaxOpenConns(50)                  // 最大打开连接数
    sqlDB.SetMaxIdleConns(10)                  // 空闲连接数
    sqlDB.SetConnMaxLifetime(30 * time.Minute) // 连接最大生命周期
    sqlDB.SetConnMaxIdleTime(5 * time.Minute)  // 空闲连接最大存活
}
```

**为什么这能减少故障**：
- `ConnMaxLifetime` 防止长期使用已经被 MySQL 端 `wait_timeout` 关闭的连接
- `MaxIdleConns` 避免频繁创建/销毁连接带来的 overhead
- `MaxOpenConns` 限制并发连接数，防止打满 MySQL 的 `max_connections`

#### 方案 C：健康检查 + 优雅降级

```go
// pkg/healthcheck/mysql.go
package healthcheck

import (
    "context"
    "sync/atomic"
    "time"

    "gorm.io/gorm"
)

type MySQLHealthChecker struct {
    db        *gorm.DB
    healthy   atomic.Bool
    interval  time.Duration
}

func NewMySQLHealthChecker(db *gorm.DB) *MySQLHealthChecker {
    hc := &MySQLHealthChecker{
        db:       db,
        interval: 5 * time.Second,
    }
    hc.healthy.Store(true)
    return hc
}

func (hc *MySQLHealthChecker) Start(ctx context.Context) {
    ticker := time.NewTicker(hc.interval)
    defer ticker.Stop()
    for {
        select {
        case <-ctx.Done():
            return
        case <-ticker.C:
            sqlDB, err := hc.db.DB()
            if err != nil {
                hc.healthy.Store(false)
                continue
            }
            pingCtx, cancel := context.WithTimeout(ctx, 2*time.Second)
            err = sqlDB.PingContext(pingCtx)
            cancel()
            hc.healthy.Store(err == nil)
        }
    }
}

func (hc *MySQLHealthChecker) IsHealthy() bool {
    return hc.healthy.Load()
}
```

调度器侧使用：

```go
func (s *Scheduler) scheduleLoop(ctx context.Context) {
    for {
        // MySQL 不健康时，跳过本轮调度，避免无意义的错误日志刷屏
        if !s.mysqlHealthChecker.IsHealthy() {
            s.logger.Warn("MySQL 不健康，跳过本轮调度")
            time.Sleep(5 * time.Second) // 等待恢复
            continue
        }
        // ... 正常调度逻辑
    }
}
```

## 二、Redis — 分布式锁（P1）

### 2.1 故障影响分析

Redis **仅被 V2 架构使用**，提供分片级分布式锁：

| 受影响模块 | 代码位置 | 具体影响 |
|------------|----------|----------|
| V2 调度器 | `schedulerv2.go` → `ShardingLoopJob` | 无法获取分片锁，整个调度暂停 |
| V2 补偿器 | `sharding.go` → `ShardingCompensatorV2` | 无法获取锁，补偿暂停 |

**注意**：V1 架构完全不依赖 Redis。如果 Redis 挂了，V1 调度器/补偿器不受影响。

### 2.2 现有机制

```go
// ioc/dlock.go — 零容错
func InitDLockClient(cmd redis.Cmdable) dlock.Client {
    return dlockredis.NewClient(cmd)
}
```

没有连接重试，没有健康检查。

### 2.3 兜底方案

#### 方案 A：锁获取失败时自动降级

V2 的 `ShardingLoopJob` 已有部分容错——锁获取失败会跳过该分片继续下一个。但 Redis 整体不可用时，所有分片都会跳过，等于全面停摆。

**改进**：检测到 Redis 连续不可用时，切换到 V1 的单节点调度模式（无锁）：

```go
// pkg/loopjob/shardingloopjob.go
type ShardingLoopJob struct {
    // ...
    consecutiveFailures atomic.Int32
    degradeThreshold    int32 // 连续失败多少次后降级
    degraded            atomic.Bool
}

func (s *ShardingLoopJob) runShard(ctx context.Context, shardIdx int) error {
    lock, err := s.lockClient.NewLock(ctx, s.lockKey(shardIdx), s.lockExpiry)
    if err != nil {
        failures := s.consecutiveFailures.Add(1)
        if failures >= s.degradeThreshold {
            s.degraded.Store(true)
            s.logger.Error("Redis 连续不可用，降级为无锁模式",
                "failures", failures)
            // 降级：直接执行，不加锁（牺牲互斥性保活性）
            return s.fn(ctx, shardIdx)
        }
        return err
    }
    // 锁获取成功，重置计数
    s.consecutiveFailures.Store(0)
    s.degraded.Store(false)
    defer lock.Unlock(ctx)
    return s.fn(ctx, shardIdx)
}
```

## 三、Kafka — 异步上报（P2）

### 3.1 故障影响分析

Kafka 故障影响**相对最小**，因为项目有 gRPC 直接上报作为备选路径：

| 受影响模块 | 代码位置 | 具体影响 |
|------------|----------|----------|
| 执行结果异步上报 | `execution_service.go:sendCompletedEvent()` | 异步通道丢失，但 gRPC 直接上报不受影响 |
| 消费端 | `report_event_consumer.go` | 消费停止，积压的消息等 Kafka 恢复后继续消费 |

### 3.2 现有机制

```go
// internal/service/task/execution_service.go
func (s *executionService) sendCompletedEvent(ctx context.Context, exec domain.TaskExecution) {
    // ... 构建事件
    if err := s.completedEventProducer.Produce(ctx, evt); err != nil {
        s.logger.Error("发送完成事件失败", elog.FieldErr(err))
        // 仅日志，不影响主流程 ✅
    }
}
```

**已有优点**：Kafka 上报失败不阻塞主流程。  
**不足**：没有生产端重试、没有死信队列、消费端失败直接丢弃。

### 3.3 兜底方案

#### 方案 A：生产端本地缓冲 + 重试

```go
// pkg/mqx/buffered_producer.go
type BufferedProducer struct {
    inner      Producer
    buffer     chan Event
    maxRetries int
    logger     *elog.Component
}

func NewBufferedProducer(inner Producer, bufferSize int) *BufferedProducer {
    bp := &BufferedProducer{
        inner:      inner,
        buffer:     make(chan Event, bufferSize), // 内存缓冲
        maxRetries: 3,
    }
    go bp.drainLoop()
    return bp
}

func (bp *BufferedProducer) Produce(ctx context.Context, evt Event) error {
    err := bp.inner.Produce(ctx, evt)
    if err != nil {
        // 发送失败，放入缓冲等待重试
        select {
        case bp.buffer <- evt:
            bp.logger.Warn("Kafka 发送失败，已放入重试缓冲", "event", evt.Key)
        default:
            bp.logger.Error("Kafka 发送失败且缓冲已满，事件丢失", "event", evt.Key)
        }
        return err
    }
    return nil
}

func (bp *BufferedProducer) drainLoop() {
    for evt := range bp.buffer {
        for i := 0; i < bp.maxRetries; i++ {
            ctx, cancel := context.WithTimeout(context.Background(), 5*time.Second)
            err := bp.inner.Produce(ctx, evt)
            cancel()
            if err == nil {
                break
            }
            time.Sleep(time.Duration(i+1) * time.Second) // 线性退避
        }
    }
}
```

#### 方案 B：消费端错误处理增强

```go
// internal/event/reportevt/report_event_consumer.go
func (c *ReportEventConsumer) Consume(ctx context.Context, msg *sarama.ConsumerMessage) error {
    var evt ReportEvent
    if err := json.Unmarshal(msg.Value, &evt); err != nil {

            "error", err)
        return err // 框架会根据配置决定重试策略
    }
    return nil
}
```

## 四、etcd — 服务发现（P1）

### 4.1 故障影响分析

| 受影响模块 | 代码位置 | 具体影响 |
|------------|----------|----------|
| Executor 注册 | `registry/etcd/registry.go:Register()` | 新节点无法注册到调度器 |
| 服务发现 | `registry/etcd/registry.go:Subscribe()` | Watch 中断，无法感知节点变化 |
| 已有节点 | lease 到期自动清除 | **已注册节点的 lease 无法续期**，会被逐出 |

**关键**：etcd 挂了之后，已缓存的节点列表仍可用（内存中的 `endpoints`），但无法感知变化。如果 lease 到期被清除，则所有节点信息丢失。

### 4.2 现有机制

```go
// pkg/grpc/registry/etcd/registry.go
func (r *Registry) Register(ctx context.Context, si registry.ServiceInstance) error {
    sess, err := concurrency.NewSession(r.client, concurrency.WithTTL(int(r.ttl.Seconds())))
    // ... lease 自动续期由 concurrency.Session 内部处理
    // 但没有处理 etcd 断连后的重连逻辑
}
```

### 4.3 兜底方案

#### 方案 A：本地缓存 + 过期兜底

```go
// pkg/grpc/registry/etcd/registry.go — 增加本地缓存
type Registry struct {
    // ...
    cachedEndpoints sync.Map          // 本地缓存
    cacheExpiry     time.Duration     // 缓存过期时间
    lastUpdateTime  atomic.Value      // 最后更新时间
}

func (r *Registry) ListServices(ctx context.Context, serviceName string) ([]registry.ServiceInstance, error) {
    instances, err := r.listFromEtcd(ctx, serviceName)
    if err != nil {
        // etcd 不可用，使用本地缓存
        if cached, ok := r.cachedEndpoints.Load(serviceName); ok {
            lastUpdate := r.lastUpdateTime.Load().(time.Time)
            if time.Since(lastUpdate) < r.cacheExpiry {
                r.logger.Warn("etcd 不可用，使用缓存的服务列表",
                    "serviceName", serviceName,
                    "cacheAge", time.Since(lastUpdate))
                return cached.([]registry.ServiceInstance), nil
            }
            // 缓存过期但 etcd 也不可用，宁可用过期数据也不停摆
            r.logger.Error("etcd 不可用且缓存已过期，使用过期缓存",
                "serviceName", serviceName)
            return cached.([]registry.ServiceInstance), nil
        }
        return nil, err
    }

    // 更新缓存
    r.cachedEndpoints.Store(serviceName, instances)
    r.lastUpdateTime.Store(time.Now())
    return instances, nil
}
```

#### 方案 B：注册重试 + 自动重注册

```go
func (r *Registry) Register(ctx context.Context, si registry.ServiceInstance) error {
    for {
        err := r.doRegister(ctx, si)
        if err == nil {
            return nil
        }
        r.logger.Error("注册失败，等待重试", "error", err)
        select {
        case <-ctx.Done():
            return ctx.Err()
        case <-time.After(5 * time.Second):
        }
    }
}

// 监听 session 过期，自动重注册
func (r *Registry) watchSessionAndReRegister(ctx context.Context, si registry.ServiceInstance) {
    for {
        select {
        case <-ctx.Done():
            return
        case <-r.session.Done(): // session 过期或断连
            r.logger.Warn("etcd session 过期，尝试重注册")
            if err := r.Register(ctx, si); err != nil {
                r.logger.Error("重注册失败", "error", err)
            }
        }
    }
}
```


### 6.3 改进对比

| 改动 | 效果 |
|------|------|
| 全部加 `restart: always` | 进程崩溃后 Docker 自动拉起 |
| Redis/etcd 加健康检查 | scheduler 不会在依赖还没就绪时启动 |
| `start_period` | 避免慢启动时被误判为不健康 |
| 内存限制 | 防止 OOM 被系统杀掉 |
| Redis AOF 持久化 | 重启后数据不丢 |

## 八、总结

```
当前状态：
  ✅ Prometheus 降级 — 做得好
  ✅ gRPC+Kafka 双通道上报 — 天然冗余
  ⚠️ MySQL — 核心单点，零运行时容错
  ⚠️ Redis — V2 依赖无降级
  ⚠️ etcd — 无重连无缓存
  ⚠️ Docker Compose — 无重启策略，部分无健康检查

推荐路径：
  第一步：Docker Compose 加固 + MySQL 连接池（半天搞定，立竿见影）
  第二步：DAO 重试 + MySQL 健康检查（核心加固）
  第三步：Redis/etcd 降级方案（根据实际故障频率决定是否投入）
```

核心原则：**宁可降级运行，也不完全停摆**。Prometheus 那块的降级逻辑就是好的示范，把同样的思路推广到其他组件就行。
