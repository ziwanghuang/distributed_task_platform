## 一、服务注册与发现

### 1.1 基于 etcd 的注册中心

采用 etcd 作为服务注册中心，利用其 Raft 协议保证强一致性，结合**租约（Lease）+ Watch** 机制实现秒级节点感知。

```go
// pkg/grpc/registry/etcd/registry.go
type Registry struct {
    sess   *concurrency.Session  // etcd 会话（带租约）
    client *eetcd.Component
}

// Register 注册服务实例（绑定租约，宕机后自动注销）
func (r *Registry) Register(ctx context.Context, si registry.ServiceInstance) error {
    val, _ := json.Marshal(si)
    _, err := r.client.Put(ctx, r.instanceKey(si), string(val),
        clientv3.WithLease(r.sess.Lease()))
    return err
}

// Subscribe 订阅服务变化（Watch 机制，事件驱动）
func (r *Registry) Subscribe(name string) <-chan registry.Event {
    ch := r.client.Watch(ctx, r.serviceKey(name), clientv3.WithPrefix())
    return eventChan
}
```

### 1.2 服务实例模型

```go
// pkg/grpc/registry/types.go
type ServiceInstance struct {
    Name         string  // 服务名称（如 "executor"）
    Address      string  // 服务地址（如 "192.168.1.100:8080"）
    ID           string  // 实例 ID（如 "node-01"）
    Weight       int64   // 权重（用于加权负载均衡）
    InitCapacity int64   // 初始容量
    MaxCapacity  int64   // 最大容量
    IncreaseStep int64   // 扩容步长
    GrowthRate   float64 // 增长率
}
```

### 1.3 设计亮点

| 特性 | 实现方式 | 业务价值 |
|------|---------|---------|
| **租约机制** | TTL=60s，宕机后自动注销 | 无僵尸节点，无需手动清理 |
| **Watch 机制** | 实时监听 PUT/DELETE 事件 | 秒级感知节点上下线 |
| **强一致性** | Raft 协议 | 避免脑裂，元数据可靠 |
| **容量模型** | Weight / MaxCapacity 等字段 | 支持动态扩缩容 |

**故障恢复流程**：

```
执行器节点宕机
→ 租约过期（60 秒内）
→ etcd 自动删除节点信息
→ 调度器 Watch 到 DELETE 事件
→ 更新本地路由表
→ 后续任务不再调度到该节点
总恢复时间：< 60 秒
```

## 二、自定义 gRPC 负载均衡器

### 2.1 核心实现

自研 `routingBalancer`，在 gRPC 原生负载均衡基础上增加**排除节点**和**指定路由**能力，适配任务重调度、故障转移等分布式场景。

```go
// pkg/grpc/balancer/routing_round_robin.go
type routingBalancer struct {
    subConnMap map[resolver.Address]balancer.SubConn  // 地址→连接
    nodeIDMap  map[resolver.Address]string            // 地址→节点 ID
    readySCs   map[balancer.SubConn]string            // 就绪连接集合
}

// Pick 选择目标节点
func (p *routingPicker) Pick(info balancer.PickInfo) (balancer.PickResult, error) {
    // 1. 检查是否指定了目标节点（任务转移场景）
    if specificNodeID := getSpecificNodeID(info.Ctx); specificNodeID != "" {
        return p.pickSpecificNode(specificNodeID)
    }

    // 2. 检查是否需要排除某个节点（故障转移场景）
    excludedNodeID := getExcludedNodeID(info.Ctx)

    // 3. 从可用节点中轮询选择（排除故障节点）
    for _, sc := range p.readySCs {
        nodeID := p.scToNodeID[sc]
        if nodeID != excludedNodeID {
            return balancer.PickResult{SubConn: sc}, nil
        }
    }

    return balancer.PickResult{}, balancer.ErrNoSubConnAvailable
}
```

### 2.2 使用场景

```go
// 场景 1：任务重调度 — 排除故障节点
ctx := balancer.WithExcludedNodeID(ctx, failedNodeID)
client.Execute(ctx, task)

// 场景 2：任务转移 — 指定目标节点
ctx := balancer.WithSpecificNodeID(ctx, targetNodeID)
client.Execute(ctx, task)
```

### 2.3 设计优势

- **精准路由**：通过 Context 传递路由信息，不侵入业务代码
- **动态连接管理**：自动建立新连接，自动清理失效连接
- **故障转移 < 10 秒**：排除故障节点后立即调度到健康节点

## 三、基于 Prometheus 的智能调度

### 3.1 核心实现

```go
// internal/service/picker/prometheus_picker.go
type PrometheusPicker struct {
    promClient     v1.API
    jobName        string
    topNCandidates int           // 从前 N 个候选节点中随机选择
    timeWindow     time.Duration // 指标查询时间窗口
}

// Pick 智能选择执行节点
func (p *PrometheusPicker) Pick(ctx context.Context, task domain.Task) (string, error) {
    // 1. 构建 Prometheus 查询（CPU 优先策略示例）
    query := fmt.Sprintf(`
        topk(%d,
            avg_over_time(
                (1 - avg(rate(node_cpu_seconds_total{mode="idle"}[1m])) by (instance))
                [%s]
            )
        )
    `, p.topNCandidates, p.timeWindow)

    // 2. 查询 Prometheus
    result, _, err := p.promClient.Query(ctx, query, time.Now())

    // 3. 解析结果，获取前 N 个低负载节点
    nodeIDs := p.parseQueryResult(result)

    // 4. 从候选节点中随机选择（避免集中到单节点）
    selectedNodeID := nodeIDs[rand.Intn(len(nodeIDs))]

    return selectedNodeID, nil
}
```

### 3.2 多维度调度策略

| 策略类型 | Prometheus 查询 | 适用场景 |
|---------|----------------|---------|
| **CPU 优先** | `1 - node_cpu_idle` | CPU 密集型任务（Spark 计算） |
| **内存优先** | `node_memory_usage / node_memory_total` | 内存密集型任务（大数据聚合） |
| **任务数优先** | `task_execution_running_count` | 均衡任务分布 |
| **综合评分** | `cpu×0.5 + mem×0.3 + task×0.2` | 通用场景 |

### 3.3 TopN 随机选择算法

传统方式直接选最优节点，容易导致"羊群效应"。本系统采用 **TopN 随机**：

1. 从 Prometheus 查出所有节点负载
2. 选出**前 N 个最优节点**（默认 N=3）
3. 从中**随机选择一个**

**效果对比**：

| 指标 | 轮询调度 | 智能调度 | 提升 |
|------|---------|---------|------|
| 资源利用率 | 60% | 85% | +42% |
| 平均任务耗时 | 120s | 80s | -33% |
| 节点负载方差 | 高 | 低 | 显著 |

### 3.4 任务类型感知调度

```go
// 根据任务类型动态调整调度权重
type TaskTypeAwarePicker struct {
    scorer *LoadScorer
}

func (p *TaskTypeAwarePicker) Pick(ctx context.Context, task domain.Task) (string, error) {
    switch task.Type {
    case "spark":
        return p.pickByCPUAndMemory(ctx) // CPU + 内存优先
    case "datax":
        return p.pickByNetworkBandwidth(ctx) // 网络带宽优先
    case "flink":
        return p.pickByLatency(ctx) // 低延迟优先
    default:
        return p.pickByScore(ctx) // 综合评分
    }
}
```

## 四、负载自适应调度

### 4.1 核心实现

当某个调度器节点性能低于集群平均水平时，自动降低其调度频率，避免慢节点拖累整个集群。

```go
// pkg/loadchecker/cluster.go
type ClusterLoadChecker struct {
    nodeID             string
    promClient         v1.API
    thresholdRatio     float64 // 性能阈值（1.2 = 超过平均值 20%）
    slowdownMultiplier float64 // 降频乘数
}

// Check 检查集群负载
func (c *ClusterLoadChecker) Check(ctx context.Context) (time.Duration, bool) {
    // 1. 获取当前节点的调度耗时
    currentDuration := GetExecutionTime(ctx)

    // 2. 查询集群平均调度耗时
    avgDuration, _ := c.queryClusterAverageTime(ctx)

    // 3. 计算性能比例
    ratio := float64(currentDuration) / float64(avgDuration)

    // 4. 如果性能低于平均水平，降低调度频率
    if ratio > c.thresholdRatio {
        backoffDuration := time.Duration(float64(currentDuration) * c.slowdownMultiplier)
        return backoffDuration, false // 需要休眠
    }

    return 0, true // 可以继续调度
}
```

### 4.2 设计价值

- **自动识别慢节点**：对比当前节点与集群平均性能
- **自适应降频**：慢节点自动降低调度频率，快节点保持高频
- **防止雪崩**：避免慢节点被压垮，保护数据库等共享资源

    waitDuration time.Duration  // 限流时的等待时间

func (l *LimiterChecker) Check(ctx context.Context) (time.Duration, bool) {

### 5.2 自动熔断（gobreaker）

```go
type CircuitBreakerInvoker struct {
    breakers sync.Map // map[string]*gobreaker.CircuitBreaker
    invoker  invoker.Invoker
}

func (cbi *CircuitBreakerInvoker) Invoke(ctx context.Context, req *InvokeRequest) error {
    key := req.ExecutorAddr

    cb, _ := cbi.breakers.LoadOrStore(key, gobreaker.NewCircuitBreaker(gobreaker.Settings{
        Name:        key,
        MaxRequests: 3,                   // 半开状态最多尝试 3 次
        Interval:    10 * time.Second,    // 统计窗口
        Timeout:     60 * time.Second,    // 熔断超时
        ReadyToTrip: func(counts gobreaker.Counts) bool {
            failureRatio := float64(counts.TotalFailures) / float64(counts.Requests)
            return counts.Requests >= 3 && failureRatio >= 0.6 // 失败率 ≥ 60% 触发熔断
        },
        OnStateChange: func(name string, from, to gobreaker.State) {
            log.Printf("Circuit breaker %s: %s -> %s", name, from, to)
        },
    }))

    _, err := cb.(*gobreaker.CircuitBreaker).Execute(func() (interface{}, error) {
        return nil, cbi.invoker.Invoke(ctx, req)
    })
    return err
}
```

**熔断状态机**：

```
Closed（正常）  →  失败率 ≥ 60%  →  Open（熔断）
Open（熔断）    →  60 秒后       →  HalfOpen（半开）
HalfOpen（半开）→  连续 3 次成功  →  Closed（恢复）
HalfOpen（半开）→  任意失败      →  Open（重新熔断）
```

**业务价值**：
- 避免雪崩效应：故障节点自动隔离
- 快速失败：不浪费资源在不可用节点
- 自动恢复：无需人工干预

## 六、四重补偿机制

### 6.1 架构概览

```
任务失败 → 判断失败类型
  ├── 网络抖动 → 重试补偿器（指数退避重试）
  ├── 节点宕机 → 重调度补偿器（选择健康节点）
  ├── 任务超时 → 中断补偿器（发送中断信号）
  └── 分片聚合 → 分片补偿器（聚合子任务状态）
```

### 6.2 四种补偿器详解

| 补偿器 | 处理状态 | 触发场景 | 处理策略 |
|--------|---------|---------|---------|
| **重试补偿器** | `FAILED_RETRYABLE` | 网络抖动、临时性错误 | 指数退避重试，最多 3 次 |
| **重调度补偿器** | `FAILED_RESCHEDULED` | 执行器宕机 | 排除故障节点，选择健康节点重新调度 |
| **中断补偿器** | `RUNNING`（超时） | 执行时间超过 `max_execution_seconds` | 发送中断信号，释放资源 |
| **分片补偿器** | 分片父任务 | 子任务全部完成 | 聚合状态：全成功→成功；有失败→失败 |

### 6.3 设计原则

1. **职责分离**：每个补偿器专注一种异常场景，降低复杂度
2. **自动恢复**：补偿器定期扫描异常任务，自动触发恢复，无需人工干预
3. **可配置**：重试次数、重试间隔、超时时间、扫描间隔均可配置
4. **互不影响**：单个补偿器失败不影响其他补偿器

## 七、工作流 DSL 引擎

### 7.1 DSL 语法设计

基于 ANTLR4 实现声明式 DSL，支持 DAG 编排：

```
# 顺序执行
task1 -> task2 -> task3

# 并行执行（所有任务都成功才继续）
task1 && task2 && task3 -> task4

# 选择执行（任意任务成功即可继续）
task1 || task2 || task3 -> task4

# 条件分支
task1 -> task2 ? task3 : task4

# 复杂工作流（电商 ETL 示例）
order_sync && user_sync && product_sync
-> data_cleaning
-> (order_transform && user_transform && product_transform)
-> data_aggregate
-> data_quality_check
? (daily_report && weekly_report && monthly_report)
: alert_notification
```

### 7.2 实现流程

```
DSL 字符串
→ ANTLR4 词法分析（Lexer）
→ 语法分析（Parser）
→ AST（抽象语法树）
→ 访问者模式遍历
→ 生成执行计划
→ 执行引擎调度
```

### 7.3 为什么选择 ANTLR4

| 方案 | 优点 | 缺点 | 结论 |
|------|------|------|------|
| 手写解析器 | 灵活 | 复杂，易出错 | ❌ |
| 正则表达式 | 简单 | 不支持嵌套语法 | ❌ |
| **ANTLR4** | 功能强大，错误恢复好 | 学习成本 | ✅ 采用 |

## 八、分库分表架构

### 8.1 分片策略（8 库 16 表）

```go
// 分片策略
type ShardingStrategy struct {
    dbCount    int  // 8 个数据库
    tableCount int  // 16 张表
}

func (s *ShardingStrategy) Route(taskID int64) (dbIndex, tableIndex int) {
    // 第一步：计算表索引（保证数据均匀分布）
    tableIndex = int(taskID % int64(s.tableCount))      // taskID % 16

    // 第二步：计算库索引
    dbIndex = int((taskID / int64(s.tableCount)) % int64(s.dbCount))  // (taskID / 16) % 8

    return dbIndex, tableIndex
}
```

### 8.2 为什么选择 8 库 16 表

| 考虑因素 | 分析 |
|---------|------|
| **数据量** | 预估 3 年内任务量 5000 万，每表 ~312 万（性能最佳区间） |
| **并发能力** | 8 个库提供 8 倍连接数，支撑更高并发 |
| **运维成本** | 分片数适中，不会给运维带来过大负担 |

### 8.3 跨分片查询优化

- **单表查询**：直接路由到目标分片，延迟 < 50ms
- **全表扫描**：`ShardingLoopJob` 并发扫描所有分片，聚合结果
- **热点数据**：Redis 缓存，命中率 95%+

### 8.4 性能对比

| 指标 | 单表（1000 万） | 分库分表（8×16） | 提升 |
|------|---------------|----------------|------|
| 查询延迟 | 500ms | 50ms | **10 倍** |
| 写入 TPS | 1,000 | 8,000 | **8 倍** |
| 索引大小 | 2GB | 128MB（单表） | **16 倍** |

## 九、双重锁机制

### 9.1 乐观锁（第一道防线）

```sql
-- 任务抢占：CAS 操作，只有一个调度器能抢占成功
UPDATE task
SET status = 'PREEMPTED',
    version = version + 1,
    scheduler_node_id = 'node-1',
    preempted_at = NOW()
WHERE id = 123
  AND version = 5
  AND status = 'ACTIVE'
  AND next_execution_time <= NOW()
```

**特点**：性能高，适合低冲突的任务抢占场景。

### 9.2 分布式锁（第二道防线）

```go
// 补偿器场景使用 Redis 分布式锁
lock := redis.SetNX(ctx,
    fmt.Sprintf("task:lock:%d", taskID),
    nodeID,
    60*time.Second)

if !lock {
    return errors.New("task is locked by another node")
}
defer redis.Del(ctx, fmt.Sprintf("task:lock:%d", taskID))
```

**特点**：防止补偿器重复处理，保证互斥性。

### 9.3 协同工作

| 场景 | 使用的锁 | 原因 |
|------|---------|------|
| 任务抢占 | 乐观锁 | 冲突少，CAS 性能高 |
| 补偿器处理 | 分布式锁 | 多节点补偿器需要互斥 |
| 状态变更 | 状态机 + 乐观锁 | 严格控制状态流转 |
